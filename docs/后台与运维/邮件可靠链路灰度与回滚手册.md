# 邮件可靠链路灰度与回滚手册

本文的邮件可靠链路与统一资产事务共享领取状态机、流水和服务发现约束；当前实现边界见[统一资产事务与奖励交付运行边界](../游戏服与接入层/统一资产事务与奖励交付运行边界.md)。

## 适用范围

本文只覆盖 `mail-service -> chat-server` 在线通知和 `mail-service -> game-server` 附件领取。Core NATS 仍是允许丢失的在线提示通道，PostgreSQL 邮件与领取工作流、game-server 背包与幂等结果分别是权威事实。不引入 JetStream，不改造 session kick、metrics、GM 广播或其他服务事件。

阶段 10 已于 2026-07-15 完成 PostgreSQL、Redis、Core NATS、mail-service、chat-server 和双 game-server 的真实故障注入验收。`tests/mail/mail-reliability-fault-drill.test.mjs` 共 11 项测试全部通过，覆盖正常闭环和十类故障场景。本文仍是部署和回滚操作契约，不能用单纯进程启动替代故障验收。

## 部署顺序

1. 先应用同时包含 `mail_notification_outbox`、`mail_claim_workflows`、`mail_admin_operation_audit` 和 `character_inventory_grants` 的数据库初始化契约。升级期间不得删除已有列、状态或幂等记录。
2. 部署全部 game-server，使其支持 `mail_claim:<mail_id>`、确定性指纹、事务幂等发放以及 3009/3010 结果查询。确认 `game-server.admin` endpoint、Redis v2 在线路由和 admin token 可用。
3. 部署全部 chat-server。滚动窗口内显式设置 `CHAT_MAIL_ACCEPT_LEGACY_PAYLOAD=true` 和一个未来的 `CHAT_MAIL_LEGACY_COMPAT_UNTIL_EPOCH_SECONDS`；先证明 v1 和旧 payload 均可解析，再继续生产者升级。
4. 部署 mail-service，先保持 `MAIL_CLAIM_NEW_REQUESTS_ENABLED=false`、`MAIL_CLAIM_RECOVERY_ENABLED=true`。验证 `/healthz` 的两个开关、outbox 发布、服务发现、game 结果查询和存量恢复。
5. 所有前置消费者健康后，把所有 mail-service 实例的 `MAIL_CLAIM_NEW_REQUESTS_ENABLED` 切为 `true`。逐实例核对 registry metadata 中 chat 的通知契约/legacy 开关和 mail 的领取/恢复开关；实例配置不能混用不同领取入口值超过一次滚动发布时间窗。
6. 观察至少一个完整业务高峰和最长配置退避窗口。确认无旧生产者后，将 chat legacy 开关切为 `false`；两条 NATS subject 继续保留供 v1 双路去重，不随 payload 兼容一起删除。

## 灰度观测

使用相同时间窗口对比灰度组与基线组，不按 player、mail、event 或 request ID 建高基数标签。

| 目标 | 指标与判定 |
|------|------|
| 通知发布 | `mail_outbox_published / (mail_outbox_published + mail_outbox_retries + mail_outbox_terminal)` 不低于基线；同时观察 backlog 和 oldest age |
| 在线推送 | `mail_notification_pushed / mail_notification_received` 与在线率共同解释；`mail_notification_offline_skipped` 不是业务失败，`mail_notification_queue_failed` 和 `mail_notification_parse_failed` 不应异常上升 |
| 旧格式清空 | 切换关闭后 `mail_notification_legacy_rejected=0`；出现非零说明仍有旧生产者或延迟消息，不能删除兼容代码 |
| 领取积压 | `mail_claim_long_running`、`mail_claim_manual_review_backlog` 和数据库各状态数量不持续增长 |
| 幂等 | `inventory_grant_idempotent_hit_total` 可在超时恢复时增加；`inventory_grant_fingerprint_conflict_total` 或 `mail_claim_fingerprint_conflicts` 任一非零都暂停灰度 |
| 恢复 | `mail_claim_recovery_duration_ms` 和 `mail_claim_recovery_unknown_age_ms` 不超过基线/告警阈值，query unavailable 与 lease takeover 不持续增长 |

通知的业务成功仍以邮件/outbox 原子落库为准，不能用在线 push 比率反向回滚邮件。game 幂等命中代表重试收敛，不等于重复发奖。

## 回滚 Runbook

1. 在所有 mail-service 实例设置 `MAIL_CLAIM_NEW_REQUESTS_ENABLED=false`，滚动后逐实例检查 `/healthz` 返回 `mail_claim_new_requests_enabled=false`、`mail_claim_recovery_enabled=true`。新领取应返回 `503 MAIL_CLAIM_INTAKE_DISABLED`，已有工作流只能返回持久状态，不能产生新的 game grant 调用。
2. 查询有效租约和未完成状态。有效玩家 lease 要等待结束或过期；恢复 lease 可由仍启用的 worker 在过期后接管。不要直接清空 lease 字段，不要把邮件从 `claiming` 改回 `read/unread`。

```sql
SELECT status, count(*)
FROM mail_claim_workflows
WHERE status <> 'claimed'
GROUP BY status
ORDER BY status;

SELECT count(*) FILTER (WHERE lease_expires_at > current_timestamp) AS active_claim_leases,
       count(*) FILTER (WHERE recovery_lease_expires_at > current_timestamp) AS active_recovery_leases
FROM mail_claim_workflows
WHERE status <> 'claimed';
```

3. 保持 game-server 的幂等发放、3009/3010 查询、`character_inventory_grants` 和 admin endpoint 在线；保持至少一个兼容 mail-service recovery worker。只允许回滚到仍认识冻结快照、相同 request ID、全部工作流状态和 query-first 恢复的版本。
4. 对 `reconciliation_pending/manual_review` 使用受保护的 mail operations 查询和调度。不得直接写 `claimed`，不得绕过 game 查询发奖。指纹冲突先停止普通恢复并核对固定 fixture。
5. 只有所有非 `claimed` 工作流为零，且 game 幂等记录保留期覆盖领取工作流保留期，才允许设置 `MAIL_CLAIM_RECOVERY_ENABLED=false`。配置仍要求新领取为 false；启动时若发现存量 backlog，服务会以 `MAIL_CLAIM_RECOVERY_DISABLE_BLOCKED` 拒绝启动。
6. 通知侧可独立回滚 mail producer，但 chat-server 应保留 v1 消费。回滚 chat-server 前必须确认 mail producer payload 与目标消费者兼容；NATS 丢通知不回滚邮件。

回滚验证至少记录：开关健康响应、上述 SQL、有效租约归零时间、最后一次恢复结果、game 幂等查询证据、outbox backlog 和相关日志错误码。不要记录 token、邮件正文或完整附件。

## 删除兼容代码门槛

必须同时满足以下条件，才能删除旧扁平 payload 解析，而不是只切关闭开关：

- 所有 chat-server 实例已升级，registry/metrics 中只出现支持 v1 的 build version，且 metadata 的 `mail_notification_contract_version=1`；mail-service metadata 的通知契约和领取/恢复开关与部署计划一致。
- 所有 mail-service 生产者都只写 v1 信封；`mail_notification_outbox.payload` 中不存在缺少 `event_type/version/mail` 的未清理记录。
- 从最后一个旧 mail producer 摘除并确认 registry 中不再出现旧 build version 开始，观察期至少覆盖一个完整的预期业务高峰；同时不得短于 chat metrics 的一个完整上报窗口（当前 5 秒）和 `CHAT_MAIL_NOTIFY_DEDUP_TTL_SECS`（默认 300 秒）中的较长者。整个观察期 `mail_notification_legacy_rejected=0`，日志中没有 `legacy_payload_disabled`。Core NATS 不持久化延迟消息，这段观察用于发现仍存活或误回滚的旧 producer，不是等待 NATS 清空历史消息。
- 两条 subject 上的 v1 双路去重已验证；删除的是旧 payload parser，不是 legacy player subject。
- 删除旧领取兼容前，`mail_claim_workflows` 无旧版本无法识别的状态，所有实例都支持当前冻结快照与 query-first 恢复，并且不存在非 `claimed` 记录。

旧 outbox 静态核对示例：

```sql
SELECT count(*) AS legacy_active_payloads
FROM mail_notification_outbox
WHERE status NOT IN ('sent', 'terminal')
  AND NOT (
    payload ? 'event_type'
    AND payload ? 'version'
    AND payload ? 'mail'
  );
```

## 正式环境变量

以下是本链路的可靠性变量；通用日志、数据库、Redis、NATS、ticket 和 service identity 仍按三个服务各自 `.env.example` 配置。

### chat-server

| 变量 | 默认值 | 正式限制 |
|------|------|------|
| `CHAT_MAIL_ACCEPT_LEGACY_PAYLOAD` | local/development `true`；strict 或 `DISCOVERY_REQUIRED=true` 未设置为 `false` | 严格策略下开启必须同时设置未来截止时间；值只接受 `true/false/1/0` |
| `CHAT_MAIL_LEGACY_COMPAT_UNTIL_EPOCH_SECONDS` | 无 | 严格策略下开启旧兼容时必填未来 Unix 秒；到期后运行中立即拒绝旧 payload |
| `CHAT_MAIL_NOTIFY_MAX_PAYLOAD_BYTES` | `4096` | 必须大于 0，解析前限制 |
| `CHAT_MAIL_NOTIFY_DEDUP_CAPACITY` | `10000` | 必须大于 0，仅进程内 |
| `CHAT_MAIL_NOTIFY_DEDUP_TTL_SECS` | `300` | 必须大于 0 |
| `CHAT_MAIL_NOTIFY_RECONNECT_BASE_MS` | `1000` | 必须大于 0 |
| `CHAT_MAIL_NOTIFY_RECONNECT_MAX_MS` | `30000` | 实际值不小于 base |

### mail-service

| 变量 | 默认值 | 正式限制 |
|------|------|------|
| `MAIL_CLAIM_NEW_REQUESTS_ENABLED` | `true` | 回滚时先切 `false`；值只接受 `true/false/1/0` |
| `MAIL_CLAIM_RECOVERY_ENABLED` | `true` | `false` 要求新领取关闭且数据库无非 `claimed` workflow，否则配置或启动失败 |
| `MAIL_CLAIM_LEASE_MS` | `30000` | `1000..300000` |
| `MAIL_CLAIM_RECOVERY_POLL_INTERVAL_MS` | `5000` | `100..60000` |
| `MAIL_CLAIM_RECOVERY_BATCH_SIZE` | `20` | `1..100` |
| `MAIL_CLAIM_RECOVERY_LEASE_MS` | `60000` | `5000..300000` |
| `MAIL_CLAIM_RECOVERY_BACKOFF_BASE_MS` | `1000` | `100..60000` |
| `MAIL_CLAIM_RECOVERY_BACKOFF_MAX_MS` | `300000` | `1000..3600000` 且不小于 base |
| `MAIL_CLAIM_RECOVERY_MAX_ATTEMPTS` | `12` | `1..100`，达到后进入 manual review |
| `MAIL_CLAIM_RECOVERY_SHUTDOWN_TIMEOUT_MS` | `10000` | `100..60000`，超时后的 lease 只能过期接管 |
| `MAIL_OUTBOX_POLL_INTERVAL_MS` / `MAIL_OUTBOX_BATCH_SIZE` / `MAIL_OUTBOX_LEASE_MS` | `5000` / `20` / `30000` | 范围见 `.env.example` |
| `MAIL_OUTBOX_MAX_ATTEMPTS` | `8` | `1..100`，不得开始 max+1 次发布 |
| `MAIL_OUTBOX_BACKOFF_BASE_MS` / `MAIL_OUTBOX_BACKOFF_MAX_MS` / `MAIL_OUTBOX_BACKOFF_JITTER_RATIO` | `1000` / `60000` / `0.2` | max 不小于 base，jitter `0..1` |
| `MAIL_OUTBOX_SENT_RETENTION_DAYS` / `MAIL_OUTBOX_TERMINAL_RETENTION_DAYS` | `7` / `30` | `1..3650` |
| `MAIL_OUTBOX_CLEANUP_INTERVAL_MS` | `3600000` | `10000..86400000` |
| `MAIL_OUTBOX_CLEANUP_BATCH_SIZE` | `500` | `1..10000` |
| `MAIL_RETENTION_DAYS` / `MAIL_CLAIM_WORKFLOW_RETENTION_DAYS` / `MAIL_GAME_GRANT_RETENTION_DAYS` | `400` / `400` / `400` | mail 与 game 保留期不能短于 workflow |
| `MAIL_CLAIM_ALERT_WINDOW_MINUTES` | `10` | `1..1440` |
| `MAIL_CLAIM_ALERT_FAILURE_RATE_PERCENT` | `20` | `1..100` |
| `MAIL_CLAIM_ALERT_LONG_RUNNING_MINUTES` | `15` | `1..10080` |
| `MAIL_CLAIM_ALERT_MANUAL_REVIEW_COUNT` | `1` | `1..100000` |
| `MAIL_OPERATIONS_TOKEN` / `MAIL_HIGH_RISK_TOKEN` | development 占位值 | production 必须为互不复用的非默认强凭证 |
| `GAME_ADMIN_TOKEN` / `GAME_ADMIN_ACTOR` | development 占位值 / service identity | production 使用服务身份；日志不得输出 token |
| `GAME_ADMIN_CONNECT_TIMEOUT_MS` / `GAME_ADMIN_WRITE_TIMEOUT_MS` / `GAME_ADMIN_READ_TIMEOUT_MS` | 各 `3000` | 正整数 |
| `GAME_ADMIN_MAX_RESPONSE_BYTES` | `1048576` | 正整数，超限按结果未知/契约失败处理 |

strict/test/staging/production 必须启用 registry 和 `DISCOVERY_REQUIRED`，移除 `GAME_SERVER_ADMIN_HOST/PORT`；玩家 `targetInstanceId` 在 strict 环境被拒绝。game-server 必须配置 PostgreSQL、Redis、`GAME_ADMIN_TOKEN`、稳定 `SERVICE_INSTANCE_ID` 和可发现的 `admin` endpoint，且幂等记录保留期不得短于 mail workflow。

## 本地验证命令

不启动外部依赖的静态/单元验证：

```powershell
npm test --workspace mail-service
cargo test --manifest-path apps/chat-server/Cargo.toml
cargo test --manifest-path apps/game-server/Cargo.toml
npx tsc --noEmit -p apps/mail-service/tsconfig.json
npm run check:proto
node tools/run-node-tests.js tests/characters/db-init-characters.test.mjs
```

真实本地联调需要 PostgreSQL 以及 Redis `6379`、Core NATS `4222`、game-server 玩家/admin、chat-server 和 mail-service。取得用户确认并准备三个服务 `.env` 后，才可执行类似：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/check-env.ps1
npm run db:init --workspace mail-service
powershell -ExecutionPolicy Bypass -File scripts/dev-stack.ps1 -WithChat -NoAuth -NoProxy -NoAdminApi -NoAdminWeb -NoMetricsCollector
npm run dev:mail
```

`dev-stack.ps1` 不负责启动 PostgreSQL 或 mail-service；两者必须由操作者单独准备。重复执行阶段 10 时应使用名称匹配 `myserver_mail_acceptance...` 的专用 PostgreSQL 数据库，并设置 `TEST_DATABASE_URL` 后运行 `node --test tests/mail/mail-reliability-fault-drill.test.mjs`；脚本会拒绝其他数据库名并在结束时删除专用数据库。不能仅凭进程启动宣称故障验收通过。
