# 管理权限与高风险操作治理设计

## 定位

本文定义 MyServer 控制面的目标安全模型，覆盖 `admin-api`、`admin-web`、`game-server` admin TCP、`game-proxy` admin HTTP，以及由管理操作触发的 Core NATS 控制事件。

当前代码仍是第一阶段静态 role、scoped token 和分散审计的基线；本文描述后续治理功能完成后直接生效的目标态，不将目标态误写为当前已实现行为。

## 已确认实施策略

- 本功能在正式上线前一次性切换，不保留旧 role 推导、旧 admin token、旧内部管理认证或旧写接口的运行时兼容分支。
- 现有管理员、审计和测试数据不要求迁移或修复；开发和测试数据库可按新 schema 重建并写入新的 seed。
- 不建设通用组织审批工作流。首期只保留操作记录上的审批状态和审批证据字段，供高风险操作接入人工审批时使用。
- development 和 test 使用独立 bootstrap 管理员、测试权限和测试签名密钥，但必须走与 production 相同的授权、预检、断言和幂等代码路径，不提供关闭授权的 bypass。
- 这是一项控制面 breaking change。所有管理类单测、GM 样例、rollout 演练和内部管理口测试必须在同一交付中改用新契约；玩家协议、房间、战斗和普通 `mock-client` 场景不在兼容范围内。

## 范围与非目标

范围包括后台管理员登录后的读写操作、跨实例 GM/NATS 操作、game-server 配置和 Room 控制、game-proxy 维护/rollout/路由控制，以及 MyForge 的受控任务入口。

公告和邮件服务已有的独立 service token 不在本次直接切换范围内；它们不得伪装为后台管理员权限，后续接入时复用本文的操作断言和审计契约。

本文不定义组织架构、多人会签 UI、任意业务数据恢复或自动回滚。恢复由明确的补偿操作、备份证据和人工 runbook 完成。

## 概念与授权模型

### 稳定标识

- `permission`：稳定的权限键，例如 `gm.send_item`、`proxy.rollout.write`、`game.config.write`。
- `resource` / `action`：被保护的资源与动作，例如 `character/grant_item`、`proxy/start_rollout`。
- `scope`：权限可操作的世界、服务、实例、角色集合、字段或最大批量数。没有匹配 scope 时默认拒绝。
- `operation_id`：一次管理操作的唯一主键。
- `request_id`：调用方为一次可重试请求提供的唯一值。同一 actor、action、target 和 payload hash 的重复提交必须返回首次结果；相同 `request_id` 但语义不同返回冲突。
- `trace_id`：贯穿 admin-api、NATS 与下游管理口的诊断关联标识。

### 数据模型

权限目录由控制面维护并以稳定键发布；前端不再维护独立的 role-to-permission 矩阵。授权分配、范围和生命周期存入 auth 数据库：

- `admin_roles`、`admin_permissions`、`admin_role_permissions`、`admin_account_roles`：角色和权限的权威关系。
- `admin_permission_grants`：针对单个管理员的额外授权、范围、授权人、理由、生效和失效时间。
- `admin_operation_requests`：请求、payload hash、actor、权限快照、目标摘要、状态、结果和唯一 `request_id`。
- `admin_operation_previews`：预检摘要、nonce、失效时间、目标/参数 hash 和审批状态。
- `admin_operation_approvals`：`not_required`、`pending`、`approved`、`rejected` 等预留状态及最小证据，不构成通用审批引擎。
- `admin_breakglass_grants`：独立紧急权限、限定动作/范围、最短必要 TTL、激活理由和撤销记录。
- `admin_audit_events`：只追加的结构化审计事件。
- `admin_operation_outbox`：已落库操作向 NATS 或内部服务可靠投递所需的待投递记录。

`admin_accounts.role`、现有 `admin_audit_logs` 和 `security_audit_logs` 不再作为新模型的授权或审计权威来源。一次性切换前的开发与测试库可重建；迁移文件仍须使用正式 SQLx 迁移，不能绕过数据库迁移体系。

### 初始风险目录

| 风险级别 | 操作类别 | 示例 | 强制要求 |
| --- | --- | --- | --- |
| 低 | 只读查询 | 审计、玩家、资产流水、监控、全局 ID 查询 | JWT、permission、scope、查询限额和审计查询访问记录 |
| 中 | 单目标可逆写入 | 非封禁玩家状态、单角色属性/称号/流派调整、普通任务取消 | `request_id`、理由、范围校验、前后摘要和追加审计 |
| 高 | 资产、处罚、全局或服务状态变更 | 发物品、封禁、踢人、全服广播、维护模式、配置 reload、rollout、路由修改、Room Transfer | 中级要求加预检、短时 nonce、影响摘要、审批状态和可靠执行记录 |
| 紧急 | 例外权限或不可逆高影响动作 | 紧急资产纠正、break-glass、允许危险权限的 MyForge 任务、批量处罚、服务关闭 | 高级要求加独立 permission、短时授权、双重确认、强审计、告警和恢复证据 |

资产、封禁、配置、服务控制和批量操作不能因为当前 controller 使用了不同路由或下游而绕过上述分级。只读操作不进入预检执行流程，但仍必须按权限和范围读取。

## 统一执行协议

所有控制面写操作使用统一操作 API 或其等价内部服务接口，旧的直接写 endpoint 在切换时删除。

1. 客户端提交 action、target、payload、`request_id` 和理由。
2. admin-api 解析有效 permission 和 scope，执行输入、字段脱敏、目标数量和服务/世界范围校验。
3. 高风险或紧急操作创建预检记录，返回影响摘要、参数/目标 hash、短时 nonce、审批状态和过期时间。
4. 客户端确认后携带同一 `request_id` 与 nonce 执行。服务端再次校验 actor、scope、payload hash、nonce、审批状态和目标当前状态。
5. admin-api 在同一事务中写操作状态和 outbox；重复请求返回首次终态或正在执行状态，不能触发第二次下游副作用。
6. 下游结果、NATS 投递结果和补偿信息回写操作记录，并追加审计事件。

建议状态机为 `preflighted -> approved | executing -> succeeded | failed | execution_uncertain | cancelled`。`execution_uncertain` 只能通过查询下游幂等结果或显式补偿收敛，不能假定重试安全。

所有控制面写操作都要求 `request_id` 和非空理由；高风险和紧急操作额外要求有效预检 nonce。缺少、过期、重复使用或与 payload hash 不符的 nonce 必须拒绝，不能降级为旧调用。

## 内部管理断言与跨实例投递

admin-api 是唯一向 `game-server`、`game-proxy` 和管理 NATS 消费者下发管理员身份的服务。它为每次下游写操作签发短时 `AdminOperationAssertion`，字段至少包括：

- `operation_id`、`request_id`、`trace_id`、issuer、key version、签发和过期时间。
- 操作人、有效 permission、scope 摘要、目标摘要和 payload hash。
- 下游 service、实例或世界绑定，防止断言被转发到其他资源。
- 非对称签名。首期使用 Ed25519，并在 `packages/proto` 定义稳定的内部控制消息和版本字段。

game-server admin TCP 和 game-proxy admin HTTP 在状态改变前验证签名、过期时间、目标绑定、permission 与本地路由要求，并按 `request_id` 去重。它们不再接受可单独完成写操作的通用 admin token 或任意 actor 字符串。NATS 控制事件同样携带断言或等价签名 envelope，消费者只处理一次并将 `request_id` 写入幂等记录。

本地开发调试通过 bootstrap 管理员和开发私钥生成同样的断言；测试使用固定 test key。私钥、测试 token、真实 token 和环境文件均不得进入 Git。

## 审计、失败与恢复

每个审计事件至少记录操作人、permission 快照、scope 摘要、`operation_id`、`request_id`、`trace_id`、理由、目标、结果、时间和来源 IP。关键操作额外保存最小必要的前后状态摘要、关联资产流水或下游幂等记录。

审计中不保存密码、token、私钥、完整 ticket、完整 Room Transfer payload、完整 MyForge prompt 或不必要的玩家隐私。广播与业务 payload 记录内容 hash、受控摘要和必要的公开标识，而不是无边界原文。

高风险和紧急操作在无法持久记录预检、执行状态或审计时必须在副作用前拒绝。副作用已经进入下游但最终状态写入失败时，记录 `execution_uncertain` 和可靠 outbox，由查询或补偿流程收敛，不能静默返回成功。

可逆操作使用新的补偿命令，例如解除封禁、关闭维护模式、恢复上一条已验证路由或配置版本。不得直接修改或删除历史审计。不可逆操作必须在预检中显示备份标识、影响范围、恢复命令和风险摘要。

紧急操作要求独立 `breakglass.activate` 权限和操作级紧急 permission。授权必须限定目标和动作、使用不超过 15 分钟的 TTL、记录理由并触发安全告警；不能由普通 `admin` 权限隐式获得。

## 限流、告警与查询

按管理员、permission 和目标范围设置控制面限流。批量操作保存游标、最大目标数、暂停、取消和部分失败状态。异常频率、权限拒绝、nonce 重放、审计写入失败、break-glass 启用和下游断言校验失败必须产生日志、指标和安全告警。

审计查询支持管理员、permission、action、目标、`request_id`、`trace_id`、时间、结果和风险级别筛选，并限制时间窗口、分页和导出数量。管理员账号不得拥有修改或删除自己审计记录的数据库权限。

## 实施边界

一次性切换交付仍可按代码模块拆分提交，但任何可部署版本必须同时具备以下内容：

1. auth 数据库的新 schema、bootstrap 管理员和测试 seed。
2. admin-api 的 policy、operation、审计、outbox 和签名断言服务。
3. admin-web 的有效权限读取、预检、确认、执行和错误状态。
4. game-server、game-proxy、相关 NATS 消费者对新断言和 `request_id` 的校验及去重。
5. 现有管理类单测、GM 样例、rollout/故障演练迁移到新测试 helper。
6. 管理后台、安全、协议和应急文档同步。

主要代码落点预计为：

- `db/migrations/auth/`、`db/seeds/`：权限、操作、审计和开发/测试 bootstrap。
- `apps/admin-api/src/auth/`、`apps/admin-api/src/gm/`、`players/`、`maintenance/`、`admins/`、`myforge/`、`admin-store.js`：统一 policy 和操作执行入口。
- `apps/admin-web/src/auth/`、`api/`、`views/`：移除静态权限矩阵，改用服务端下发的有效 capability 和操作状态。
- `packages/proto/`、`apps/game-server/src/admin_server/`、`apps/game-proxy/src/admin_server/`：签名操作断言、权限/目标校验和下游去重。
- `tests/admin/`、`tests/ops/`、各服务管理口测试：新的身份、签名、预检和执行 fixture。

真实高风险联调仍必须在执行前列出 Redis、PostgreSQL、NATS、目标 game-server/game-proxy 实例和可能影响，并按仓库协作约定等待用户确认。

## 完成定义

- 每一条管理写路径均可解析到 permission、scope、`request_id` 和追加审计记录。
- 高风险操作可预览、确认、幂等执行，并在异常时留下可恢复的状态和证据。
- 下游管理口只接受短时、目标绑定的操作断言，不能通过旧 token 或 actor 字符串扩大权限。
- development、test、预发和 production 不存在关闭授权的 bypass；差异只限于独立 bootstrap 身份、密钥和环境数据。
