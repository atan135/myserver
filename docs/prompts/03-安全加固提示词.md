# 安全加固提示词

## 审阅前可修改项

- TLS 方案：`开发环境自签证书 / 正式环境 CA 证书`
- 内网边界：`单机本地开发`
- 认证方案：`HTTP token + game ticket；内部 service token，正式环境 mTLS`
- 密钥管理方式：`.env`
- 审计保留周期：`30 天`
- 限流策略：`IP + 账号 + token 多维限流`

## 正式提示词

```text
你现在作为安全架构师兼后端工程师，针对当前“通用游戏后端服务器框架”设计并落地第一批安全加固措施。请先检查当前仓库和已有实现，再开始工作。

系统背景：
1. HTTP 登录服务器使用 Node.js + Express，对外 JSON
2. 游戏服务器优先使用 Rust，通过原生 TCP 与玩家通信
3. 玩家协议优先采用“自定义 TCP 包头 + Protobuf 消息体”
4. 游戏服还需要与 HTTP 服务以及管理工具交互
5. 房间规模为 5-10 人

版本与环境约束：
1. Node.js：22.20.0
2. Express：5.x
3. Rust：1.94.0
4. TLS 方案：开发环境自签证书，正式环境 CA 证书
5. 凭证与密钥管理：HTTP token + game ticket，密钥通过 .env 管理，正式环境建议接入 mTLS
6. 网络边界与部署方式：Windows 10 本地开发 / 单机部署
如仓库已有更明确实现，请优先遵循现有工程实际。

你的任务：
1. 审查当前方案中的主要安全风险
2. 给出按优先级排序的安全加固建议
3. 直接落地第一批必要的安全措施
4. 为暂时不实现的高级安全能力留出扩展点

必须重点覆盖以下方面：
1. HTTP 登录安全
2. TCP 接入安全
3. 内部服务调用安全
4. 管理控制面安全
5. 配置与参数热更新安全
6. 日志与审计安全
7. 密钥与敏感信息管理
8. 基础 DoS / 滥用防护

至少需要考虑并尽可能落地的内容：
1. HTTP 输入校验
2. 登录限流、连接限流、消息频率限制
3. 账号或游客登录后的短时 game ticket 设计
4. TCP 未鉴权连接白名单限制
5. 消息长度限制、消息类型白名单、非法包处理
6. 基础重放防护或可追踪机制
7. 管理接口鉴权、授权、审计
8. 内网控制面的访问限制
9. 环境变量和密钥不入库
10. 明确哪些接口需要 TLS / mTLS

约束要求：
1. 不要把“未来理想方案”当成“当前已完成”
2. 第一版不要实现任意代码热更新
3. 如果需要热更新，只允许参数或配置级热更新，并要求具备权限控制、审计记录和回滚思路
4. 不要为了安全而引入明显过重、无法落地的复杂组件，优先做当前阶段最值得做的措施

输出要求：
1. 先列出安全问题清单，按严重程度排序
2. 再说明你准备落地哪些改动
3. 然后实施改动
4. 最后说明剩余风险、暂未覆盖项和后续建议
5. 对每一项已实现的安全措施，给出验证方法

如果安全加固会影响现有协议、接口或配置，请同步更新文档和测试脚本。
```
