# 架构设计提示词

## 审阅前可修改项

请先按你的本地环境补充或修正以下版本与约束：

- Node.js 版本：` v22.20.0`
- Express 版本：`<待确认，例如 5.x>`
- Rust 版本：`rustc 1.94.0 (4a4ef493e 2026-03-02)`
- Cargo 版本：`cargo 1.94.0 (85eff7c80 2026-01-15)`
- Protobuf 版本：`proto3`
- MySQL 版本：`MariaDB 10.5.29`
- Redis 版本：`5.0.14.1`
- 操作系统：`本地Windows 11 / 生产环境Ubuntu 22.04`
- 部署形态：`本地开发 / 单机部署`

## 正式提示词

```text
你现在作为资深后端架构师，帮助我为一个“通用游戏后端服务器框架”输出第一版架构设计方案。请先阅读当前仓库，不要臆测现状。

项目目标：
1. 提供一个 HTTP 登录服务器
2. 提供一个 TCP 游戏服务器
3. 支持 MySQL 和 Redis
4. 支持无真实客户端依赖的手动校验和自动化校验
5. 支持后续扩展管理工具、配置热更新和安全加固

明确技术边界：
1. HTTP 登录服务器必须使用 Node.js + Express
2. HTTP 对外接口使用 JSON
3. TCP 游戏服务器优先使用 Rust 实现；如你认为 C++ 更合适，可以对比，但默认优先 Rust
4. 玩家与游戏服务器通过原生 TCP 长连接通信
5. 游戏服务器除与玩家通信外，还需要与 HTTP 服务交互，并支持管理工具修改参数、触发管理操作
6. 房间规模目标是每个 room 5-10 人

版本与环境约束：
1. Node.js：22.20.0
2. Express：5.x
3. Rust：1.94.0
4. Protobuf：proto3，protoc 3.19.5
5. MySQL：MariaDB 10.5.29
6. Redis：Redis 5.0.14.1
7. 操作系统与运行环境：Windows 10 10.0.19044.7058，PowerShell 5.1，本地开发 / 单机部署
如果当前仓库或环境中已经存在更明确的信息，请以实际内容为准，并指出与本提示词不一致的地方。

请你先完成架构设计，不要直接开始大规模编码。设计内容必须明确回答以下问题：
1. 为什么 Rust 或 C++ 更适合当前 TCP 游戏服
2. 为什么最终推荐的技术栈适合“通用框架 + 小房间长连接 + 后续扩展”
3. HTTP 服、游戏服、MySQL、Redis、管理控制面的职责边界是什么
4. HTTP 服与游戏服之间应采用什么内部通信方式
5. 管理工具应如何安全地修改游戏服参数或触发管理操作
6. 玩家通信协议应如何设计
7. 游戏服如何处理连接、会话、鉴权、心跳、断线、重连、幂等、限流、审计和可观测性
8. 哪些能力应该在第一版实现，哪些只做扩展点预留

协议与通信要求：
1. HTTP 对外统一使用 JSON
2. 玩家链路使用原生 TCP
3. 优先采用“自定义 TCP 包头 + Protobuf(proto3)消息体”的方案，除非你有更强理由推翻它
4. 管理控制面不要和玩家 TCP 通道混用
5. 内部通信请比较 gRPC、内部 HTTP/JSON、自定义 TCP 控制通道，并给出推荐

安全要求：
1. 必须从第一版开始考虑安全，不接受“后续再补”
2. 登录、内部服务调用、管理控制面都要有清晰鉴权方案
3. TCP 连接建立后必须先鉴权，未鉴权连接只能访问极少数白名单消息
4. 需要考虑消息长度限制、连接频率限制、请求频率限制、重放防护、非法消息处理、权限隔离
5. 管理接口要有审计和变更记录思路
6. 第一版不做任意代码热更新；如需热更新，只允许参数或配置级热更新，并给出未来 Lua/WASM sandbox 预留建议

输出要求：
1. 先总结当前仓库现状
2. 再给出架构图或文字化架构说明
3. 再给出技术选型对比和最终建议
4. 明确列出第一版范围、后续范围、已知风险和默认假设
5. 给出建议目录结构
6. 给出下一阶段实施计划，但先不要直接写大量代码

如果你发现我给出的要求有冲突，请明确指出并提出替代方案。
```
