# 首版落地提示词

## 审阅前可修改项

- Node.js 版本：`22.20.0`
- Express 版本：`5.x`
- Rust 版本：`1.94.0`
- Tokio 版本：`1.x`
- rustls 版本：`0.23.x`
- Protobuf 工具链：`proto3 + protoc 3.19.5`
- ORM / SQL 访问方案：`mysql2`
- 配置方式：`.env`

## 正式提示词

```text
你现在作为资深实现工程师，在当前仓库中直接落地“通用游戏后端服务器框架”的第一版最小可运行版本。开始前请先检查当前仓库，避免臆测。

架构前提：
1. HTTP 登录服务器使用 Node.js + Express
2. 玩家与游戏服务器通过原生 TCP 长连接交互
3. TCP 游戏服务器优先使用 Rust 实现
4. HTTP 使用 JSON
5. 玩家 TCP 消息优先采用“自定义包头 + Protobuf(proto3)消息体”
6. MySQL 和 Redis 本地已存在，可直接接入

版本与环境信息：
1. Node.js：22.20.0
2. Express：5.x
3. Rust：1.94.0
4. Tokio：1.x
5. rustls：0.23.x
6. Protobuf：proto3，protoc 3.19.5
7. MySQL：MariaDB 10.5.29
8. Redis：Redis 5.0.14.1
如仓库内已有更准确信息，请按仓库实际情况执行并说明差异。

请在确认架构方案后，直接实现第一版最小可运行闭环，至少包括：
1. HTTP 登录服基础骨架与启动方式
2. 一个最小登录闭环，例如游客登录或账号登录
3. 登录成功后签发 game ticket 或其他适合 TCP 鉴权的短时凭证
4. Rust TCP 游戏服基础骨架与启动方式
5. TCP 建连、拆包、封包、鉴权、心跳、基础消息路由、统一错误码
6. MySQL 基础连接和最小用户表
7. Redis 基础连接和最小会话或 ticket 存储
8. 一个最小内部控制面闭环，例如：
   - 查询游戏服状态
   - 查询在线连接数
   - 查询房间数量
   - 修改某项房间或服务器参数
9. 配置管理、日志、错误处理、环境变量示例
10. README、协议说明、启动说明、数据库初始化说明

实现原则：
1. 第一版优先最小可运行，不要一开始过度抽象
2. 结构要清晰，保留后续扩展空间
3. 玩家通道和管理控制面不要混用
4. 不要依赖真实客户端才能验证
5. 所有关键模块都要有最小可观测性，例如关键日志和错误码
6. 如果发现我当前需求无法一次性全部实现，请优先保证主链路闭环

建议你按以下顺序工作：
1. 检查仓库现状
2. 输出实施计划
3. 创建目录结构
4. 实现 HTTP 登录服
5. 实现 Rust TCP 游戏服
6. 实现内部控制面
7. 实现数据库与缓存接入
8. 补齐 README 与脚本
9. 给出逐阶段验证方式

每完成一个阶段，请明确告诉我：
1. 新增了什么
2. 还缺什么
3. 我该如何在本地验证

如果没有明显阻塞，不要停留在讨论层面，直接开始实现。
```
