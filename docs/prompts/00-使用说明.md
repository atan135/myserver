# 使用说明

这份说明用于帮助你按正确顺序使用以下 4 份提示词，并在使用前补齐必要的本地环境信息。

相关文件：

- [01-架构设计提示词.md](./01-架构设计提示词.md)
- [02-首版落地提示词.md](./02-首版落地提示词.md)
- [03-安全加固提示词.md](./03-安全加固提示词.md)
- [04-测试与校验提示词.md](./04-测试与校验提示词.md)
- [版本清单.md](./版本清单.md)

## 这 4 份提示词是什么关系

它们是一个完整任务的分阶段拆分，不是互相独立的 4 份随机模板。

推荐理解方式：

- `01`：先定架构和边界
- `02`：再做第一版可运行实现
- `04`：尽快补测试和校验闭环
- `03`：在主链路稳定后做安全强化
- `04`：安全加固后再次回归测试

推荐使用顺序：

1. `01-架构设计提示词`
2. `02-首版落地提示词`
3. `04-测试与校验提示词`
4. `03-安全加固提示词`
5. 再次使用 `04-测试与校验提示词` 做回归

原因：

- 没有 `01`，后续实现容易反复推翻
- 没有 `02`，安全和测试都缺少落点
- 没有第一版 `04`，你很难判断 `03` 是否把主流程改坏
- `03` 不应该脱离实际工程单独先做

## 你现在该先做什么

你现在最适合先做的是：

1. 先看 [版本清单.md](./版本清单.md)
2. 确认里面的“实际版本”和“建议版本”是否符合你的预期
3. 再把其中的确定项同步回填到各提示词中的 `<待确认>` 区块
4. 优先先补 `01` 和 `02`
5. 再把 `01` 提示词交给 Codex

建议顺序：

1. 先核对 `版本清单`
2. 再填 `01`
3. 再填 `02`
4. 如果你已经确定测试工具，再填 `04`
5. 如果你已经确定 TLS、鉴权、密钥管理方式，再填 `03`

## 每份提示词在使用前至少要补什么

### 01-架构设计提示词

最少补这些：

- Node.js 版本
- Express 版本
- Rust 版本
- Protobuf 版本
- MySQL 版本
- Redis 版本
- 操作系统
- 部署形态

这一份的目标不是写代码，而是让 Codex 先把这些问题讲清楚：

- HTTP 服和 TCP 游戏服怎么拆
- 为什么选 Rust 或 C++
- 玩家协议怎么定
- HTTP 服和游戏服怎么通信
- 管理控制面怎么设计
- 第一版做什么，不做什么

完成标志：

- 产出一份你认可的架构方案
- 关键技术选型已经明确
- 第一版范围和后续范围已区分清楚

### 02-首版落地提示词

最少补这些：

- Node.js 版本
- Express 版本
- Rust 版本
- Tokio 版本
- rustls 版本
- Protobuf 工具链版本
- MySQL 访问方案
- Redis 访问方案
- 配置方式

这一份的目标是让 Codex 开始真正落地。

你希望它至少产出：

- HTTP 登录服骨架
- Rust TCP 游戏服骨架
- 一个最小登录闭环
- 一个最小 TCP 鉴权和心跳闭环
- 一个最小内部控制面
- README 和启动说明

完成标志：

- 项目可以本地启动
- HTTP 和 TCP 主链路能跑通
- 可以不依赖真实客户端进行基础验证

### 03-安全加固提示词

最少补这些：

- TLS 方案
- 内网边界
- 认证方案
- 密钥管理方式
- 审计保留周期
- 限流策略

这一份的目标是对已有实现做“第一批值得做的安全加固”，不是无限展开。

你希望它重点处理：

- 登录安全
- TCP 接入安全
- 内部控制面安全
- 参数热更新安全
- 审计和敏感信息保护

完成标志：

- 已识别并排序主要安全风险
- 至少落地一批高优先级安全措施
- 文档和验证方法同步更新

### 04-测试与校验提示词

最少补这些：

- HTTP 测试工具
- TCP mock client 方案
- Node 测试框架
- Rust 测试框架
- 压测工具
- CI 或本地执行方式

这一份的目标是建立“无客户端依赖”的验证体系。

你希望它至少覆盖：

- HTTP 登录测试
- game ticket 获取
- TCP 建连与鉴权
- 心跳
- 基础消息收发
- 管理控制面调用
- 异常路径测试

完成标志：

- 存在手工验证命令或脚本
- 存在自动化测试命令
- 存在 mock client
- 出问题时可以快速定位

## 实际使用方式

推荐操作方式：

1. 先在本地确认 [版本清单.md](./版本清单.md)
2. 再把某一份提示词中的 `<待确认>` 信息补齐
3. 然后把整份提示词交给 Codex
4. 等 Codex 完成当前阶段后，再进入下一份提示词
5. 不建议 4 份提示词一次性同时丢给 Codex

不建议的用法：

- 一次性把 4 份都发过去
- 在 `01` 还没稳定时直接执行 `03`
- `02` 已经改了架构，但 `01` 和 `04` 没同步

## 建议的阶段检查点

在每个阶段结束时，你可以手动检查这 3 件事：

1. 当前阶段目标是否已经完成
2. 文档是否和实现保持一致
3. 下一阶段依赖的信息是否已经明确

你可以用下面这套简单判断：

- `01` 结束后：我是否已经认可当前架构和技术选型
- `02` 结束后：我是否已经能本地启动并跑通主链路
- `04` 结束后：我是否已经能重复验证主链路和异常路径
- `03` 结束后：我是否已经补上当前阶段最重要的安全措施

## 一条实用建议

你在填写 `<待确认>` 时，优先填“当前本机真实可用版本”，而不是填“理论上最想用的版本”。

原因很简单：

- 你现在的目标是尽快做出第一版闭环
- 真实环境版本决定了工具链能不能立刻跑起来
- 理论最优版本可以在后续升级，不应该阻塞首版落地
