# 测试与校验提示词

## 审阅前可修改项

- HTTP 测试工具：`curl`
- TCP mock 工具：`Node.js 脚本`
- 自动化测试框架：`node:test + cargo test`
- 压测工具：`k6`
- CI 环境：`本地执行`

## 正式提示词

```text
你现在作为测试工程师兼后端工程师，为当前“通用游戏后端服务器框架”建立一套不依赖真实客户端的手动校验与自动化校验体系。请先检查当前仓库已有测试、脚本、文档和启动方式，不要臆测现状。

系统背景：
1. HTTP 登录服务器使用 Node.js + Express，对外返回 JSON
2. 游戏服务器优先使用 Rust，通过 TCP 长连接与玩家通信
3. 玩家协议优先采用“自定义 TCP 包头 + Protobuf 消息体”
4. 游戏服还需要支持内部控制面
5. MySQL 和 Redis 参与主流程

版本与工具约束：
1. HTTP 测试工具：curl
2. TCP mock client 方案：Node.js 脚本
3. Node 测试框架：node:test
4. Rust 测试框架：cargo test
5. 压测工具：k6
6. CI 或本地执行方式：本地执行
如果仓库内已有现成工具链，请优先复用。

你的目标：
1. 建立可重复执行的手动验证方案
2. 建立自动化测试方案
3. 建立基础冒烟测试
4. 建立 HTTP 接口测试
5. 建立 TCP 协议测试
6. 建立内部控制面测试
7. 提供基础压测或容量验证建议

必须覆盖的验证链路：
1. HTTP 登录
2. 获取 game ticket
3. 建立 TCP 连接
4. 发送鉴权消息
5. 发送心跳
6. 收发至少一类基础业务消息
7. 调用管理接口修改参数
8. 验证修改结果生效
9. 验证异常路径，例如：
   - 非法 ticket
   - 未鉴权发业务消息
   - 超长消息
   - 心跳超时
   - 频率超限

交付要求：
1. 提供手动验证命令或脚本
2. 提供自动化测试命令
3. 提供 mock client 或脚本化测试客户端
4. 提供运行前置条件说明
5. 提供测试数据初始化方法
6. 提供成功结果与失败结果的判定方式
7. 尽量提升可观测性，帮助定位失败原因

输出顺序要求：
1. 先总结当前已有测试基础
2. 再给出测试策略
3. 然后实现脚本、测试和文档
4. 最后给出完整执行步骤

请优先复用项目已有脚本和工具，不要无意义引入新框架。如果没有明显阻塞，请直接开始实现。
```
