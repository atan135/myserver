# 服务发现迁移基线

本文是统一服务发现迁移的 M0 基线，只盘点当前代码中的注册事实。后续还需要继续补充：消费发现状态、固定配置扫描、统一入口/准入/本地 fallback 口径。

端口和地址只记录本地默认监听值或默认发布值，用于理解现状；不要把这些默认值当作测试或线上推荐直连配置。

## 当前注册状态

当前 Redis service registry 的通用数据模型来自 `packages/service-registry`：

- Rust 服务使用 `packages/service-registry/src/client.rs` 的 `RegistryClient`，实例写入 `service:<service>:instances:<instance>` 的 `data` 字段，心跳写入 `heartbeat:<service>:<instance>`，默认心跳 TTL 为 30 秒。
- Node 服务各自有本地 `src/registry-client.js`，但复用 `packages/service-registry/node/registry-schema.js` 生成 payload 和 key。它们同样写 `data` 字段和 `heartbeat:<service>:<instance>`。
- Node 周边服务配置里有 `REGISTRY_ENABLED`/`DISCOVERY_REQUIRED` 用于发现侧校验，但自身注册启动代码当前直接调用 `registryClient.register()`；注册失败只记录 `startup.registry_failed`，不会因为默认 `REGISTRY_ENABLED=false` 自动跳过注册。

### Heartbeat 与 deregister

| 服务 | heartbeat 行为 | deregister 行为 |
|---|---|---|
| `game-server` | `register()` 时写一次 heartbeat；随后 `start_heartbeat_task()`，间隔来自 `REGISTRY_HEARTBEAT_INTERVAL`，默认 10 秒，TTL 30 秒 | `server::run` 返回后 abort heartbeat task，再 `deregister()` 删除实例和 heartbeat key |
| `game-proxy` | `register()` 时写一次 heartbeat；随后 `start_heartbeat_task()`，当前未调用 `with_heartbeat_interval`，使用 registry 包默认 10 秒，TTL 30 秒 | `proxy_server::run` 返回后 abort heartbeat task，再 `deregister()` 删除实例和 heartbeat key |
| `chat-server` | `register()` 时写一次 heartbeat；随后 `start_heartbeat_task()`，间隔来自 `REGISTRY_HEARTBEAT_INTERVAL`，默认 10 秒，TTL 30 秒 | 主循环返回后释放相关任务，再 `deregister()` 删除实例和 heartbeat key |
| `match-service` | `register()` 时写一次 heartbeat；随后 `start_heartbeat_task()`，间隔来自 `REGISTRY_HEARTBEAT_INTERVAL`，默认 10 秒，TTL 30 秒 | `server::run` 返回后 abort heartbeat task，再 `deregister()` 删除实例和 heartbeat key |
| `auth-http` | `main.ts` 调用 `startHeartbeat(10)`；Node client 立即 `setex` 一次，之后每 10 秒续写，TTL 30 秒 | `closeNestApp` 先 `stopHeartbeat()`，再 `deregister()` 删除实例和 heartbeat key |
| `mail-service` | `main.ts` 调用 `startHeartbeat(10)`；立即写 heartbeat，之后每 10 秒续写，TTL 30 秒 | `closeNestApp` 先 `stopHeartbeat()`，再 `deregister()` 删除实例和 heartbeat key |
| `announce-service` | `main.ts` 调用 `startHeartbeat(10)`；立即写 heartbeat，之后每 10 秒续写，TTL 30 秒 | `closeNestApp` 先 `stopHeartbeat()`，再 `deregister()` 删除实例和 heartbeat key |
| `admin-api` | `main.ts` 调用 `startHeartbeat(10)`；立即写 heartbeat，之后每 10 秒续写，TTL 30 秒 | `closeNestApp` 先 `stopHeartbeat()` 和 `stopDiscoveryRefresh()`，再 `deregister()` 删除实例和 heartbeat key |

### metrics-collector 不注册的代码依据

`apps/metrics-collector/src/registry-client.js` 的 `maybeRegisterService()` 默认返回 `registered: false`，原因文本是：它消费 NATS metrics 并写 Redis snapshots，但没有可连接服务 endpoint。若设置 `SERVICE_REGISTRY_REGISTER=true`，该函数会直接抛错，说明在暴露真实 `http`/`tcp`/`local_socket` endpoint 之前不支持注册。

因此 `metrics-collector` 当前不是服务发现生产者。它写入的 `metrics:*` 和 `metrics:heartbeat:*` key 是指标快照与指标新鲜度，不是 service registry 实例记录。

补充口径：

- “严格模式”在当前实现中主要由 `DISCOVERY_REQUIRED=true` 或 `NODE_ENV` / `APP_ENV` 为 `production` / `test` 触发；`rollout-three-process-drill.ps1` 还把 `staging` / `testing` 视作严格环境。
- Node 服务的旧直连变量在非本地或严格发现下会被忽略并输出 warning；设置 `DISALLOW_LEGACY_DIRECT_CONFIG=true` 时，相关变量存在会直接拒绝启动。
- 多实例 admin 控制链路的当前边界是：读/聚合类操作可以遍历所有 endpoint，写入到单个 game-server 的操作需要显式 `targetInstanceId`。

### 6. 扫描结论

- `.env.test.example` 和 `.env.production.example` 当前只保留 `REGISTRY_ENABLED=true`、`DISCOVERY_REQUIRED=true` 和 `REGISTRY_URL`，未发现测试/生产示例中残留 `GAME_PROXY_HOST`、`MATCH_SERVICE_ADDR`、`GAME_SERVER_ADMIN_HOST`、`GAME_SERVER_ADMIN_PORT`、`UPSTREAM_LOCAL_SOCKET_NAME`、`GAME_SERVER_INTERNAL_SOCKET_NAME` 这类消费者直连配置。
- `.env.example` 中仍保留的消费者直连变量已经标注为 `Local fallback only` 或默认注释，当前口径是本地 registry disabled 且 discovery 非严格时才可用；严格环境会忽略、告警或通过 `DISALLOW_LEGACY_DIRECT_CONFIG=true` 拒绝。
- 可见 `.env` 主要是本机开发覆盖值；其中 `apps/chat-server/.env` 的 `GAME_HOST` / `GAME_PORT` 未在当前 `apps/chat-server/src` 中发现读取路径，属于旧本地残留风险，需要后续清理或确认是否仍有外部脚本依赖。
- `scripts` 和 `tools/mock-client` 保留了大量 `127.0.0.1` 默认值，但主要用于本地栈绑定、端口探测、registry fixture / dry-run、帮助文本和 manual drill；灰度主脚本已明确 strict/test/production 必须通过 registry discovery 解析 endpoint。
- 按本次检索口径，未发现未标注的测试/线上消费者直连残留；仍需注意文档中的默认端口表和历史设计说明可能被误读，后续文档收敛应继续统一为“默认监听值 / 本地 fallback / registry-first”三类表述。

| Rust `packages/service-registry` | `RegistryClient::new()` 接收调用方传入 URL；包内默认 key prefix 为 `REGISTRY_KEY_PREFIX -> REDIS_KEY_PREFIX -> ""` | `default_key_prefix()` 和 `with_key_prefix()` 都作用于 `service:<name>:instances:<id>`、`heartbeat:<name>:<id>` 与发现缓存 key | 被 Rust 生产者/消费者复用。 |
| `game-server` | `REGISTRY_URL -> REDIS_URL -> redis://127.0.0.1:6379` | `REGISTRY_KEY_PREFIX -> REDIS_KEY_PREFIX -> ""` | 注册自身；发现 `match-service.grpc`；registry key prefix 实际生效。 |
| `game-proxy` | `REGISTRY_URL -> REDIS_URL -> redis://127.0.0.1:6379` | `REGISTRY_KEY_PREFIX -> REDIS_KEY_PREFIX -> ""` | 注册自身；发现 `game-server.proxy-local`；registry key prefix 实际生效。 |
| `chat-server` | `REGISTRY_URL -> REDIS_URL -> redis://127.0.0.1:6379` | `REGISTRY_KEY_PREFIX -> REDIS_KEY_PREFIX -> ""` | 注册自身；当前不发现其他服务；registry key prefix 仅用于自身注册。 |
| `match-service` | `REGISTRY_URL -> REDIS_URL -> redis://127.0.0.1:6379` | `REGISTRY_KEY_PREFIX -> REDIS_KEY_PREFIX -> ""` | 注册自身；发现 `game-server.internal`；registry key prefix 实际生效。 |
| Node 服务 `auth-http` / `admin-api` / `mail-service` / `announce-service` | 当前 `config.js` 没有独立 `registryUrl` 字段，registry client 使用应用已创建的 Redis client；该 Redis client 读取 `REDIS_URL -> redis://127.0.0.1:6379` | `REGISTRY_KEY_PREFIX ?? REDIS_KEY_PREFIX ?? ""` | 4 个服务注册自身；`auth-http`、`admin-api`、`mail-service` 还消费 discovery；registry key prefix 实际生效。风险是模板里有 `REGISTRY_URL`，但 Node 运行时仍需要部署层把同一个 Redis 地址提供给 `REDIS_URL`，否则 registry 连接不会单独使用 `REGISTRY_URL`。 |
| `metrics-collector` | 不注册、不发现；只读取 `REDIS_URL` 写 metrics snapshot | 保留 `registryKeyPrefix = REGISTRY_KEY_PREFIX ?? REDIS_KEY_PREFIX ?? ""` 字段，但 `maybeRegisterService()` 默认不写 registry，`server.js` 的 `metrics:*` key 也不套该 prefix | `registryKeyPrefix` 当前是保留字段，不是实际 service registry 使用者。 |

`REDIS_KEY_PREFIX` 是业务 Redis key 的隔离前缀，和 `REGISTRY_KEY_PREFIX` 可以相同也可以分开。当前实际使用业务 Redis prefix 的路径包括：

- `auth-http`：session、ticket、ticket version、限流、动态黑名单、维护模式、metrics 和 global id lease。
- `game-server`：ticket / ticket version 校验相关 key。
- `game-proxy`：ticket 校验、动态黑名单、共享维护模式；route store 另有 `PROXY_ROUTE_STORE_KEY_PREFIX`，未设置时 fallback 到 `REDIS_KEY_PREFIX`。
- `chat-server`：ticket 校验、在线路由 key。
- `match-service`：Redis runtime store 入口传入 `redis_key_prefix`；匹配运行时另有 `MATCH_RUNTIME_KEY_PREFIX`。
- `admin-api`：后台 session、admin store、共享维护模式。
- `mail-service`：ticket 校验、聊天在线路由查询、global id lease。
- `announce-service`：ticket 校验、global id lease。
- `metrics-collector`：当前 metrics key 固定为 `metrics:*` / `metrics:heartbeat:*`，不使用 `REDIS_KEY_PREFIX`。

### 6. 结论与风险

- 测试/生产 strict discovery 入口已经形成统一口径：8 个 registry 参与服务都设置 `REGISTRY_ENABLED=true`、`DISCOVERY_REQUIRED=true`，并用 `REGISTRY_URL` + `REGISTRY_KEY_PREFIX` 表达 registry Redis 入口和环境隔离。
- Rust 服务的 registry URL fallback 链路已经统一为 `REGISTRY_URL -> REDIS_URL -> 本地默认`；Node 服务当前 registry 操作复用 `REDIS_URL` 创建的 Redis client，`REGISTRY_URL` 只出现在模板口径里。测试/线上若只设置 `REGISTRY_URL` 而未同步设置 `REDIS_URL`，Node 服务会继续连 `REDIS_URL` 或本地默认，这是需要部署层规避或后续代码收敛的差异。
- `REGISTRY_KEY_PREFIX` fallback 到 `REDIS_KEY_PREFIX` 或空字符串的口径已在 Rust 与 Node 注册/发现客户端统一；生产模板默认空 prefix，仅适合独占 Redis。共享 Redis 时必须显式设置 prefix，例如按环境、集群或 namespace 隔离。
- `NATS_URL` 和 `DATABASE_URL` 属于基础设施入口，不是服务发现入口。测试/生产覆盖模板当前没有覆盖这两个值，实际环境必须提供真实 NATS / PostgreSQL 地址。
- 服务实例 ID 的唯一性仍依赖部署层。代码默认值和 `.env.example` 固定值适合单机本地开发，不适合多副本测试/线上；部署系统应按服务名、环境、pod / task / process id 或端口生成唯一 `SERVICE_INSTANCE_ID`。

## 迁移前后准入口径基线

本节定义统一服务发现迁移的准入边界。核心目标是：本地开发可以保留显式 fallback，测试、预发和线上不允许 `REGISTRY_ENABLED=false`，也不允许把旧直连变量当作服务间入口。

### 1. 当前严格环境判定

| 位置 | 严格条件 | 当前行为 |
|---|---|---|
| Node 服务 `auth-http` / `admin-api` / `mail-service` / `announce-service` | `DISCOVERY_REQUIRED=true`，或 `NODE_ENV` / `APP_ENV` 为 `production` / `test` | `config.js` 将 `registryDiscoveryRequired` 置为 true；若 `REGISTRY_ENABLED=false`，配置加载阶段直接抛错。`auth-http`、`admin-api`、`mail-service` 还会在严格或非本地 fallback 场景忽略旧直连变量并输出 warning。 |
| Rust 服务 `game-server` / `game-proxy` / `match-service` / `chat-server` | `DISCOVERY_REQUIRED=true`，或 `NODE_ENV` / `APP_ENV` 为 `production` / `test` | 配置加载阶段要求 `REGISTRY_ENABLED=true`。`game-server`、`match-service`、`chat-server` 通过 panic 拒绝；`game-proxy` 通过配置校验返回错误。 |
| `scripts/rollout-three-process-drill.ps1` | `DiscoveryRequired=true`，或 `EnvironmentName` / `APP_ENV` / `NODE_ENV` 匹配 `prod` / `production` / `staging` / `test` / `testing` | 默认 `RegistryEnabled=true`、`DiscoveryRequired=true`。严格环境必须从 registry 解析 `auth-http.internal`、`game-proxy.admin`、old/new `game-server.admin`，local fallback 会被 preflight 拒绝。 |
| `metrics-collector` | 无 service registry 严格发现判定 | 当前不注册也不消费 service registry。`SERVICE_REGISTRY_REGISTER=true` 会直接报错，避免在没有真实 endpoint 时伪注册。 |

### 2. `REGISTRY_ENABLED=false` 的当前处理

| 服务 / 类型 | 配置加载阶段 | 启动注册阶段 | 运行发现阶段 |
|---|---|---|---|
| Node `auth-http` | 严格发现下拒绝 `REGISTRY_ENABLED=false`；本地非严格可保留 `GAME_PROXY_HOST` / `GAME_PROXY_PORT` fallback | 启动时始终尝试 `registryClient.register()`；注册失败只记录 `startup.registry_failed`，当前不会让进程退出 | 登录返回 `game-proxy.client` 时，若 required discovery 且 registry disabled / 缺 endpoint，会抛 `SERVICE_DISCOVERY_UNAVAILABLE`。 |
| Node `admin-api` | 严格发现下拒绝 `REGISTRY_ENABLED=false`；本地非严格可保留 game-server / game-proxy admin fallback | 注册失败只记录 `startup.registry_failed`；`startDiscoveryRefresh()` 只有注册 try 成功后才启动 | GM / 监控控制面在 required discovery 下发现禁用或缺 endpoint 会明确失败；多实例写操作还要求 `targetInstanceId`。 |
| Node `mail-service` | 严格发现下拒绝 `REGISTRY_ENABLED=false`；本地非严格可保留 game-server admin fallback | 注册失败只记录 `startup.registry_failed` | 发放附件链路在 required discovery 下发现禁用或缺 endpoint 会明确失败；多实例时要求显式目标。 |
| Node `announce-service` | 严格发现下拒绝 `REGISTRY_ENABLED=false` | 注册失败只记录 `startup.registry_failed` | 当前不消费 registry discovery；准入主要覆盖自身注册配置。 |
| Rust `game-server` | 严格发现下拒绝 `REGISTRY_ENABLED=false` | registry client 创建失败或注册失败在严格环境 fail-fast；非严格只记录错误继续运行 | 发现 `match-service.grpc` 在 required discovery 下失败会 panic / fail-fast；运行中 rediscovery 失败会保留现有 client 到下个 tick。 |
| Rust `game-proxy` | 严格发现下拒绝 `REGISTRY_ENABLED=false` | registry client 创建失败或注册失败在严格环境 fail-fast；非严格只记录错误继续运行 | 启动必须发现至少一个 `game-server.proxy-local`；后续 watch 失败会把已知路由标为 unavailable，而不是回退到 `internal` 或旧直连。 |
| Rust `match-service` | 严格发现下拒绝 `REGISTRY_ENABLED=false` | registry client 创建失败或注册失败在严格环境 fail-fast；非严格只记录错误继续运行 | 创建房间解析 `game-server.internal`，required discovery 下 registry disabled、无 endpoint 或无合格候选都会返回明确失败。 |
| Rust `chat-server` | 严格发现下拒绝 `REGISTRY_ENABLED=false` | registry client 创建失败或注册失败在严格环境 fail-fast；非严格只记录错误继续运行 | 当前不消费 registry discovery。 |
| `rollout-three-process-drill.ps1` | 参数 / 环境解析后若 required discovery 且 registry disabled，直接失败 | 不负责注册服务，只读取 registry snapshot | required discovery 下 registry 读取失败、endpoint 缺失、多实例歧义或 strict 环境 fallback 都会失败。 |
| `metrics-collector` | 不读取 `REGISTRY_ENABLED` / `DISCOVERY_REQUIRED` 作为准入条件 | 默认不注册；强制注册会报错 | 不消费 registry discovery。 |

### 3. 迁移阶段准入口径

| 阶段 | 准入口径 | 不允许项 |
|---|---|---|
| 迁移前 / 当前本地开发 | `NODE_ENV` / `APP_ENV` 为 development / local，且未设置 `DISCOVERY_REQUIRED=true` 时，可以使用 `.env.example` 中标注的 `Local fallback only` 配置。fallback 只用于单机调试和 manual drill。 | 不应把 `.env.example` 默认 host / port 复制到测试、预发或线上作为服务间入口。 |
| 迁移中 / 测试和线上准入 | 测试、预发、线上必须设置 `REGISTRY_ENABLED=true` 和 `DISCOVERY_REQUIRED=true`；消费者必须按 registry endpoint name 发现目标服务，注册失败、发现失败、缺 endpoint、目标实例歧义都应让对应启动、请求或演练明确失败。 | 不允许 `REGISTRY_ENABLED=false`；不允许依赖 `GAME_PROXY_HOST`、`MATCH_SERVICE_ADDR`、`GAME_SERVER_ADMIN_HOST`、`UPSTREAM_LOCAL_SOCKET_NAME`、`GAME_SERVER_INTERNAL_SOCKET_NAME` 等旧直连变量完成测试/线上链路。 |
| 迁移完成 / 关闭旧直连 | 对仍保留 legacy 变量解析的服务设置 `DISALLOW_LEGACY_DIRECT_CONFIG=true`，把旧直连变量从“严格环境忽略 + warning”升级为“配置加载失败”。测试/生产模板当前已覆盖 `auth-http`、`admin-api`、`game-server`、`game-proxy`、`match-service`、`mail-service`。 | 不允许旧直连变量以注释外的活动配置存在；不允许新增消费者绕过 service registry 直连其他服务。 |

### 4. 已落地 enforcement 点

- 配置加载：8 个 registry 参与服务在 `DISCOVERY_REQUIRED=true` 或 `NODE_ENV` / `APP_ENV=test|production` 下都会拒绝 `REGISTRY_ENABLED=false`。
- 本地 fallback 隔离：`auth-http`、`admin-api`、`mail-service`、`match-service`、`game-proxy` 的 fallback 只在非严格本地场景生效；严格环境会忽略并告警，或通过 `DISALLOW_LEGACY_DIRECT_CONFIG=true` 直接拒绝。
- 发现失败：`auth-http` 登录发现 `game-proxy.client`、`game-server` 发现 `match-service.grpc`、`game-proxy` 初始发现 `game-server.proxy-local`、`match-service` 发现 `game-server.internal`、`admin-api` / `mail-service` 发现 admin endpoint、rollout 脚本发现控制面 endpoint，都已有 required discovery 下的明确失败路径。
- 注册失败：Rust `game-server`、`game-proxy`、`chat-server`、`match-service` 在严格环境下 registry client 创建或注册失败会 fail-fast；非严格环境只记录错误继续运行。
- rollout 准入：`rollout-three-process-drill.ps1` 把 `staging` / `testing` 也视作严格环境，并拒绝 strict/test/production 下使用 local fallback endpoint。
- metrics 边界：`metrics-collector` 明确不注册、不消费 service registry，强制注册会失败，避免形成伪 endpoint。

### 5. 剩余风险 / 后续验证项

- Node 服务启动注册失败当前只记录 `startup.registry_failed`，不会像 Rust 服务一样在严格环境 fail-fast；迁移中需要补齐或用部署健康检查兜底，避免服务启动但 registry 不可见。
- Node 服务模板已有 `REGISTRY_URL`，但运行时 registry client 复用 `REDIS_URL`；测试/线上部署必须同步设置 `REDIS_URL`，后续可考虑代码层统一读取 `REGISTRY_URL`。
- `DISALLOW_LEGACY_DIRECT_CONFIG=true` 还没有覆盖 `chat-server` 和 `announce-service`，原因是当前没有对外发现消费链路；若后续新增消费者，需要同时补 legacy 变量清单与测试。
- `metrics-collector` 没有 strict discovery 准入，因为它不是 service registry 参与者；如果未来暴露 HTTP 或控制面 endpoint，需要重新定义注册与 fail-fast 口径。
- 多副本 `SERVICE_INSTANCE_ID` 唯一性仍靠部署层注入；准入检查应验证 registry 中同服务实例 ID 没有覆盖，且 admin 写操作目标必须明确。
- 后续验证应把 `.env.test.example` / `.env.production.example`、配置单元测试、rollout strict dry-run 和 registry fixture 检查纳入常规准入，防止旧直连变量或 `REGISTRY_ENABLED=false` 回流。

## 本地开发 fallback 与日志基线

本节定义本地开发阶段允许保留的 fallback 口径：fallback 只能用于 `local` / `development` 且 discovery 非严格的单机联调、手工 drill 或 mock-client 调试；日志、返回 endpoint 或报告中必须能看出它不是 registry 结果，最低要求是出现 `source=registry|fallback` 与 `reason=registry_disabled|fallback_used|fallback_forbidden|registry_error|endpoint_missing` 之一，或脚本报告中明确出现 `local-fallback` / `fallback_used` 语义。

### 1. 允许 fallback 的链路与条件

| 链路 | 允许 fallback 的条件 | registry disabled 与发现缺失的区别 | 测试/线上边界 |
|---|---|---|---|
| `auth-http -> game-proxy.client` 登录返回 | `REGISTRY_ENABLED=false` 且 `DISCOVERY_REQUIRED=false`，并且本地 fallback 配置可构造 `services.game`；或者 registry enabled 但 `game-proxy.client` 缺失且 discovery 非严格 | registry disabled 时直接返回本地 `GAME_PROXY_HOST` / `GAME_PROXY_PORT`；registry enabled 但 endpoint 缺失时仅在非严格下回退。若无法构造本地 endpoint，只记录 forbidden，不编造地址 | `test` / `production` / `DISCOVERY_REQUIRED=true` 下禁用；缺 registry 或缺 endpoint 会抛 `SERVICE_DISCOVERY_UNAVAILABLE` |

### 2. 当前日志与 metrics 字段

| 链路 | 当前是否满足明确标注 | 已有字段 / 口径 | 缺口 |
|---|---|---|---|
| `auth-http -> game-proxy.client` | 基本满足 | `registry.discovery_disabled` 使用 `source=registry`、`reason=registry_disabled`；fallback 使用 `source=fallback`、`reason=fallback_used`；无法 fallback 使用 `reason=fallback_forbidden`。Node discovery metric 会把 `fallback_used` 计入 `fallback_used_total` | 只覆盖 `game-proxy.client`；可选 side endpoint 缺失不单独 fail-fast，也不编造 fallback |

| `game-proxy static upstream` | 部分满足 | `proxy_server.rs` 在静态 upstream 生效时记录 metric `fallback_used`，日志包含 `source=fallback`、`reason=fallback_used`、`upstream_server_id` 和 socket；`main.rs` 另有纯文本 `service registry disabled; using local static upstream fallback` | `main.rs` 纯文本日志没有结构化 `source/reason` 字段，后续可补齐为同一事件口径 |

补充说明：

- 共享库已经有统一 metrics kind：Rust `packages/service-registry/src/discovery_metrics.rs` 和 Node `packages/service-registry/node/registry-schema.js` 都会把 `source=fallback` + `reason=fallback_used` 汇总为 `fallback_used_total`，并把 `registry_disabled`、`fallback_forbidden`、`registry_error` 汇总为 `discovery_failure`。
- 本地 fallback 的日志应该优先包含 service、endpoint、instance id、source、reason 和目标地址 / socket；当前 Node admin / auth 与 Rust discovery 链路大多已经满足，脚本和少数 Rust 纯文本日志仍需要收敛。
- 测试、预发和线上准入不以 fallback 成功作为通过条件；这些环境应验证 `REGISTRY_ENABLED=true`、`DISCOVERY_REQUIRED=true`、endpoint 存在、目标实例无歧义，以及没有启用 `Local fallback only` 变量。

### 3. 剩余风险 / 后续要求

- Node 与 Rust 日志字段还没有完全统一：Rust registry disabled fallback 常用 `reason=registry_disabled`，但 metrics 记录为 `fallback_used`；Node 多数直接用 `reason=fallback_used`。后续可以统一为 `source=fallback`、`reason=fallback_used`，并额外加 `cause=registry_disabled|endpoint_missing|registry_error`。
- `game-proxy` 在 `main.rs` 的 registry disabled 提示仍是纯文本，建议补结构化字段，避免只靠 human-readable message 判断 fallback。
- rollout 脚本已经有 `local-fallback` 模式和 preflight warning，但字段名与服务日志不同；后续可在报告中补 `reason=fallback_used` 和 `cause=registry_disabled|discovery_not_required`。
- mock-client 的直接参数应继续保持 local-only 文案；如果接入 CI 或验收脚本，应默认走登录响应 / registry 结果，并把手工 `--host` / `--port` 标成 `source=manual-local`。
- 新增服务发现消费者时，必须同步补充 `.env.example` 的 `Local fallback only` 注释、严格环境单测、fallback 日志字段和 required discovery 失败路径。
