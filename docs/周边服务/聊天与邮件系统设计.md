# 聊天系统 + 邮件系统 统一架构设计

## 1. 核心定位

聊天系统和邮件系统有高度相似的架构模式：

| 特性 | 聊天 | 邮件 |
|------|------|------|
| 发送者 → 接收者 | ✅ | ✅ |
| 离线存储 | ✅ (离线消息) | ✅ (未读邮件) |
| 持久化存储 | ✅ | ✅ |
| 群组模式 | ✅ (群聊) | ❌ |
| 实时性 | ✅ | 部分支持：新邮件通过 Core NATS 通知在线玩家 |
| 频率/连接限制 | 已有单连接入站限流、单 IP / 单账号本实例连接数限制，默认关闭 | ❌ |

**结论**：两者可以共用一套消息存储层，但在业务逻辑上保持独立。

当前仓库实现状态说明：

- `chat-server` 已作为独立 Rust TCP 服务落地
- `chat-server` 认证阶段已复用 game ticket，并检查签名、过期时间、Redis `ticket:<sha256(ticket)>` 归属和 `player-ticket-version:<playerId>`；Redis key 均受 `REDIS_KEY_PREFIX` 影响
- `chat-server` 的生产 `TICKET_SECRET` 必须与 `auth-http` / `game-server` 的 ticket 签发和校验侧保持一致，且不能使用默认值或 `.env.example` 占位值
- `chat-server` 已支持认证后的单连接入站消息频率限制，`CHAT_MSG_RATE_MAX=0` 默认关闭；启用后在业务 handler 前返回 `ErrorRes(MSG_RATE_EXCEEDED)`，不断开连接
- `chat-server` 已支持 ticket 校验通过后的单账号 / 单 IP 本实例连接数限制，`CHAT_MAX_CONNECTIONS_PER_PLAYER=0` 与 `CHAT_MAX_CONNECTIONS_PER_IP=0` 默认关闭；超限时在注册 session 和 online route 前返回失败 `ChatAuthRes` 并关闭新连接
- 当前在线聊天推送仍使用本实例 `player_id -> sender` 内存 session map；同一账号多连接时推送路由沿用现有覆盖行为，连接数限制计数单独维护
- `chat-server` 可在统一 service registry 中注册 schema v2 的 `chat-server.tcp` endpoint，`visibility=internal`，metadata 包含 `service_instance_id` / `instance_id`、`online_route_ttl_secs`、`build_version`、`zone` 等；严格环境下 `DISCOVERY_REQUIRED=true` 要求 `REGISTRY_ENABLED=true`
- `chat:online:{playerId}` 仍是玩家在线实例路由，写入 Redis 并受 `REDIS_KEY_PREFIX` 影响，TTL 默认来自 `CHAT_ONLINE_ROUTE_TTL_SECS=60`；它不并入 service registry，也不由 registry heartbeat 替代
- `chat-server` 当前不消费 service registry。它自身可以注册 TCP endpoint，但邮件在线通知仍使用 Redis 在线路由 + Core NATS + 本实例内存 session map
- `mail-service` 已作为独立 Node.js HTTP 服务落地，并已实现附件领取
- `mail-service` 玩家读邮件、邮件详情、标记已读和领取附件已复用 game ticket 做第一阶段 HTTP 鉴权，并检查 ticket 签名、过期时间、Redis `ticket:<sha256(ticket)>` 归属和 `player-ticket-version:<playerId>`；Redis key 均受 `REDIS_KEY_PREFIX` 影响
- `mail-service` 系统发信入口 `POST /api/v1/mails` 已通过 `MAIL_SERVICE_TOKEN` 做内部 service token 鉴权
- `mail-service` 可在统一 service registry 中注册 schema v2 的 `mail-service.http` endpoint，`visibility=internal`；endpoint metadata 包含服务身份、`build_version`、`zone`，实例 metadata 还包含 `player_auth_required`、`service_token_enabled` 等当前实现信息
- `mail-service` 发送在线通知时先读取 Redis `chat:online:{playerId}`，命中后发布到实例级 NATS subject `myserver.mail.notify.instance.<instance_id_token>`；缺路由或查询失败时回退到 legacy `myserver.mail.notify.<player_id_token>`。service registry 不替代 NATS，也不替代 chat 在线路由
- `mail-service` 领取附件调用 `game-server.admin` 时通过 registry 发现 endpoint；`GAME_SERVER_ADMIN_HOST` / `GAME_SERVER_ADMIN_PORT` 只允许 development/local 且 discovery 非 required、registry disabled 的本地 fallback。此时固定 endpoint 只是传输地址，仍必须读取 Redis v2 在线权威路由并携带 generation/token，不能把合成标签 `local-fallback` 当作 game-server 实例身份。strict/test/prod 中 registry disabled、发现不到 endpoint 或旧直连变量存在都会明确失败，不静默直连
- 多 `game-server` 实例时，`grantMailAttachments()` 使用 game-server 维护的 Redis 在线权威路由选择实例并与 registry endpoint 交叉验证；玩家不能指定目标。`targetInstanceId` 只保留为 development/local 且 discovery 非 required 的受限调试能力；registry disabled 的单机 fallback 下只接受当前 route 的真实 `instance_id`，不接受 `local-fallback` 伪身份
- `announce-service` 已作为独立 Node.js HTTP 服务落地，支持公告 CRUD、有效公告查询、Redis 注册与 NATS metrics 上报
- `metrics:heartbeat:*` 或 NATS metrics 只是指标 / 事件新鲜度通道，不是 registry heartbeat，不能替代 service discovery
- “聊天与邮件共用同一套存储层”目前仍未实现
- 离线聊天当前更接近“历史可查询”，未实现“登录后自动补发离线消息”的完整闭环

## 2. 当前消息系统架构

当前仓库没有把聊天、邮件、公告统一挂到 `game-proxy` 后面，也没有实现共用 `message-store`。实际链路是：

```
Client
  -> auth-http 登录，获取 game-proxy 入口；允许的非生产场景可选获取 services.chat / services.mail / services.announce
  -> chat-server TCP + Protobuf 聊天协议（可选内部 endpoint，不是生产默认直连入口）
  -> mail-service HTTP 邮件接口（可选内部 endpoint，不是生产默认直连入口）
  -> announce-service HTTP 公告接口（可选内部 endpoint，不是生产默认直连入口）

mail-service
  -> Redis: chat:online:{playerId}
  -> Core NATS: myserver.mail.notify.<player_id_token>
  -> Core NATS: myserver.mail.notify.instance.<instance_id_token>
  -> chat-server mail_subscriber
  -> 在线聊天 TCP 连接推送 MailNotifyPush

mail-service
  -> service registry: discover game-server.admin
  -> game-server admin (GrantItemsReq / GrantItemsRes)
  -> player inventory updated

chat-server -> PostgreSQL chat_messages / chat_groups / chat_group_members
mail-service -> PostgreSQL mails 或内存存储
announce-service -> PostgreSQL announcements 或内存存储
```

`game-proxy` 当前只承担游戏流量接入，不负责聊天、邮件或公告 HTTP/TCP 转发。`auth-http` 可选暴露 `chat-server.tcp` / `mail-service.http` / `announce-service.http` 内部 endpoint 时，必须在允许的非生产场景并且通过 registry 发现；生产环境和正式客户端默认不返回这些内网 endpoint。

## 3. 聊天系统设计

### 3.1 支持范围

- **单聊**：用户A ↔ 用户B
- **群聊**：用户群组内共享消息
- **聊天与房间解耦**：聊天不依赖游戏房间
- **历史查询**：支持私聊 / 群聊历史消息分页查询
- **邮件通知**：订阅 `myserver.mail.notify.*` 并向在线聊天连接推送 `MailNotifyPush`
- **认证边界**：首包必须是 `ChatAuthReq`，ticket 被单张 revoke 或 owner 不匹配时拒绝认证
- **服务发现边界**：自身可注册 `chat-server.tcp` 内部 endpoint，但当前不通过 service registry 发现其他服务；玩家在线路由继续写 Redis `chat:online:{playerId}`

### 3.2 群组管理

群组是独立的，需要：

```rust
pub struct ChatGroup {
    pub group_id: String,
    pub name: String,
    pub owner_id: String,
    pub members: Vec<String>,       // 成员列表
    pub created_at: i64,
}
```

群聊操作：
- `CREATE_GROUP` - 创建群组
- `JOIN_GROUP` - 加入群组
- `LEAVE_GROUP` - 离开群组
- `DISMISS_GROUP` - 解散群组（仅群主）

### 3.3 聊天协议

```proto
// 单聊
message ChatPrivateReq {
  string target_id = 1;    // 接收者用户ID
  string content = 2;
}

message ChatPrivateRes {
  bool ok = 1;
  string error_code = 2;
  string msg_id = 3;
}

// 群聊
message ChatGroupReq {
  string group_id = 1;     // 群组ID
  string content = 2;
}

message ChatGroupRes {
  bool ok = 1;
  string error_code = 2;
  string msg_id = 3;
}

// 消息推送
message ChatPush {
  string msg_id = 1;
  string chat_type = 2;    // private / group
  string sender_id = 3;
  string sender_name = 4;
  string content = 5;
  int64 timestamp = 6;
  string target_id = 7;     // 单聊时为接收者ID
  string group_id = 8;      // 群聊时为群组ID
}

// 群组操作
message GroupCreateReq { string name = 1; }
message GroupJoinReq { string group_id = 1; }
message GroupLeaveReq { string group_id = 1; }
```

### 3.4 消息存储

聊天消息 `msg_id` 使用全局 ID 字符串格式 `msg_<base32>`，聊天群 `group_id` 使用 `grp_<base32>`。历史旧格式不保留兼容生成逻辑。

```sql
-- 消息存储表
CREATE TABLE chat_messages (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    msg_id varchar(64) UNIQUE NOT NULL,     -- 消息唯一ID
    chat_type smallint NOT NULL,            -- 1=私聊, 2=群聊
    sender_id varchar(64) NOT NULL,
    content text NOT NULL,
    created_at bigint NOT NULL,

    -- 索引用于查询
    target_id varchar(64),                  -- 私聊时为对方用户ID
    group_id varchar(64)                    -- 群聊时为群组ID
);

CREATE INDEX IF NOT EXISTS idx_chat_messages_sender ON chat_messages (sender_id);
CREATE INDEX IF NOT EXISTS idx_chat_messages_target ON chat_messages (target_id);
CREATE INDEX IF NOT EXISTS idx_chat_messages_group ON chat_messages (group_id);
CREATE INDEX IF NOT EXISTS idx_chat_messages_created ON chat_messages (created_at);

-- 群组表
CREATE TABLE chat_groups (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    group_id varchar(64) UNIQUE NOT NULL,
    name varchar(128) NOT NULL,
    owner_id varchar(64) NOT NULL,
    created_at bigint NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_chat_groups_owner ON chat_groups (owner_id);

-- 群组成员表
CREATE TABLE chat_group_members (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    group_id varchar(64) NOT NULL,
    player_id varchar(64) NOT NULL,
    joined_at bigint NOT NULL,
    CONSTRAINT uk_chat_group_members_group_player UNIQUE (group_id, player_id)
);

CREATE INDEX IF NOT EXISTS idx_chat_group_members_player ON chat_group_members (player_id);
```

### 3.5 离线消息

- 用户上线时查询 `created_at > last_login_time` 的消息
- 推送离线消息给用户

当前实现尚未做“上线后自动补发离线消息”，但已经支持客户端主动通过 `ChatHistoryReq` 查询私聊或群聊历史。

## 4. 邮件系统设计

### 4.1 与聊天系统共用部分

邮件系统可以复用聊天系统的：
- 消息存储表结构
- 用户收件箱查询逻辑
- 离线未读推送机制

当前仓库实现说明：
- `mail-service` 已独立落地为 Node.js HTTP 服务
- 新邮件通知通过 Redis 在线路由和 Core NATS 下发：优先按 `chat:online:{playerId}` 路由到 `myserver.mail.notify.instance.<instance_id_token>`，缺路由时回退到 `myserver.mail.notify.<player_id_token>`
- 附件领取由 `mail-service` 发现并调用内网 `game-server.admin` endpoint 完成真实发奖；调用 `AdminAuthReq` 时使用 service actor，兼容 game-server 控制面审计
- 当前未与 `chat-server` 共用同一套存储表

### 4.2 邮件特有字段

邮件 `mail_id` 使用全局 ID 字符串格式 `mail_<base32>`，与通知 outbox、附件领取和审计幂等键保持同一业务 ID。

```sql
CREATE TABLE mail_messages (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    mail_id varchar(64) UNIQUE NOT NULL,
    sender_type varchar(32) NOT NULL,       -- system / player
    sender_id varchar(64) NOT NULL,         -- 玩家侧看到的发件人ID；系统固定为 "system"
    sender_name varchar(128),               -- 玩家侧看到的发件人名称
    receiver_id varchar(64) NOT NULL,       -- 收件人
    subject varchar(256),                   -- 邮件主题
    content text,                           -- 邮件正文
    attachments jsonb,                      -- 附件 (物品/货币奖励)
    created_by_type varchar(32) NOT NULL,   -- system / admin / player / service
    created_by_id varchar(64),              -- 实际触发方，用于审计
    created_by_name varchar(128),           -- 实际触发方展示名
    is_read boolean DEFAULT false,          -- 是否已读
    created_at bigint NOT NULL,
    expires_at bigint                       -- 过期时间 (可选)
);

CREATE INDEX IF NOT EXISTS idx_mail_messages_receiver ON mail_messages (receiver_id);
CREATE INDEX IF NOT EXISTS idx_mail_messages_created ON mail_messages (created_at);
```

建议约定：
- 面向玩家展示的发件人，固定使用 `sender_type / sender_id / sender_name`
- 面向后台审计的操作者，固定使用 `created_by_type / created_by_id / created_by_name`
- 系统奖励邮件不需要真实可登录账号，系统发件人统一保留值 `sender_id = "system"`

### 4.3 当前实现协议

当前邮件能力对外通过 HTTP 暴露，核心接口如下：

```http
POST /api/v1/mails                         # 内部发信，要求 MAIL_SERVICE_TOKEN
GET  /api/v1/mails?status=<status>&limit=<n>&offset=<n>
GET  /api/v1/mails/:mailId
PUT  /api/v1/mails/:mailId/read
POST /api/v1/mails/:mailId/claim
```

玩家接口默认要求 `Authorization: Bearer <game_ticket>` 或 `X-Game-Ticket`。服务端以 ticket 中认证出的 `playerId` 作为唯一玩家身份；query/body 中的 `player_id` 可以省略，若传入则只能与认证身份一致，否则返回拒绝。`MAIL_PLAYER_AUTH_REQUIRED=false` 仅用于本地兼容调试，生产环境必须为 `true`。

其中附件领取接口的当前行为为：
- 玩家身份来自 game ticket；请求体 `player_id` 可省略，只允许与认证身份一致
- 只能读取或领取属于自己的邮件，邮件详情也会校验 `to_player_id`
- 已过期邮件不可领取
- 当前真实发奖只支持 `attachments[].type = "item"`
- `mail-service` 首次领取时会在事务内创建唯一 `mail_claim_workflows` 记录，冻结角色、规范化附件快照、canonical SHA-256 指纹和稳定 `claim_request_id = mail_claim:<mail_id>`，同时把邮件置为 `claiming`
- 明确未执行的临时错误记录为 `retryable_failure`，永久业务错误记录为 `permanent_failure`；两者都保留同一工作流和冻结快照，玩家修正条件后重试时不会生成第二个 request id
- 请求写出后发生读超时、断线或响应无法验证时记录为 `reconciliation_pending`，返回 202 且玩家重试不会再次调用发奖；只有后续对账能够决定 `claimed` 或重试
- 每次真实发奖尝试持有短租约，完成和失败更新都必须匹配当前 lease token；过期租约可被接管，旧持有者的晚到响应不能覆盖新状态
- 发奖目标通过 service registry 发现 `game-server.admin`。`GAME_SERVER_ADMIN_HOST` / `GAME_SERVER_ADMIN_PORT` 仅为 development/local 且 discovery 非 required、registry disabled 的本地 fallback；该模式使用唯一固定 admin 地址和 Redis 当前 v2 权威 route，route 的真实实例 ID 与 generation/token 仍随请求交由 game-server 校验。strict/test/prod 中禁用旧直连 fallback，registry disabled 或 endpoint 缺失会返回明确错误
- `game-server` 在玩家鉴权成功后把当前角色、实例和 session 所有权写入 Redis 短 TTL 在线路由；`mail-service` 只接受该服务端路由，并与 registry 中健康的 `game-server.admin` endpoint 交叉验证后发放
- strict/test/staging/production 不接受领取 body 的 `targetInstanceId`；development/local 且 discovery 非 required 时仅保留显式实例调试入口。registry enabled 时目标必须存在于服务发现结果并等于当前 route owner；registry disabled 的单机 fallback 下目标必须等于当前 route 的真实 `instance_id`
- 当前没有可验证的跨实例离线资产单写 owner。正式多实例环境中角色离线、在线路由缺失或路由实例已摘流时返回可重试 `MAIL_CLAIM_ROUTE_UNAVAILABLE`，不得 hash、随机或选择发现列表第一个实例写背包
- 发奖请求使用稳定幂等键 `requestId = mail_claim:<mail_id>`，`source = "mail-claim"`，并在 `AdminAuthReq` 中带合法 service actor（默认 `mail-service` 或服务实例标识规范化结果），以兼容 `GAME_ADMIN_AUDIT_REQUIRE_ACTOR=true`
- 返回体保留 `claimed`、`already_claimed`、`status`、`read_at`、`claimed_at` 兼容字段，并增加 `claim_status`、`request_id`、`attempts`、`retryable`、`error_category` 和 `result_state`
- `claim_status` 明确区分 `claimed`（200）、`processing`（202）、`retryable_failure`（503）、`permanent_failure`（通常 422）和 `reconciliation_pending`（202）；202 表示继续等待当前工作流，不能提示玩家重新开始领取
- 工作流可以保存最多 512 字符的内部错误摘要，供后续受保护的管理查询使用；玩家响应只返回稳定公开错误码、分类、结果状态和固定安全文案，不回显内部 endpoint、Redis/数据库/socket 错误、token-like 文本或持久化原始摘要
- 重复领取是幂等的，不会重复发放道具

创建邮件时，`mail-service` 会把邮件记录和新邮件通知 outbox 写入同一个 PostgreSQL 事务；通知发布失败时保留 outbox 待后台重试。

发信入口支持 `Authorization: Bearer <MAIL_SERVICE_TOKEN>`、`X-Service-Token` 或 `X-Admin-Token`，不接受 query/body token。`MAIL_SERVICE_TOKEN` 是 `mail-service` 的上游调用凭证，不复用下游 `GAME_ADMIN_TOKEN`。

当前附件领取链路：

```text
client
  -> POST /api/v1/mails/:mailId/claim
  -> mail-service
  -> service registry: game-server.admin
  -> game-server admin (GrantItemsReq / GrantItemsRes)
  -> player inventory updated
  -> mail status = claimed
```

### 4.4 三服务可靠交互契约

本节是 `mail-service`、`chat-server` 与 `game-server` 邮件链路的稳定契约。契约中的“当前实现”用于说明改造起点；标记为“目标契约”的字段和状态是后续实现必须收敛到的行为，不表示当前代码已经全部具备。

#### 4.4.1 权威数据与成功边界

| 数据或动作 | 权威方 | 成功边界 |
|------|------|------|
| 邮件内容、未读/已读状态、附件快照和领取进度 | `mail-service` PostgreSQL | 邮件记录和通知 outbox 在同一事务提交后，邮件创建即业务成功 |
| 在线新邮件提示 | `chat-server` 当前有效 session | 只做尽力在线推送，不属于邮件创建事务，也不改变邮件状态 |
| 角色背包和附件发放结果 | `game-server` PostgreSQL | 背包变更与幂等结果在同一事务提交后，资产发放才成功 |

必须遵守以下边界：

- Core NATS 只承担在线提示。`publish` 成功只表示消息已交给 Core NATS，不表示 `chat-server`、玩家 session 或客户端已收到。
- `chat-server` 离线、玩家离线、session 队列已满、Redis 在线路由缺失或 NATS 丢消息，都不得回滚邮件，也不得把邮件创建接口改成失败。
- `mail-service` 不直接写角色背包，也不能根据调用超时或连接断开推断“game-server 没有发放”。
- 只有 `game-server` 返回与本次 `request_id + request_fingerprint` 匹配的首次成功或幂等成功，`mail-service` 才能把邮件更新为 `claimed`。
- 在线背包 push 是资产事务提交后的派生提示。push 失败不得回滚已经提交的资产，也不得把确定成功降级为结果未知。

#### 4.4.2 当前邮件创建与在线通知时序

当前代码锚点：

| 环节 | 当前实现 |
|------|------|
| 邮件与 outbox 原子写入 | `apps/mail-service/src/db-store.js` 的 `createMailWithNotificationOutbox()` |
| outbox 扫描、租约、重试 | `apps/mail-service/src/mails/mails.service.ts` 的 `processPendingNotificationOutbox()` 及 `db-store.js` 对应方法 |
| Redis 在线路由和 NATS 发布 | `apps/mail-service/src/pubsub-client.js` 的 `PubSubClient` |
| NATS 消费和在线 push | `apps/chat-server/src/mail_subscriber.rs` |

```text
上游服务
  -> POST /api/v1/mails（MAIL_SERVICE_TOKEN）
  -> mail-service 校验发信参数并生成 mail_id
  -> PostgreSQL transaction
       INSERT mails
       INSERT mail_notification_outbox(payload)
     COMMIT
  -> 尝试立即领取一条 outbox
  -> Redis GET chat:online:{player_id}
       命中：myserver.mail.notify.instance.<instance_id_token>
       未命中/查询失败：myserver.mail.notify.<player_id_token>
  -> Core NATS publish
       成功：outbox 标记 sent
       失败：outbox 标记 failed，等待重试
  -> chat-server 同时订阅实例 subject 和 legacy player subject
  -> 解析 payload
  -> 当前 session 存在时发送 MailNotifyPush；否则跳过
```

当前实现已经保证 PostgreSQL 模式下邮件和 outbox 同事务，但仍有一个需要收敛的响应窗口：事务提交后，创建接口会同步调用一次 outbox 扫描；如果领取 outbox 或更新 outbox 状态本身抛出异常，HTTP 调用可能返回错误，而邮件实际已经落库。目标契约要求数据库事务提交后始终按邮件创建成功响应，通知处理只能影响 outbox 指标和日志。

#### 4.4.3 当前附件领取持久状态机与恢复边界

当前代码锚点：

| 环节 | 当前实现 |

| 玩家身份、邮件归属和附件校验 | `apps/mail-service/src/mails/mails.service.ts` 的 `claim()` |
| 工作流预占、租约 fencing、失败记录和 `claiming/claimed` 条件更新 | `apps/mail-service/src/db-store.js` 的 `reserveMailClaimWorkflow()`、`recordMailClaimWorkflowFailure()` 和 `completeMailClaimWorkflow()` |
| game-server endpoint 发现和 admin TCP 请求 | `apps/mail-service/src/game-admin-client.js` |

| 背包变更与幂等记录事务 | `apps/game-server/src/core/player/player_manager.rs`、`db_player_store.rs` |

```text
玩家
  -> POST /api/v1/mails/:mailId/claim（game ticket）
  -> mail-service 从认证上下文取得 player_id 和 character_id
  -> 校验邮件归属、过期时间和附件格式
  -> PostgreSQL transaction + row lock
       INSERT mail_claim_workflows（唯一 mail_id/request_id、冻结附件、指纹、角色、attempt=1、lease）
       UPDATE mails unread/read -> claiming
  -> 发现 game-server.admin endpoint
  -> AdminAuthReq(service actor)
  -> GrantItemsReq(request_id + mail_id + request_fingerprint + trace_id)
  -> game-server 独立重算指纹，原子写背包和 character_inventory_grants
  -> GrantItemsRes(ok=true, applied=true|false, 首次结果摘要)
  -> mail-service 以当前 lease_token 条件更新 workflow -> claimed、mails -> claimed
```

当前分阶段实现状态如下，后续实现不得忽略剩余缺口：

- `mail-service` 已禁止 strict/test/staging/production 从领取 HTTP body 覆盖 `targetInstanceId`；development/local 调试目标也必须通过 registry endpoint 校验。角色、邮件、附件、`request_id`、source、指纹和内部目标均从认证上下文或持久邮件生成，不能由玩家覆盖。
- `game-server` 已按 4.4.5 实现 canonical JSON + SHA-256 独立重算，并在 `character_inventory_grants` 中保存请求指纹和首次结果；相同 `request_id` 的角色、附件、绑定状态或 source 冲突会返回 `REQUEST_FINGERPRINT_CONFLICT`。
- game admin 客户端已补齐 `mail_id`、canonical SHA-256 指纹和 32 位 `trace_id`，并校验响应包 sequence、消息类型、大小限制、错误码以及 `GrantItemsRes` 的 `request_id + request_fingerprint + trace_id + result_summary`。连接建立失败或明确 `ROUTE_UNAVAILABLE/not_applied` 时会重新发现权威实例，重试保持同一请求身份；请求写出后的读超时、protobuf 解码失败或成功证据不匹配统一标记为 `RESULT_UNKNOWN/unknown`，不在客户端内盲目重发。
- `mail-service` 已使用独立 `mail_claim_workflows` 持久化状态机。一封邮件只有一个工作流和一个 request id；附件、角色和指纹在首次预占时冻结，后续邮件内容修改、过期或硬删除都不改变已经开始的发放请求。工作流表不使用指向 `mails` 的级联删除外键，确保硬删除后仍保留资产侧对账证据。
- `processing` 尝试使用 `lease_owner + lease_token + lease_expires_at` fencing；玩家并发请求返回 202 processing，过期尝试沿用原工作流接管并递增 attempts，旧 token 不能写入失败或成功。
- 明确 `not_applied` 按稳定分类持久化为 `retryable_failure` 或 `permanent_failure`，邮件继续保持 `claiming`；`RESULT_UNKNOWN/unknown` 持久化为 `reconciliation_pending`，不会释放或由玩家请求再次发奖。
- game-server 已在受 admin token 保护的内部通道提供按 `request_id` 查询首次结果的接口；`mail-service` 的工作流记录已经具备恢复所需的 request id、指纹、冻结快照、角色、attempts、租约、最后错误和 trace，但尚未接入结果查询，也尚未实现启动/周期恢复 worker。

工作流字段和邮件状态的职责分离如下：

| 持久状态 | 含义 | 玩家再次领取 |
|------|------|------|
| `mails.status = unread/read` | 尚未开始领取 | 校验邮件并创建唯一工作流 |
| `mails.status = claiming` + workflow `processing` | 当前发奖尝试持有有效租约或等待租约恢复 | 有效租约返回 202；过期租约以同 request id 接管 |
| workflow `retryable_failure` | game-server 明确保证未执行，条件可重试 | 以同角色、冻结快照和 request id 新建一次 attempt |
| workflow `permanent_failure` | 业务条件当前不满足，错误原因已保存 | 玩家/配置修正后仍以同工作流重试 |
| workflow `reconciliation_pending` | 请求可能已提交，结果未知 | 返回 202，不再次发奖，等待恢复 worker 查询 |
| `mails.status = claimed` + workflow `claimed` | 首次成功或幂等成功证据已确认 | 返回幂等已领取结果 |

#### 4.4.4 邮件通知事件 v1

目标通知使用版本化 JSON 事件信封。新旧格式兼容窗口内，`chat-server` 同时接受当前扁平 payload 和下述 v1 信封；新生产者只发布一份事件，不能为了兼容同时向两个 subject 发布同一事件。

```json
{
  "event_id": "mail.notify:mail_01ABC",
  "event_type": "mail.created",
  "version": 1,
  "occurred_at": 1783931896000,
  "player_id": "plr_01ABC",
  "mail": {
    "mail_id": "mail_01ABC",
    "title": "系统奖励",
    "from_player_id": "system",
    "from_name": "系统",
    "mail_type": "system",
    "created_at": 1783931896000
  },
  "trace_id": "0123456789abcdef0123456789abcdef"
}
```

| 字段 | 类型与约束 | 语义 |
|------|------|------|
| `event_id` | 非空字符串，最多 128 字节 | 一条通知事件的稳定 ID；首次写 outbox 时确定，所有发布重试保持不变。存量 outbox 升级可使用 `mail.notify:<mail_id>` 确定性补齐 |
| `event_type` | 固定 `mail.created` | 未知类型拒绝处理，但不能退出订阅任务 |
| `version` | 整数，当前为 `1` | 未知版本记录兼容失败并跳过 |
| `occurred_at` | Unix epoch 毫秒整数 | 邮件与 outbox 原子创建的业务发生时间，不是某次重试发布时间 |
| `player_id` | 非空字符串，最多 128 字节 | 通知目标；必须与 outbox 的 `to_player_id` 一致 |
| `mail.mail_id` | 非空字符串，最多 64 字节 | 邮件权威 ID |
| `mail.title` | 字符串，最多 256 字节 | push 摘要，不包含正文和附件 |
| `mail.from_player_id` | 非空字符串，最多 128 字节 | 展示发件人；系统邮件固定为 `system` |
| `mail.from_name` | 字符串，最多 128 字节 | 展示名称 |
| `mail.mail_type` | 非空字符串，最多 32 字节 | 邮件类型 |
| `mail.created_at` | Unix epoch 毫秒整数 | 邮件创建时间，供 `MailNotifyPush` 使用 |
| `trace_id` | 32 位小写十六进制字符串 | 由 `mail-service` 生成并持久化；重试保持不变，只用于关联日志和指标 |

`event_id` 用于 `chat-server` 的有界短期去重，不是永久 inbox。玩家离线时不保存事件，不在重新上线后补投；客户端应通过邮件列表读取权威数据。

`chat-server` 的消费边界如下：

- 灰度期间同时接受既有扁平 payload 与 v1 信封。只有 v1 信封具有 `event_id`，旧 payload 不伪造事件 ID，也不承诺跨消息幂等。
- 实例定向 subject 与 legacy player subject 共用同一个进程内去重器；默认容量 `10000`、TTL `300` 秒，按首次接收时间过期，容量满时确定性淘汰最早事件。对应配置为 `CHAT_MAIL_NOTIFY_DEDUP_CAPACITY` 和 `CHAT_MAIL_NOTIFY_DEDUP_TTL_SECS`。
- `CHAT_MAIL_NOTIFY_MAX_PAYLOAD_BYTES` 默认 `4096`。消费者必须先检查原始 payload 字节数，再解析 JSON，并按本节字段上限检查 UTF-8 字节长度；非法 payload 的日志只记录路由类型、错误码和 payload 大小，不记录 payload 或玩家、邮件、事件标识。
- 通知只尝试写入目标玩家当前有效的 chat session 队列。玩家离线、队列已满或连接已关闭都安全跳过，不写回邮件状态；同账号旧连接退出时不得删除覆盖它的新 session 或新 session 的在线路由。
- NATS 连接或订阅中断后重新建立两个订阅，退避由 `CHAT_MAIL_NOTIFY_RECONNECT_BASE_MS` 和 `CHAT_MAIL_NOTIFY_RECONNECT_MAX_MS` 控制；服务停机时会取消订阅任务并在有界时间内 drain NATS 客户端。
- 消费指标按上报窗口聚合 `mail_notification_received`、`mail_notification_parse_failed`、`mail_notification_version_rejected`、`mail_notification_deduplicated`、`mail_notification_pushed`、`mail_notification_offline_skipped` 和 `mail_notification_queue_failed`，不使用 `player_id`、`mail_id` 或 `event_id` 作为标签。

通知 outbox 的运行状态固定为：

```text
pending -> sending -> sent
                   -> failed -> sending
                   -> terminal
```

- `sending` 使用数据库租约，持久化 `lease_owner / lease_token / locked_until`；完成、失败和终止更新必须匹配当前 `lease_token`。租约过期后其他 `mail-service` 实例可以接管，因此发布语义是至少一次，`chat-server` 仍需按 `event_id` 做有界去重。
- `attempts` 表示已经开始的发布次数，第 `MAIL_OUTBOX_MAX_ATTEMPTS` 次允许发布；该次失败或发布结果因进程崩溃而未知时转为 `terminal`，不得开始第 `max_attempts + 1` 次发布。
- `failed` 只表示 Core NATS 临时发布失败，按带抖动且有上限的指数退避重试。
- 永久无效 payload 不继续重试，直接转为 `terminal` 并保存最多 512 字节的错误摘要。
- `sent` 只表示 Core NATS `publish` 已成功返回，不表示 `chat-server`、玩家 session 或客户端确认收到。
- 默认保留已发布记录 7 天、终止记录 30 天，后台按批清理；分别由 `MAIL_OUTBOX_SENT_RETENTION_DAYS`、`MAIL_OUTBOX_TERMINAL_RETENTION_DAYS` 和清理批次配置控制。
- 运行指标覆盖积压数、最老事件年龄、发布延迟、重试数、终止数和过期租约接管数。

#### 4.4.5 附件发放请求 v1

附件领取工作流的稳定幂等键固定为：

```text
request_id = mail_claim:<mail_id>
source     = mail-claim
```

目标 admin 请求在当前 `GrantItemsReq` 基础上补齐以下语义字段。JSON 兼容入口沿用 camelCase；Protobuf 字段使用对应 snake_case 名称。

```json
{
  "requestId": "mail_claim:mail_01ABC",
  "mailId": "mail_01ABC",
  "characterId": "chr_01ABC",
  "items": [
    { "itemId": 1001, "count": 2, "binded": false }
  ],
  "requestFingerprint": "sha256:0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef",
  "source": "mail-claim",
  "reason": "claim mail mail_01ABC",
  "traceId": "0123456789abcdef0123456789abcdef"
}
```

| 字段 | 类型与约束 | 稳定性与信任边界 |
|------|------|------|
| `requestId` | `mail_claim:<mail_id>`，最多 128 字节 | 在首次领取、超时恢复、人工对账和所有跨实例重试中保持不变，由 `mail-service` 生成 |
| `mailId` | 邮件 ID，最多 64 字节 | 来自 `mail-service` 权威邮件记录，不接受玩家覆盖 |
| `characterId` | 非空角色 ID，最多 128 字节 | 来自已验证 game ticket 和服务端角色关系，不接受请求 body 覆盖 |
| `items` | 1 个以上标准化物品项 | 来自预占时冻结的邮件附件快照，不在恢复时重新读取可变请求参数 |
| `requestFingerprint` | `sha256:` 加 64 位小写十六进制 | 由 `mail-service` 按下述规范计算，`game-server` 必须独立重算并比较 |
| `source` | 固定 `mail-claim` | 参与指纹，其他值拒绝 |
| `reason` | 最多 512 字节 | 审计展示，不参与指纹，不得影响幂等判定 |
| `traceId` | 32 位小写十六进制 | 首次预占时生成并持久化，恢复重试保持不变，不参与指纹 |

附件项标准化和指纹算法固定如下，Node.js 与 Rust 必须使用同一测试 fixture 验证：

1. 只接受 `type` 缺省或为 `item` 的附件；把 `id` / `item_id` / `itemId` 统一为十进制 `item_id`，把 `binded` 缺省为 `false`。
2. 校验 `item_id` 为正的 int32、`count` 为正的 uint32、`binded` 为布尔值。
3. 按 `item_id` 升序、`binded` 的 `false` 在 `true` 前排序；相同 `item_id + binded` 合并并对 `count` 做溢出检查。原始附件顺序不影响指纹。
4. 构造无额外空白、UTF-8 编码且键顺序固定的 JSON：

```json
{"mail_id":"mail_01ABC","character_id":"chr_01ABC","source":"mail-claim","items":[{"item_id":1001,"count":2,"binded":false}]}
```

5. 对上述字节计算 SHA-256，编码为 `sha256:<lowercase_hex>`。

`request_id`、`reason`、`trace_id` 和目标实例不进入指纹。`mail_id` 已隐含在 `request_id` 中，但仍显式进入指纹，便于 game-server 查询结果时校验完整业务意图。相同 `request_id` 只有在已存记录的 `request_fingerprint` 完全一致时才能返回幂等成功；任何字段冲突都返回 `REQUEST_FINGERPRINT_CONFLICT`，不得再次发放，也不得把邮件标记为 `claimed`。

#### 4.4.6 附件发放权威路由与调用安全

`game-server` 当前维护独立于 `game-proxy` Room/重连偏好路由的在线资产路由。`game-proxy` 的 `CharacterRouteRecord` 用于接入和 Room owner 选择，不是离线资产写 owner，不能被 `mail-service` 当作安全离线分片依据。

在线路由使用以下 Redis 契约：

```text
key   = <REDIS_KEY_PREFIX>game:online-route:<sha256(character_id)>
value = { version: 2, character_id, instance_id, session_id, authority_generation, authority_token }
TTL   = max(60, 3 * HEARTBEAT_TIMEOUT_SECS)
owner = <REDIS_KEY_PREFIX>game:online-route-owner:<sha256(character_id)>
owner TTL = 3 * route TTL
generation = <REDIS_KEY_PREFIX>game:online-route-generation:<sha256(character_id)>
fence = <REDIS_KEY_PREFIX>game:online-route-fence:<sha256(character_id)>
fence TTL = owner TTL
```

- 只有 game ticket 完整校验成功的连接才能申请权威 generation。Redis Lua 通过 `INCR generation` 产生跨实例可比较的单调代际，并原子写入 `generation:authority_token` fence；申请新代际时立即使旧 route/owner 失效。各 game-server 进程自己的 `session_id` 只用于本地 registry fencing，不能作为跨实例顺序。
- route 发布必须以刚申请的 fence 做 Lua CAS；如果另一个实例已经申请更高代际，延迟的旧发布返回 superseded，不能进入本实例可写 registry。route、owner witness 和 fence 的刷新、恢复、删除也都要求同一 generation/token，旧 owner 不能覆盖、续期或删除新 owner。
- owner TTL 长于 route 但有界，给 route 单独 eviction/过期保留恢复窗口；generation 计数器不参与在线存活判断，用于保证 Redis 正常运行周期内代际单调。Redis 重启导致 route/owner/fence 丢失时，存量连接不得按本地 session 猜测 owner，必须重新认证。
- 刷新区分 `Refreshed`、`Missing` 和 `OwnershipChanged`。只有本实例 registry 仍指向当前 session、fence 与 owner witness 都精确匹配时才能恢复 missing route；检测到 ownership changed、无法证明 owner 或本地 session 已被替换时，旧连接立即从 `PlayerRegistry` 注销并安全断开，不再作为本实例可写会话运行。
- `mail-service` 读取路由后重新扫描 registry，只有 `instance_id` 对应健康且 visibility 为 admin 的 `game-server.admin` endpoint 才允许调用；路由缺失、格式错误、Redis 不可用或实例已摘流统一归入可重试路由失败。
- `mail-service` 把本次解析到的 `authority_generation + authority_token` 随发放请求传给目标实例。`game-server` 在创建 UID、写背包或写幂等成功记录之前，必须同时验证请求 fencing、本地 `PlayerRegistry` handle 以及 Redis route/owner/fence 仍完全一致。目标已切换时返回 `MAIL_CLAIM_ROUTE_MISMATCH / ROUTE_UNAVAILABLE / not_applied`；调用方重新解析 endpoint 和新 fencing，但保持原 `request_id`、指纹和 `trace_id`。
- development/local 的显式调试实例也不能绕过 fencing；调试目标必须同时是 Redis 当前权威 owner。
- 资产事务成功后的 `ItemObtainPush` 和 `InventoryUpdatePush` 直接发送到 `PlayerRegistry` 中当前 session，不依赖角色是否已经进入 Room；旧 session 不会成为推送目标。push 失败仍遵守 4.4.1，不回滚资产事务。
- admin 调用使用 `GAME_ADMIN_TOKEN` 和规范化 service actor，分别限制连接、写入和读取超时，并限制单响应最大字节数。日志只包含实例、`request_id`、`trace_id`、错误码和调用阶段，不记录 token、完整附件或邮件正文。

#### 4.4.7 发放结果与邮件状态决策

目标 game-server 结果必须能够表达以下四种状态：

| 结果 | 含义 | mail-service 动作 |
|------|------|------|
| 首次成功 | 本次请求已经原子提交资产和幂等记录 | 条件更新 `claiming -> claimed` |
| 幂等成功 | 同一 `request_id + request_fingerprint` 的首次成功记录已存在，返回首次结果摘要 | 条件更新 `claiming -> claimed`，不得再次发放 |
| 明确未执行 | game-server 明确保证资产事务没有提交 | 保持同一领取工作流；按错误类别重试、等待玩家修正或进入人工处理 |
| 结果未知 | 请求可能已经到达并提交，但调用方没有获得可验证结果 | 保持 `claiming`/待对账，先按 `request_id` 查询，不能释放为 `unread/read` |

game-server 返回成功时至少关联 `request_id`、`request_fingerprint`、`applied`（首次为 `true`、幂等命中为 `false`）和首次结果摘要。仅有 `ok=true, applied=false` 而没有匹配指纹证据，不足以让邮件进入 `claimed`。

#### 4.4.8 稳定错误分类

三服务内部错误统一携带 `error_code`、`error_category`、`result_state` 和 `retryable`。`error_code` 保留具体原因；调用方依据稳定分类做状态转换，不解析错误文本。

`result_state` 只允许：

- `not_applied`：权威服务明确保证资产未提交。
- `applied`：存在匹配 `request_id + request_fingerprint` 的成功记录。
- `unknown`：无法证明已提交或未提交。

| `error_category` | 代表错误 | `result_state` | 自动处理 |
|------|------|------|------|
| `INVALID_REQUEST` | `INVALID_CHARACTER_ID`、`EMPTY_ITEMS`、`INVALID_ITEM_COUNT`、`INVALID_REQUEST_ID`、`INVALID_FINGERPRINT` | `not_applied` | 不自动重试；记录安全/契约错误 |
| `PERMISSION_DENIED` | game ticket 无效、邮件不属于玩家、admin service actor/token 无效 | `not_applied` | 不重试；HTTP 返回 401/403，记录受限审计 |
| `ROUTE_UNAVAILABLE` | `SERVICE_DISCOVERY_UNAVAILABLE`、`GAME_SERVER_ADMIN_ENDPOINT_NOT_FOUND`、权威角色实例不可确定 | `not_applied` | 保持领取工作流并退避重试，不随机选择 endpoint |
| `TIMEOUT` | registry 查询超时、连接建立超时且确认请求尚未写出 | `not_applied` | 保持同一 `request_id` 退避重试 |
| `RETRYABLE_FAILURE` | game-server 明确返回过载、数据库事务失败或临时内部错误且保证未提交 | `not_applied` | 保持同一请求和冻结快照退避重试 |
| `PERMANENT_FAILURE` | `ITEM_NOT_FOUND`、不支持的附件类型、背包容量不足、`REQUEST_FINGERPRINT_CONFLICT` | `not_applied` | 停止自动发放；等待玩家修正、配置修复或人工处理。指纹冲突必须安全告警 |
| `RESULT_UNKNOWN` | 请求写出后的读超时、连接重置、响应损坏、game-server 提交后崩溃、结果查询暂不可用 | `unknown` | 进入待对账；先查询结果，绝不释放领取预占 |

目标内部错误响应至少包含以下字段；Protobuf 使用对应 snake_case 字段，JSON 兼容入口使用 camelCase。不能只返回自由文本 `message`。

```json
{
  "ok": false,
  "requestId": "mail_claim:mail_01ABC",
  "errorCode": "GAME_SERVER_RESPONSE_TIMEOUT",
  "errorCategory": "RESULT_UNKNOWN",
  "resultState": "unknown",
  "retryable": true,
  "traceId": "0123456789abcdef0123456789abcdef"
}
```

玩家 HTTP 接口不暴露内部 endpoint、token 或底层数据库错误。领取工作流尚未持久化时，`INVALID_REQUEST` 返回 400，`PERMISSION_DENIED` 返回 401/403，`ROUTE_UNAVAILABLE` / `TIMEOUT` / `RETRYABLE_FAILURE` 返回 503/504。领取已经持久化后，自动重试或待对账状态返回 202，并给出稳定业务状态和同一 `request_id`；`PERMANENT_FAILURE` 返回 409 或 422。`RESULT_UNKNOWN` 必须返回 202，不能返回暗示“可以重新开始一次领取”的普通失败。

同一个底层“超时”必须按发生位置分类：连接建立前超时属于 `TIMEOUT/not_applied`；请求可能写入后的响应超时属于 `RESULT_UNKNOWN/unknown`。通知链路的 Redis/NATS/chat push 错误只更新通知 outbox 或指标，不进入附件领取错误状态。

#### 4.4.9 权威路由和客户端输入边界

- 正式环境的领取 HTTP 请求不能选择 game-server。`targetInstanceId` 仅可作为 local/development 调试字段；strict/test/prod 收到时必须忽略或拒绝，不能参与 endpoint 选择。
- `mail-service` 必须根据服务发现和角色权威路由选择 `game-server.admin`。在线角色优先命中其权威实例；离线角色只有在架构能保证单写一致性时才选择确定性可写实例。
- development/local 使用 registry disabled 的固定 admin fallback 时，`local-fallback` 仅表示 endpoint 来源，不是 game-server 身份。无调试目标时使用唯一固定 endpoint 加当前 Redis v2 route；有调试目标时只接受 route 的真实 `instance_id`。两种路径都必须携带 route generation/token，由 game-server 在资产事务前重新校验。
- 找不到权威实例、存在多个候选且无法证明唯一写入方、实例摘流或迁移中时，返回 `ROUTE_UNAVAILABLE`。不得随机选择健康 endpoint 尝试资产写入。
- 玩家不能通过 body 覆盖 `player_id`、`character_id`、`mail_id`、附件、`request_id`、`source`、指纹或内部实例。HTTP path、认证上下文和邮件权威记录是这些字段的唯一来源。
- `trace_id` 是关联标识，不是认证或幂等凭证；日志不得输出 ticket、admin token、完整邮件正文或无界附件 payload。

#### 4.4.10 非目标和兼容边界

本轮邮件可靠交互明确不包含：

- 不引入 JetStream；Core NATS 继续作为允许丢失的在线通知通道。
- 不建设全仓库通用 outbox/inbox SDK，也不改造 metrics、session kick、GM 广播和其他服务事件。
- 不让 `chat-server` 保存邮件、补发离线邮件通知、确认邮件送达或参与附件领取状态。
- 不让 `mail-service` 直接操作 game-server 背包表，也不让 game-server 修改邮件状态。
- 不把短期 `event_id` 去重升级成永久 inbox；邮件列表和 PostgreSQL 邮件记录仍是离线权威来源。
- 不要求三服务同一时刻升级。部署顺序必须先让 `chat-server` 兼容新旧通知、让 `game-server` 支持指纹幂等和结果查询，再启用新的 `mail-service` 生产与恢复逻辑。

### 4.5 邮件类型

- **系统邮件**：由系统/管理员发送，如奖励发放、公告通知
- **玩家邮件**：玩家之间发送（可选功能）

## 5. 公告系统设计

### 5.1 定位

公告系统独立于 `auth-http`，因为 `auth-http` 只负责登录认证。

当前仓库实现说明：

- 已落地独立 `announce-service`
- 当前对外使用 HTTP 接口，而不是独立 TCP / protobuf 公告协议
- 当前支持：
  - `GET /api/v1/announcements`，默认要求 `ANNOUNCE_READ_TOKEN` 或玩家 game ticket
  - `GET /api/v1/announcements/:announceId`，默认要求 `ANNOUNCE_READ_TOKEN` 或玩家 game ticket
  - `POST /api/v1/announcements`，要求 `ANNOUNCE_ADMIN_TOKEN`
  - `PUT /api/v1/announcements/:announceId`，要求 `ANNOUNCE_ADMIN_TOKEN`
  - `DELETE /api/v1/announcements/:announceId`，要求 `ANNOUNCE_ADMIN_TOKEN`
- `ANNOUNCE_ADMIN_TOKEN` 支持 `Authorization: Bearer <token>` 和 `X-Admin-Token: <token>` header，不支持 query token；生产环境禁止空值或开发默认值
- `ANNOUNCE_READ_TOKEN` 支持 `Authorization: Bearer <token>`、`X-Read-Token` 和 `X-Service-Token` header；玩家 game ticket 支持 `Authorization: Bearer <game_ticket>` 和 `X-Game-Ticket`，并校验签名、过期、Redis ticket 归属和 `player-ticket-version:<playerId>`；不支持 query token
- `ANNOUNCE_READ_AUTH_REQUIRED=false` 只允许本地/内网兼容调试；生产环境强制为 `true`，并拒绝默认/弱 `ANNOUNCE_ADMIN_TOKEN`、默认/弱 `TICKET_SECRET` 和默认/弱或与 admin token 相同的 `ANNOUNCE_READ_TOKEN`
- `announce-service` 默认仍是内网能力服务，不是生产公网入口；只读查询第一阶段鉴权已落地，但如果临时公网暴露，仍需要网关、TLS、用途隔离/换票、RBAC 和审计查询继续兜底
- 当前存储支持：
  - `DB_ENABLED=true` 时使用 PostgreSQL 持久化
  - `DB_ENABLED=false` 时回退到内存存储
- 已验证 PostgreSQL 模式下公告在服务重启后仍可查询
- 已通过 `tools/mock-client` 完成 `announce-list/get/create/update/delete` 联调验证

### 5.2 建议放置位置

**方案A**：独立 `announce-service`
- 完全独立，职责单一
- 仅本地开发或手工调试 fallback 可被客户端 HTTP 直连；测试、预发和线上不以固定 host/port 直连作为准入路径，应经 registry endpoint、API/BFF、网关或服务端入口收敛

**方案B**：放在 `game-proxy` 内
- 与 `game-proxy` 部署在一起
- 减少服务数量

**推荐方案A**，原因：
- 公告更新频率低，但逻辑可能扩展（如定时公告、区域公告）
- 独立服务更容易单独扩缩容
- 职责清晰，不与代理混淆

### 5.3 公告协议

```proto
message AnnouncementListReq {
  string locale = 1;              -- 语言
  int32 priority = 2;             -- 优先级筛选
}

message AnnouncementListRes {
  repeated Announcement announcements = 1;
}

message Announcement {
  string id = 1;
  string title = 2;
  string content = 3;
  int32 priority = 4;
  int64 start_time = 5;
  int64 end_time = 6;
  string type = 7;               -- popup / banner /滚屏
  string target_group = 8;       -- 目标玩家群体
}

message AnnouncementPushReq {     -- 管理员推送
  string title = 1;
  string content = 2;
  int32 priority = 3;
  int64 duration = 4;            -- 持续时间(秒)
  string target_group = 5;
}
```

### 5.4 公告存储

公告 `announce_id` 使用全局 ID 字符串格式 `ann_<base32>`，HTTP CRUD、后台展示和 mock-client 参数均使用该 ID。

```sql
CREATE TABLE announcements (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    announce_id varchar(64) UNIQUE NOT NULL,
    locale varchar(32) NOT NULL DEFAULT 'default',
    title varchar(256) NOT NULL,
    content text NOT NULL,
    priority integer DEFAULT 0,
    announce_type varchar(32) DEFAULT 'banner', -- popup / banner / scroll
    target_group varchar(128) DEFAULT 'all',    -- all / vip / new_user 等
    start_time timestamptz NOT NULL,
    end_time timestamptz NOT NULL,
    created_at timestamptz NOT NULL DEFAULT current_timestamp,
    updated_at timestamptz NOT NULL DEFAULT current_timestamp
);

CREATE INDEX IF NOT EXISTS idx_announcements_locale ON announcements (locale);
CREATE INDEX IF NOT EXISTS idx_announcements_priority ON announcements (priority);
CREATE INDEX IF NOT EXISTS idx_announcements_time ON announcements (start_time, end_time);
```

## 6. 服务职责划分

| 服务 | 职责 |
|------|------|
| `auth-http` | 登录、认证、token 发放（保持现状） |
| `game-proxy` | KCP 接入、流量代理 |
| `game-server` | 游戏逻辑、房间系统 |
| `chat-server` | 聊天（单聊、群聊）、历史消息查询、邮件通知推送 |
| `mail-service` | 邮件系统、附件领取、Core NATS 通知 |
| `announce-service` | 公告系统 |

## 7. 客户端获取公告的流程

```
1. 客户端启动
2. 请求 auth-http 登录
3. 登录成功后，可从返回的 `services.announce` 获取公告服务地址
4. HTTP 请求 announce-service 获取公告列表，并携带 game ticket；如果经内部网关读取，也可由网关携带 `ANNOUNCE_READ_TOKEN`
5. announce-service 返回当前有效公告
6. 客户端展示公告
```

当前仓库实际情况补充：

- `auth-http` 在允许暴露内部服务 endpoint 的非生产场景，可通过 registry 把 `chat-server.tcp`、`mail-service.http`、`announce-service.http` 写入统一的 `services` 对象
- 生产环境和正式客户端默认不暴露 chat/mail/announce 内网 endpoint；应经游戏入口、服务端入口、API/BFF 或网关收敛
- 同时仍保留旧的 `gameProxyHost/gameProxyPort` 字段兼容现有客户端
- 当前公告获取链路是“登录后主动 HTTP 拉取”，不是 `game-proxy` 自动推送

```json
{
  "services": {
    "announce": {
      "host": "127.0.0.1",
      "port": 9004,
      "protocol": "http"
    }
  }
}
```

以下是未实现的备选方案，不是当前链路：

```
1. 客户端连接 game-proxy 时
2. game-proxy 自动推送当前有效公告
```
