# 服务注册中心设计

## 1. 文档定位

本文描述 MyServer 当前已经落地的 Redis service registry 实现和统一服务发现迁移后的准入口径。早期“独立 HTTP Registry API”“只注册 host/port”“通过 `metadata.internal_socket` 发现内部 socket”等设计已经不是当前主路径。

当前事实以代码为准，主要实现位置：

| 范围 | 代码位置 |
|---|---|
| Rust registry 类型与客户端 | `packages/service-registry/src/types.rs`、`packages/service-registry/src/client.rs` |
| Node registry schema / discovery helper | `packages/service-registry/node/registry-schema.js` |
| 迁移基线盘点 | `docs/周边服务/服务发现迁移基线.md` |
| 整体架构口径 | `docs/总览/整体架构.md` |

固定 host、port、socket 示例只表示本地默认监听或本地开发 fallback，不是测试、预发、线上跨服务直连建议。测试、预发、线上消费者必须通过 service registry endpoint name 和 instance id 解析目标。

---

## 2. 当前数据模型：schema v2

### 2.1 Redis Key

注册中心当前直接写 Redis，不提供独立 HTTP Registry 服务。

| 数据 | Key | 内容 |
|---|---|---|
| 实例 payload | `service:{service}:instances:{instance}` | Redis Hash，`data` 字段保存 JSON payload |
| 实例心跳 | `heartbeat:{service}:{instance}` | String，TTL 续租，默认 30 秒 |

实际 key 可能带 `REGISTRY_KEY_PREFIX` 前缀。共享 Redis 时必须按环境或 namespace 显式配置前缀，避免不同环境实例互相发现。

### 2.2 实例 Payload

实例 payload 当前使用 `schema_version=2`，核心字段如下：

| 字段 | 说明 |
|---|---|
| `schema_version` | 当前固定为 `2` |
| `id` | 实例唯一 ID，部署层必须保证同服务内唯一 |
| `name` | 服务名，例如 `game-server` |
| `host` / `port` / `admin_port` / `local_socket` | 兼容字段，保留给旧读取路径和 v1 归一化；新消费者应读取 `endpoints` |
| `endpoints` | schema v2 的主路径，描述同一实例可被不同消费者使用的多个端点 |
| `tags` / `weight` / `metadata` | 标签、权重和实例级元数据 |
| `registered_at` | 注册时间戳 |
| `healthy` | 实例级健康标记 |

Rust 和 Node schema 都会归一化 payload。显式 v2 payload 必须提供合法 `endpoints`；旧 v1 payload 只在兼容读取时由 `host` / `port` / `admin_port` / `local_socket` 补齐默认 endpoint。

### 2.3 Endpoint 字段

`endpoints[]` 中每个 endpoint 的字段和语义：

| 字段 | 说明 |
|---|---|
| `name` | 稳定逻辑名称，消费者按它选择目标能力 |
| `protocol` | `http`、`tcp`、`udp`、`kcp`、`grpc`、`local_socket` |
| `host` | 网络 endpoint 的发布地址；`local_socket` endpoint 必须为空 |
| `port` | 网络 endpoint 的端口；`local_socket` endpoint 必须为 `0` |
| `socket` | `local_socket` endpoint 的 socket 名称或路径；网络 endpoint 必须为空 |
| `visibility` | `public`、`internal`、`admin`、`local` |
| `metadata` | endpoint 级元数据，例如 build version、zone、instance id |
| `healthy` | endpoint 级健康标记 |

服务注册时会避免发布 `0.0.0.0`、`::`、空字符串这类通配监听地址；本地默认会归一化为 `127.0.0.1`，部署环境应注入可被消费者访问的 advertised host。

### 2.4 当前 Endpoint Name

| Endpoint name | 当前主要发布者 | 语义 |
|---|---|---|
| `client` | `game-proxy`、`game-server` | 客户端或兼容玩家入口；当前正式游戏入口使用 `game-proxy.client` |
| `client-tcp-fallback` | `game-proxy` | 本地 mock / 调试使用的 TCP fallback 玩家入口 |
| `http` | `auth-http`、`admin-api`、`mail-service`、`announce-service` | HTTP 服务入口，visibility 由服务职责决定 |
| `tcp` | `chat-server` | 聊天 TCP 内部入口 |
| `grpc` | `match-service` | 匹配服务 gRPC 内部入口 |
| `admin` | `game-server`、`game-proxy` | 内部控制面入口 |
| `internal` | `auth-http`、`game-server` | 服务内部入口；`game-server.internal` 当前是 local socket，用于 match-service 建房 |
| `proxy-local` | `game-server` | `game-proxy` 转发到 `game-server` 的本机 local socket |

消费者不得把 endpoint name 退化成“找任意 host/port”。例如 `game-proxy` 只消费 `game-server.proxy-local`，不会回退到 `game-server.internal` 或 legacy `client` endpoint；管理链路只消费 `admin` 且要求 admin visibility。

## 3. 当前服务注册状态

以下服务当前都会注册 schema v2 endpoint：

| 服务 | 注册状态 | 当前 endpoint |
|---|---|---|
| `game-server` | 已注册 | `client` / `tcp` / `internal`；`admin` / `tcp` / `admin`；`internal` / `local_socket` / `local`；`proxy-local` / `local_socket` / `local` |
| `game-proxy` | 已注册 | `client` / `kcp` / `public`；`client-tcp-fallback` / `tcp` / `public`；`admin` / `http` / `admin` |
| `chat-server` | 已注册 | `tcp` / `tcp` / `internal` |
| `match-service` | 已注册 | `grpc` / `grpc` / `internal` |
| `mail-service` | 已注册 | `http` / `http` / `internal` |
| `announce-service` | 已注册 | `http` / `http` / `internal` |
| `auth-http` | 已注册 | `http` / `http` / `public`；`internal` / `http` / `internal` |
| `admin-api` | 已注册 | `http` / `http` / `admin` |

`metrics-collector` 当前不注册 service registry 实例，也不发布可连接 endpoint。它消费 Core NATS metrics 并写 Redis `metrics:*` 与 `metrics:heartbeat:*` 快照；这些 key 不是 service registry schema。

## 4. 已完成的统一发现链路

当前已经完成迁移并按 endpoint name 发现的主链路：

| 消费者 | 发现目标 | endpoint | 当前行为 |
|---|---|---|---|
| `auth-http` | `game-proxy` | `client` | 登录 / 签票响应中的 `services.game` 来自 registry；兼容字段 `gameProxyHost` / `gameProxyPort` 仍保留 |
| `auth-http` | `game-server` | `admin` | 内部 game-server 管理调用通过 registry 解析 admin endpoint |
| `admin-api` | `game-server` | `admin` | GM、玩家控制、运行时配置等通过 registry 解析；多实例写操作要求明确 `targetInstanceId` |
| `admin-api` | `game-proxy` | `admin` | 监控和 rollout 控制面通过 registry 解析，可聚合多个 proxy 实例 |
| `mail-service` | `game-server` | `admin` | 邮件附件发放通过 registry 解析；多实例场景要求明确目标实例 |
| `game-server` | `match-service` | `grpc` | 匹配客户端通过 `match-service.grpc` 解析 tonic gRPC 地址，并支持重发现 |
| `game-proxy` | `game-server` | `proxy-local` | 接入代理只消费 `game-server.proxy-local` local socket endpoint |
| `match-service` | `game-server` | `internal` | 创建 matched room 时按 `game-server.internal` local socket endpoint 选择游戏服实例 |

补充链路：

- `auth-http` 可在显式允许的非生产配置下把 `chat-server.tcp`、`mail-service.http`、`announce-service.http` 放入登录响应的 `services` 对象；生产默认不暴露这些内部 endpoint。
- `chat-server` 和 `announce-service` 当前主要是注册生产者，不消费 service registry 去访问其它服务。

## 5. Strict Discovery 准入口径

测试、预发、线上统一要求：

```bash
REGISTRY_ENABLED=true
DISCOVERY_REQUIRED=true
```

准入规则：

| 规则 | 当前口径 |
|---|---|
| registry 必须开启 | `DISCOVERY_REQUIRED=true` 或 strict 环境下，服务配置必须拒绝 `REGISTRY_ENABLED=false` |
| 消费者必须按 endpoint 发现 | 跨服务访问不能依赖旧直连变量拼 host/port/socket |
| 发现失败要明确失败 | registry 不可用、目标 endpoint 缺失、无健康实例、目标实例歧义时，应让启动、请求或演练明确失败 |
| 多实例写操作必须明确目标 | `admin-api` / `mail-service` 对单个 `game-server.admin` 的写操作不能在多 endpoint 下随机选中 |
| local fallback 只限本地 | 只有 `APP_ENV` / `NODE_ENV` 为 `local` 或 `development` 且 discovery 非 required 时，才允许使用本地 fallback |
| 关闭 legacy 变量 | 设置 `DISALLOW_LEGACY_DIRECT_CONFIG=true` 后，旧直连变量存在即配置失败 |

本地 fallback 变量只能用于 development / local 的单机联调或手工 drill，例如：

| 变量 | 只允许的语境 |
|---|---|
| `GAME_PROXY_HOST` / `GAME_PROXY_PORT` | `auth-http` 在 registry disabled 且 discovery 非 required 时回填本地 `services.game` |
| `GAME_SERVER_ADMIN_HOST` / `GAME_SERVER_ADMIN_PORT` | `auth-http`、`admin-api`、`mail-service` 本地管理口 fallback |
| `MATCH_SERVICE_ADDR` | `game-server` 本地直连 `match-service` fallback |
| `UPSTREAM_LOCAL_SOCKET_NAME` | `game-proxy` 本地静态 upstream fallback |
| `GAME_SERVER_INTERNAL_SOCKET_NAME` / `GAME_INTERNAL_SOCKET_NAME` | `match-service` 本地建房 socket fallback |

这些变量不能出现在测试、预发、线上跨服务准入路径中。固定端口表和 `.env.example` 中的值只用于本地默认监听或 fallback 示例。

## 6. 心跳、注销与健康过滤

注册中心通过实例级 heartbeat 过滤可用实例：

1. 服务注册时写入 `service:{service}:instances:{instance}` 的 `data` 字段。
2. 服务启动 heartbeat，写入 `heartbeat:{service}:{instance}`，默认 TTL 30 秒，常见续租间隔 10 秒。
3. 发现逻辑扫描 `service:{service}:instances:*`，只返回 heartbeat key 仍存在且 payload 能归一化的实例。
4. 正常关闭时删除实例 key 和 heartbeat key。

需要区分：

| Key 类型 | 用途 |
|---|---|
| `heartbeat:{service}:{instance}` | service registry 实例续租 |
| `metrics:heartbeat:{service}` / `metrics:heartbeat:{service}:{instance}` | metrics 新鲜度快照 |

二者不是同一套健康机制，不能互相替代。

## 7. Redis Route Store 与 Registry 边界

`game-proxy` 的 Redis route store 和 service registry 共用 Redis 基础设施，但 schema 不同：

| 能力 | Key / Channel | 说明 |
|---|---|---|
| service registry | `service:*:instances:*`、`heartbeat:*` | 服务实例注册、发现和心跳 |
| proxy route store | `proxy:route-store:state`、`proxy:route-store:updates` | proxy 上游、房间、玩家路由状态快照与 reload 通知 |

`game-proxy -> game-server.proxy-local` 的发现来自 service registry；发现到的 upstream 再同步进 route store。route store 不是服务发现 schema 的替代品。

## 8. 监控边界与未完成项

当前已经存在 metrics 上报、`metrics-collector` 采集、Redis 快照和 `admin-api` 基础读取链路，但这不等于运维观测 M10 已完成。

尚未在本设计中声明完成的 M10 项包括：

- dashboard
- alerts
- 审计化巡检
- 容量指标

后续如要把发现成功率、fallback 使用、endpoint 缺失、容量水位纳入准入，需要另行补 dashboard / alert / audit 文档和实现。

## 9. 历史设计边界

以下内容仅作为历史背景，不再作为当前实现依据：

| 历史设想 | 当前状态 |
|---|---|
| 独立 HTTP Registry API | 当前没有独立 registry HTTP 服务，服务直接读写 Redis |
| 只靠实例级 `host` / `port` / `admin_port` / `local_socket` | 当前主路径是 schema v2 `endpoints` |
| 通过实例级 metadata 保存内部 socket 作为发现主路径 | 当前通过 `game-server.internal` 和 `game-server.proxy-local` endpoint 发现 |
| 周边服务未统一注册 | `mail-service`、`announce-service`、`auth-http`、`admin-api` 当前都注册 schema v2 endpoint |
| `auth-http` 只返回静态 game proxy 地址 | 当前 `services.game` 主路径来自 `game-proxy.client` 发现，旧字段只是兼容 |

新增或修改服务发现消费者时，需要同步更新本文、`docs/周边服务/服务发现迁移基线.md` 和对应配置校验，确保 strict discovery、endpoint name、fallback 日志与测试覆盖一致。
