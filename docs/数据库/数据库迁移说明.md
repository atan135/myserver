# 数据库迁移现状

## 文档定位

本文记录当前仓库 PostgreSQL 初始化与 migration runner 的事实状态。仓库数据库方案已切换到 PostgreSQL，历史旧库体系不再作为当前开发目标。

## 当前能力

- `db/init.sql` 是本地开发和空库 bootstrap 的主要初始化脚本，按 `psql` 语义创建各服务数据库与表结构。
- `db/migrations/0001_create_schema_migrations.sql` 已创建 `schema_migrations` 元表。
- `tools/db-migrate.js` 已支持 migration 文件名、顺序、重复版本和 checksum 校验。
- `tools/db-migrate.js --dry-run` / `--list` 可在不连接数据库的情况下打印执行顺序和 checksum。
- 真实执行会读取 `DATABASE_URL`，应用未执行 migration，并检查已应用 migration 的名称和 checksum 是否漂移。
- `scripts/db-migrate.ps1` 提供 PowerShell 封装。
- 根脚本已提供：
  - `npm run check:migrations`
  - `npm run db:migrate:dry-run`
  - `npm run db:migrate`

## 后续范围

以下工作应按 PostgreSQL 继续推进：

- 将 `db/init.sql` 完整拆分为版本化初始 schema migration。
- 为已有开发库或测试库设计 baseline 标记策略。
- 编写 PostgreSQL 增量升级、回滚和数据修复脚本。
- 将 Node 服务启动期的零散 schema 初始化全部改造成 migration 驱动。

## 当前使用建议

本地空库仍可使用：

```powershell
psql -U postgres -f db/init.sql
```

检查 migration 文件元数据：

```powershell
npm run check:migrations
npm run db:migrate:dry-run
```

新的业务表开发应避免继续散落在服务启动期 DDL 中，优先沉淀为 PostgreSQL migration。后续仍需细化 schema 目录、baseline 规则、升级顺序、回滚/修复策略和 CI 检查命令。
