# 数据库部署准入说明

## 定位

`tools/db-deploy.js` 是五库 migration 的部署编排入口。它只编排 `auth -> game -> chat -> announce -> mail` 的数据库检查和 SQLx migration，不启动、停止、重启或引流任何服务，也不会自动执行恢复、回滚或删除业务数据库。

部署 gate 的版本化声明位于 `db/config/deploy-gate.json`。该文件固定每库关键表、静态服务兼容 migration 范围和可选 readiness URL 的环境变量名；它不是运行中服务的健康证明。实际 readiness 只有在 postflight 显式传入 `--check-readiness` 后才会发送 HTTP GET，未配置或未检查时报告会分别标记为 `not-configured` 或 `not-checked`，绝不会称服务已经启动或健康。

当前仍存在历史服务启动期 DDL 的治理项。本 gate 负责部署前后数据库事实、版本兼容声明和已有 readiness endpoint 的检查，不把该现状伪装为“所有服务已迁离 DDL”；服务侧迁移完成前，仍须按实际代码和专题设计审阅发布范围。

## 入口

| 目的 | 命令 | 是否写数据库 |
|------|------|--------------|
| 静态 CI 校验 | `npm run db:deploy -- validate --environment ci` | 否 |
| 部署前检查 | `npm run db:deploy -- preflight --environment production` | 否 |
| 执行五库 migration | `npm run db:deploy -- apply --environment production --actor release-operator` | 是，按顺序停止 |
| 部署后数据库检查 | `npm run db:deploy -- postflight --environment production --check-readiness --require-readiness` | 否，readiness 仅 HTTP GET |
| Windows PowerShell 入口 | `powershell -ExecutionPolicy Bypass -File .\scripts\db-deploy.ps1 -Command preflight -Environment production` | 由命令决定 |
| 仓库静态 CI | `npm run db:ci` | 否 |

`apply` 固定执行 `preflight -> 五库 up -> postflight`。preflight、任何一库 migration 或 postflight 失败都会返回稳定的既有数据库退出码并停止后续库和服务步骤。输出为单行 JSON，包含 `phase`、逐库 report、`stopped` 和人工恢复步骤；不得解析底层 PostgreSQL 或 SQLx 原始文本。

每个 `up` 通过 `tools/db-lock-runner.js` 先以与 SQLx `Migrate::lock` 相同的 database-derived advisory key 执行 `pg_try_advisory_lock`，持锁运行 SQLx CLI child，最后在 `finally` 释放。锁不可用时返回 `5` 且不会启动 SQLx child。SQLx CLI 0.8.6 的 `migrate run` 不会自行调用这个锁，因此 `bin/sqlx.exe migrate run` 不是受支持的部署入口；部署和运维必须使用本仓库的 `db` 或 `db-deploy` 命令。

## Preflight 与 Postflight

preflight 对每个数据库检查：

- 本地 migration 文件、SQLx history、pending version、未知 history、checksum 和失败 history。
- 与 SQLx 相同 database name 派生的 PostgreSQL advisory lock 是否可用；使用 `pg_try_advisory_lock` 探测后立即释放，不等待、不执行 DDL。
- pending 不可逆 migration 是否具有 `MYSERVER_DB_DEPLOY_BACKUP_<DATABASE>_ID` 与 `MYSERVER_DB_DEPLOY_BACKUP_<DATABASE>_CHECKSUM`。报告只记录证据是否存在和格式是否有效，不输出证据值。
- `db/config/deploy-gate.json` 中静态服务兼容范围是否覆盖本次本地目标 migration。该字段明确带有 `runtimeObserved: false`。
- SQLx `validate` 是否在受控制品、目录和 history 上通过。

postflight 在 history 没有 pending、checksum 合法且 advisory lock 可用后继续检查：

- 目标 migration 已写入 history，且 `db drift` 对受审阅 target catalog 没有未批准漂移。target/actual `manifest_sha256` 都按 `[object_kind, object_identity, definition_sha256]` 计算，干净 schema 的两个值必须相同；报告标签 `object_name` 不影响完整性信号。
- `deploy-gate.json` 配置的关键表存在。
- 只有显式 `--check-readiness` 时才访问配置的 readiness URL。HTTP 2xx 且 JSON `ok` 不是 `false` 才标记 `healthy`；不可达、非 2xx 或 `ok: false` 会阻断 postflight。`--require-readiness` 还会拒绝未配置 endpoint。

失败后不要手工修改 `_sqlx_migrations`、继续后续服务发布或自动重试事务外 DDL。按 JSON 的 `recovery` 和 migration 文件的 `Recovery command` 处理：锁冲突等待持有者释放；checksum/history 问题恢复经审阅的文件或 history；不可逆变更先核验备份；漂移通过新的受审阅 migration 或精确 allowance 处理。

## 环境变量与审批

迁移连接仍使用[数据库迁移体系设计](./数据库迁移体系设计.md)定义的 `MYSERVER_DB_MIGRATION_<DATABASE>_{URL,USER,PASSWORD}`。部署入口不读取服务 `DATABASE_URL`，不将 URL、密码或完整备份信息写入命令行、JSON 或日志。

生产或预发的不可逆 pending migration 必须先由人工审批确认以下证据：

- 每个受影响库的 backup artifact ID 和 SHA-256 或 SHA-512 checksum，通过 `MYSERVER_DB_DEPLOY_BACKUP_<DATABASE>_ID` 和 `MYSERVER_DB_DEPLOY_BACKUP_<DATABASE>_CHECKSUM` 注入。
- migration 文件中的 Backup point、Recovery command、Risk summary 和已演练恢复步骤。
- 静态兼容范围覆盖的服务版本，以及旧消费者已经退出或仍保持 expand/migrate 兼容的证据。
- postflight 所需实际 readiness URL。环境变量名称在 `deploy-gate.json` 中固定，例如 `MYSERVER_DB_DEPLOY_AUTH_HTTP_READINESS_URL`；URL 不得携带凭据、query 或 fragment。

CI 的 `.github/workflows/database-migration.yml` 只运行不连接 PostgreSQL 的静态 gate 和 unit test。受保护的 integration runner 如需执行五库空库重建，必须有本地受控 SQLx/psql 制品和独立 PostgreSQL 凭据，且在审批后显式运行：

```powershell
$env:MYSERVER_DB_DEPLOY_TEMPORARY_REBUILD = "1"
$env:MYSERVER_DB_DEPLOY_TEMPORARY_POSTGRES_URL = "postgresql://postgres@localhost:5432/postgres"
$env:MYSERVER_DB_DEPLOY_TEMPORARY_POSTGRES_PASSWORD = "<secret-manager-injected>"
npm run db:ci:rebuild
```

`db:ci:rebuild` 还要求命令内的 `--confirm-temporary-rebuild stage6-temporary-rebuild`，并只接受 `localhost`、`127.0.0.1` 或 `::1` 的 `5432/postgres` bootstrap URL。它只创建 `myserver_stage6_` 前缀、随机 token 的临时库；名称只有在本次 `CREATE DATABASE` 成功后才登记 cleanup，五库完成或失败后都会 terminate 这些已创建库的连接并在 `finally` 中 drop。任何清理失败都会保留非零结果和临时库名。它不读取、写入或删除 `myserver_auth`、`myserver_game`、`myserver_chat`、`myserver_announce`、`myserver_mail` 等业务库。

`npm run db:stage7:drill` 是独立的受控失败与恢复演练，不属于 CI 默认路径。它需要 `MYSERVER_STAGE7_RUN_POSTGRES=1`、本机 `localhost:5432/postgres` bootstrap URL 和受保护密码环境变量，且只创建/清理 `myserver_stage7_` 临时库。演练会验证空库/重复执行、lock-runner 对独立 SQLx worker 的互斥和释放后重试、实际 500ms DDL lock timeout 后的未应用/释放重试、checksum、SQL failure、受控连接中断和 Stage 4 expand/contract 恢复；它不会启动服务、探测 readiness 或接触业务库。

若部署环境需要采集迁移事件，操作者在执行 `db up` 或 deployment apply 前显式设置 `MYSERVER_DB_MIGRATION_METRICS_ENABLED=1`，并通过 `MYSERVER_DB_MIGRATION_METRICS_NATS_URL` 或既有 `NATS_URL` 指向 Core NATS。producer 发送到 `myserver.metrics.db-migration.*`，由已部署的 `metrics-collector` 写入 Redis；事件含数据库、目标/已应用/尝试版本、outcome 和错误类别，不含连接串、密码、SQL 或原始错误。emitter 只接收 allowlist 中的 Windows 运行变量、metrics 开关/timeout、metrics-specific NATS URL、`NATS_URL` fallback 和事件 JSON；`NATS_URL` 是唯一允许的通用 transport URL，Redis、数据库、业务服务、Stage7/deploy 和通用凭据变量不会传入 child。指标通道不可用只会在 CLI report 标为 `unavailable`，不会改变 migration、安全审计或部署 gate 的结果。完整指标边界与验证见[数据库迁移体系设计](./数据库迁移体系设计.md)。

## Development 与 Production

development 的常规重置仍使用 `scripts/reset-dev-data.ps1 -Confirm`，它会明确删除五个本地开发业务库；不要把受保护的 `db:ci:rebuild` 当成开发数据 reset。开发环境可先运行 `preflight` 和 `postflight` 验证已有库，但 endpoint 探测必须由操作者显式启用，且本工具不会启动 dev stack。

production 和预发的建议顺序为：先以只读 preflight 验证 history、锁、备份和兼容范围；完成变更审批后运行一次 `apply`；再使用 `postflight --check-readiness --require-readiness` 证明数据库和已配置 endpoint。只有 postflight 通过后，外部发布系统才可以继续后续服务启动或流量步骤；这些步骤不属于数据库工具。
