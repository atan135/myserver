# 数据库初始化说明

## 文档定位

本文记录当前仓库 PostgreSQL 初始化与本地重置流程。数据库迁移体系已进入建设阶段；工具选型、对象归属和目标规范见[数据库迁移体系设计](./数据库迁移体系设计.md)。在迁移骨架和基线迁移完成前，本页命令仍是本地开发的过渡入口，不代表生产升级方案。

## 当前能力

- `db/init.sql` 是本地开发和空库 bootstrap 的主要过渡脚本，按 `psql` 语义创建各服务数据库与表结构。
- `scripts/reset-dev-data.ps1` 用于清空并重建本地 PostgreSQL 开发库，必须显式传入 `-Confirm`。
- 当前初始化覆盖 `myserver_auth`、`myserver_game`、`myserver_chat`、`myserver_announce` 和 `myserver_mail`。
- 当前尚未落地迁移状态表、migration runner 或根级数据库迁移脚本；不能提前把设计态命令用于数据库。

## 建设中范围

以下工作应按 PostgreSQL 继续推进：

- 将 `db/init.sql` 完整拆分为版本化初始 schema migration。
- 为已有开发库、测试库或线上库设计 baseline 标记策略。
- 编写 PostgreSQL 增量升级、回滚和数据修复脚本。
- 将 Node 服务启动期的零散 schema 初始化全部改造成 migration 驱动。

## 当前使用建议

初始化本地空库：

```powershell
psql -U postgres -f db/init.sql
```

清空并重建本地开发库：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\reset-dev-data.ps1 -Confirm
```

`reset-dev-data.ps1` 只处理 PostgreSQL，不清理 Redis、NATS、日志、上传文件、缓存或本地进程。需要完整清理本地开发状态时，应按具体依赖另行处理。

迁移基线落地前，新的业务表开发仍需同步更新 `db/init.sql` 和相关测试，且不得继续增加新的服务启动期 DDL。迁移基线落地后，以版本化 migration 为唯一结构变更入口，`db/init.sql` 和服务本地初始化脚本按迁移体系设计中的兼容/退役策略处理。

## 邮件可靠链路数据库边界

邮件跨服务链路同时依赖根 `db/init.sql` 与 `apps/mail-service/db/init.sql` 中一致的 mail schema，以及 game 数据库中的 `character_inventory_grants`：

- `mail_notification_outbox` 保存版本化通知、发布租约、尝试次数和终止结果；邮件与 outbox 必须同事务创建。
- `mail_claim_workflows` 保存冻结附件、稳定 `claim_request_id`、指纹、玩家/恢复租约和 `processing/retryable_failure/permanent_failure/reconciliation_pending/manual_review/claimed` 状态。
- `mail_admin_operation_audit` 是 append-only 管理审计，不允许 UPDATE/DELETE/TRUNCATE。
- `character_inventory_grants` 与背包事务原子写入，以 `request_id` 唯一并保存角色、指纹和首次结果摘要。

当前开发期仍可删库重建，但三服务滚动升级或回滚时不能删除这些表、列、状态和幂等记录。回滚前先关闭新领取，然后查询非 `claimed` 工作流和有效租约；只要存在存量工作流，就必须保留兼容 recovery worker、game-server 3009/3010 查询和 game 幂等表。`MAIL_CLAIM_RECOVERY_ENABLED=false` 的新版本会在启动时执行同一安全检查并以 `MAIL_CLAIM_RECOVERY_DISABLE_BLOCKED` 拒绝遗弃 backlog。

完整 SQL、保留期和操作顺序见 [邮件可靠链路灰度与回滚手册](../后台与运维/邮件可靠链路灰度与回滚手册.md)。该手册不替代未来的增量 migration；进入不可删库环境前，仍需把这些 schema 变更转换为版本化迁移。
