# 数据库初始化说明

## 文档定位

本文记录当前仓库 PostgreSQL 初始化与本地重置流程。数据库迁移体系已建立骨架和空库基线，并完成五库临时 PostgreSQL 的 migration/catalog 对比；工具选型、对象归属和目标规范见[数据库迁移体系设计](./数据库迁移体系设计.md)。部署前后 gate、CI 空库演练与生产审批边界见[数据库部署准入说明](./数据库部署准入说明.md)。运行时存量库 baseline 的 allowlist 仍故意为空，因此本页命令不代表生产升级方案。

## 当前能力

- `db/init.sql` 是兼容保留的过渡脚本；它仍可用于历史本地环境，但不再是新增 Schema 的入口。
- `db/bootstrap/development.sql` 只创建五个开发数据库；`db/migrations/<database>/` 是五库 Schema 的唯一结构来源；`db/seeds/development/` 只放 local origin/world 等显式开发数据。
- `scripts/reset-dev-data.ps1` 用于清空并重建本地 PostgreSQL 开发库，必须显式传入 `-Confirm`，且只接受 `development` 和 localhost migration URL。
- `tools/db-deploy.js` 的 `rebuild-check` 是 CI/部署 gate 的受保护临时空库演练，只允许 `myserver_stage6_` 前缀临时库并 finally 清理；它不会删除五个开发业务库，不能代替 reset。
- 当前初始化覆盖 `myserver_auth`、`myserver_game`、`myserver_chat`、`myserver_announce` 和 `myserver_mail`。
- `db/config/sqlx-cli.json` 已登记并验证 Windows x64 的受控 SQLx CLI 0.8.6 制品及 SHA-256；工具只接受 `bin/sqlx.exe` 的登记制品，缺失、版本不符或 hash 不符时以稳定错误码失败，不会下载、安装或回退到 PATH。
- 五库临时空库已完成 `migration -> catalog` 与当前 `db/init.sql` 对照；受测试注入的审阅 allowlist 已验证存量 baseline history 写入和 SQLx 校验。运行时 allowlist 仍为空，实际存量库、预发和生产库会继续安全拒绝，直到对应环境 fingerprint 经人工审阅后登记。

## 后续范围

以下工作应按 PostgreSQL 继续推进：

- 采集、人工审阅并登记预发、生产和其他存量环境各自的 catalog fingerprint；在登记前继续拒绝 baseline。
- 编写 PostgreSQL 增量升级、回滚和数据修复脚本。
- 将 Node 服务启动期的零散 schema 初始化全部改造成 migration 驱动。

## 当前使用建议

初始化或重置本地开发业务库：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\reset-dev-data.ps1 -Confirm
```

清空并重建本地开发库：

`reset-dev-data.ps1` 依次运行 `db/bootstrap/development.sql`、`db up --database all --actor local-reset` 和 `db/seeds/development/auth-local-world.sql`。它只使用 manifest 中的受控 SQLx CLI 制品并校验 hash；不会下载、安装或回退到 PATH。制品缺失、未登记或 hash 不符时会安全失败，留下空开发库供排查。

`reset-dev-data.ps1` 只处理 PostgreSQL，不清理 Redis、NATS、日志、上传文件、缓存或本地进程。需要完整清理本地开发状态时，应按具体依赖另行处理。

需要在不启动服务的前提下检查已有 development 数据库时，使用部署 gate 的只读 preflight；只有配置 endpoint 且显式传入 `--check-readiness` 才会探测服务：

```powershell
npm run db:deploy -- preflight --environment development
npm run db:deploy -- postflight --environment development --check-readiness
```

迁移基线已拆出后，新增业务表开发只更新版本化 migration 和相关测试，且不得继续增加新的服务启动期 DDL。`db/init.sql` 已完成一次五库临时空库 migration/catalog 对比，现仅作为历史兼容与 catalog 对照来源；常规本地初始化已经从该入口退役。只有在依赖它的历史环境、静态测试和服务本地 SQL 均迁离后，才可移除该兼容文件；服务本地初始化脚本也不得再新增 Schema 语义。

## 邮件可靠链路数据库边界

邮件跨服务链路同时依赖根 `db/init.sql` 与 `apps/mail-service/db/init.sql` 中一致的 mail schema，以及 game 数据库中的 `character_inventory_grants`：

- `mail_notification_outbox` 保存版本化通知、发布租约、尝试次数和终止结果；邮件与 outbox 必须同事务创建。
- `mail_claim_workflows` 保存冻结附件、稳定 `claim_request_id`、指纹、玩家/恢复租约和 `processing/retryable_failure/permanent_failure/reconciliation_pending/manual_review/claimed` 状态。
- `mail_admin_operation_audit` 是 append-only 管理审计，不允许 UPDATE/DELETE/TRUNCATE。
- `character_inventory_grants` 与背包事务原子写入，以 `request_id` 唯一并保存角色、指纹和首次结果摘要。

当前开发期仍可删库重建，但三服务滚动升级或回滚时不能删除这些表、列、状态和幂等记录。回滚前先关闭新领取，然后查询非 `claimed` 工作流和有效租约；只要存在存量工作流，就必须保留兼容 recovery worker、game-server 3009/3010 查询和 game 幂等表。`MAIL_CLAIM_RECOVERY_ENABLED=false` 的新版本会在启动时执行同一安全检查并以 `MAIL_CLAIM_RECOVERY_DISABLE_BLOCKED` 拒绝遗弃 backlog。

完整 SQL、保留期和操作顺序见 [邮件可靠链路灰度与回滚手册](../后台与运维/邮件可靠链路灰度与回滚手册.md)。该手册不替代未来的增量 migration；进入不可删库环境前，仍需把这些 schema 变更转换为版本化迁移。
