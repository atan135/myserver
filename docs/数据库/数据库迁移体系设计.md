# 数据库迁移体系设计

## 文档定位

本文是 MyServer PostgreSQL 迁移体系的当前设计基线，记录 2026-07-18 的 Schema 来源盘点、对象归属、工具选型和后续实现约束。具体实现和命令以仓库代码为准；基线已拆分，并已完成五库临时 SQLx/PostgreSQL 演练。运行时存量库 baseline 仍保持拒绝状态，原因是 allowlist 故意未登记任何环境 fingerprint，而非演练尚未完成。

本设计只治理数据库结构版本、基线、部署准入和漂移检测，不调整游戏业务模型，也不把业务数据修复或大表异步回填塞进 DDL migration。

## 当前事实

### Schema 来源

当前数据库结构同时来自以下入口，并非真正的单一来源：

| 来源 | 作用 | 当前问题 |
|------|------|----------|
| `db/init.sql` | 使用 `psql` 创建五个数据库并初始化目标结构 | 同时包含数据库 bootstrap、五库 DDL 和 local origin/world 基础数据；依赖 `\connect`，不能作为单库 migration 直接执行 |
| `apps/announce-service/db/init.sql` | 公告服务独立初始化 | 与根 SQL 的 `world_id` 列和索引不一致，并包含演示公告 |
| `apps/mail-service/db/init.sql` | 邮件服务独立初始化 | 与根 SQL 的 `world_id` 列和索引不一致 |
| 六个服务的启动期 DDL | 服务连接后执行 `CREATE/ALTER/INDEX/FUNCTION/TRIGGER` | 无版本、checksum 和全局锁；不同启动路径会生成不同结构 |
| `scripts/reset-dev-data.ps1` | 删除并重建五个本地开发库 | 使用 development bootstrap + migration + seed；仅接受 localhost URL，受 `-Confirm` 保护 |
| 服务初始化、种子和测试脚本 | 初始化管理员、origin/world、测试账号或测试数据库 | Schema、基础数据和测试数据边界未完全分离 |

根 SQL 和服务 SQL 都没有显式 `CREATE SCHEMA` 或 `SET search_path`，所有未限定对象当前都落在连接角色默认 `search_path` 下，通常是 `public`。根 SQL 也没有稳定设置数据库或对象 owner；物理 owner 是实际执行 `CREATE DATABASE`/DDL 的登录角色，本地示例通常为 `postgres`。后续迁移必须固定 `public` 为首期受管 Schema，并显式区分迁移凭据和应用运行凭据。

### 数据库与连接拓扑

| 数据库 key | 默认数据库 | 受管 Schema | 逻辑所有者 | 运行时连接和消费者 |
|------------|------------|-------------|------------|--------------------|
| `auth` | `myserver_auth` | `public` | `auth-http` 负责账号和登录审计；`admin-api` 负责后台、MyForge、全局 ID/世界元数据和监控归档 | `auth-http:DATABASE_URL`；`admin-api:DATABASE_URL`；二者都存在写操作 |
| `game` | `myserver_game` | `public` | `game-server` 负责游戏运行时、角色和资产对象；`admin-api` 对 GM 对象有受控写权限 | `game-server:DATABASE_URL`；`auth-http:GAME_DATABASE_URL/AUTH_GAME_DATABASE_URL`；`admin-api:GAME_DATABASE_URL/ADMIN_GAME_DATABASE_URL` |
| `chat` | `myserver_chat` | `public` | `chat-server` | `chat-server:DATABASE_URL` |
| `announce` | `myserver_announce` | `public` | `announce-service` | `announce-service:DATABASE_URL` |
| `mail` | `myserver_mail` | `public` | `mail-service` | `mail-service:DATABASE_URL`；邮件发奖通过 game-server 管理协议，不直接连接游戏库 |

这里的“逻辑所有者”决定谁可以新增或修改该组对象。PostgreSQL 没有跨数据库 join：`auth-http` 和 `admin-api` 分别建立 auth/game 两个连接池，跨库业务由应用层组合，不是在一个 SQL 事务中访问两库。`game-proxy`、`match-service` 和 `metrics-collector` 当前不直接连接 PostgreSQL。

### 对象清单

以下清单以根 `db/init.sql` 为主，并单独标注服务启动后才出现或与根 SQL 冲突的对象。主键、唯一约束、CHECK 和外键属于表定义，阶段 3 生成基线时仍必须逐项比较，不能因为本节聚焦表、索引、函数和触发器而忽略。

#### chat

根 SQL 在 `myserver_chat.public` 定义 3 张表和 6 个显式索引，无函数和触发器，全部由 `chat-server` 所有：

| 表 | 根 SQL 索引 | Rust 启动期索引 |
|----|-------------|-----------------|
| `chat_messages` | `idx_sender`、`idx_target`、`idx_group`、`idx_created` | `idx_chat_messages_sender`、`idx_chat_messages_target`、`idx_chat_messages_group`、`idx_chat_messages_created` |
| `chat_groups` | `idx_owner` | `idx_chat_groups_owner` |
| `chat_group_members` | `idx_player` | `idx_chat_group_members_player` |

两套索引的列相同但名称不同。根 SQL 允许 `chat_messages.target_id/group_id` 为 NULL，Rust 启动期新建表时使用 `NOT NULL DEFAULT ''`；群成员唯一约束名称也不同。当前数据库最终结构取决于先运行根初始化还是先启动服务，存量基线必须识别这两类指纹，不能静默收紧空值约束或删除重复索引。

#### announce

根 SQL 在 `myserver_announce.public` 定义 1 张表、4 个显式索引、1 个函数和 1 个触发器，全部由 `announce-service` 所有：

- 表：`announcements`。
- 根索引：`idx_announcements_locale`、`idx_announcements_world_id`、`idx_announcements_priority`、`idx_announcements_time`。
- 函数：`set_current_timestamp_updated_at()`。
- 触发器：`trg_announcements_updated_at`。

服务本地 SQL 和启动期 DDL 没有 `announcements.world_id` 与 `idx_announcements_world_id`，其余核心对象一致。只有服务本地 SQL 会写入 `announce_test_001/002` 演示公告，根 SQL 和启动期 DDL 都不会写演示数据。

#### mail

根 SQL 在 `myserver_mail.public` 定义 4 张表、13 个显式索引、1 个函数和 2 个触发器，全部由 `mail-service` 所有：

| 表 | 显式索引 |
|----|----------|
| `mails` | `idx_mails_to_player_id`、`idx_mails_world_id`、`idx_mails_status`、`idx_mails_created_at`、`uk_mails_delivery_request_id` |
| `mail_claim_workflows` | `idx_mail_claim_workflows_recovery`、`idx_mail_claim_workflows_player`、`uk_mail_claim_workflows_recovery_lease_token` |
| `mail_notification_outbox` | `idx_mail_notification_outbox_pending`、`idx_mail_notification_outbox_to_player_id`、`idx_mail_notification_outbox_retention` |
| `mail_admin_operation_audit` | `idx_mail_admin_operation_target`、`idx_mail_admin_operation_actor` |

`reject_mail_admin_operation_audit_mutation()`、`trg_mail_admin_operation_audit_immutable` 和 `trg_mail_admin_operation_audit_no_truncate` 共同保护管理审计不可修改，并额外执行 `REVOKE UPDATE, DELETE, TRUNCATE ... FROM PUBLIC`。

服务本地 SQL 和启动期 DDL 没有 `mails.world_id` 与 `idx_mails_world_id`。其余主要对象保持同步，但三份 SQL 仍是需要消除的重复来源。

## 工具方案比较

所有候选方案都必须由部署任务调用，Node/Rust 服务本身不在启动路径自动执行 migration。Node 和 Rust 的“均可调用”指二者所在部署环境都能通过统一 PowerShell/CLI 入口运行同一个数据库工具，而不是在两个语言栈分别嵌入迁移库。

| 方案 | 事务与并发锁 | checksum / baseline | 漂移能力 | Windows、运行时和许可 | 结论 |
|------|--------------|---------------------|----------|------------------------|------|
| SQLx CLI + 仓库锁包装 | PostgreSQL migration 默认在事务中执行；`-- no-transaction` 可显式用于事务外 DDL；SQLx CLI 0.8.6 的 `migrate run` 本身不调用 `Migrate::lock`，因此 `tools/db-lock-runner.js` 必须在 CLI child 外持有同一 PostgreSQL advisory lock 避免同库并发迁移 | `_sqlx_migrations` 记录版本、描述、成功状态、checksum、执行时间和耗时；可检测已执行文件被改写。没有安全的存量库 baseline 命令，必须由包装层在 catalog 指纹匹配后显式写入 history | `sqlx migrate info` 仅比较本地 migration 与 history；不等于实时 Schema 漂移，阶段 5 仍需仓库自有 catalog 快照比较 | 由现有 Rust 工具链通过 `cargo install sqlx-cli` 安装；需固定 `sqlx-cli` 与 `sqlx` 的兼容版本。迁移文件是普通 PostgreSQL SQL，Node `pg` 服务可直接消费 | **选用执行内核**。现有 Rust 服务已使用 SQLx，保留跨语言 SQL 单一事实来源，不引入 Prisma ORM 模型或 Java/第三方运行时 |
| Flyway Community CLI | PostgreSQL 版本化 migration 默认事务执行；支持显式非事务 migration；使用数据库锁并可配置锁重试 | `flyway_schema_history` 提供显式 `baseline`、`migrate`、`info`、`validate` | `validate` 检查历史/文件漂移，不等于实时 Schema 漂移 | 官方 Windows CLI；作为外部 Redgate 工具需固定版本并复核功能/许可边界 | 不选。执行与基线能力成熟，但会引入独立 CLI/运行时治理；当前 Rust + Node.js 服务均可共享 SQLx SQL migration，收益不足以覆盖新增工具链成本 |
| Liquibase Community CLI | changeset 默认事务执行，可用 `runInTransaction` 调整；`DATABASECHANGELOGLOCK` 负责并发互斥 | `DATABASECHANGELOG` 记录 changeset、author、MD5、执行顺序和部署 ID；`changelog-sync` 可建立基线 | 支持 diff 类命令，但准确目标、输出和部分高级检查仍需额外约束 | Windows CLI，Java 工具链；Community/商业能力边界和 changelog 格式需要持续治理 | 不选。功能足够，但 changeset/changelog/rollback 元数据明显重于当前按库 SQL 文件模型，执行耗时和实际执行身份还需补充 |
| Atlas CLI | SQL migration 可按文件事务执行，PostgreSQL 使用数据库锁；`atlas.sum` 保护目录完整性 | revision 表记录执行状态；支持 baseline；目录 checksum 使用 SHA-256 | Schema inspect/diff 是强项，适合目标态比较 | Windows 单文件 CLI；开源 CLI 与云/商业能力需固定版本并审查边界 | 不选。漂移能力有优势，但执行历史字段和离线 CI/商业边界需要额外适配；首期仍要自己定义稳定报告，不足以抵消换一套 HCL/目录语义的成本 |
| 仓库自研 Node + `pg` runner | 可实现单文件事务和 `pg_advisory_lock` | 可自定义 SHA-256 与 baseline | 全部自研 | Node 已是仓库基础运行时，无额外二进制和许可 | 不选作执行内核。提交 `360eeae` 删除的旧 runner 只有单库、单事务和基础 checksum，没有多库所有权、存量指纹、并发锁、失败状态、非事务 migration 和漂移治理；直接恢复会重新承担生产迁移器的正确性风险 |

### 选型结论

采用“SQLx CLI + 仓库薄包装层”方案：

1. SQLx CLI 只负责单数据库版本化 DDL 的排序、事务、history 和 checksum 检查；不作为服务运行时依赖，不由服务启动流程自动执行。SQLx CLI 0.8.6 的 `migrate run` 不调用其 `Migrate::lock` 实现，不能把直接调用 CLI 视为并发安全。
2. `tools/` 下的 Node 包装层负责五库选择、配置检查、稳定退出码、JSON 结果归一化、URL 脱敏、存量 Schema 指纹预检、显式 baseline history 写入、阶段 5 的漂移报告和 migration runner 互斥；不重新实现 SQL 执行器。
3. `up` 通过 `tools/db-lock-runner.js` 在独立 `pg` session 中获取 `pg_try_advisory_lock(0x3d32ad9e * CRC-32/ISO-HDLC(current_database()))`，持锁执行 SQLx child 后再释放。所有迁移应用必须走 `tools/db.js` 或 `tools/db-deploy.js`，直接运行 `bin/sqlx.exe migrate run` 会绕过该互斥层，属于不受支持的操作。
4. `scripts/db.ps1` 是 Windows 入口，转发到 `tools/db.js`。执行器只接受仓库根 `bin/sqlx.exe` 的受批准预编译制品：`db/config/sqlx-cli.json` 受版本控制地固定版本、目标平台、`artifactUrl`、SHA-256 和 `provisioned` 状态，调用前逐字节校验；不回退到 `PATH`，不信任环境变量提供的 hash，也不在部署现场执行 `cargo install`。
5. 版本升级必须重新确认 `sqlx-cli` 与各 Rust 服务 `sqlx` crate 的兼容范围，以及 PostgreSQL 支持范围。Node 服务继续使用 `pg` 消费已迁移 Schema，不引入 Prisma ORM 或第二份声明式 Schema。
6. SQLx CLI 没有可直接用于本仓库的存量库 baseline 流程。`db baseline` 只能在 catalog 指纹完全匹配受支持基线时，通过独立、显式、带审计的包装层命令写入 `_sqlx_migrations`；不得执行空库 DDL，也不得跳过指纹检查。
7. 已发布 migration 不可改写。checksum 不匹配必须失败；合法修正必须新增 migration，禁止通过手工修改 `_sqlx_migrations` 覆盖 history。

## 目录与版本规范

后续阶段按以下目标布局实施；目录尚未创建不表示可以使用其他临时结构绕过本规范。

```text
db/
├── bootstrap/                    # 创建数据库/角色的特权入口，不进入 SQLx history
├── config/
│   └── databases.json           # database key、目录、默认库名、逻辑 owner；不含凭据
├── migrations/
│   ├── auth/
│   ├── game/
│   ├── chat/
│   ├── announce/
│   └── mail/
├── schema/                       # catalog 查询、受审阅 target manifest 与精确环境差异策略
├── backfills/                    # 独立、可暂停/续跑的数据回填任务，不由 SQLx 执行
└── seeds/
    └── development/              # local world、测试账号、演示公告等可选开发数据
```

版本化文件使用 SQLx 格式：

```text
YYYYMMDDHHMMSS_lower_snake_case.sql
```

约束如下：

- 时间戳使用 UTC，精确到秒；在同一数据库目录内必须唯一并严格递增。发生同秒冲突时由后提交者顺延版本，禁止改写已合并或已部署文件。
- 文件名和 SQL migration 元数据使用 ASCII；文件首部注释至少记录逻辑 owner、兼容阶段（`expand`/`migrate`/`contract`）和不可逆风险。
- 每个文件只能连接并修改其所在目录对应的一个数据库，不得包含 `\connect`、`CREATE DATABASE`、凭据、测试数据或其他数据库的限定对象。
- 首个基线 migration 使用相同版本规则和 `initial_schema` 描述，不使用特殊可变文件名。
- 首期禁止 repeatable migration 管理表、函数或权限，避免重复执行掩盖已发布历史；变更继续使用新的版本化文件。
- 事务外 SQL 必须由文件元数据和包装层显式声明，默认拒绝；禁止通过 SQL 内自行 `COMMIT` 逃逸事务。

每个数据库使用自己的 `public._sqlx_migrations`，互不共享版本序列。SQLx history 记录已执行 migration 的版本、描述、成功状态、checksum、执行时间和耗时；数据库执行身份由包装层审计日志记录，migration 作者和审核者由 Git 历史追踪，不混为一个字段。

## 数据库身份与凭据

### 环境变量

迁移入口使用独立变量，不复用服务 `DATABASE_URL` 作为生产默认值：

| 用途 | URL | 用户 | 密码 |
|------|-----|------|------|
| 特权 bootstrap | `MYSERVER_DB_BOOTSTRAP_URL` | `MYSERVER_DB_BOOTSTRAP_USER` | `MYSERVER_DB_BOOTSTRAP_PASSWORD` |
| auth migration | `MYSERVER_DB_MIGRATION_AUTH_URL` | `MYSERVER_DB_MIGRATION_AUTH_USER` | `MYSERVER_DB_MIGRATION_AUTH_PASSWORD` |
| game migration | `MYSERVER_DB_MIGRATION_GAME_URL` | `MYSERVER_DB_MIGRATION_GAME_USER` | `MYSERVER_DB_MIGRATION_GAME_PASSWORD` |
| chat migration | `MYSERVER_DB_MIGRATION_CHAT_URL` | `MYSERVER_DB_MIGRATION_CHAT_USER` | `MYSERVER_DB_MIGRATION_CHAT_PASSWORD` |
| announce migration | `MYSERVER_DB_MIGRATION_ANNOUNCE_URL` | `MYSERVER_DB_MIGRATION_ANNOUNCE_USER` | `MYSERVER_DB_MIGRATION_ANNOUNCE_PASSWORD` |
| mail migration | `MYSERVER_DB_MIGRATION_MAIL_URL` | `MYSERVER_DB_MIGRATION_MAIL_USER` | `MYSERVER_DB_MIGRATION_MAIL_PASSWORD` |

URL 使用标准 PostgreSQL DSN；SQLx CLI 通过 `DATABASE_URL` 接收完整 DSN，因此包装层只通过子进程环境变量注入，绝不将其写入命令行、报告或日志。生产、预发和 CI 从 secret manager 或受保护环境变量注入；禁止写入仓库 `.env.example` 的真实值或配置文件。本地兼容应用 URL 必须由显式开关启用，不能静默回退。

目标权限模型：

- bootstrap 身份只用于创建数据库和角色，不执行普通 migration。
- 每个数据库建立 NOLOGIN owner role；migration 身份可切换到对应 owner 并执行 DDL。
- 应用运行身份只获得所需 Schema usage、DML、sequence 和函数权限，不拥有表，不执行 DDL。
- `auth-http`/`admin-api` 和 game 库的跨服务写权限按表授权；逻辑所有者仍是唯一 migration 审核方。
- migration 连接固定 `search_path=public,pg_catalog`，并在执行前验证当前数据库名，防止 URL 指错库。

### 脱敏边界

日志和 JSON 报告允许输出 database key、脱敏后的 host、port、数据库名、migration 版本、文件名、checksum、耗时和错误类别。以下内容不得输出：

- URL userinfo、密码、token、私钥、客户端证书内容和完整 secret 环境变量。
- 带业务参数的动态 SQL、数据修复 payload 和可能含玩家数据的查询结果。
- SQLx CLI 子进程完整环境变量快照或含完整 DSN 的诊断输出。

错误日志可以输出 SQLSTATE、migration 文件和不含参数的语句位置；若底层错误回显连接串，包装层必须先脱敏再落盘。

## 命令与退出码契约

统一入口最终至少暴露：

```text
db status   --database <auth|game|chat|announce|mail|all>
db up       --database <...>
db validate --database <...>
db baseline --database <...> --expected-fingerprint <sha256>
db drift    --database <auth|game|chat|announce|mail|all> --environment <name>
db backfill-status --database <key> --task <task-id>
db backfill-run|backfill-pause|backfill-resume --database <key> --task <task-id> --actor <identity>
```

`all` 按 `auth -> game -> chat -> announce -> mail` 顺序执行并逐库报告，不承诺跨数据库原子性；任一库失败立即停止后续库。稳定退出码由包装层定义，不能把外部 CLI 的单一非零码直接暴露给 CI：

| 退出码 | 含义 |
|--------|------|
| `0` | 成功，目标库全部达到期望状态 |
| `2` | 参数、database key 或本地配置错误 |
| `3` | 连接、TLS 或认证失败 |
| `4` | migration 命名、顺序、history 或 checksum 校验失败 |
| `5` | 并发锁获取超时或已有迁移执行中 |
| `6` | migration SQL 执行失败；数据库状态必须在报告中标注 |
| `7` | baseline 指纹或实时 Schema 漂移检查失败 |
| `8` | SQLx CLI 缺失、版本不匹配或当前版本能力不受支持 |

`status` 只读；`validate` 必须同时检查本地目录和数据库 history；`up` 默认先 validate；`baseline` 必须单库执行，不支持 `all`，也不允许跳过指纹检查。`drift` 可以按 `all` 顺序执行，但必须显式提供部署环境名；`backfill-*` 必须单库、单 task 执行，除 `backfill-status` 外均需 actor。

阶段 2 已提供 `node tools/db.js <status|up|validate> --database <key|all>` 和 `scripts/db.ps1`。所有命令先通过 `psql` 只读检查 `public._sqlx_migrations` 和已有 user table，因此 `status` 不会意外创建 history；没有 history 的 `status` 返回未初始化状态，`validate` 返回 `4`。调用 `up` 必须携带 `--actor` 或 `MYSERVER_DB_MIGRATION_ACTOR`；SQLx history 记录版本、描述、checksum 和执行耗时，包装层在同一数据库的 `public._myserver_migration_audit` 持久记录 actor、操作、开始/完成时间、结果和 history 摘要，审计写入失败时 `up` 必须失败。发现已有 user table 但 history 缺失时立即以 `7` 熔断，不执行 SQLx，也不会写入 history，因此存量库必须等待阶段 3 catalog 指纹 baseline；空库可由 SQLx 创建自己的 history。已有 history 时，`validate` 和 `up` 先逐行比对本地 source 的 version、description、SHA-384 checksum 与 success，再调用 SQLx `migrate info`；不能只依赖 CLI 的展示输出。`up` 最后通过受控 lock runner 执行 `migrate run`，该 runner 持有同一数据库的 advisory lock，锁不可用返回 `5` 而不启动 SQLx child。包装层只返回错误类别，不回显 SQLx/psql 原始错误或 SQL 参数。预编译制品必须在 `db/config/sqlx-cli.json` 中受版本控制地登记 `artifactUrl`、SHA-256 和 `provisioned: true`；当前 Windows x64 的 SQLx CLI 0.8.6 制品已登记并完成 hash、版本和临时 PostgreSQL 演练验证。制品缺失、版本不符或 hash 不符时命令返回 `8`，不接受环境变量提供的 hash，也不会从 PATH 下载、安装或回退。

阶段 3 已建立五个 `20260718161350_initial_schema.sql` 基线、development bootstrap 和独立 local seed，并提供 `db/schema/catalog-snapshot.sql` 的 canonical catalog 输入和 `baseline-allowlist.json` 的受版本控制 allowlist。五库临时 PostgreSQL 演练已验证：空库 migration 后的 catalog 与当前 `db/init.sql` 对应库段相同；受测试注入的审阅 allowlist 可以成功记录 baseline history，并由 SQLx 校验 checksum。allowlist 当前为 schema `2`：每个已审阅 fingerprint entry 必须同时包含 `sha256`、14 位 UTC `version` 和由已审阅 SQLx 文件名导出的 ASCII `description`。CLI 先校验该目标版本/描述存在于本地严格递增的 migration 目录且未超过目录最高版本，再读取实时 catalog、稳定排序为 UTF-8 JSON 并计算 SHA-256；两者完全匹配后，才在 SQLx 同一 PostgreSQL advisory lock 内创建 history 和 audit。baseline 只写入批准目标版本及之前的 history，并把 `target_version` 和 `target_description` 写入审计摘要，绝不把目录中后续 migration 标记为已执行。重复 baseline（已存在 `_sqlx_migrations`）安全拒绝。当前运行时 allowlist 没有任何 fingerprint，因此实际存量库、预发和生产环境的 CLI 一律拒绝 baseline；成功路径只可在测试通过依赖注入的审阅 allowlist 覆盖，不能通过环境变量或命令参数启用。不得提供 `--force`、未知指纹放行或自动修补入口。

SQLx 兼容契约来自本机 crates.io registry 的官方 `sqlx-core-0.8.6`（lock checksum `ee6798b1838b6a0f69c007c133b8df5866302197e404e8b6ee8ed3e3a5e68dc6`）和 `sqlx-postgres-0.8.6`：`src/migrate/migration.rs` 以 migration SQL UTF-8 bytes 的 SHA-384 作为 checksum；`src/migrate/source.rs` 将简单 `.sql` 文件去掉 `.sql` 并把 description 中 `_` 替换为空格；`src/migrate.rs` 定义 `_sqlx_migrations(version BIGINT PRIMARY KEY, description TEXT, installed_on TIMESTAMPTZ DEFAULT now(), success BOOLEAN, checksum BYTEA, execution_time BIGINT)`，写入时 `success=TRUE`、`execution_time=-1`。`sqlx-postgres` 的 `Migrate::lock` 会通过 `pg_advisory_lock(0x3d32ad9e * CRC-32/ISO-HDLC(current_database()))` 获取 session lock，但 SQLx CLI 0.8.6 的 `migrate::run` 手动 apply migration 而不调用该方法；因此包装层用相同公式在 lock runner 中补齐互斥。baseline 不执行 migration SQL，因此保留 SQLx 初始 `-1` execution time，而非虚构 DDL 耗时。

## 基线和漂移处理原则

阶段 3 不能机械复制根 SQL 后宣称完成。首个 baseline 要同时覆盖以下现实：

1. 根 `db/init.sql` 初始化后的标准对象。
2. 服务已经启动过、因此存在额外表、复合/重复索引或不同空值约束的开发库。
3. auth/game 两库各自可能存在 `metrics_archive` 的事实。
4. announce/mail 本地初始化缺少 `world_id`，chat 本地启动产生另一组索引的事实。

做法是先生成标准化 catalog 指纹，明确支持的存量变体。`db/schema/catalog-snapshot.sql` 返回 `object_kind`、`object_name`、`object_identity` 和 `definition`；baseline fingerprint 继续只使用前三字段语义中的 kind/name/definition，稳定排序并序列化为 UTF-8 JSON 后计算 SHA-256，避免 PostgreSQL OID 和输出顺序影响指纹。`object_identity` 只用于阶段 5 将同一对象的定义变化与对象缺失/额外对象区分开。五库临时 PostgreSQL 已生成并比对过标准空库 catalog，但这些演练指纹不是预发或生产环境的放行凭据。完全匹配某一受支持且人工审阅的 `sha256 + version + description` entry 才允许 baseline；任何未知表、列、约束、函数或触发器差异都返回退出码 `7`。基线只记录“该存量库已达到哪个已知版本”，不执行修补、不删除重复对象、不收紧约束，也不跳过 entry 目标之后的 migration。运行时 allowlist 目前为空，不能默认支持根 `init.sql` 或任何现有环境；测试 fixture 只在单元测试中通过依赖注入提供，不是可部署配置。

空库 baseline 以完整服务启动后所需的有效结构为目标，首期可以保留兼容但暂时无消费者的对象，避免借迁移体系建设偷偷改变业务语义。无用副本、重复索引和约束收紧必须在后续独立 migration 中按 expand/migrate/contract 处理。

## DDL 与业务数据边界

SQLx migration 负责：

- Schema、表、列、约束、索引、函数、触发器、sequence 和数据库权限的版本化变更。
- 为结构正确性必需、规模有严格上界且环境无关的元数据；此类数据必须幂等并在评审中明确说明。
- 小规模、确定性、能在 migration 事务和锁预算内完成的结构兼容填充，但默认仍应拆到独立回填任务。

SQLx migration 不负责：

- 玩家账号、角色、背包、邮件、公告、审计或事件的业务纠错。
- local world、测试账号、演示公告和测试夹具等环境数据。
- 大表全量 UPDATE、跨库搬运、远程服务调用、文件扫描或需要限速/暂停/重试的操作。
- 依赖当前服务业务逻辑才能解释的数据转换。

业务数据修复使用受审计的运维命令；大表异步回填放在 `db/backfills/<database>/<task>/`，使用独立状态表记录游标、批次、尝试、暂停和完成状态。回填可以由 deployment 编排在 expand 与 contract 之间，但成功状态不写入 `_sqlx_migrations`，也不能靠伪造 migration 版本表示完成。

## 阶段 5：数据回填与 Schema 漂移

阶段 5 的目标 Schema 不是从待检环境自动接受或覆盖。`db/schema/drift-policy.json` 为每个 database key 绑定 `db/schema/targets/<database>.json`、目标 SQLx migration 的版本、描述和 SHA-384 checksum。target manifest 是受版本控制的对象清单，包含 table、column、constraint、index、trigger 和 function 的稳定 identity 与 definition SHA-256；它必须由该版本 migration 在明确命名的空临时数据库中生成、审阅 diff 后随 migration 提交。新 migration 改变受管对象时，必须同步更新目标 manifest，不能用线上 catalog 或 `db drift` 输出直接重写它。

`node tools/db.js drift --database <key|all> --environment <name>` 只读读取 catalog，输出单行稳定 JSON。每库报告 target/actual catalog digest、对象数，以及 `target_missing`、`actual_extra`、`definition_change`、`allowed` 和 `unapproved` 列表。`manifest_sha256` 使用稳定有序的 `[object_kind, object_identity, definition_sha256]` 语义三元组；`object_name` 仅为报告标签，不参与该 digest，因此 version-controlled compact target 与干净实时 catalog 的 manifest digest 可直接比较。未允许漂移返回 `7`，连接失败返回 `3`，本地 policy 或 target 不完整/不匹配返回 `2`；CI 和部署脚本只消费这些稳定字段和退出码，不解析 PostgreSQL 原始错误文本。

允许环境差异只登记在 policy 的 `allowances` 中。每条记录必须精确绑定 database、单一 `scope.environment`、差异方向、object kind、object identity、对应的 expected/actual definition SHA-256 和非空审阅理由；不支持 `*`、范围数组、仅按名称放行或无 digest 放行。allowance 只可解释明确已审阅的当前差异，不能掩盖后续手工 DDL 或另一环境的变化。

大表回填的版本控制定义位于 `db/backfills/<database>/<task-id>/task.json` 与同目录 `batch.sql`。目录名、CLI `--task` 和 `task.json.id` 必须完全一致。task 固定 owner、目标 migration version、整数 cursor 初值、batch/max batch、每批最小延迟、statement timeout 和单条参数化 `WITH` SQL；SQL 使用 `$1` 接收上次 cursor、`$2` 接收 batch size，并返回 `next_cursor` 与 `processed_rows`。运行时在各业务库独立维护 `_myserver_backfill_state` 和 `_myserver_backfill_audit`：前者保存 task revision、cursor、状态、批次/行数和失败计数，后者记录 batch、pause、resume 与 failure。每批使用短事务和 task advisory lock 提交，延迟在提交后发生；pause 在下一批前生效，resume 从已提交 cursor 继续。已启动 task 的 definition digest、owner 或目标版本变化会失败关闭，必须建立新的审阅 task id。失败 batch 回滚并写入脱敏 failure audit；后续 `backfill-run` 对仍为 `failed` 的 state 必须返回非零执行失败和机器可读 state/reason，运维修复后显式 `resume` 才允许再次执行。该状态表和审计表被 catalog 查询排除，且永远不写入 `_sqlx_migrations`。

## 阶段 4：前向兼容、事务和恢复策略

生产变更默认遵循 `expand -> migrate -> contract`。一次发布只进入一个兼容阶段：`expand` 只增加旧服务可忽略的结构，`migrate` 由已发布的新服务完成双读写、回填或切换，`contract` 只在旧版本已退出且恢复条件满足后删除旧结构。不得把新字段、旧字段删除和数据重写塞进同一个 migration。

阶段 4 后新增的每个 migration 都必须在 SQL 文件首部使用下列 ASCII 元数据。现有五个 `20260718161350_initial_schema.sql` 是阶段 3 已登记的历史基线，保留原有三行元数据以避免改写 SQLx checksum；CLI 只在文件名、逻辑 owner 和已登记 SHA-384 内容同时匹配时将它们识别为 legacy 例外，并仍使用下文的默认超时预算。任何改写会失去该例外并被严格格式校验拒绝；新文件不适用该例外。

```sql
-- Logical owner: auth-http
-- Compatibility phase: expand
-- Irreversible risk: none
-- Transaction: required
-- Lock timeout: 5s
-- Statement timeout: 60s
-- Backup point: not-required
-- Recovery command: SQLx rolls back the transaction; correct the migration and rerun db up.
```

`tools/db.js validate` 和 `db up` 会在调用 SQLx 前校验：逻辑 owner、阶段、风险分类、事务模式、超时、备份点和恢复命令都必须存在；timeout 只能为正的 `ms`、`s` 或 `min`，且不超过受版本控制的 `db/config/migration-safety.json` 上限。`Irreversible risk` 只能是 `none`、`data-loss`、`data-rewrite` 或 `external-state`。非 `none` 的 migration 必须额外提供命名 `Backup point`、非空 `Risk summary` 和可执行的 `Recovery command`；可逆 migration 的 `Backup point` 必须写 `not-required`。`expand` 还会被静态拒绝删列、改类型、改名和其他明显破坏性 DDL，避免用阶段标签掩盖不兼容变更。SQL 文件不得自行执行 `BEGIN`、`COMMIT` 或 `ROLLBACK`。

变更模板如下；其中表名、列名、备份标识和恢复命令必须替换为实际经评审的值：

| 变更 | expand | migrate | contract / 恢复边界 |
|------|--------|---------|----------------------|
| 增列 | 先新增可空或带兼容默认值的列，不在同一发布设为 `NOT NULL` | 新旧服务双读写；大表回填进入 `db/backfills/` | 验证新列完整性后再加约束；删除旧列另开不可逆 contract |
| 改名 | 以“新增目标列”代替直接 `RENAME COLUMN` | 新版本回退时仍写旧列，完成双写和校验 | 所有旧服务退出后删除旧列；必须备份并声明 data-loss 风险 |
| 删列 | 不做删除 | 先删除应用读取和写入路径，确认 rollout/readiness | 单独 `contract` 文件，备份点、恢复命令和风险说明必填 |
| 改类型 | 新增目标类型列，保留旧列 | 分批回填和双写，校验转换错误率 | 删除旧列或收紧类型属于不可逆 contract；禁止在长事务中直接全表 rewrite |
| 索引 | 小索引可事务内 `CREATE INDEX` | 无需数据切换 | 大索引使用 `CREATE INDEX CONCURRENTLY` 的显式事务外模板；失败后检查无效索引并按恢复命令清理 |
| 约束 | 先增加兼容约束；外键/CHECK 优先 `NOT VALID` | 应用修复和回填不合规数据，分批验证 | `VALIDATE CONSTRAINT` 后才收紧 `NOT NULL` 或删除兼容约束；删除/收紧需要 contract 审查 |

事务外操作只允许 PostgreSQL 确实不支持事务的并发索引类动作。文件的**第一个字节**必须是 SQLx 0.8.6 的精确指令 `-- no-transaction`，随后填写 `Transaction: no-transaction` 和受 allowlist 限制的 `Non-transaction reason`。CLI 会拒绝两者不一致、缺失、理由与 SQL 操作不匹配，或把该字段写在事务内 migration 的文件。模板见 `db/migrations/templates/`。

```sql
-- no-transaction
-- Logical owner: auth-http
-- Compatibility phase: expand
-- Irreversible risk: none
-- Transaction: no-transaction
-- Non-transaction reason: create-index-concurrently
-- Lock timeout: 5s
-- Statement timeout: 5min
-- Backup point: not-required
-- Recovery command: Inspect pg_index.indisvalid; drop the invalid index concurrently, then rerun db up.
CREATE INDEX CONCURRENTLY idx_example_new_column ON example_table (new_column);
```

`db/config/migration-safety.json` 将默认预算固定为 `lock_timeout=5s`、`statement_timeout=60s`，最大值为 `15s` 和 `5min`。SQLx 一次 `migrate run` 的连接只能接收一组 `PGOPTIONS`，因此 CLI 对同一 source 目录、本次 SQLx 调用参与校验的 migration 取各自已批准预算的**最大值**，再硬性夹紧到全局上限；例如 `60s` migration 与声明 `5min` 的并发索引同批运行时，二者都使用 `5min`。这保证长迁移的审核预算会实际生效，同时意味着同批较短 migration 不会得到更低的逐文件超时保证；需要更短隔离预算时必须拆分部署批次。连接 URL 中的 `options` 或 `options[...]` 被拒绝，不能借 DSN 覆盖锁或语句超时。SQLx 仍是唯一 DDL 执行器，不在包装层实现第二个 migration runner。

失败恢复遵循以下固定规则：事务内 migration 失败时 SQLx 回滚该文件的事务，停止后续数据库和服务部署，修正后以原版本文件重新执行；已经发布的文件不得改写，发现 checksum 不匹配按退出码 `4` 处理。事务外 migration 失败时不得自动重试或假定回滚，先按文件 `Recovery command` 检查对象状态（并发建索引重点检查 `pg_index.indisvalid`）、清理半成品或从命名备份点恢复，再重新执行。锁等待超时返回退出码 `5`，语句超时或其他 DDL 失败返回退出码 `6`；二者都必须保留 SQLx history 和审计记录作为人工恢复输入，而不是继续下一个数据库。

不可逆 contract 的部署前记录必须包含：已验证的备份产物标识和校验、恢复命令及其演练结果、仍在运行的服务版本范围、旧字段/旧索引消费者已清零的证据和明确的风险摘要。恢复命令只能引用受保护的凭据环境或 secret manager，不能把 DSN、密码或业务数据写进 migration 文件、CLI 输出或 Git。

滚动兼容样例位于 `tests/fixtures/db/stage4-rollout/`：它先以旧形态创建表，再通过 SQLx 执行 additive expand，模拟旧服务继续读写旧列以及新服务使用新增列；随后演练 contract 的备份和恢复路径。该演练只允许名称以 `myserver_stage4_` 开头的本机临时数据库，结束时无条件删除，不会写入 `myserver_*` 业务库。

## 阶段 6：部署与 CI 准入

阶段 6 的正式说明见[数据库部署准入说明](./数据库部署准入说明.md)。根 `tools/db-deploy.js` 将五库部署固定为 `preflight -> apply -> postflight` 的串行 gate：preflight 检查 pending/history/checksum、SQLx advisory lock、不可逆 backup evidence 和版本化静态 service compatibility；postflight 检查已执行版本、target catalog drift、关键对象和操作者显式配置的 readiness endpoint。任一阶段失败以既有稳定退出码停止后续数据库与服务步骤，并输出人工恢复路径，不自动回滚、drop 业务库或启动服务。

`db/config/deploy-gate.json` 中的 compatibility 是“本次目标 migration 被哪些服务版本范围声明支持”的静态事实，报告必须标记 `runtimeObserved: false`。实际服务 readiness 只在 postflight 的 `--check-readiness` 下以 HTTP GET 探测，未配置或未探测不能视为健康。现存服务启动期 DDL 的清理仍是服务侧治理工作；部署 gate 不得以数据库 schema 检查替代它。

CI 的常规 Windows workflow 只运行不连数据库的静态检查和 Node test，并同时监听 `tests/fixtures/db/**`，因此 Stage 4/7 fixture 的 SQL 或安全头修改也会触发 gate。真实空库 rebuild 使用独立 localhost:5432 `postgres` bootstrap URL、`MYSERVER_DB_DEPLOY_TEMPORARY_REBUILD=1`、命令内确认值和 `myserver_stage6_` 前缀 guard；五库临时库在 finally 中清理，任何清理失败保持非零并报告名称。它绝不使用或删除五个 `myserver_*` 业务库。

## 阶段 7：验证与演练

`npm run db:ci` 默认只运行 Stage 7 的 fixture、guard 和退出码静态测试，不连接 PostgreSQL。真实演练必须由操作者显式注入本机受保护凭据，并同时设置 `MYSERVER_STAGE7_RUN_POSTGRES=1`：

```powershell
$env:MYSERVER_STAGE7_RUN_POSTGRES = "1"
$env:MYSERVER_STAGE7_POSTGRES_URL = "postgresql://postgres@localhost:5432/postgres"
$env:MYSERVER_STAGE7_POSTGRES_PASSWORD = "<secret-manager-injected>"
npm run db:stage7:drill
```

该入口只接受 `localhost`、`127.0.0.1` 或 `::1` 的 `5432/postgres` bootstrap URL，拒绝 PostgreSQL `options` 覆盖，并只会创建随机 token 加后缀的 `myserver_stage7_` 临时数据库。每个名字只有本次 `CREATE DATABASE` 成功后才会登记 cleanup；无论成功或失败，finally 都会先终止该临时库连接，再删除本次创建的库。它绝不读取、写入或删除 `myserver_auth`、`myserver_game`、`myserver_chat`、`myserver_announce`、`myserver_mail` 等业务库，也不启动服务或探测 readiness。

真实演练覆盖五库空库到当前 history、重复 `up`、SQLx advisory lock holder 对独立 runner 的互斥及释放后重试、checksum source 不一致、事务内 SQL 失败、由受控 backend terminate 触发的连接中断，以及 Stage 4 fixture 的 expand 兼容、contract 删除和备份表恢复。它还会先创建 Stage 7 base table，再用独立连接持有该表的 `ACCESS EXCLUSIVE` DDL lock，使后续 `ALTER TABLE` 在 migration 文件的 `Lock timeout: 500ms` 内实际超时；演练确认被阻塞列和后续 sentinel 都未创建，释放锁后同一 migration 可成功重试。失败场景必须分别返回既有 `4`、`5`、`6`、`3` 退出码，并确认后续 fixture migration 未执行。

迁移 CLI 成功 `up` 的单行 JSON audit 现在包含 `migrationVersions` 与 `targetMigrationVersion`；数据库 `_myserver_migration_audit.history_summary` 还包含稳定的 `versions=<version,...>`，可与 drift JSON 的 `target.migration.version` 交叉定位具体迁移版本。`up` 同时具备实际 Core NATS metrics producer：设置 `MYSERVER_DB_MIGRATION_METRICS_ENABLED=1` 后，`tools/db-migration-metrics.js` 会向 `myserver.metrics.db-migration.<base64url-instance>` 发布现有 `metrics-collector` 已消费的 `{service, instance_id, bucket, timestamp, metrics}` 信封；`MYSERVER_DB_MIGRATION_METRICS_NATS_URL` 优先于 `NATS_URL`，`MYSERVER_DB_MIGRATION_METRICS_TIMEOUT_MS` 默认 1000ms、范围 100-5000ms。每个事件以 `migration-<database>-<target-version>` 作为 instance，metrics 字段固定包含 `database_key`、`target_migration_version`、`applied_migration_versions`、`attempted_migration_versions`、`outcome` 和安全的 `error_category`，不携带 DSN、密码、原始 SQL 或数据库错误文本。emitter child 从显式 allowlist 构造环境，只保留 Windows/Node 必需运行变量、上述 metrics 配置、事件 JSON 和 `NATS_URL` fallback；`NATS_URL` 是唯一允许的通用 transport URL，绝不继承 `REDIS_URL`、数据库/服务 URL、PG*、Stage7/deploy 或 token/secret/password 变量。

metrics producer 位于 migration/audit 结果之后，使用受时限的 best-effort publish。NATS 不可用、超时或 adapter 无法启动时，CLI 只在安全 JSON 中写 `metrics.state=unavailable`，不修改既有迁移退出码、不回滚已完成 migration、也不伪造“已采集”状态；恢复后可由 `_myserver_migration_audit`、SQLx history 和 CLI audit 重新定位版本。`metrics-collector` 会把成功送达的事件写入 `metrics:db-migration:<instance>:<bucket>` Redis hash；`tests/db/stage7-verification.test.mjs` 以 fake NATS 和真实 collector `writeMetrics` 验证该 adapter，无需启动 NATS、Redis 或业务服务。默认未启用时状态为 `disabled`，Stage 7 drill 会如实报告 `runtimeMetrics.delivery`，不会把 stdout JSON、静态兼容声明或 readiness 伪装成指标采集。

## 后续阶段约束

- 阶段 2 先固定 Rust toolchain、`sqlx-cli` 版本与安装方式，再实现薄包装层；不得复制旧 `tools/db-migrate.js` 作为完成品。
- 阶段 3 分离 `CREATE DATABASE` bootstrap、五库 baseline、开发 seed，并对上述差异形成机器可读指纹。
- 阶段 4 默认采用 expand/migrate/contract；不可逆和事务外 migration 必须有备份点、恢复命令、超时和锁预算。
- 阶段 5 的 Schema drift 比较必须覆盖表、列、索引、约束、函数和触发器；SQLx `migrate info` 不能替代该检查。
- 阶段 6 在所有服务移除启动期 DDL 前，必须先提供版本/readiness 检查和滚动兼容验证。
- 阶段 7 的真实 PostgreSQL 演练只对明确命名的临时数据库执行，继续遵守用户确认和清理保护。
