# 数据库迁移体系设计

## 文档定位

本文是 MyServer PostgreSQL 迁移体系的当前设计基线，记录 2026-07-18 的 Schema 来源盘点、对象归属、工具选型和后续实现约束。具体实现和命令以仓库代码为准；在迁移骨架和基线迁移完成前，现有初始化命令仍按[数据库初始化说明](./数据库初始化说明.md)执行。

本设计只治理数据库结构版本、基线、部署准入和漂移检测，不调整游戏业务模型，也不把业务数据修复或大表异步回填塞进 DDL migration。

## 当前事实

### Schema 来源

当前数据库结构同时来自以下入口，并非真正的单一来源：

| 来源 | 作用 | 当前问题 |
|------|------|----------|
| `db/init.sql` | 使用 `psql` 创建五个数据库并初始化目标结构 | 同时包含数据库 bootstrap、五库 DDL 和 local origin/world 基础数据；依赖 `\connect`，不能作为单库 migration 直接执行 |
| `apps/announce-service/db/init.sql` | 公告服务独立初始化 | 与根 SQL 的 `world_id` 列和索引不一致，并包含演示公告 |
| `apps/mail-service/db/init.sql` | 邮件服务独立初始化 | 与根 SQL 的 `world_id` 列和索引不一致 |
| 六个服务的启动期 DDL | 服务连接后执行 `CREATE/ALTER/INDEX/FUNCTION/TRIGGER` | 无版本、checksum 和全局锁；不同启动路径会生成不同结构 |
| `scripts/reset-dev-data.ps1` | 删除并重建五个本地开发库 | 直接调用根 `db/init.sql`，只适用于可删除的本地数据 |
| 服务初始化、种子和测试脚本 | 初始化管理员、origin/world、测试账号或测试数据库 | Schema、基础数据和测试数据边界未完全分离 |

根 SQL 和服务 SQL 都没有显式 `CREATE SCHEMA` 或 `SET search_path`，所有未限定对象当前都落在连接角色默认 `search_path` 下，通常是 `public`。根 SQL 也没有稳定设置数据库或对象 owner；物理 owner 是实际执行 `CREATE DATABASE`/DDL 的登录角色，本地示例通常为 `postgres`。后续迁移必须固定 `public` 为首期受管 Schema，并显式区分迁移凭据和应用运行凭据。

### 数据库与连接拓扑

| 数据库 key | 默认数据库 | 受管 Schema | 逻辑所有者 | 运行时连接和消费者 |
|------------|------------|-------------|------------|--------------------|
| `auth` | `myserver_auth` | `public` | `auth-http` 负责账号和登录审计；`admin-api` 负责后台、MyForge、全局 ID/世界元数据和监控归档 | `auth-http:DATABASE_URL`；`admin-api:DATABASE_URL`；二者都存在写操作 |
| `game` | `myserver_game` | `public` | `game-server` 负责游戏运行时、角色和资产对象；`admin-api` 对 GM 对象有受控写权限 | `game-server:DATABASE_URL`；`auth-http:GAME_DATABASE_URL/AUTH_GAME_DATABASE_URL`；`admin-api:GAME_DATABASE_URL/ADMIN_GAME_DATABASE_URL` |
| `chat` | `myserver_chat` | `public` | `chat-server` | `chat-server:DATABASE_URL` |
| `announce` | `myserver_announce` | `public` | `announce-service` | `announce-service:DATABASE_URL` |
| `mail` | `myserver_mail` | `public` | `mail-service` | `mail-service:DATABASE_URL`；邮件发奖通过 game-server 管理协议，不直接连接游戏库 |

这里的“逻辑所有者”决定谁可以新增或修改该组对象。PostgreSQL 没有跨数据库 join：`auth-http` 和 `admin-api` 分别建立 auth/game 两个连接池，跨库业务由应用层组合，不是在一个 SQL 事务中访问两库。`game-proxy`、`match-service` 和 `metrics-collector` 当前不直接连接 PostgreSQL。

### 对象清单

以下清单以根 `db/init.sql` 为主，并单独标注服务启动后才出现或与根 SQL 冲突的对象。主键、唯一约束、CHECK 和外键属于表定义，阶段 3 生成基线时仍必须逐项比较，不能因为本节聚焦表、索引、函数和触发器而忽略。

#### chat

根 SQL 在 `myserver_chat.public` 定义 3 张表和 6 个显式索引，无函数和触发器，全部由 `chat-server` 所有：

| 表 | 根 SQL 索引 | Rust 启动期索引 |
|----|-------------|-----------------|
| `chat_messages` | `idx_sender`、`idx_target`、`idx_group`、`idx_created` | `idx_chat_messages_sender`、`idx_chat_messages_target`、`idx_chat_messages_group`、`idx_chat_messages_created` |
| `chat_groups` | `idx_owner` | `idx_chat_groups_owner` |
| `chat_group_members` | `idx_player` | `idx_chat_group_members_player` |

两套索引的列相同但名称不同。根 SQL 允许 `chat_messages.target_id/group_id` 为 NULL，Rust 启动期新建表时使用 `NOT NULL DEFAULT ''`；群成员唯一约束名称也不同。当前数据库最终结构取决于先运行根初始化还是先启动服务，存量基线必须识别这两类指纹，不能静默收紧空值约束或删除重复索引。

#### announce

根 SQL 在 `myserver_announce.public` 定义 1 张表、4 个显式索引、1 个函数和 1 个触发器，全部由 `announce-service` 所有：

- 表：`announcements`。
- 根索引：`idx_announcements_locale`、`idx_announcements_world_id`、`idx_announcements_priority`、`idx_announcements_time`。
- 函数：`set_current_timestamp_updated_at()`。
- 触发器：`trg_announcements_updated_at`。

服务本地 SQL 和启动期 DDL 没有 `announcements.world_id` 与 `idx_announcements_world_id`，其余核心对象一致。只有服务本地 SQL 会写入 `announce_test_001/002` 演示公告，根 SQL 和启动期 DDL 都不会写演示数据。

#### mail

根 SQL 在 `myserver_mail.public` 定义 4 张表、13 个显式索引、1 个函数和 2 个触发器，全部由 `mail-service` 所有：

| 表 | 显式索引 |
|----|----------|
| `mails` | `idx_mails_to_player_id`、`idx_mails_world_id`、`idx_mails_status`、`idx_mails_created_at`、`uk_mails_delivery_request_id` |
| `mail_claim_workflows` | `idx_mail_claim_workflows_recovery`、`idx_mail_claim_workflows_player`、`uk_mail_claim_workflows_recovery_lease_token` |
| `mail_notification_outbox` | `idx_mail_notification_outbox_pending`、`idx_mail_notification_outbox_to_player_id`、`idx_mail_notification_outbox_retention` |
| `mail_admin_operation_audit` | `idx_mail_admin_operation_target`、`idx_mail_admin_operation_actor` |

`reject_mail_admin_operation_audit_mutation()`、`trg_mail_admin_operation_audit_immutable` 和 `trg_mail_admin_operation_audit_no_truncate` 共同保护管理审计不可修改，并额外执行 `REVOKE UPDATE, DELETE, TRUNCATE ... FROM PUBLIC`。

服务本地 SQL 和启动期 DDL 没有 `mails.world_id` 与 `idx_mails_world_id`。其余主要对象保持同步，但三份 SQL 仍是需要消除的重复来源。

## 工具方案比较

所有候选方案都必须由部署任务调用，Node/Rust 服务本身不在启动路径自动执行 migration。Node 和 Rust 的“均可调用”指二者所在部署环境都能通过统一 PowerShell/CLI 入口运行同一个数据库工具，而不是在两个语言栈分别嵌入迁移库。

| 方案 | 事务与并发锁 | checksum / baseline | 漂移能力 | Windows、运行时和许可 | 结论 |
|------|--------------|---------------------|----------|------------------------|------|
| Flyway Community CLI | PostgreSQL 版本化 migration 默认事务执行；支持显式非事务 migration；使用数据库锁并可配置锁重试 | `flyway_schema_history` 记录版本、描述、脚本、checksum、执行者、时间、耗时和成功状态；有显式 `baseline`、`migrate`、`info`、`validate` | `validate` 检查历史/文件漂移，不等于实时 Schema 漂移；阶段 5 仍需仓库自有 catalog 快照比较 | 官方有 Windows CLI；作为外部 Redgate 工具必须固定版本、校验下载物并在升级时复核 Community 功能和许可；包装层不假设系统 Java | **选用执行内核**。历史字段与本清单最匹配，生产失败/锁/baseline 语义成熟 |
| Liquibase Community CLI | changeset 默认事务执行，可用 `runInTransaction` 调整；`DATABASECHANGELOGLOCK` 负责并发互斥 | `DATABASECHANGELOG` 记录 changeset、author、MD5、执行顺序和部署 ID；`changelog-sync` 可建立基线 | 支持 diff 类命令，但准确目标、输出和部分高级检查仍需额外约束 | Windows CLI，Java 工具链；Community/商业能力边界和 changelog 格式需要持续治理 | 不选。功能足够，但 changeset/changelog/rollback 元数据明显重于当前按库 SQL 文件模型，执行耗时和实际执行身份还需补充 |
| Atlas CLI | SQL migration 可按文件事务执行，PostgreSQL 使用数据库锁；`atlas.sum` 保护目录完整性 | revision 表记录执行状态；支持 baseline；目录 checksum 使用 SHA-256 | Schema inspect/diff 是强项，适合目标态比较 | Windows 单文件 CLI；开源 CLI 与云/商业能力需固定版本并审查边界 | 不选。漂移能力有优势，但执行历史字段和离线 CI/商业边界需要额外适配；首期仍要自己定义稳定报告，不足以抵消换一套 HCL/目录语义的成本 |
| 仓库自研 Node + `pg` runner | 可实现单文件事务和 `pg_advisory_lock` | 可自定义 SHA-256 与 baseline | 全部自研 | Node 已是仓库基础运行时，无额外二进制和许可 | 不选作执行内核。提交 `360eeae` 删除的旧 runner 只有单库、单事务和基础 checksum，没有多库所有权、存量指纹、并发锁、失败状态、非事务 migration 和漂移治理；直接恢复会重新承担生产迁移器的正确性风险 |

### 选型结论

采用“Flyway Community CLI + 仓库薄包装层”方案：

1. Flyway 只负责单数据库版本化 DDL 的排序、事务、锁、历史、checksum、baseline 和 validate。
2. `tools/` 下的 Node 包装层负责五库选择、配置检查、稳定退出码、JSON 结果归一化、URL 脱敏、存量 Schema 指纹预检和阶段 5 的漂移报告；不重新实现 SQL 执行器。
3. `scripts/` 下的 PowerShell 入口负责 Windows 友好调用，优先寻找仓库根 `bin/` 中固定版本的 Flyway，再回退到 `PATH`。`bin/` 当前被 Git 忽略，因此仓库必须提供固定版本/哈希清单和安装脚本或 CI 安装步骤，不能提交本机二进制。
4. 版本升级必须重新确认 PostgreSQL 支持范围、Community 功能边界和许可条款。不能因为开发机当前安装了 Java 就把系统 Java 作为隐含部署前提。
5. 不启用 `baselineOnMigrate`。存量库只能在指纹完全匹配受支持基线时，通过独立、显式、带审计的 baseline 命令标记。
6. 不使用 Flyway `repair` 自动覆盖已部署 checksum。已发布 migration 不可改写；合法修正必须新增 migration。

## 目录与版本规范

后续阶段按以下目标布局实施；目录尚未创建不表示可以使用其他临时结构绕过本规范。

```text
db/
├── bootstrap/                    # 创建数据库/角色的特权入口，不进入 Flyway history
├── config/
│   └── databases.json           # database key、目录、默认库名、逻辑 owner；不含凭据
├── migrations/
│   ├── auth/
│   ├── game/
│   ├── chat/
│   ├── announce/
│   └── mail/
├── schema/                       # 标准化 catalog 快照及允许的环境差异
├── backfills/                    # 独立、可暂停/续跑的数据回填任务，不由 Flyway 执行
└── seeds/
    └── development/              # local world、测试账号、演示公告等可选开发数据
```

版本化文件使用 Flyway 格式：

```text
VYYYYMMDDHHMMSS__lower_snake_case.sql
```

约束如下：

- 时间戳使用 UTC，精确到秒；在同一数据库目录内必须唯一并严格递增。发生同秒冲突时由后提交者顺延版本，禁止改写已合并或已部署文件。
- 文件名和 SQL migration 元数据使用 ASCII；文件首部注释至少记录逻辑 owner、兼容阶段（`expand`/`migrate`/`contract`）和不可逆风险。
- 每个文件只能连接并修改其所在目录对应的一个数据库，不得包含 `\connect`、`CREATE DATABASE`、凭据、测试数据或其他数据库的限定对象。
- 首个基线 migration 使用相同版本规则和 `__baseline` 描述，不使用特殊可变文件名。
- 首期禁止 repeatable migration 管理表、函数或权限，避免重复执行掩盖已发布历史；变更继续使用新的版本化文件。
- 事务外 SQL 必须由文件元数据和包装层显式声明，默认拒绝；禁止通过 SQL 内自行 `COMMIT` 逃逸事务。

每个数据库使用自己的 `public.flyway_schema_history`，互不共享版本序列。Flyway history 中的 `installed_by` 是数据库执行身份；migration 作者和审核者由 Git 历史追踪，不混为一个字段。

## 数据库身份与凭据

### 环境变量

迁移入口使用独立变量，不复用服务 `DATABASE_URL` 作为生产默认值：

| 用途 | URL | 用户 | 密码 |
|------|-----|------|------|
| 特权 bootstrap | `MYSERVER_DB_BOOTSTRAP_URL` | `MYSERVER_DB_BOOTSTRAP_USER` | `MYSERVER_DB_BOOTSTRAP_PASSWORD` |
| auth migration | `MYSERVER_DB_MIGRATION_AUTH_URL` | `MYSERVER_DB_MIGRATION_AUTH_USER` | `MYSERVER_DB_MIGRATION_AUTH_PASSWORD` |
| game migration | `MYSERVER_DB_MIGRATION_GAME_URL` | `MYSERVER_DB_MIGRATION_GAME_USER` | `MYSERVER_DB_MIGRATION_GAME_PASSWORD` |
| chat migration | `MYSERVER_DB_MIGRATION_CHAT_URL` | `MYSERVER_DB_MIGRATION_CHAT_USER` | `MYSERVER_DB_MIGRATION_CHAT_PASSWORD` |
| announce migration | `MYSERVER_DB_MIGRATION_ANNOUNCE_URL` | `MYSERVER_DB_MIGRATION_ANNOUNCE_USER` | `MYSERVER_DB_MIGRATION_ANNOUNCE_PASSWORD` |
| mail migration | `MYSERVER_DB_MIGRATION_MAIL_URL` | `MYSERVER_DB_MIGRATION_MAIL_USER` | `MYSERVER_DB_MIGRATION_MAIL_PASSWORD` |

URL 使用不含 userinfo 的标准 PostgreSQL DSN，例如 `postgresql://db.example:5432/myserver_auth?sslmode=require`；包装层负责转换为 Flyway JDBC URL。用户名/密码单独注入。生产、预发和 CI 从 secret manager 或受保护环境变量注入；禁止写入仓库 `.env.example` 的真实值、Flyway 配置文件、命令行参数、报告或日志。本地兼容应用 URL 必须由显式开关启用，不能静默回退。

目标权限模型：

- bootstrap 身份只用于创建数据库和角色，不执行普通 migration。
- 每个数据库建立 NOLOGIN owner role；migration 身份可切换到对应 owner 并执行 DDL。
- 应用运行身份只获得所需 Schema usage、DML、sequence 和函数权限，不拥有表，不执行 DDL。
- `auth-http`/`admin-api` 和 game 库的跨服务写权限按表授权；逻辑所有者仍是唯一 migration 审核方。
- migration 连接固定 `search_path=public,pg_catalog`，并在执行前验证当前数据库名，防止 URL 指错库。

### 脱敏边界

日志和 JSON 报告允许输出 database key、脱敏后的 host、port、数据库名、migration 版本、文件名、checksum、耗时和错误类别。以下内容不得输出：

- URL userinfo、密码、token、私钥、客户端证书内容和完整 secret 环境变量。
- 带业务参数的动态 SQL、数据修复 payload 和可能含玩家数据的查询结果。
- Flyway 子进程完整命令行或完整环境变量快照。

错误日志可以输出 SQLSTATE、migration 文件和不含参数的语句位置；若底层错误回显连接串，包装层必须先脱敏再落盘。

## 命令与退出码契约

统一入口最终至少暴露：

```text
db status   --database <auth|game|chat|announce|mail|all>
db up       --database <...>
db validate --database <...>
db baseline --database <...> --expected-fingerprint <sha256>
```

`all` 按 `auth -> game -> chat -> announce -> mail` 顺序执行并逐库报告，不承诺跨数据库原子性；任一库失败立即停止后续库。稳定退出码由包装层定义，不能把外部 CLI 的单一非零码直接暴露给 CI：

| 退出码 | 含义 |
|--------|------|
| `0` | 成功，目标库全部达到期望状态 |
| `2` | 参数、database key 或本地配置错误 |
| `3` | 连接、TLS 或认证失败 |
| `4` | migration 命名、顺序、history 或 checksum 校验失败 |
| `5` | 并发锁获取超时或已有迁移执行中 |
| `6` | migration SQL 执行失败；数据库状态必须在报告中标注 |
| `7` | baseline 指纹或实时 Schema 漂移检查失败 |
| `8` | Flyway 缺失、版本不匹配或当前版本/许可能力不受支持 |

`status` 只读；`validate` 必须同时检查本地目录和数据库 history；`up` 默认先 validate；`baseline` 必须单库执行，不支持 `all`，也不允许跳过指纹检查。

## 基线和漂移处理原则

阶段 3 不能机械复制根 SQL 后宣称完成。首个 baseline 要同时覆盖以下现实：

1. 根 `db/init.sql` 初始化后的标准对象。
2. 服务已经启动过、因此存在额外表、复合/重复索引或不同空值约束的开发库。
3. auth/game 两库各自可能存在 `metrics_archive` 的事实。
4. announce/mail 本地初始化缺少 `world_id`，chat 本地启动产生另一组索引的事实。

做法是先生成标准化 catalog 指纹，明确支持的存量变体。完全匹配某一受支持指纹才允许 baseline；任何未知表、列、约束、函数或触发器差异都返回退出码 `7`。基线只记录“该存量库已达到哪个已知版本”，不执行修补、不删除重复对象、不收紧约束。

空库 baseline 以完整服务启动后所需的有效结构为目标，首期可以保留兼容但暂时无消费者的对象，避免借迁移体系建设偷偷改变业务语义。无用副本、重复索引和约束收紧必须在后续独立 migration 中按 expand/migrate/contract 处理。

## DDL 与业务数据边界

Flyway migration 负责：

- Schema、表、列、约束、索引、函数、触发器、sequence 和数据库权限的版本化变更。
- 为结构正确性必需、规模有严格上界且环境无关的元数据；此类数据必须幂等并在评审中明确说明。
- 小规模、确定性、能在 migration 事务和锁预算内完成的结构兼容填充，但默认仍应拆到独立回填任务。

Flyway migration 不负责：

- 玩家账号、角色、背包、邮件、公告、审计或事件的业务纠错。
- local world、测试账号、演示公告和测试夹具等环境数据。
- 大表全量 UPDATE、跨库搬运、远程服务调用、文件扫描或需要限速/暂停/重试的操作。
- 依赖当前服务业务逻辑才能解释的数据转换。

业务数据修复使用受审计的运维命令；大表异步回填放在 `db/backfills/<database>/<task>/`，使用独立状态表记录游标、批次、尝试、暂停和完成状态。回填可以由 deployment 编排在 expand 与 contract 之间，但成功状态不写入 `flyway_schema_history`，也不能靠伪造 migration 版本表示完成。

## 后续阶段约束

- 阶段 2 先固定 Flyway 版本/哈希与安装方式，再实现薄包装层；不得复制旧 `tools/db-migrate.js` 作为完成品。
- 阶段 3 分离 `CREATE DATABASE` bootstrap、五库 baseline、开发 seed，并对上述差异形成机器可读指纹。
- 阶段 4 默认采用 expand/migrate/contract；不可逆和事务外 migration 必须有备份点、恢复命令、超时和锁预算。
- 阶段 5 的 Schema drift 比较必须覆盖表、列、索引、约束、函数和触发器；Flyway `validate` 不能替代该检查。
- 阶段 6 在所有服务移除启动期 DDL 前，必须先提供版本/readiness 检查和滚动兼容验证。
- 阶段 7 的真实 PostgreSQL 演练只对明确命名的临时数据库执行，继续遵守用户确认和清理保护。
