# 外部客户端接入说明

## 1. 定位

MyServer 仓库只维护服务端、协议包、后台、联调工具和本地脚本。正式游戏客户端已迁移到独立仓库 `mybevy`，不作为 MyServer monorepo 的子目录维护。

外部客户端对背包、邮件领取和提交后 push 的接入边界见[统一资产事务与奖励交付运行边界](../游戏服与接入层/统一资产事务与奖励交付运行边界.md)；客户端不得构造道具、附件、奖励来源或领取 request ID。

本机开发示例路径可以是：

```text
C:\project\mybevy
```

该路径只用于个人开发示例，不能作为文档、脚本或 CI 的固定依赖。其他环境应按实际 clone 路径配置。

## 2. 路径配置

需要让脚本或本地工具访问外部客户端时，统一使用环境变量：

```powershell
$env:MYSERVER_CLIENT_ROOT = "C:\project\mybevy"
```

优先使用环境变量 `MYSERVER_CLIENT_ROOT`；环境变量未设置时，检查工具可读取仓库根目录 `local_help.txt` 中的 `MYSERVER_CLIENT_ROOT=...` 做本机只读协议定位。两者都不可用时，本仓库只使用 `tools/mock-client` 作为服务端联调边界。

`scripts/online-lockstep-reconcile.ps1 -Client mybevy` 的运行时路径解析更严格：显式 `-ClientRoot` 优先，其次读取 Process 级 `MYSERVER_CLIENT_ROOT`，最后读取 User 级同名变量；该 wrapper 不读取 `local_help.txt`。因此本机只要 User 或 Process 级变量指向有效 `mybevy` 根目录，wrapper 就不需要再补路径参数。当前 Cargo 仍以 `../../MyServer/packages/authority-core` 和 `../../MyServer/packages/sim-core` 引用共享 crate，所以“只设置环境变量”仅对 `mybevy` 与 `MyServer` 保持当前相邻目录布局的情况足够；任意非相邻布局还必须同步调整 `mybevy/project/Cargo.toml` 的 path dependency，或改用后续版本化依赖。

## 3. 仓库边界

- `mybevy`：正式游戏客户端，实现真实输入、渲染、预测、重连、资源和用户体验。
- `tools/mock-client`：服务端联调和回归验证工具，用于覆盖登录、房间、移动、战斗、rollout 等协议流程。
- Unity 历史 demo：已从 MyServer 仓库删除，不再参与协议同步、常规联调或测试准入。

## 4. 客户端接入契约

外部客户端应按以下服务边界接入：

- 登录入口：`auth-http`，本地默认 `http://127.0.0.1:3000`
- 游戏入口：优先连接 `auth-http` 返回的 `gameProxyHost/gameProxyPort` 或 `services.game`
- 本地主链路：`game-proxy:4000`；TCP fallback 调试端口通常是 `14000`
- 直连调试：`game-server:7000` 仅用于定位游戏服协议或房间逻辑问题，不作为正式客户端入口
- 协议事实源：`packages/proto`

客户端应实现并持续校验的关键流程：

- 密码账号注册：`POST /api/v1/auth/register`；如果返回 `pendingReview=true`，客户端应提示等待审核，不进入游戏连接流程
- 登录或游客登录只获取账号 access token：`POST /api/v1/auth/login` / `POST /api/v1/auth/guest-login`，注册审核关闭时注册成功也返回同类账号登录态
- 使用 access token 调用 `GET /api/v1/characters` 拉取当前账号角色列表；角色列表中的 `character_id_short`、`display_discriminator`、`same_name_hint` 用于同名角色展示区分，角色名不作为唯一键
- 无角色或玩家主动创建角色时，调用 `POST /api/v1/characters`，请求体包含 `name` 和可选 `appearance`；普通玩家单账号最多 `6` 个有效角色，角色名允许重复
- 账号 / 选角页如需查看角色基础资料，调用 `GET /api/v1/characters/{character_id}/profile`；返回基础信息、四属性、装备称号、职业阶位、同名区分信息和生命周期空态 / 来源标记
- 玩家删除角色调用 `POST /api/v1/characters/delete`，只做软删除并返回恢复窗口和硬删冷却信息；软删除角色不会出现在普通角色列表、不可选角、不可补发 game ticket，也不可查询普通资料页
- 玩家恢复角色调用 `POST /api/v1/characters/restore`，必须在恢复窗口内且恢复后不超过普通账号有效角色数量上限
- 进入游戏前必须调用 `POST /api/v1/characters/select`，请求体包含 `character_id`；该接口返回 character-bound game ticket、角色信息和服务地址
- ticket 快过期但账号 access token 仍有效时，可调用 `POST /api/v1/game-ticket/issue` 并传 `character_id` 补发同一角色的 game ticket
- 使用 character-bound game ticket 连接 `game-proxy` 并发送 `AuthReq`；缺少 `characterId` 的旧 ticket 当前会被 `auth-http` / `game-proxy` / `game-server` 拒绝
- 当前 mybevy 可暂不发送 `AuthReq.client_protocol_version`，服务端将缺失字段按 legacy v1 接受；在服务端提高最低协议版本前，mybevy 必须先发送显式版本，并对 `AuthRes` 的 `CLIENT_PROTOCOL_VERSION_TOO_OLD`、`upgrade_message` 和可选 `upgrade_url` 提供升级体验。完整策略见[协议版本协商与发布边界](./协议版本协商与发布边界.md)
- 房间加入、离开、准备、开始、断线重连和观战
- 持续发送移动/帧输入，只提交意图，不提交权威结果
- 进入游戏后如需展示角色四属性，通过 `GetCharacterElementsReq` / `GetCharacterElementsRes` 查询当前 ticket 绑定角色的 `affinity` / `mastery`
- 使用四属性成长道具仍发送现有 `ItemUseReq(item_uid)`；服务端根据 `ItemTable.csv` 的 `UseEffect=CharacterElementChange` 决定是否调用四属性成长服务，成功后客户端应消费 `CharacterElementsChangePush(1505)` 或重新拉取 `GetCharacterElementsReq`，不要本地预测永久四属性变化。
- 进入游戏后如需展示角色称号，通过 `GetCharacterTitlesReq(1417)` / `GetCharacterTitlesRes(1418)` 查询称号定义、拥有状态和 `equipped_title`
- 需要切换主展示称号时，发送 `EquipCharacterTitleReq(1419)`，成功后使用 `EquipCharacterTitleRes(1420).equipped_title` 更新角色名牌、资料页和社交展示
- 需要展示职业 / 流派阶位时，通过 `GetCharacterDisciplinesReq(1421)` / `GetCharacterDisciplinesRes(1422)` 查询当前 ticket 绑定角色的 `discipline_id`、`tier`、`points` 和 `active`
- 需要正式学习职业 / 流派时，发送 `LearnCharacterDisciplineReq(1425)`，请求体只包含 `discipline_id`；服务端用当前 ticket 绑定角色校验条件并通过 `LearnCharacterDisciplineRes(1426)` 返回学习结果、定义摘要和实际消耗的道具
- 需要激活或停用已学习职业 / 流派时，发送 `SetCharacterDisciplineActiveReq(1427)`；服务端按 `MAX_ACTIVE_DISCIPLINES` 上限校验，并通过 `SetCharacterDisciplineActiveRes(1428)` 返回职业列表、`active_skill_pool` 和本次解锁称号
- 需要切换当前激活职业 / 流派时，发送 `SwitchCharacterDisciplineReq(1429)`；服务端停用其他 active 职业后激活目标，并通过 `SwitchCharacterDisciplineRes(1430)` 返回新状态
- 需要系统入口推进职业 / 流派 points 时，发送 `AddCharacterDisciplinePointsReq(1431)`；`points_delta` 必须为正数，服务端根据 `DisciplineTable.TierRules.tiers[].min_points` 自动计算阶位，并通过 `AddCharacterDisciplinePointsRes(1432)` 返回结果
- 需要由任务、成就、活动、排行榜或世界事件触发正式成长奖励时，发送 `ApplyCharacterProgressReq(1433)`；请求体只包含 `progress_id`，服务端按 `CharacterProgressTable.csv` 校验条件并通过 `ApplyCharacterProgressRes(1434)` 返回 `applied`、`sourceType/sourceId` 和奖励摘要
- 角色状态变化 push 包括 `CharacterElementsChangePush(1505)`、`CharacterTitleChangePush(1506)`、`CharacterDisciplineChangePush(1507)`；push 只投递当前连接 ticket 绑定角色，`CharacterPushMeta.sequence/revision` 按角色单调递增，客户端应按 `characterId + revision` 去重并按 `sequence` 保序应用。
- `CharacterTitleChangePush.meta.action` 会覆盖 `grant`、`revoke`、`equip`、`unequip`、`expire`、`renew`、`auto_unlock`；客户端统一用推送里的 `titles` 和 `equipped_title` 覆盖本地称号状态，不要只按单个 `title` 做增量猜测。
- 场景区域、机关、NPC 分支和天气 / 区域 / 世界事件上下文当前由服务端 `SceneCatalog` 判定。客户端后续接入场景交互协议时只消费服务端返回的 `prompt_key`、`condition_status`、`condition_reason` 和临时 `effective_elements`，不得在请求体中提交可冒用的条件结果或本地计算的临时属性。
- 战斗和属性面板后续展示应区分永久 `affinity/mastery`、当前 `effective_elements`、以及服务端由有效属性派生的 `combat_stats/health`。称号当前只作为展示、身份和条件来源，不给客户端面板提供直接战斗属性。
- `arena.robot_sync` 使用 `RoomJoinReq.policy_id = "robot_sync_room"`，通过 `PlayerInputReq(action="robot_move")` 提交机器人输入，并消费 `FrameBundlePush` 做客户端确定性 replay
- 共享帧同步 demo 使用 `RoomJoinReq.policy_id = "lockstep_sim_demo"`，通过 `PlayerInputReq(action="sim_input", payload_json=...)` 提交移动 / 停止 / 朝向 / 技能释放意图，并消费 `RoomSnapshot.game_state` 中的 `initialSnapshot`、`lastFrame` 和 `observerFrame.lastFrame` 做恢复、hash 和事件对账。该入口是 MyServer 侧共享 `sim-core` 验证契约，不代表完整生产玩法 UI / 动画 / 预测已完成。
- 处理 `ServerRedirectPush`，断线后重新鉴权并执行 `RoomJoinReq` 或 `RoomReconnectReq`；若本地保存了最近已应用的角色状态 push sequence，应在 `RoomReconnectReq.last_character_push_sequence` 中带回
- 生产环境只依赖 `auth-http` 和 `game-proxy`；聊天、公告、邮件和匹配能力应通过服务端入口、游戏协议或后续 BFF 收敛，不作为生产客户端直连内部服务的默认模型

游戏 TCP/KCP 连接上，响应包和服务器 push 可能交错到达。真实客户端不要假设“发送 `RoomJoinReq` 后收到的下一包一定是 `RoomJoinRes`”；应按 `messageType + seq` 关联请求响应，中间的 `RoomFrameRatePush`、`RoomStatePush`、`FrameBundlePush`、`ServerRedirectPush` 和角色状态 push 作为异步事件进入各自处理队列。四属性、称号、职业阶位查询、正式学习 / 激活 / 停用 / 切换 / points 推进和正式进度奖励结果及后续状态推送都应作为异步状态更新写入角色状态缓存 / UI store，不应阻塞式假设下一包就是对应响应。断线重连后，客户端先按最新快照重新查询四属性、称号和职业；同时可把最近已应用的 `CharacterPushMeta.sequence` 放入 `RoomReconnectReq.last_character_push_sequence`，服务端会在重连成功后按当前 ticket 绑定角色回放近期 outbox，补偿包带 `snapshotCompensation=true`。服务端当前仅在本进程保留最近 push outbox 作为补偿契约，尚未提供跨实例持久离线投递。`tools/mock-client` 的 `character-room-join`、`character-elements-debug`、`character-titles-debug`、`character-disciplines-debug`、`character-discipline-learn`、`character-discipline-activate`、`character-discipline-deactivate`、`character-discipline-switch`、`character-discipline-points` 和 `character-progress-apply` 场景也按这个规则等待目标响应或角色 push，跳过先到达的无关包。

身份边界：

- `playerId` / 协议里的 `player_id` 表示账号玩家 ID，用于 session、封禁、logout、改密、ticket version、kick/revoke、mail ownership 等账号级链路。
- `characterId` / `character_id` 表示本次选中的游戏内角色 ID，进入 game ticket payload，并作为房间、匹配、输入、移动、战斗、背包和 transfer 的玩法主体。
- `RoomMember.character_id`、`RoomSnapshot.owner_character_id`、`FrameInput.character_id`、`MovementSnapshotPush.target_character_ids`、`MovementRejectPush.character_id`、`CreateMatchedRoomReq.character_ids` 均为角色 ID；`RoomReconnectReq` 不携带可指定目标角色的字段，重连身份来自已鉴权 ticket，`last_character_push_sequence` 只作为当前角色 push 补偿游标。
- 背包、仓库、装备、道具使用、丢弃、Buff 状态和后续货币均作用于当前 ticket 绑定角色；客户端背包请求体不能加入可冒用的 `character_id` 或账号 `player_id`。
- 角色四属性、称号、职业阶位查询 / debug、正式职业学习、激活、停用、切换、points 推进和正式进度奖励场景只作用于当前 ticket 绑定角色；客户端不能在这些请求体中新增可冒用的 `character_id`。
- 管理后台 `GET /api/v1/players/characters/{characterId}/profile` 和 `GET /api/v1/players/characters/{characterId}/titles` 是运营只读校验入口，只面向 `admin-api` Bearer token 和 `players.read` 权限。正式游戏客户端不应直连这些后台接口；联调工具可用 `tools/mock-client --scenario admin-character-readonly-check` 核对后台只读视图与玩家协议状态是否一致。

P1 四属性边界：

- 已落地角色 `affinity` / `mastery` 查询协议和服务端统一永久变更入口。
- `affinity` 表示四属性倾向比例，总和固定为 `10000`；`mastery` 表示长期掌握度，总和不固定。
- 已落地道具使用调用四属性变更入口：`ItemUseReq` 使用配置为 `UseEffect=CharacterElementChange` 的物品时，服务端会按配置提交 `affinity` / `mastery` delta，并把来源写入 `character_element_logs`；如果校验失败或 DB 不可用，道具不消耗并返回稳定错误码。
- 装备道具四属性当前只在服务端 `effective_item_elements` accessor 中汇总，未新增客户端协议字段；需要展示装备元素时应等待后续背包详情 / 面板协议扩展，或由配置表结合背包实例字段统一设计。
- `ItemTable.csv` 中 `DisciplineConditionKey` 和 `TitleUnlockSource` 可表达道具作为职业 / 流派学习条件和称号解锁来源；正式学习当前以 `DisciplineTable.csv` 的 item 条件为准，支持成功学习时消耗道具，称号自动触发和完整客户端 UI 闭环仍待后续扩展。
- `DebugApplyCharacterElementChangeReq` 需要玩家 ticket 加 `GAME_ADMIN_TOKEN`，仅用于测试 / GM 调试；mock-client 参数为 `--element-debug-token`，生产真实客户端不应暴露该入口给普通玩家。
- 场景上下文已可临时影响服务端 `effective_elements` 计算；这不是永久成长，客户端展示时不得把它写入角色基础 `affinity` / `mastery`，也不得替代 `GetCharacterElementsReq` 的长期属性状态。
- 统一有效属性计算已在服务端组合永久四属性、当前激活职业 / 流派、装备道具、Buff、场景上下文和系统临时修正；战斗实体可用该结果派生血量和攻击 / 防御 / 速度 / 暴击等战斗属性。客户端仍只消费服务端结果，不本地决定战斗属性。
- 称号第一阶段不提供直接战斗属性；`TitleTable.Effects` 会拒绝战斗 / 四属性效果配置。后续如开放称号战斗效果，需要先同步协议字段、称号效果白名单和战斗设计。
- 当前背包持久化已按 `character_id` 隔离，不保存传统 `level`；装备和道具使用不再以角色等级作为判定来源。

P2 / P3 称号与职业阶位边界：

- 已落地 `GetCharacterTitlesReq/Res`、`EquipCharacterTitleReq/Res`、`GetCharacterDisciplinesReq/Res`、受控 `DebugCharacterTitleReq/Res`、正式 `LearnCharacterDisciplineReq/Res`、`SetCharacterDisciplineActiveReq/Res`、`SwitchCharacterDisciplineReq/Res`、`AddCharacterDisciplinePointsReq/Res` 和 `ApplyCharacterProgressReq/Res`，消息号为 `1417-1434`。
- 称号列表响应返回配置表全部称号定义及当前角色拥有状态；客户端应按 `owned`、`equipped`、`expired`、`hidden`、`limited` 控制展示。
- 装备称号展示以 `equipped_title` 为准；成功装备后应更新角色名牌、资料页、队伍 / 社交条目等可见位置。
- 正式职业 / 流派学习只传 `discipline_id`，不传 `character_id`、条件结果或 debug token；服务端根据当前连接角色校验 `affinity`、`mastery`、已学习职业阶位、称号和道具条件。
- 正式激活、停用、切换和 points 推进不需要 debug token；服务端按 `MAX_LEARNED_DISCIPLINES` 默认 `8`、`MAX_ACTIVE_DISCIPLINES` 默认 `2` 校验，并在成功后写 `character_discipline_logs`、触发职业来源称号解锁检查。
- `active_skill_pool` 只来自当前 `active=true` 职业 / 流派的 `DisciplineTable.SkillPool`；客户端技能 UI 和后续战斗请求不要把全部已学习职业都视为可用技能来源。
- `ApplyCharacterProgressReq` 是当前最小正式进度入口：客户端提交服务端下发或约定的 `progress_id`，服务端负责校验任务 / 成就 / 活动 / 排行榜 / 世界事件条件、写入 `source_type/source_id` 并发放奖励；客户端只消费响应和随后重新拉取的称号 / 职业 / 四属性状态，不在本地决定奖励内容。
- 任务 / 事件、NPC 好感、组织身份、地区声望、场景位置和世界状态等条件可以配置表达；对应运行时数据未提供时，场景 / 职业 / 进度判断都会返回稳定 unsupported 语义，不应在客户端本地判定通过。
- `DebugCharacterTitleReq` 需要玩家 ticket 加 `GAME_ADMIN_TOKEN` 或 `MYSERVER_CHARACTER_TITLE_DEBUG_TOKEN`，仅用于测试 / GM 调试；mock-client 参数为 `--title-debug-token`，生产真实客户端不应暴露该入口给普通玩家。
- 主动升级消耗模型、完整任务列表 / 活动面板 / 排行榜结算 UI 和跨实例持久离线 push 补偿仍是后续阶段。

`tools/mock-client` 已支持 `character-list`、`character-create`、`character-profile`、`character-delete`、`character-restore`、`character-select`、`character-login-auth`、`character-room-join`、`character-elements-debug`、`character-titles-debug`、`character-disciplines-debug`、`character-discipline-learn`、`character-discipline-activate`、`character-discipline-deactivate`、`character-discipline-switch`、`character-discipline-points`、`character-progress-apply`、`character-duplicate-name`、`character-limit` 场景，以及 `--character-id`、`--character-name`、`--auto-create-character`、`--create-character-if-missing`、`--element-affinity-*-delta`、`--element-mastery-*-delta`、`--element-debug-token`、`--title-id`、`--discipline-id`、`--discipline-tier`、`--discipline-points`、`--progress-id`、`--title-debug-token` 等参数。无角色账号直接执行需要进游戏的场景会提示先创建或选择角色，不会绕过选角进入游戏；正式职业和正式进度场景只把业务 ID 编入请求，角色身份来自 ticket；角色 debug / 职业 / 进度场景会监听并摘要输出 `1505-1507` 角色 push；`character-restore` 因普通列表隐藏已删除角色，必须显式传 `--character-id`。

## 5. 共享帧同步外部客户端交接门禁

本节是外部 `mybevy` 接入 `arena.lockstep_sim` 的 MyServer 侧交接与复验入口。MyServer 仓库只维护服务端契约、联调前置条件和验收记录引用；客户端实现仍位于外部仓库。共享帧同步的 headless、单/双客户端、重连、观战、GUI smoke、失败归档和回归证据统一以 [共享帧同步调试验证 Checklist](../游戏服与接入层/checklists/共享帧同步调试验证_checklist.md) 为验收入口；旧 [mybevy 共享帧同步接入 Checklist](../游戏服与接入层/checklists/mybevy共享帧同步接入_checklist.md) 只保留接入阶段的历史记录。

### 5.1 外部仓库修改门禁

修改外部客户端仓库前必须同时满足：

- 先读取环境变量 `MYSERVER_CLIENT_ROOT`；未设置时，再读取 MyServer 仓库根目录 `local_help.txt` 中的 `MYSERVER_CLIENT_ROOT=...`。
- 校验目标路径存在、可访问，并且确认为外部 `mybevy` 仓库，而不是 MyServer 子目录或历史客户端目录。
- 向用户列出准备修改的外部仓库路径、涉及文件范围、计划命令和预期影响，并获得明确允许。
- 未获得允许时，只允许在 MyServer 仓库内记录契约或做只读核对，不修改外部仓库、不运行真实客户端、不启动外部客户端联调。

### 5.2 MyServer 契约清单

共享帧同步 demo 的当前服务端入口：

- room policy ID：`lockstep_sim_demo`，客户端通过 `RoomJoinReq.policy_id = "lockstep_sim_demo"` 进入。
- 上行输入：`PlayerInputReq(action = "sim_input", payload_json = "...")`。
- 当前没有专用 `SimInputReq` / `SimSnapshotPush` / `SimHashPush` Protobuf；第一阶段 wire 契约仍是 `PlayerInputReq.payload_json` 与 `RoomSnapshot.game_state` JSON。

`sim_input` payload schema：

```json
{
  "version": 1,
  "seq": 1,
  "commands": [
    { "type": "move", "dirX": 1000, "dirY": 0, "speed": 6000 },
    { "type": "stop" },
    { "type": "face", "dirX": 707, "dirY": -707 },
    { "type": "castSkill", "skillId": 1, "targetEntityId": 9000 }
  ]
}
```

字段语义：

- `version` 固定为 `1`；不兼容版本必须明确失败。
- `seq` 是客户端输入序号；同角色同帧冲突选择以 `seq` 和原始顺序为依据。
- `commands` 当前最多 `8` 条；payload 当前最大 `2048` bytes。
- `move` / `face` 使用 `dirX`、`dirY`，范围为 `-1000..=1000`，并且 `dirX^2 + dirY^2 <= 1000000`；`move` 的零方向非法。
- `move.speed` 可选，单位为 raw milli-units per second；当前服务端接受 `1..=12000`。
- `stop` 表示停止移动；`face` 只改朝向；`castSkill.skillId` 必须大于 `0`。
- `castSkill.targetEntityId` 可省略；省略表示 `SkillTarget::None`，非零值表示 `SkillTarget::Entity`。当前在线 wire 不接收位置目标或方向目标。
- 客户端只能提交输入意图，不得提交 `entityId`、命中、伤害、治疗、Buff、死亡、最终坐标、`finalState`、`stateHash` 或事件结果。当前服务端对 payload 和 command 使用严格 JSON 字段集合，未知权威字段会被拒绝为 `INVALID_SIM_INPUT_JSON`。

`RoomSnapshot.game_state` 的恢复与对账结构：

- 顶层包含 `logicType`、`roomId`、`worldFrame`、`tickRate`、`initialSnapshot`、`lastFrame`、`observerFrame` 和诊断字段。
- `initialSnapshot` 是可恢复初始快照，包含 `schema = "myserver.lockstep-sim.initial-snapshot.v1"`、`schemaVersion = 1`、`roomId`、`startFrame`、`tickRate`、`configVersion`、`configHash`、`simSchemaVersion`、`rngSeed`、`stateHash`、`snapshot`、`entities` 和 `controlBindings`。客户端以 `initialSnapshot.snapshot` 恢复 `SimWorld`，并用 `stateHash.hex` 校验快照 hash。
- `lastFrame` 是最近一帧 `SimFrameEnvelope`；`observerFrame.lastFrame` 是面向观战 / 恢复读取的最近 envelope。客户端对账时优先读取 `observerFrame.lastFrame`，没有时读取顶层 `lastFrame`。
- `SimFrameEnvelope` 的 `schema` 为 `myserver.lockstep-sim.frame-envelope.v1`，`schemaVersion = 1`，包含 `roomId`、`frame`、`tickRate`、`configVersion`、`configHash`、`simSchemaVersion`、`stateHash`、`eventCount`、`events`、`eventSummaries`、`inputSources`、`debugSummary` 和 `debugState`。
- `configVersion` 必须大于 `0`，`simSchemaVersion` 必须是客户端支持的 `sim-core` schema，`configHash` 不得为空。客户端恢复 `initialSnapshot` 后，应将这三个字段作为本地 replay 的契约基线；后续每个 `lastFrame` / `observerFrame.lastFrame` 都必须与基线完全一致，任何漂移都应明确失败，不得静默继续 replay。
- `stateHash.hex` 是该帧应用后 `SimWorld` 的权威状态 hash；hash 不包含 `events`、`inputSources` 或 `debugSummary`。
- `events` 是该帧 `sim-core` 输出的事件流，只读用于表现和对账，不写回 `SimWorld`，也不参与下一帧 hash。
- `eventSummaries` 是稳定字段摘要，每个元素包含 `schemaVersion`、`kind`、`frame`、`sourceEntityId`、`targetEntityId`、`skillId`、`buffId`、`amount`、`sequence`；`eventCount` 必须与 `events` 数量一致。`observerFrame` 包含 `worldFrame`、`stateHash`、`lastEventCount`、`lastEventSummaries` 和 `lastFrame`，用于只读恢复诊断。
- `inputSources` 用于区分真实输入、合成空输入和重复上一帧输入；`debugSummary` 只用于诊断输入数量、合成输入数量、事件数量和实体数量，`debugState` 只用于轻量实体状态排查，二者都不是权威模拟输入。
- 在线 replay 从 `FrameBundlePush.snapshot.game_state` 读取上述 snapshot / envelope，并用同一包的 `FrameBundlePush.inputs` 还原该帧输入；恢复后继续 replay 应以服务端 `stateHash.hex` 为准判断是否 mismatch。

当前实现边界：

- `lockstep_sim_demo` 是独立 room policy，不替换 `robot_sync_room`、`movement_demo` 或 `combat_demo`。
- `sim-core` 只承担确定性移动 / 战斗计算；房间网络、ticket 鉴权、输入等待、缺输入补偿、`FrameBundlePush` 广播、`RoomSnapshot.game_state` 生成和观战 / 重连恢复仍在 `game-server`。
- `robot_sync_room` 仍是 `arena.robot_sync` 的轻量输入转发 / 客户端 replay 样例；`movement_demo` 仍是旧服务端权威位移与校正样例；`combat_demo` 仍是旧战斗 ECS、事件和 snapshot 对照样例。
- 暂不支持或暂不作为本阶段验收：生产部署、复杂物理、NavMesh、生产 AOI、跨服迁移、完整 CSV 技能 / Buff 到 `sim-core` 映射、正式玩法 UI / 动画 / 预测、产品化回滚和生产入口全链路验收。当前 `arena.lockstep_sim` direct local online 与 GUI smoke 只属于调试验收。

常见错误码：

- 上行输入拒绝通过 `PlayerInputRes.ok = false` 和 `PlayerInputRes.errorCode` 返回，常见值包括 `INVALID_SIM_INPUT_ACTION`、`SIM_INPUT_PAYLOAD_TOO_LARGE`、`INVALID_SIM_INPUT_JSON`、`UNSUPPORTED_SIM_INPUT_VERSION`、`SIM_INPUT_COMMAND_COUNT_TOO_LARGE`、`SIM_INPUT_DIR_OUT_OF_RANGE`、`SIM_INPUT_MOVE_DIR_ZERO`、`SIM_INPUT_SPEED_OUT_OF_RANGE`、`SIM_INPUT_SKILL_ID_OUT_OF_RANGE`、`SIM_INPUT_TARGET_ENTITY_ID_OUT_OF_RANGE`。
- 快照或 envelope 不可用时，客户端应拒绝开始 replay 或进入明确错误态；常见诊断值包括 `UNSUPPORTED_SIM_SNAPSHOT_SCHEMA`、`INVALID_SIM_SNAPSHOT_ROOM_ID`、`INVALID_SIM_SNAPSHOT_TICK_RATE`、`SIM_SNAPSHOT_CONFIG_HASH_MISMATCH`、`SIM_SNAPSHOT_FRAME_MISMATCH`、`SIM_SNAPSHOT_RNG_SEED_MISMATCH`、`SIM_SNAPSHOT_HASH_ENVELOPE_MISMATCH`、`SIM_SNAPSHOT_ENTITIES_MISMATCH`、`INVALID_SIM_SNAPSHOT_CONTROL_BINDING`、`UNSUPPORTED_SIM_FRAME_ENVELOPE_SCHEMA`、`SIM_FRAME_ROOM_MISMATCH`、`INVALID_SIM_FRAME_TICK_RATE`、`SIM_FRAME_HASH_FRAME_MISMATCH`。
- `RoomSnapshot.game_state.lastError` 只作为服务端房间逻辑诊断字段；客户端不得把它当作可恢复模拟输入。

### 5.3 外部客户端联调启动门禁

启动真实 `mybevy` 联调前，必须先向用户列出并等待确认：

- MyServer 依赖：Redis、PostgreSQL、Core NATS，以及本次是否需要 `match-service`。
- MyServer 服务：正式客户端链路至少包含 `auth-http`、`game-proxy`、`game-server`；如只做本地定位，可说明是否临时直连 `game-server:7000`，但直连不作为正式客户端入口。
- 端口：以 `apps/port.txt` 和各服务 `.env.example` / 本地配置为准，重点确认 `auth-http:3000`、`game-proxy:4000`、`game-proxy` TCP fallback、`game-server:7000`、`game-server admin:7500`、Redis、PostgreSQL 和 NATS。
- 外部仓库路径：来自 `MYSERVER_CLIENT_ROOT` 或 `local_help.txt`，并说明是否会修改外部仓库。
- 命令：列出将执行的 MyServer 启动命令、`tools/lockstep-client` dry-run / online 命令、`mybevy` 客户端启动命令或双客户端脚本。
- 预期影响：本机会占用端口、写入日志、创建测试账号 / 角色 / room、写入 Redis / PostgreSQL 测试数据；清理只针对本次可识别数据，不执行宽泛 reset 或 flush。

用户确认前，不启动 Redis、PostgreSQL、NATS、MyServer 服务、真实 `tools/lockstep-client online`、真实 `mybevy` 客户端或双客户端脚本。

### 5.4 联调结果记录边界

外部客户端联调结果可以作为 MyServer 交接验收记录引用，记录内容应包含日期、MyServer commit / diff 状态、外部 `mybevy` 路径和 commit / diff 状态、启动服务、端口、命令、room、policy、frame、`stateHash.hex`、events、mismatch 结果和日志位置。

这些结果只用于证明 MyServer 契约可被外部客户端消费；完成状态、验证命令、人工目检结论和归档证据统一写入正式的 [共享帧同步调试验证 Checklist](../游戏服与接入层/checklists/共享帧同步调试验证_checklist.md)，不再分散到旧接入清单。

### 5.5 `arena.lockstep_sim` 调试入口

`mybevy` 已提供以下互补入口，它们不能互相冒充证据：

每条 JSONL 均带 `scene=arena.lockstep_sim`、room、policy、player、frame、`serverHash`、`localHash`、mismatch、inputs、entities、events 和 `replayRecovery`。mismatch 时还可带 `mybevy.lockstep.mismatch-comparison` v1；真实 online 若服务端没有下发某侧完整快照，比较项会显式标记 `not_available`，不得用本地推测补成服务端证据。

### 5.6 wrapper 复跑命令

所有命令从 MyServer 根目录执行，`<run-id>` 使用 3 到 40 位小写字母、数字或连字符，且每次必须唯一。ticket 只通过环境变量名传递，或由 `-ProvisionDevTickets` 在 wrapper 进程内临时生成；命令、日志和报告不得写 ticket 值。

只读 plan，不创建目录、不读取 ticket、不启动进程、不联网：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\online-lockstep-reconcile.ps1 `
  -Client mybevy -Check all `
  -StartDevStack -ProvisionDevTickets `
  -RunId <run-id>
```

离线 dry-run 会调用 Cargo 并写 run 目录，但不打开网络；mybevy 的四个 check 此时都转为离线 fixture packet/telemetry 入口：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\online-lockstep-reconcile.ps1 `
  -DryRun -Client mybevy -Check all `
  -RunId <run-id>
```

dry-run 不启动 GUI，也不生成四个 visual report/screenshot，因此这些 artifact 应为 `not-applicable`。这四项不能作为 dry-run 的视觉验收证据，只有真实 `-Execute -Check visual-smoke` 才应要求它们全部 `present`。

真实单客户端、双客户端、重连/观战和 GUI smoke 一键执行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\online-lockstep-reconcile.ps1 `
  -Execute -Client mybevy -Check all `
  -StartDevStack -ProvisionDevTickets `
  -RunId <run-id> `
  -RedisKeyPrefix "lockstep-online:<run-id>:"
```

如果不需要 GUI，可把 `-Check all` 改为 `-Check single-client,dual-client,reconnect-observer`。若使用外部正式 ticket，不传 `-ProvisionDevTickets`，只设置 `MYSERVER_LOCKSTEP_TICKET` 和需要第二身份时的 `MYSERVER_LOCKSTEP_OBSERVER_TICKET`；不要把值写进命令。`-Check` 的有效值为 `single-client`、`dual-client`、`reconnect-observer`、`visual-smoke`，或 `all`。

强制离线的 synthetic mismatch 诊断夹具：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\online-lockstep-reconcile.ps1 `
  -DiagnosticFixture -Client mybevy `
  -RunId <run-id>
```

`-DiagnosticFixture` 不能与 `-Execute`、`-DryRun`、`-SelfTest`、`-StartDevStack`、`-ProvisionDevTickets` 或 `-SkipTicketRedisPreflight` 组合。它的 `report.status=failed` 是预期注入结果；wrapper 只有确认 frame 3 mismatch、三类 diff 完整、适用 artifact 无缺失后才以进程 exit 0 结束并写入 `provenance.verified=true`。该报告不能用来证明 MyServer online 对账通过。

MyServer 自身的移动、近战和 observer recovery 对账使用默认 `-Client lockstep-client`，`-Check` 有效值为 `move`、`melee`、`observer` 或 `all`：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\online-lockstep-reconcile.ps1 `
  -Execute -Client lockstep-client -Check all `
  -StartDevStack -ProvisionDevTickets `
  -RunId <run-id> `
  -RedisKeyPrefix "lockstep-online:<run-id>:"
```

### 5.7 依赖、ownership 与证据目录

`-StartDevStack` 的最小本地 online 集合是 Redis `6379`、Core NATS `4222`、`game-server` player `7000` 和 admin `7500`。wrapper 强制其自有 `game-server` 使用 `DB_ENABLED=false`，不访问 PostgreSQL，也不启动 `auth-http` 或 `game-proxy`；只有外部账号/ticket 路径或正式入口链路另行需要这些服务。执行真实 online 前仍必须向用户确认依赖、端口、ticket 来源、外部仓库、命令和影响范围。

每个非 plan run 写入 `logs/lockstep-online/<run-id>/`：

- `report.json`：总报告，同时包含 `artifacts`、`triage`、`diagnosticIndex`、命令、ownership 和 cleanup。
- `<stage>.stdout.log` / `<stage>.stderr.log`：lockstep-client 输出或 mybevy JSONL/错误输出。
- `mybevy-online-report.json`、`mybevy-online.png`、`offline-fixture-report.json`、`mybevy-offline-fixture.png`：仅 `visual-smoke` 适用。
- `owned-services/`：仅保存本次 run 确认拥有并已按 PID/启动时间停止的服务日志；复用的 Redis/NATS 不会被伪装为 owned artifact。

`artifacts.items[].status` 只有 `present`、`not-applicable`、`missing`；`not-applicable` 不是失败，execute 或 diagnostic fixture 的适用项出现 `missing` 表示采集不完整。失败时先读 `triage.errorCode`、`failureStage`、`firstMismatchFrame`、双方 hash 和 input/entity/event diff，再按 `triage.relatedArtifacts` 打开 JSONL 或服务日志，并用 `diagnosticIndex` 的匹配项继续排查。在线服务端快照确实不可得时，diff 为 `not_available`，不得写成 `complete`。

`-Execute` 无论成功或失败都会在 `finally` 中尝试精确 cleanup：先停止 run-owned game-server，再 compare-delete 本 run 的 ticket/registry 精确 key，最后停止 run-owned NATS/Redis 并检查 owned ports。复用服务和外部 ticket key 不会被停止或删除；禁止用 `FLUSH*`、通配 key 删除、宽泛 `dev-stack -Stop` 或按端口杀进程替代 ownership 校验。中断恢复的详细命令见 `tools/lockstep-client/README.md` 的 “Reports and cleanup”。

### 5.8 自动化与人工 smoke 边界

自动化已经覆盖：固定移动/停止/朝向、近战与基础技能事件、Buff/DoT fixture、单客户端、双客户端、重连、观战、snapshot 恢复、同帧 server/local hash、实体 fixed 坐标、inputs、events、旧 `arena.robot_sync` 与 authority/MyServer 构建回归，以及失败报告的首 mismatch 定位。

仍需人工目检：实际窗口中的角色移动与朝向、HUD 文本可读性、技能/命中反馈、伤害数字、Buff/DoT 标记、死亡姿态和整体视觉观感。GUI smoke 自动生成报告和截图只证明证据已采集，不能替代人对图片的验收；离线视觉 fixture 中的 Buff/DoT/死亡也不能冒充真实 online 服务端事件。

## 6. MyBevy Robot Sync 联调

`mybevy` 当前已接入 `arena.robot_sync`。服务端对应能力是：

- room policy：`robot_sync_room`
- RoomLogic：`apps/game-server/src/gameroom/robot_sync_room/mod.rs`
- mock-client 场景：`npm run flow:mock-client -- --scenario robot-sync-room` 或 `node tools/mock-client/src/index.js --scenario robot-sync-room`

本地 MyServer 完整栈：

```powershell
Set-Location C:\project\MyServer
powershell -ExecutionPolicy Bypass -File .\scripts\dev-stack.ps1 -WithMatch
```

`-WithMatch` 当前是本地完整栈联调的必要条件之一，否则 `game-server` 可能因为发现不到 `match-service.grpc` 而无法正常服务。

mybevy 单客户端 MyServer 模式示例：

```powershell
Set-Location C:\project\mybevy\project
$env:MYBEVY_START_SCENE="arena.robot_sync"
$env:ROBOT_SYNC_AUTHORITY_MODE="myserver"
$env:AUTHORITY_PLAYER_ID="robot-player-a"
$env:AUTHORITY_MYSERVER_GUEST_ID="robot-guest-a"
$env:AUTHORITY_MYSERVER_ROOM="robot-sync-room"
$env:AUTHORITY_MYSERVER_POLICY="robot_sync_room"
$env:MYSERVER_TRANSPORT="tcp"
$env:MYSERVER_GAME_HOST="127.0.0.1"
$env:MYSERVER_TCP_FALLBACK_PORT="17002"
cargo run -- --window-profile phone-small --window-scale 50%
```

mybevy 双客户端脚本：

```powershell
Set-Location C:\project\mybevy
.\scripts\start-robot-sync-two-clients.ps1 -SkipBuild -HostAddress 127.0.0.1 -Port 17002
```

端口说明：

- 客户端正式链路应接 `game-proxy`，不是直连 `game-server:7000`。
- TCP fallback 端口由 `game-proxy` 配置决定；默认口径通常是 `PROXY_PORT + 10000`，但本机 `.env` 可覆盖为 `17002`。
- mybevy 脚本会把 `-HostAddress` 写入 `MYSERVER_GAME_HOST`，把 `-Port` 写入 `MYSERVER_TCP_FALLBACK_PORT`；KCP transport 时写入 `MYSERVER_KCP_PORT`。

验收关注：

- 服务端日志或 mock-client 输出能看到 `room_id`、`policy_id=robot_sync_room`、账号 `player_id`、`character_id`、`guest_id`、`frame_id`、`action=robot_move`、`input_delay_frames`。
- 输入拒绝时返回明确 `PlayerInputRes.errorCode`，例如 `INVALID_ROBOT_MOVE_ACTION`、`INVALID_ROBOT_MOVE_JSON`、`ROBOT_MOVE_DIR_OUT_OF_RANGE`、`ROBOT_MOVE_SPEED_OUT_OF_RANGE`。
- mybevy 客户端日志 `robot sync frame applied` 中，同一 `frame_id` 的 `checksum` 和 `robots` fixed 坐标摘要在两端一致。

## 7. 协议维护

新增或修改协议时，应同步处理：

1. `packages/proto`
2. Rust / Node 服务端枚举和编解码逻辑
3. `tools/mock-client`
4. 外部 `mybevy` 客户端协议绑定
5. `docs/协议与客户端/协议设计.md` 和相关专题文档

其中，仅新增兼容字段且当前最低版本仍接受 legacy 时，mybevy 允许在后续独立版本中同步；任何提高最低版本、开始依赖新字段语义或删除旧行为的发布，必须先完成 mybevy 显式协商与 UI 处理。

MyServer 协议变更提交前必须执行 `npm run check:proto`。该命令在 Windows PowerShell 与 GitHub Actions 中一致运行候选基线、生成漂移、已发布兼容、路由、fixture 和版本策略检查；本机需要 Node.js、稳定版 Rust/Cargo 和已安装的 Node workspace 依赖，不需要全局 `protoc`。CI 不配置 `MYSERVER_CLIENT_ROOT`，所以不会要求外部 mybevy 工作区；本机配置该变量时，检查器只读核对协议绑定，仍不会修改外部仓库。
