# 协议兼容性基线治理

## 范围与事实源

`packages/proto` 是 MyServer Protobuf 的唯一事实源。当前盘点和所有权记录在 `packages/proto/compatibility/inventory.json`：

- `game.proto`：玩家游戏包，以及 game-server 的内部控制消息；由 `game-server` 和 `game-proxy` 共同负责。
- `admin.proto`：game-server 管理控制包；由 `game-server` 负责。
- `match.proto`：match-service 与 game-server 间 gRPC；由两个服务共同负责。
- `chat.proto`：chat-server 内网消息；由 `chat-server` 负责。

仓库当前没有位于 `packages/proto` 外的活动 `.proto` 定义。`tools/proto-compatibility-baseline.js --check` 会核对这一点；历史归档文档不属于活动协议来源。

Rust 生成消费者和 Node.js 手写消费者也记录在清单中。玩家消息号和 mock-client 的手写编解码尚未收敛为生成物，继续由 `tools/check-mock-client-protocol.js` 检查。外部 `mybevy` 只能通过 `MYSERVER_CLIENT_ROOT` 做只读核对；修改外部仓库仍需满足《外部客户端接入说明》的授权门禁。

## 基线格式

`packages/proto/compatibility/baseline.json` 是当前已发布 API 的规范化语义快照，格式为 `myserver.protobuf.compatibility-baseline/v1`。它记录：

- package、message 和 field 的名称、号码、类型、label、oneof 归属；
- message/enum 的 reserved 名称、号码和范围；
- enum 名称和值；
- service/RPC 名称、请求与响应类型及 streaming 形态。

它刻意忽略注释、空白、声明顺序和源码换行，因此 Windows 与 CI 不会因格式产生漂移。当前仓库尚未固定 `protoc` 和插件版本，所以阶段 1 使用无第三方依赖的 Node.js 规范化基线，而不是依赖本机工具版本的 descriptor set。阶段 2 可以增加已锁定工具链生成的 descriptor set；不得移除现有 JSON 基线的审计用途。

日常检查：

```powershell
node .\tools\proto-compatibility-baseline.js --check
```

## 更新授权

基线不是在普通格式化或生成步骤中自动更新的文件。任何 `packages/proto/*.proto` 的语义变化都必须在同一变更中更新 `baseline.json`，并同时满足：

1. 变更说明写明兼容性判断、影响协议分类和部署/客户端顺序。
2. 至少由受影响服务的 owner 审核；涉及 `client_gameplay` 时还要由客户端契约 owner 审核，涉及 `service_to_service` 时由两端 owner 审核。
3. 破坏性变更需要架构/发布负责人明确批准，并在后续 breaking-change 规则或受控豁免中留下原因、责任人与到期条件。
4. 审核后的提交同时包含 proto、基线和相关测试/文档。Git review 的提交记录是最终授权证据，命令行参数不能替代审核。

获批后，使用下列命令生成候选基线，随后审阅 diff：

```powershell
node .\tools\proto-compatibility-baseline.js --write `
  --reason "说明已审核的协议变更" `
  --approved-by "审核人或批准单号"
```

`--reason` 和 `--approved-by` 是有意的操作门槛，避免把基线更新误当成普通构建产物；它们不会被写入基线，也不会自行证明授权。基线变更合入前必须再次执行 `--check`。
