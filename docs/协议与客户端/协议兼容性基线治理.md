# 协议兼容性基线治理

## 范围与事实源

`packages/proto` 是 MyServer Protobuf 的唯一事实源。当前盘点和所有权记录在 `packages/proto/compatibility/inventory.json`：

- `game.proto`：玩家游戏包，以及 game-server 的内部控制消息；由 `game-server` 和 `game-proxy` 共同负责。
- `admin.proto`：game-server 管理控制包；由 `game-server` 负责。
- `match.proto`：match-service 与 game-server 间 gRPC；由两个服务共同负责。
- `chat.proto`：chat-server 内网消息；由 `chat-server` 负责。

仓库当前没有位于 `packages/proto` 外的活动 `.proto` 定义。`tools/proto-compatibility-baseline.js --check` 会核对这一点；历史归档文档与 `tests/proto/fixtures/` 下的故意破坏样例都不属于活动协议来源。

Rust 生成消费者和 Node.js 手写消费者也记录在清单中。玩家消息号和 mock-client 的手写编解码尚未收敛为生成物，继续由 `tools/check-mock-client-protocol.js` 检查。外部 `mybevy` 只能通过 `MYSERVER_CLIENT_ROOT` 做只读核对；修改外部仓库仍需满足《外部客户端接入说明》的授权门禁。

## 基线格式

`packages/proto/compatibility/baseline.json` 是当前候选 API 的规范化语义快照，格式为 `myserver.protobuf.compatibility-baseline/v1`。它记录：

- package、message 和 field 的名称、号码、类型、label、oneof 归属；
- message/enum 的 reserved 名称、号码和范围；
- enum 名称和值；
- service/RPC 名称、请求与响应类型及 streaming 形态。

它刻意忽略注释、空白、声明顺序和源码换行，因此 Windows 与 CI 不会因格式产生漂移。JSON 语义基线继续承担协议声明的可审阅差异和兼容性比较职责；固定的 `protoc-bin-vendored`、`prost-build`、`tonic-build` 生成工具链，以及生成物/descriptor 漂移检查，见[协议生成与漂移检查](./协议生成与漂移检查.md)。不得以生成物检查替代此 JSON 基线的审计用途。

候选快照保证 proto 改动会被显式审阅，但它不能单独判断破坏性变更：同一变更若同时更新 proto 和 `baseline.json`，两者会重新一致。因此发布兼容性检查永远将候选快照与独立的已发布参考比较：

- `release-baseline.json`：最后一个获准发布的语义快照，不能在普通候选更新中顺带替换；
- `release-reference.json`：关联 release id、批准人、批准日期、理由和快照 digest 的审计记录；
- `breaking-exemptions.json`：临时受控豁免。每条必须精确匹配一条规则和一个声明目标，说明原因、责任人和到期日。过期、无效、歧义或未命中的豁免都会使检查失败。

首次引入的 `release-baseline.json` 是历史协议的无回归下限，并不替代真实发布记录；获得更准确的发布版本信息后，必须通过下述晋升流程更新它。候选快照与发布参考的职责不能互换。

日常候选检查：

```powershell
node .\tools\proto-compatibility-baseline.js --check
```

发布兼容性检查：

```powershell
npm run check:proto-breaking
```

检查诊断会读取 `inventory.json` 的 `classification`。含 `client_gameplay` 的协议按 `PLAYER_CLIENT` 风险输出，意味着历史客户端和外部 `mybevy` 契约仍可能存在；纯内部协议按 `COORDINATED_INTERNAL` 输出，意味着可以协同滚动部署，但仍必须审查生产者/消费者顺序与回滚。两类未豁免破坏都阻断检查。

## 更新授权

基线不是在普通格式化或生成步骤中自动更新的文件。任何 `packages/proto/*.proto` 的语义变化都必须在同一变更中更新 `baseline.json`，并同时满足：

1. 变更说明写明兼容性判断、影响协议分类和部署/客户端顺序。
2. 至少由受影响服务的 owner 审核；涉及 `client_gameplay` 时还要由客户端契约 owner 审核，涉及 `service_to_service` 时由两端 owner 审核。
3. 破坏性变更需要架构/发布负责人明确批准，并在后续 breaking-change 规则或受控豁免中留下原因、责任人与到期条件。
4. 审核后的提交同时包含 proto、基线和相关测试/文档。Git review 的提交记录是最终授权证据，命令行参数不能替代审核。

获批后，使用下列命令生成候选基线，随后审阅 diff：

```powershell
node .\tools\proto-compatibility-baseline.js --write `
  --reason "说明已审核的协议变更" `
  --approved-by "审核人或批准单号"
```

`--reason` 和 `--approved-by` 是有意的操作门槛，避免把基线更新误当成普通构建产物；它们不会被写入基线，也不会自行证明授权。基线变更合入前必须再次执行 `--check`。

## 已发布参考晋升

只有候选已通过兼容性检查、且发布负责人已在提交或批准单中完成审核时，才可执行下列命令更新已发布参考：

```powershell
node .\tools\check-proto-breaking-changes.js --promote-release `
  --release-id "release-identifier" `
  --reason "已审核的发布兼容性说明" `
  --approved-by "发布/协议负责人或批准单号" `
  --approved-at "YYYY-MM-DD"
```

该命令会同时更新 `release-baseline.json` 与 `release-reference.json`；两个文件缺一不可。它拒绝带有未解决破坏诊断的候选，也不会在普通 `--write` 或生成步骤中运行。命令参数和 manifest 只是可审计门槛，Git 审核记录仍是授权证据。

长期兼容演进应使用新增字段、消息或 RPC，而不是通过晋升发布参考抹除破坏记录。确有短期迁移例外时，在 `breaking-exemptions.json` 添加一条最小范围的记录，并在到期前完成迁移或移除该变更。
