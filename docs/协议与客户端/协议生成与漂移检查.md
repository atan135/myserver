# 协议生成与漂移检查

`packages/proto` 是共享 Protobuf 的唯一输入源。Rust 生成物仍提交在各服务的 `src/proto/` 目录中，以便协议变更在代码审查中可见；它们不能手动编辑。

## 固定工具链

每个 Rust 消费服务的 `Cargo.toml` 都把以下 build dependency 固定为精确版本，并由相邻的 `Cargo.lock` 锁定：

- `protoc-bin-vendored = 3.2.0`
- `prost-build = 0.13.5`
- `tonic-build = 0.12.3`（仅 gRPC 消费者）

构建脚本通过 `protoc-bin-vendored` 取得当前平台的 `protoc`，不读取系统 `PATH` 中的 `protoc`。因此 Windows PowerShell 和 CI 都只需要可用的 Rust/Cargo 工具链；生成命令使用 `cargo check --locked`，不会隐式更新 lockfile 或下载未锁定工具。

## 命令

写入已提交 Rust 生成物：

```powershell
npm run generate:proto
```

在临时目录重新生成并与已提交 Rust 生成物逐字比较：

```powershell
npm run check:generated-proto
```

该命令覆盖：

- `game-server`：`game.proto`、`admin.proto`、`match.proto`
- `game-proxy`：`game.proto`
- `chat-server`：`chat.proto`
- `match-service`：`game.proto`、`match.proto`

生成输出不包含时间戳或机器绝对路径；漂移检查使用干净临时目录再次生成，所以非确定性输出也会被报告。构建失败会给出服务目标、输入 proto、输出目录以及要求的工具版本。

## Node.js 与 mock-client

仓库没有 Node.js Protobuf 生成物。`tools/mock-client` 保留手写编码/解码器和 packet message-id 常量，因此两个命令都会运行 `tools/check-mock-client-protocol.js` 校验：

- `tools/mock-client/src/messages.js` 与 `game.proto` / `chat.proto` 的字段号和 wire 类型；
- `tools/mock-client/src/constants.js`、game-server、game-proxy 以及可选 `MYSERVER_CLIENT_ROOT` 下 mybevy 的消息号映射。

## 统一门禁与 Windows 环境

提交协议相关变更前，运行统一入口：

```powershell
npm run check:proto
```

它按固定顺序检查候选基线、生成代码漂移（其中包含手写 Node.js/mock-client 检查）、已发布协议的破坏性变更、消息类型与路由、二进制 fixture 和客户端版本策略。子检查的标准输出与错误输出不会被聚合器重写，因此 CI 会保留具体的文件、消息、字段和违反规则。

Windows PowerShell 本地环境需要 Node.js 与稳定版 Rust/Cargo；先通过 `npm install` 安装 workspace 依赖。CI 使用锁文件对应的 `npm ci`。不需要全局安装 `protoc`，Rust 生成步骤通过已锁定的 `protoc-bin-vendored` 取得平台对应二进制。GitHub Actions 在 Windows runner 上运行同一条 `npm run check:proto` 命令。

`MYSERVER_CLIENT_ROOT` 是可选的只读 mybevy 协议核对路径。未设置时，mybevy 检查会跳过，MyServer 仍以 mock-client 为检查边界；CI 不要求也不会提供外部 mybevy 仓库。仓库测试会用 Node.js 标准库核对 workflow 的关键结构和对称路径过滤；GitHub Actions 本身是该 workflow YAML 语法与 schema 的最终解析器。
