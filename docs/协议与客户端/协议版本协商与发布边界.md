# 协议版本协商与发布边界

## 1. 当前策略

玩家包头的 `version=1` 是固定的 14 字节传输格式版本，不是玩法协议或客户端能力版本。本期保持包头布局、端口和传输版本不变；应用协议版本只在 `AuthReq/AuthRes` 的 Protobuf body 中协商。

策略事实源是：

- `packages/proto/compatibility/version-policy.json`：审计用元数据。
- `packages/proto/compatibility/version-policy.rs`：`game-proxy` 和 `game-server` 共同 include 的纯判断函数。
- `packages/proto/game.proto`：对外 wire contract。

当前值：

| 项目 | 值 | 含义 |
|------|----|------|
| packet header version | `1` | 固定传输格式；本策略不修改它。 |
| current client protocol version | `1` | 服务端当前可接受的最高应用协议版本。 |
| minimum client protocol version | `1` | 服务端可接受的最低应用协议版本。 |
| legacy implicit version | `1` | `AuthReq.client_protocol_version` 缺失或为 `0` 时的等效版本。 |

`AuthReq.client_protocol_version=2` 是可选新增字段。旧客户端只发送 ticket，Protobuf 解码得到 `0`，两端将其按 legacy v1 判断；因此当前发布不要求修改 `mybevy`。

`AuthRes` 无论成功或普通鉴权失败都带 `server_protocol_version` 与 `minimum_client_protocol_version`。版本拒绝额外带 `upgrade_message` 与 `upgrade_url`：

- 低于最低版本：`CLIENT_PROTOCOL_VERSION_TOO_OLD`，默认文案要求更新客户端。
- 高于当前版本：`CLIENT_PROTOCOL_VERSION_TOO_NEW`，默认文案要求稍后重试，不能把它误导成下载旧客户端。
- `upgrade_url` 默认空字符串。只有发布负责人确认页面已公开且为绝对 HTTPS URL 后才能填入；客户端不得根据包名、区域或错误码猜测下载地址。

`game-proxy` 在本地 ticket 校验前执行该检查并向正式客户端返回 `AuthRes`。它仍会原样 replay `AuthReq` 到上游；`game-server` 对直连调试和 proxy replay 也使用同一纯函数校验，避免入口间策略差异。proxy replay 的 server 结果不是第二个客户端响应。

## 2. 升级和回滚顺序

新增字段、消息或 RPC 必须遵循 expand/contract，不能原地改变已发布字段、枚举值、消息号或 RPC 的语义。

| 场景 | 正确顺序 | 回滚边界 |
|------|----------|----------|
| 新 Protobuf 字段 | 先发布能读取缺失与新字段的消费者，再发布写新字段的生产者。 | 先停止生产者写新字段，旧消费者继续忽略未知字段；不要删除字段或复用字段号。 |
| 新玩家消息 | 先部署 server/proxy 路由和兼容处理，再发布 mybevy 发送逻辑。 | 客户端停止发送新消息后才允许回退服务端；旧消息号永久保留。 |
| 新内部 RPC | 先部署提供方支持新 RPC/新可选字段，再部署调用方；所有实例通过 registry 发现 endpoint，不按固定端口直连。 | 先让调用方退回旧 RPC，再回退提供方；保留旧 RPC 到所有调用方和重试队列退役。 |
| 抬升最低客户端版本 | 先发布 `current=N, minimum=旧值` 的 server/proxy，再发布会显式发送 `N` 且能展示 `AuthRes` 升级字段的 mybevy；观测后先让 game-proxy 拒绝旧版，再滚动 game-server 拒绝旧版。 | 若拒绝比例或登录失败异常，先回退 proxy 的 minimum，随后回退 game-server；不能先让 server 拒绝而 proxy 仍向旧客户端返回本地成功。 |

当前策略是编译期常量，尚未提供独立的线上 minimum-version 开关。因此“proxy 先、server 后”的执行需要通过两个受审查的兼容发布完成：先发布 `current=N, minimum=旧值`，随后才发布抬升 minimum 的版本；不得把该操作伪装成单服务热配置。

任何语义不兼容的玩法变更必须新增字段、新消息或新 RPC，旧语义持续可读；删除前必须通过 `reserved` 和兼容基线检查。`release-baseline.json` 只能在真正对外发布后按基线治理流程 promotion，不能为了让当前改动通过检查而提前改写。

## 3. 观测与退役

`game-proxy` 和 `game-server` 的 5 秒 NATS metrics bucket 会写入以下窗口增量字段：

| 字段 | 逻辑 labels | 说明 |
|------|-------------|------|
| `client_protocol_auth_accepted_legacy_total` | `decision=accepted, version_source=legacy_implicit` | 未声明版本的旧客户端。 |
| `client_protocol_auth_accepted_current_total` | `decision=accepted, version_source=explicit` | 显式声明且处于当前支持范围的客户端。 |
| `client_protocol_auth_accepted_supported_older_total` | `decision=accepted, version_source=explicit, version_age=older` | 仍在兼容窗口内的显式旧版本。 |
| `client_protocol_auth_rejected_too_old_total` | `decision=rejected, reason=too_old` | 被最低版本拦截的客户端。 |
| `client_protocol_auth_rejected_too_new_total` | `decision=rejected, reason=too_new` | 客户端先于服务端发布。 |

现有 metrics payload 的真实维度是 service subject、`instance_id` 和上述扁平字段，不支持任意动态 label；表中的 labels 是字段名表达的固定维度，不应假设已存在通用 Prometheus label 导出。proxy 和 game-server 会对同一次正式连接分别计数，因为 proxy 会 replay 鉴权；退役判定以 `game-proxy` 实例指标为准，game-server 指标只用于直连与一致性排查。

允许移除 legacy implicit v1 的前置条件：连续 7 个完整自然日内，所有生产 `game-proxy` 实例的 `client_protocol_auth_accepted_legacy_total` 和 `client_protocol_auth_accepted_supported_older_total` 聚合均为 `0`，且 `client_protocol_auth_rejected_too_old_total` 没有异常上升；同时确认已支持的 mybevy 最低版本已覆盖正式分发渠道和强制升级提示。满足后仍先把最低版本抬升为 `2`，不删除 AuthReq/AuthRes 字段、不变更 packet header，也不复用任何旧字段号。

## 4. 验证边界

不启动服务即可验证策略：Rust 纯函数覆盖 legacy/current/too-old/too-new，Node 定向测试覆盖 policy metadata、mock-client 的显式与 legacy 编码、以及 `AuthRes` 升级字段解码。真实登录、NATS metrics 上报和 mybevy UI 提示属于后续经用户确认的联调范围。
