# 协议设计

## 1. 适用范围

本文描述当前仓库已经落地的协议现状，覆盖：

- `game-server` 玩家 TCP 协议
- `chat-server` 聊天 TCP 协议
- `game-server` admin 控制通道的消息号占用
- `game-proxy` 透传和路由所需的玩家协议消息号

额外协议说明：

- `match-service` 对外接口和 `game-server -> match-service` 回调使用 gRPC，定义在 `packages/proto/match.proto`
- `match-service -> game-server` 的 matched room 创建复用 `packages/proto/game.proto` 中的 `CreateMatchedRoomReq/CreateMatchedRoomRes`，通过 `game-server` 内部 socket 承载
- 管理控制消息结构定义在 `packages/proto/admin.proto`，消息号见本文 admin 段；它们通过独立 TCP admin 通道承载，不走玩家连接

代码基准：

- `packages/proto/game.proto`
- `apps/game-server/src/protocol/message_type.rs`
- `apps/game-proxy/src/protocol.rs`
- `packages/proto/chat.proto`

- `apps/auth-http/src/game-ticket/game-ticket.controller.ts`

## 2. 通用 TCP 包结构

`game-server`、`game-proxy` 与 `chat-server` 共享同一套包头格式。`game-proxy` 主要解析接入认证、鉴权前心跳、房间路由相关请求和部分 rollout 消息。连接完成 `AuthReq` 且代理本地校验成功前，其余业务包不会被转发到上游 `game-server`；鉴权成功并绑定上游后，其余业务包按原始包头与 body 透传：

```text
| magic(2) | version(1) | flags(1) | msgType(2) | seq(4) | bodyLen(4) | body(N) |
```

字段说明：

- `magic`: 固定值 `0xCAFE`
- `version`: 当前版本 `1`
- `flags`: 当前必须为 `0`
- `msgType`: 消息号
- `seq`: 请求序号，响应通常回填同一序号
- `bodyLen`: 消息体长度
- `body`: Protobuf 编码内容

## 3. 协议号分段规则

为避免不同服务继续复用同一段消息号，并为 `game-server` 后续功能模块预留足够空间，当前采用“按服务划大段，再按功能划 100 号段”的规则。

约束说明：

- `game-server` 后续新增功能模块，默认申请一个独立的 `xx00-xx99` 数据段
- `9000-9099` 是跨服务共享的通用错误段，属于全局保留例外

| 段 | 用途 | 当前状态 |
|----|------|----------|
| `1000-19999` | `game-server` 玩家 TCP、admin / 内部控制以及后续功能模块扩展 | 已使用/保留 |
| `20000-20999` | `chat-server` TCP 协议 | 已使用 |
| `21000-29999` | 预留给未来独立 TCP 服务 | 保留 |
| `9000-9099` | 通用错误响应 | 已使用 |

当前细分约定：

| 段 | 用途 |
|----|------|
| `1000-1099` | 游戏服鉴权/心跳 |
| `1100-1199` | 房间/对局请求响应 |
| `1200-1299` | 房间/对局推送 |
| `1300-1399` | 查询类消息 |
| `1400-1499` | 背包/属性/外观请求响应 |
| `1500-1599` | 背包/属性/外观推送 |
| `1600-1699` | room rollout / drain / transfer 控制消息 |
| `1700-1999` | 预留给 `game-server` 后续功能模块，按每模块 `100` 号段继续划分 |
| `2000-2099` | admin 基础控制消息 |
| `3000-3099` | GM 控制消息 |
| `2100-2999`、`3100-8999` | 预留给 `game-server` 后续功能模块，按每模块 `100` 号段继续划分 |
| `9000-9099` | 通用错误响应 |
| `9100-19999` | 预留给 `game-server` 后续功能模块，按每模块 `100` 号段继续划分 |
| `20000-20099` | chat 认证 |
| `20100-20199` | chat 发送/推送 |
| `20200-20299` | 群组与历史查询 |
| `20300-20399` | chat 异步通知推送 |

后续新增消息时遵循：

- 新服务必须先申请独立段位，再定义消息号
- 不再允许通过“不同端口复用同一消息号”的方式扩展协议
- 同一服务内，请求/响应尽量成对分配，推送消息单独留段

#### 通用错误

| msgType | 名称 |
|---------|------|
| `9000` | `ErrorRes` |

当前 `game-proxy` 和 `game-server` 都会在鉴权成功前拒绝非 `AuthReq` / `PingReq` 消息，并返回 `ErrorRes`，`error_code=PREAUTH_MESSAGE_NOT_ALLOWED`。代理侧错误在本地产生，不会触发上游选择、鉴权 replay 或 upstream 连接；游戏服侧错误在 dispatch 层产生，不会进入房间、移动、背包等业务 handler。

`game-proxy` 与 `game-server` 都支持单连接消息频率限制。proxy 侧使用 `PROXY_MSG_RATE_WINDOW_MS` / `PROXY_MSG_RATE_MAX`，在读到完整 packet 后、进入本地鉴权 / 预鉴权白名单 / 上游转发前检查；game-server 侧使用 `MSG_RATE_WINDOW_MS` / `MSG_RATE_MAX`，在业务 dispatch 前检查。对应最大值为 `0` 时关闭；启用后超限返回 `ErrorRes`，`error_code=MSG_RATE_EXCEEDED`，并继续保持连接。

`game-proxy` 可通过 `PROXY_REDIS_BLOCKLIST_ENABLED=true` 启用 Redis 动态黑名单。IP 命中 `${REDIS_KEY_PREFIX}security:blocklist:ip:<ip>` 时会在连接建立早期拒绝，通常不会产生协议包；账号玩家命中 `${REDIS_KEY_PREFIX}security:blocklist:player:<player_id>` 时会在本地 ticket 校验成功后返回 `AuthRes(ok=false,error_code=PLAYER_BLOCKED)`。这里的 `player_id` 是账号玩家 ID，不是游戏内 `characterId`。启用后 Redis 查询失败按 fail-closed 处理，连接早期拒绝或在 `AuthReq` 返回 `AuthRes(ok=false,error_code=BLOCKLIST_UNAVAILABLE)`。

`auth-http` 可通过 `AUTH_REDIS_BLOCKLIST_ENABLED=true` 启用同一套 Redis 动态黑名单。登录入口 IP 命中时 HTTP 返回 `IP_BLOCKED`；登录成功但账号玩家命中时不创建 session，返回 `PLAYER_BLOCKED`；`/api/v1/characters/select` 和 `/api/v1/game-ticket/issue` 账号玩家命中时不签发新 ticket。启用后 Redis 查询失败按 fail-closed 返回 `BLOCKLIST_UNAVAILABLE`。

### 4.2 关键消息结构

同一条游戏连接上，请求响应和服务端 push 可能交错到达。客户端应按 `message_type + seq` 匹配请求响应；`RoomFrameRatePush`、`RoomStatePush`、`FrameBundlePush`、`ServerRedirectPush` 等 push 没有请求序列语义，必须作为异步事件处理。角色四属性、称号、职业阶位查询、正式学习 / 激活 / 停用 / 切换 / points 推进结果和后续状态刷新也应进入异步状态更新流程。不要假设发送某个请求后读取到的下一包一定是对应响应包。

角色状态 push 使用当前已鉴权连接的 character-bound ticket 投递，服务端只从鉴权上下文取得 `character_id`，不接受客户端在请求体中指定目标角色。`CharacterPushMeta.sequence` / `revision` 按角色单调递增；客户端应按 `character_id + revision` 去重，按 `sequence` 保序应用。断线期间服务端会在本进程内保留最近角色 push outbox 用于补偿契约；`RoomReconnectReq.last_character_push_sequence` 表示客户端已应用到的当前角色 push sequence，重连成功后服务端会按当前 ticket 绑定角色回放 `sequence > last_character_push_sequence` 的近期 push，回放包的 `CharacterPushMeta.snapshot_compensation=true`。当前生产跨实例离线投递尚未扩展为持久队列，重连后客户端仍必须重新拉取 `GetCharacterElementsReq`、`GetCharacterTitlesReq`、`GetCharacterDisciplinesReq` 做快照补偿。

异步匹配规则统一为：

- HTTP 登录、角色生命周期、profile 和 admin-api 只读查询按普通 HTTP 请求 / 响应处理，失败时优先读取 JSON 中的 `error` 或 `errorCode`。
- TCP 请求必须保存本次 `seq`，只在 `messageType` 和 `seq` 同时匹配时完成对应请求；中途到达的 push 要进入异步队列。
- `1505-1507` 角色状态 push、`1201-1209` 房间 / redirect / session push、`20301` 邮件通知 push 没有请求响应语义，不使用客户端发出的 `seq`。
- 客户端收到 `ServerRedirectPush` 后应主动断开旧连接，连接目标入口后重新 `AuthReq`，再按本地状态选择 `RoomReconnectReq` 或 `RoomJoinReq`。
- `RoomReconnectReq.last_character_push_sequence` 只用于当前 ticket 绑定角色的近期角色状态 push 补偿；它不是房间帧游标，也不是跨角色查询参数。

#### `AuthReq`

- `ticket: string`

说明：

- `ticket` 必须是 `auth-http` 在角色选择或指定角色补发流程中签发的 character-bound game ticket。
- 登录 / 注册 / 游客登录只返回账号 access token，不再直接返回可进游戏的 ticket。
- 缺少 `characterId` 的旧 game ticket 没有兼容开关，`auth-http` 校验、`game-proxy` 和 `game-server` 都会拒绝。

#### `AuthRes`

- `ok: bool`
- `player_id: string`
- `error_code: string`

说明：

- `player_id` 是账号玩家 ID，对应 ticket payload 的 `playerId`，用于登录态、账号安全和 ticket version 归属。
- 游戏内角色身份来自 ticket payload 的 `characterId`；`AuthRes` 不返回 `character_id`，房间、匹配、输入、移动、战斗、背包和 transfer 链路以鉴权上下文中的 `characterId` 为主体。

#### 背包与道具请求

`ItemEquipReq`、`ItemUseReq`、`ItemDiscardReq`、`WarehouseAccessReq`、`ItemAddReq` 和 `GetInventoryReq` 都作用于当前已鉴权连接 ticket 绑定的 `characterId`。

说明：

- 请求体不接受 `player_id`、`character_id` 或账号级目标字段。
- 服务端按鉴权上下文中的 `character_id` 读取和保存背包、仓库、装备、属性、外观和 Buff。
- `GetInventoryRes` / `InventoryUpdatePush` 返回的 `inventory_items` 与 `warehouse_items` 只表示当前角色数据；同账号其他角色不会混入。
- 道具配置和背包持久化不再使用传统角色 `level` 作为装备、使用或区域判断依据。

#### `GetCharacterElementsReq`

空 body。

说明：

- 查询当前已鉴权连接 ticket 绑定的 `characterId`。
- 客户端不能在请求体里声明其他角色 ID。

#### `GetCharacterElementsRes`

- `ok: bool`
- `error_code: string`
- `character_id: string`
- `elements: CharacterElements`

`CharacterElements`：

- `affinity: ElementValues`
- `mastery: ElementValues`

`ElementValues`：

- `earth: int32`
- `fire: int32`
- `water: int32`
- `wind: int32`

说明：

- `affinity` 总和固定为 `10000`，表达 normalized 倾向比例。
- `mastery` 总和不固定，表达长期掌握度。
- `GetCharacterElementsRes.elements` 只返回角色永久四属性。装备道具、Buff、场景上下文、当前激活职业 / 流派和系统临时修正产生的 `effective_elements` 由服务端在战斗 / 场景上下文中临时计算，不能覆盖本响应中的长期 `affinity` / `mastery`。
- 后续客户端属性面板如展示“当前有效属性”或“战斗属性”，字段应明确区分 `base_elements`、`effective_elements` 和由有效属性派生的 `combat_stats` / `health`；不要把称号作为直接战斗属性来源。
- 当前查询响应通过 `messageType=1414` 和原请求 `seq` 匹配；真实客户端必须作为异步状态更新处理。

#### `DebugApplyCharacterElementChangeReq`

- `affinity_delta: ElementValues`
- `mastery_delta: ElementValues`
- `reason: string`
- `debug_token: string`

说明：

- 该入口只用于测试 / GM 调试，不是正式玩法成长入口。
- 调用方必须先用合法玩家 ticket 完成 `AuthReq`，且 `debug_token` 必须匹配 `game-server` 的 `GAME_ADMIN_TOKEN`。
- `tools/mock-client` 对应参数为 `--element-debug-token`，默认读取 `MYSERVER_CHARACTER_ELEMENT_DEBUG_TOKEN` 或 `GAME_ADMIN_TOKEN`。
- 合法变更必须保持 `affinity` 总和为 `10000`，且 `affinity` / `mastery` 结果不能为负。
- 服务端会通过统一四属性变更服务写入角色表和 `character_element_logs`，source 为 `gm` / `debug-character-elements`，operator 为当前账号玩家 ID。

#### `DebugApplyCharacterElementChangeRes`

- `ok: bool`
- `error_code: string`
- `character_id: string`
- `before: CharacterElements`
- `after: CharacterElements`

#### `PlayerInputReq`

- `frame_id: uint32`
- `action: string`
- `payload_json: string`
- `client_timestamp_ms: int64`

说明：

- `client_timestamp_ms` 与 `MoveInputReq` 使用同一套 `game-server` 可配置窗口校验；默认兼容旧客户端。
- `game-server` 会对同一角色短窗口内的重复输入、过期帧、未来帧和时间戳异常做本实例内计数与结构化日志；默认只观测不额外拒绝，配置阈值后返回 `INPUT_ANOMALY_BLOCKED`。
- `lockstep_sim_demo` 复用该消息承载共享帧同步输入。该 room policy 的 `policy_id` 固定为 `lockstep_sim_demo`，客户端提交 `PlayerInputReq(action="sim_input", payload_json=...)`，服务端只接受 `action` 等于 `sim_input` 的输入，其他 action 返回 `INVALID_SIM_INPUT_ACTION`。

`sim_input` 当前 JSON payload：

```json
{
  "version": 1,
  "seq": 1,
  "commands": [
    { "type": "move", "dirX": 1000, "dirY": 0, "speed": 6000 },
    { "type": "stop" },
    { "type": "face", "dirX": 707, "dirY": -707 },
    { "type": "castSkill", "skillId": 1, "targetEntityId": 9000 }
  ]
}
```

字段语义：

- `version` 固定为 `1`；不匹配返回 `UNSUPPORTED_SIM_INPUT_VERSION`。
- `seq` 是该 payload 内 commands 共享的输入序号。`sim-core` 对同一角色同一帧的移动类命令按最大 `seq` 生效，`seq` 相同时按原始输入顺序后者生效；技能类命令同样按最大 `seq` / 后者选择一条。
- `commands` 最多 `8` 条；超出返回 `SIM_INPUT_COMMAND_COUNT_TOO_LARGE`。payload 大小当前限制为 `2048` bytes。
- `move` 需要 `dirX`、`dirY`，可选 `speed`；`dirX/dirY` 是量化方向，分量范围 `-1000..=1000`，且 `dirX * dirX + dirY * dirY <= 1000000`。零方向 move 返回 `SIM_INPUT_MOVE_DIR_ZERO`；非法方向返回 `SIM_INPUT_DIR_OUT_OF_RANGE`。常用方向为 `(1000,0)`、`(-1000,0)`、`(0,-1000)`、`(0,1000)` 和对角 `707/-707` 组合。`speed` 单位为 raw milli-units per second，当前允许 `1..=12000`，缺省时由模拟实体当前正速度或配置默认速度决定。
- `stop` 不带额外字段，使实体进入 idle；后续缺输入不会继续移动。
- `face` 只更新朝向，不触发移动选择；`dirX/dirY` 使用同一量化方向规则，允许零方向。
- `castSkill` 需要 `skillId > 0`，当前在线 wire 只支持可选 `targetEntityId`。省略 `targetEntityId` 表示 `SkillTarget::None`，非零值表示 `SkillTarget::Entity`；`targetEntityId=0` 返回 `SIM_INPUT_TARGET_ENTITY_ID_OUT_OF_RANGE`。`sim-core` 和 offline scenario 已有 position / direction target，但 `lockstep_sim_demo` 当前在线 JSON adapter 不接收这两类目标。
- `lockstep_sim_demo` 默认玩家实体 ID 从 `1000` 开始，训练目标实体 ID 为 `9000`，默认玩家技能 ID 为 `1`。
- `sim_input` 是意图输入，不是结算上报。客户端不得提交 `entityId`、`hit`、`damage`、`buffs`、`finalState`、`stateHash` 等权威结果字段；当前 payload 和 command 都使用严格 JSON 结构，未知字段会被拒绝为 `INVALID_SIM_INPUT_JSON`。

#### `MoveInputReq`

- `frame_id: uint32`
- `input_type: MoveInputType`
- `dir_x: float`
- `dir_y: float`
- `has_client_state: bool`
- `client_x: float`
- `client_y: float`
- `client_frame_id: uint32`
- `client_timestamp_ms: int64`

说明：

- `MOVE_INPUT_TYPE_MOVE_DIR` 和 `MOVE_INPUT_TYPE_MOVE_STOP` 是移动控制输入，客户端移动期间应持续发送。
- `MOVE_INPUT_TYPE_FACE_TO` 只表示朝向变化，不会让服务端继续保持移动。
- 服务端会拒绝非有限数值或超出安全范围的方向、客户端位置字段。
- `client_timestamp_ms` 已在 `game-server` 落地可配置窗口校验，默认兼容旧客户端；字段缺失或为 `0` 时默认跳过校验，`INPUT_TIMESTAMP_REQUIRED=true` 时拒绝并返回 `INPUT_TIMESTAMP_REQUIRED`。
- 当 `client_timestamp_ms > 0` 且 `INPUT_TIMESTAMP_MAX_SKEW_MS > 0` 时，服务端会按当前 Unix 毫秒时间校验绝对偏差，超出窗口返回 `INPUT_TIMESTAMP_SKEW`；`INPUT_TIMESTAMP_MAX_SKEW_MS=0` 表示只要求字段存在、不做偏差窗口校验。

- 连续缺少真实移动控制输入达到房间策略阈值后，服务端会强制停步，并通过 `MovementSnapshotPush.reason_code = MOVEMENT_CORRECTION_REASON_CONTROL_TIMEOUT` 下发权威状态。

### 4.3 房间快照结构

#### `RoomMember`

- `is_owner: bool`
- `offline: bool`

说明：房间成员以 `character_id` 作为游戏内主体；账号玩家 ID 只保留在登录、安全、封禁、kick/revoke、mail ownership 等账号级链路。

#### `RoomSnapshot`

- `room_id: string`
- `owner_character_id: string`
- `state: string`
- `members: repeated RoomMember`
- `current_frame_id: uint32`
- `game_state: string`

`game_state` 是玩法逻辑自定义 JSON 字符串。`lockstep_sim_demo` 当前写入以下主要字段：

- `logicType`: 固定为 `lockstep_sim_demo`。
- `worldFrame`: 当前 `SimWorld.frame`。
- `tickRate`: 当前房间 tick rate，默认 `20`。
- `trainingTargetEntityId`: 当前默认 `9000`。
- `initialSnapshot`: `SimInitialSnapshot`，schema 为 `myserver.lockstep-sim.initial-snapshot.v1`，`schemaVersion=1`，包含 `roomId`、`startFrame`、`tickRate`、`configHash`、`rngSeed`、`stateHash`、可恢复的 `snapshot`、`entities` 和 `controlBindings`。客户端重连 / 初次恢复应使用 `initialSnapshot.snapshot` 恢复 `SimWorld`，并校验 `stateHash`、`rngSeed`、`configHash` 与 bindings。
- `lastFrame`: 最近一帧 `SimFrameEnvelope`，schema 为 `myserver.lockstep-sim.frame-envelope.v1`，`schemaVersion=1`，包含 `frame`、`tickRate`、`configHash`、`stateHash`、`eventCount`、`events`、`eventSummaries`、`inputSources`、`debugSummary` 和 `debugState`。
- `observerFrame`: 只读观测摘要，包含 `worldFrame`、当前 `stateHash` 和可选 `lastFrame`。在线对账工具优先读取 `observerFrame.lastFrame`，为空时回退到顶层 `lastFrame`。
- `lastHash` / `lastHashHex`、`lastEventCount`、`lastError`: 调试摘要，不作为客户端权威模拟输入。

`FrameBundlePush.snapshot` 可携带 `RoomSnapshot`；`lockstep_sim_demo` 在线 replay 通过该 snapshot 的 `game_state` 读取上述 `initialSnapshot` / frame envelope，并结合同一个 `FrameBundlePush.inputs` 重放本帧。恢复后继续 replay 时，客户端应以服务端 `stateHash.hex` 为权威对账结果，本地 hash 不一致时进入 mismatch / 恢复流程。

`SimFrameEnvelope.stateHash` 是应用该帧后世界状态 hash，格式为 `{ frame, value, hex }`，其中 `hex` 是 16 位十六进制字符串。`events` 是本帧事件输出，当前可能包括 `skill_cast`、`damage_applied`、`heal_applied`、`buff_applied`、`buff_expired`、`entity_died`、`buff_tick` 等 snake_case 事件；事件不写回 `SimWorld`，也不进入 `hash_world`，客户端需要单独对账事件列表。`eventCount` 是服务端记录的本帧事件数量；`eventSummaries` 是稳定字段摘要，当前包含 `schemaVersion`、`kind`、`frame`、`sourceEntityId`、`targetEntityId`、`skillId`、`buffId`、`amount`、`sequence`，`kind` 使用 camelCase，例如 `skillCast`、`damage`、`heal`、`buffApplied`、`buffExpired`、`buffTick`、`death`。`inputSources` 标记真实输入和合成输入，当前 source 包括 `real`、`synthesizedEmpty`、`synthesizedRepeatLast`；`debugSummary` 只用于诊断 input / event / entity 计数，`debugState` 只包含最多 32 个实体的 `entityId`、`xRaw`、`yRaw`、`hp`、`maxHp`、`alive`，二者都不是权威模拟输入。

### 4.4 房间状态约束

- 未鉴权连接仅允许 `AuthReq`、`PingReq`；`game-proxy` 与 `game-server` 都已强制该白名单
- `AuthReq` 失败后连接保持未认证状态，后续房间、移动、背包、GM、未知消息等会被鉴权前白名单拒绝
- `game-proxy` 默认 `PROXY_MAX_PREAUTH_FAILURES=3`，同一连接在鉴权成功前非法消息或鉴权失败累计达到阈值后关闭连接；配置为 `0` 表示不按失败次数断开
- `game-proxy` 默认 `PROXY_MSG_RATE_MAX=0` 不启用单连接入站消息频率限制；配置为正整数后，超频消息会收到 `MSG_RATE_EXCEEDED`，当前不断开连接，也不计入预鉴权失败次数
- `game-server` 默认 `MSG_RATE_MAX=0` 不启用单连接消息频率限制；配置为正整数后，超频消息会收到 `MSG_RATE_EXCEEDED`，当前不断开连接
- 一个连接同一时刻只能处于一个房间上下文
- `PlayerInputReq` / `MoveInputReq` 只应在允许的对局状态中发送；两者都带 `client_timestamp_ms`，`game-server` 默认兼容旧客户端，配置要求时间戳或窗口校验失败时不会进入玩法层
- 重连和观战走独立消息，不复用普通 `RoomJoinReq`

## 7. ticket 设计

`auth-http` 当前有两层凭证：

- 账号 access token：由 `POST /api/v1/auth/login`、`POST /api/v1/auth/register` 和 `POST /api/v1/auth/guest-login` 返回，用于访问 `/api/v1/auth/me`、角色列表 / 创建 / 选择和 ticket revoke 等 HTTP 接口。
- character-bound game ticket：只能在客户端选择已有角色或指定角色补发时签发，用于 `game-proxy` / `game-server` 的 `AuthReq`，也被部分内部能力查询复用。

账号登录成功只创建账号 session，不再直接返回可进游戏的 game ticket。当前客户端进游戏必须按以下流程：

```text
1. POST /api/v1/auth/login 或 /register 或 /guest-login，获取 accessToken。
2. GET /api/v1/characters，查询账号下有效角色。
3. 如需创建角色，POST /api/v1/characters，请求体包含 name 和可选 appearance。
4. 如需查看选角页资料，GET /api/v1/characters/{character_id}/profile。
5. POST /api/v1/characters/select，请求体包含 character_id，返回 character-bound game ticket 和 services。
6. 使用 ticket 连接 game-proxy，发送 AuthReq。
```

`POST /api/v1/game-ticket/issue` 也可用现有账号 access token 为指定 `character_id` 补发 ticket，适用于 ticket 快过期刷新或工具化联调；服务端会校验角色存在、归属当前账号、未删除且状态允许登录。

角色列表 / 创建 / 选择接口当前返回的角色对象使用 snake_case 字段，包括：

- `character_id`
- `character_id_short`
- `display_discriminator`
- `same_name_hint`
- `name`
- `world_id`
- `status`
- `appearance_json`
- `created_at`
- `last_login_at`
- `deleted_at`
- `position`
- `attributes`

普通玩家单账号最多 `6` 个有效角色；角色名允许重复。需要区分同名角色的客户端界面应展示 `character_id_short` 或 `display_discriminator`，不要把角色名当作唯一键。

角色生命周期和账号 / 选角相关 HTTP 接口归属 `auth-http`：

- `POST /api/v1/characters/delete`：请求体 `{ "character_id": "chr_xxx" }`，只允许当前账号删除未删除且状态可删除的角色。服务端执行软删除，返回删除后的角色对象和 `lifecycle`，不立即物理删除。
- `POST /api/v1/characters/restore`：请求体 `{ "character_id": "chr_xxx" }`，只允许当前账号在恢复窗口内恢复软删除角色，并重新校验普通账号有效角色数量上限。
- `GET /api/v1/characters/{character_id}/profile`：返回当前账号未删除且可查询角色的基础资料、四属性、装备称号、职业阶位、短 ID 和同名区分信息。装备称号和职业阶位来自 `character_titles` / `character_disciplines`；未接入或无数据时返回 `null` 和明确的 `profile_sources`。

`lifecycle` 字段包括 `state`、`deleted_at`、`restore_window_seconds`、`restore_expires_at`、`delete_cooldown_seconds`、`hard_delete_eligible_at`。默认恢复窗口和硬删冷却均为 `2592000` 秒，可分别通过 `CHARACTER_RESTORE_WINDOW_SECONDS` 和 `CHARACTER_DELETE_COOLDOWN_SECONDS` 配置。常见稳定错误码包括 `CHARACTER_NOT_FOUND`、`CHARACTER_OWNER_MISMATCH`、`CHARACTER_ALREADY_DELETED`、`CHARACTER_NOT_DELETABLE`、`CHARACTER_NOT_RESTORABLE`、`CHARACTER_RESTORE_WINDOW_EXPIRED`、`CHARACTER_LIMIT_EXCEEDED`、`CHARACTER_NOT_QUERYABLE`。

管理后台当前在 `admin-api` 提供角色只读查询入口，供运营和联调工具核对角色状态；这些入口不执行 GM 写操作：

- `GET /api/v1/players/{playerId}/characters`：查询账号下角色列表，默认包含已删除角色，可用 `includeDeleted=false` 过滤；需要 `players.read`。
- `GET /api/v1/players/characters/{characterId}/profile?logLimit=20`：查询角色基础资料、四属性、称号、职业和元素 / 称号 / 职业日志摘要；需要 `players.read`，成功和失败都会写查询审计。
- `GET /api/v1/players/characters/{characterId}/titles?logLimit=20`：查询称号、装备称号、职业和称号日志；需要 `players.read`，成功和失败都会写查询审计。

`tools/mock-client --scenario admin-character-readonly-check` 只调用上述 `profile` 和 `titles` 两个 GET 端点；管理员 token 只从 `--admin-token`、`MYSERVER_ADMIN_TOKEN` 或 `ADMIN_API_TOKEN` 读取，不内置真实 token。

管理后台当前在 `admin-api` 提供最小角色生命周期写入口，不需要 `admin-web` 页面：

- `POST /api/v1/players/{playerId}/characters`：为指定账号创建角色。请求体至少包含 `name`、`reason`，可选 `worldId` / `world_id`、`appearance`、`position`、`affinity`、`mastery`。该入口明确绕过普通账号有效角色数量上限，角色名仍允许重复。
- `POST /api/v1/players/characters/{characterId}/restore`：恢复已软删除角色。请求体至少包含 `reason`。该入口明确绕过普通账号有效角色数量上限，但仍校验角色存在、`deleted_at` 非空且 `status=deleted`。

这两个管理入口沿用 `JwtAuthGuard + RolesGuard`，权限暂挂在现有 `players.status.update`，避免新增未映射权限导致默认管理员不可用。成功和失败都会调用 `adminStore.appendAuditLog()` 写入 `admin_audit_logs`，`targetType=character`，`details` 包含 `reason`、目标账号 / 角色、`bypassCharacterLimit=true`、结果和错误码。

GM 角色状态写入口当前位于 `admin-api /api/v1/gm/characters/{characterId}` 下：

- `POST /elements`：设置 `affinity` / `mastery`，权限 `gm.character_elements.write`。
- `POST /titles`：授予、撤销、装备或卸下称号，权限 `gm.character_titles.write`。
- `POST /disciplines`：设置职业 points、阶位和 active 状态，权限 `gm.character_disciplines.write`。
- `POST /unlock-check`：触发称号解锁检查，权限 `gm.character_titles.write` 和 `gm.character_disciplines.write`。

这些 GM HTTP 写入口是后台控制面，不是正式客户端协议；正式客户端只使用玩家 TCP 的 `1413-1434` 查询 / 操作和 `1505-1507` push。

进入游戏后的实时玩法资料仍可由 `game-server` 玩家协议提供；当前 auth-http profile 用于登录后 / 选角页的角色基础资料闭环，不替代游戏连接内的四属性、称号和职业阶位异步查询 / 状态刷新。

`auth-http` 签发的 game ticket 格式：

```text
base64url(payload_json).base64url(hmac_sha256_signature)
```

当前 payload 结构：

```json
{
  "playerId": "plr_xxx",
  "characterId": "chr_xxx",
  "worldId": 0,
  "nonce": "hex_random",
  "ver": 1,
  "exp": "2026-06-25T12:15:00.000Z"
}
```

字段语义：

- `playerId`：账号玩家 ID，仍用于账号级 session、封禁、限流、logout、改密、ticket version、kick/revoke 和 mail ownership。
- `characterId`：本次游戏会话绑定的游戏内角色 ID，必填；缺失时返回 / 记录 `MISSING_CHARACTER_ID`，没有兼容开关。
- `worldId`：角色所属世界 ID，当前可选但由角色选择和补发流程随角色信息写入。
- `nonce`：单张 ticket 随机值。
- `ver`：账号玩家级 ticket version，logout / 改密 / 强制失效会递增 `player-ticket-version:<playerId>`。
- `exp`：ISO 时间字符串，超过后拒绝。

当前校验规则：

- `game-server` 使用同一 `TICKET_SECRET` 校验签名
- `game-proxy` 会先校验签名并检查 Redis ticket 记录，随后把认证包 replay 到 `game-server`
- `chat-server` 也复用同一套签名校验逻辑，并检查 Redis ticket 记录和 ticket payload 中的版本号；当前聊天身份仍以账号玩家 ID 为主
- `auth-http`、`game-proxy`、`game-server` 都强制 ticket payload 包含合法 `characterId`
- `exp` 过期则拒绝
- `game-server` 还会继续检查 Redis 中是否存在对应 ticket 记录
- `game-proxy`、`game-server`、`chat-server` 都会检查 Redis 中的 `player-ticket-version:<playerId>`，从而感知 logout / 改密等玩家级 ticket version 失效
- `game-proxy`、`game-server`、`chat-server` 都会检查 Redis 中的 `ticket:<sha256(ticket)>`，从而感知单张 ticket revoke；P0 阶段 Redis ticket 记录仍保存账号玩家 ID 作为 owner，其中 `chat-server` 对不存在或归属不匹配统一返回 `TICKET_REVOKED`
- `auth-http` 当前默认 `TICKET_TTL_SECONDS=900`。logout 和改密通过递增 `player-ticket-version:<playerId>` 统一失效该账号下所有角色的未过期旧 ticket，不枚举删除所有 ticket key；logout body 中可选的 `ticket` 仍会走单张 revoke 路径并校验归属
- 这些 Redis key 都受 `REDIS_KEY_PREFIX` 影响

Redis 相关键：

- `session:<accessToken>`
- `ticket:<sha256(ticket)>`
- `player-session:<playerId>`
- `player-ticket-version:<playerId>`

说明：上述 Redis key 均按账号玩家 ID 归属和失效；P0 没有角色级 session、角色级 kick 或角色级 revoke。

并发登录/改密踢旧连接通知已迁移为 Core NATS subject：

- `myserver.session.kick.<player_id_token>`

## 8. 维护原则

后续维护协议时，按以下顺序处理：

1. 先确认新消息属于哪个服务。
2. 在该服务所属段位内分配新的 `msgType`。
3. 同步更新协议定义、服务端枚举、`tools/mock-client`、外部 `mybevy` 客户端协议绑定和文档。
4. 如果需要新增独立 TCP 服务，先在本文中为它预留新段位，再写代码。
