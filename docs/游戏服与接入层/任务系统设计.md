# 任务系统设计

## 当前定位

当前仓库尚未落地完整任务 UI、任务列表、每日刷新、活动调度、排行榜结算或世界事件编排。本阶段只在 `game-server` 落地最小正式进度服务，用于让任务、成就、活动、排行榜和世界事件通过正式玩家协议触发角色成长和称号授予，不再依赖 P2 debug 协议。

正式入口：

- 玩家协议：`ApplyCharacterProgressReq(1433)` / `ApplyCharacterProgressRes(1434)`
- 配置表：`apps/game-server/csv/CharacterProgressTable.csv`
- 服务实现：`apps/game-server/src/core/character_progress.rs`
- 玩家持久化：`character_inventory.progress_data`

## 身份边界

`ApplyCharacterProgressReq` 只包含 `progress_id`。服务端从当前已鉴权连接的 character-bound ticket 读取 `character_id`，不接受客户端传入 `character_id`、奖励内容、条件结果或 debug token。

上游系统如任务、成就、活动、排行榜和世界事件只能选择一个已配置的 `progress_id` 触发。奖励和来源都由服务端 CSV 决定。

## 配置字段

`CharacterProgressTable.csv` 当前字段：

| 字段 | 说明 |
|------|------|
| `ProgressId` | 业务进度 ID，客户端 / 上游只提交该值 |
| `SourceType` | 稳定来源类型：`task`、`quest`、`achievement`、`activity`、`ranking`、`world_event` |
| `SourceId` | 稳定来源 ID，例如活动 ID、榜单周期 ID、世界事件 ID |
| `Conditions` | JSON 条件；可为 `"always"`、对象，或顶层数组（按 `all_of` 处理） |
| `Rewards` | JSON 奖励数组 |
| `Repeatable` | `0` 表示只发一次；重复触发返回 `applied=false` |
| `Enabled` | `0` 表示禁用 |

## 条件

已支持：

- `always`
- `affinity`
- `mastery`
- `discipline_tier`
- `title`
- `discipline`
- `item_growth`
- `all_of`
- `any_of`

`Conditions` 顶层数组等价于 `{"all_of":[...]}`，加载校验和正式触发运行时按同一语义处理。

条件只在服务端校验。条件失败返回 `CHARACTER_PROGRESS_CONDITION_NOT_MET`，不发放奖励。

## 奖励

已支持：

- `affinity`：调用 `CharacterElementService.apply_change`
- `mastery`：调用 `CharacterElementService.apply_change`
- `discipline_points`：调用 `DisciplineService.add_points_for_identity`
- `title`：调用 `TitleService.grant_for_identity`
- `discipline_eligibility`：写入 `PlayerData.progress.discipline_learning_eligibilities`，可被 `DisciplineTable.LearnConditions` 的 `discipline_eligibility` / `progress_eligibility` 条件消费

奖励发放前会做配置预检，避免前半段成长已写入、后半段才发现明显配置错误。四属性、职业和称号仍以各自服务为最终校验和日志来源。

## 称号来源和限时规则

称号授予必须保留 `source_type` / `source_id`。当前进度服务会把 `CharacterProgressTable.csv` 的 `SourceType` / `SourceId` 传给 `TitleService`、`DisciplineService` 和 `CharacterElementService`，并写入 `PlayerData.progress.reward_logs`。

限时称号必须由显式业务配置带 `expires_at` 授予。`TitleUnlockService` 自动检查器会跳过 `Limited=1` 的称号，不会无过期时间授予限时称号。示例：

- `TitleTable.csv`：`9101` 夏季竞逐者，`Limited=1`
- `CharacterProgressTable.csv`：`activity_summer_ranker`、`ranking_arena_weekly_top` 在 title 奖励中显式带 `expires_at`

## 查询和排查

运营查询当前仍以既有后台只读能力和数据库日志为主：

- 称号当前状态和最近称号日志：`GET /api/v1/players/characters/:characterId/titles`
- 称号授予日志：`character_title_logs`
- 职业 points / 阶位日志：`character_discipline_logs`
- 四属性变化日志：`character_element_logs`
- 进度幂等和奖励摘要状态：`character_inventory.progress_data`

当前后台还没有独立任务 / 成就 / 活动页面，也没有正式进度日志筛选 API。后续如果补后台页面，应按 `progress_id`、`source_type`、`source_id`、`reward_type`、`reward_id`、`status` 查询 `progress_data.reward_logs`，并交叉展示称号 / 职业 / 四属性日志。

## Mock Client

`tools/mock-client` 提供 `character-progress-apply` 场景：

```bash
node tools/mock-client/src/index.js --scenario character-progress-apply \
  --http-base-url http://127.0.0.1:3000 \
  --host 127.0.0.1 --port 14000 \
  --login-name test001 --password Passw0rd! \
  --character-id chr_0000000000001 \
  --progress-id achievement_first_forge \
  --json-output
```

该命令需要真实服务栈运行；按协作约定，启动 PostgreSQL、Redis、NATS、auth-http、game-proxy 或 game-server 前应先由用户确认。
