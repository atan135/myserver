# 统一资产阶段 8：奖励来源模块接入

本文记录阶段 8 的已实现边界。当前代码和配置优先；本文件不将未落地的商店、制作、场景掉落或完整战斗结算描述为已开放玩家功能。

## 统一来源契约

`apps/game-server/src/core/reward_source.rs` 是普通物品奖励的内部来源适配入口。来源模块只能构造服务端业务对象和标准化物品意图，然后通过 `RewardSourceService` 调用 `RewardDeliveryService`；不能修改 `PlayerData.inventory`、分配物品 UID、自己创建奖励邮件或把客户端物品列表转发为奖励。

| 业务来源 | `AssetOriginType` | 规范 `origin_id` | 默认交付策略 |
| --- | --- | --- | --- |
| 成就 | `achievement` | `achievement:<achievement-claim-id>` | `PREFER_INVENTORY` |
| 任务 | `quest` | `task:<task-settlement-id>` | `PREFER_INVENTORY` |
| 任务链/委托 | `quest` | `quest:<quest-settlement-id>` | `PREFER_INVENTORY` |
| 战斗 | `battle` | `battle:<server-settlement-id>` | `PREFER_INVENTORY` |
| 场景拾取 | `scene_pickup` | `scene_pickup:<drop-entity-id>` | `PREFER_INVENTORY` |
| 活动 | `activity` | `activity:<activity-settlement-id>` | `PREFER_INVENTORY` |
| 排行榜 | `ranking` | `ranking:<ranking-batch-and-rank-id>` | `MAIL_ONLY` |
| 世界事件 | `world_event` | `world_event:<world-event-settlement-id>` | `PREFER_INVENTORY` |

来源 ID 必须是服务端生成的、无空白的业务主键。`RewardSourceClaim` 使用角色 ID、来源种类和规范来源 ID 构造稳定 request ID；同一角色的同一来源重复请求复用已完成的来源状态，而不是再调用交付服务。
已经发放过的来源 ID 不得重新绑定另一份奖励；规则变更必须生成新的结算/领取 ID。

来源状态只在 `RewardDeliveryService` 返回 `applied` 后完成。direct、mail-only 和容量 fallback 邮件均是 `applied`；`INVENTORY_REQUIRED` 的容量失败和结果未知不会完成来源状态，以便使用原 request ID 安全重试。

## 已接入界面路径

正式玩家协议中的界面领取仍是 `ApplyCharacterProgressReq { progress_id }`。它只提交 `progress_id`；角色身份来自已鉴权会话，来源类型、来源 ID、条件、奖励内容和交付策略由 `CharacterProgressTable` 与服务端逻辑解析。

`character_progress` 和 CSV 校验均调用 `RewardSource::from_character_progress`，因此任务、成就、活动、排行榜和世界事件的现有配置必须先映射为上述稳定来源 ID。当前 CharacterProgressTable 中的奖励类型是属性、称号、流派点和学习资格，尚无物品奖励行；后续新增物品奖励必须经 `RewardSourceService -> RewardDeliveryService`，不能在进度服务中直接 grant。

## 战斗、拾取和交换边界

- `BattleServerResult` 只能由 crate 内服务端模拟结果构造，再转换为 `RewardSourceClaim`。玩家协议没有携带战利品列表或“结算成功”字段。
- `SceneDrop` 的 `prepare_pickup` 校验掉落实体 ID、所有权、距离和未移除状态。`remove_after_delivery` 只接受已提交的 source delivery；容量 fallback 邮件已可靠托管时可以移除，未提交或未知结果必须保留掉落。
- 商店、制作、兑换尚没有正式玩家入口或货币余额实现。`InventoryRequiredExchange` 是后续模块必须使用的服务器内部契约：同一个 `AssetCommand` 包含材料 `Consume` 与产物 `Grant`，固定 `AllOrNothing` 和 `INVENTORY_REQUIRED`，没有邮件 fallback。货币扣款接入前不得伪造货币结算。

## 验证边界

离线测试覆盖每个来源的重复请求和来源状态重放；普通来源通过真实 `RewardDeliveryService` 验证 direct 与容量 fallback，排行榜验证 `MAIL_ONLY`。这些是领域层组合测试，不依赖 PostgreSQL、Redis、NATS 或运行中的游戏服务。

真实服务联调应在有可调用的战斗结算、场景掉落或物品型进度配置后，配合 PostgreSQL 的 delivery/outbox 表和 mail-service dispatcher 进行。当前没有这样的正式入口，启动服务不能额外验证该阶段的核心行为。
