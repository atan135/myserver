## 1. 核心结论

因果留影与风门远行必须拆成两个独立领域：

- **因果留影领域**负责内容上传、展示、可见性、收藏、举报、审核、推荐、热度和纪元归档。
- **风门远行领域**负责角色状态、来源限制、目的地准入、吟唱、受击打断、资源预加载、容量、路由、迁移和安全落位。

两者只通过稳定的 `impression_id -> anchor_key` 关系连接。留影可以帮助玩家发现目标，但不能携带或授予肉身抵达权限。

以下结论作为后续开发的硬边界：

1. 留影、收藏、分享链接和客户端请求不得把裸坐标作为长期目的地身份。
2. 客户端不得提交目标 Room、game-server 实例或最终落点。
3. 远行开始和完成时必须分别执行服务端准入与容量校验。
4. 最终落点由当前锚点 revision、场景导航数据和动态占用共同计算。
5. 留影可见性与远行准入是两套独立判断。
6. 跨 Room 角色迁移不能复用当前“整个 Room Transfer”语义。
7. `game-server` 是肉身移动的最终权威；内容服务和推荐服务不能批准角色落位。
8. 第一阶段只支持同一世界、当前纪元、公共稳定锚点，不开放跨世界肉身远行和任意精确落点。

早期设想中的“风景卡片携带绝对 Vector3，并传送到摄影师精确机位”不进入正式实现，也不提供兼容协议。

## 2. 当前实现基线

### 2.1 已有基础

当前仓库已经具备以下可复用能力：

- character-bound game ticket，连接上下文包含 `character_id` 和可选 `world_id`。
- `RoomManager + RoomRuntimePolicy + RoomLogic` 房间运行框架。
- `persistent_world` 房间策略，可表达常驻、不按空房销毁和 AOI 参数。
- `movement_demo` 的服务端权威二维移动、静态可行走校验、校正和恢复快照。
- `SceneTable.csv`、`SceneSpawnPoint.csv`、`SceneRegion.csv` 及场景条件 DSL。
- `SceneQuery::is_walkable`、`clamp_position` 和区域准入查询。
- Room 断线重连、最近输入、快照和基础迁移 payload。
- `game-proxy` 的 room route、character route、Redis route store 和 redirect/reconnect 基础设施。
- Redis 服务注册、在线角色 route fencing 和 PostgreSQL 角色数据。
- `admin-api + admin-web` 控制面、RBAC 和审计基础。

### 2.2 尚未形成闭环的部分

当前实现不能被视为已经支持大世界远行，主要差距如下：

- `PersistentWorldLogic` 目前只记录生命周期日志和 tick 次数，没有移动、战斗、区域、锚点或持久世界状态。
- `movement_demo` 在角色加入时使用场景默认出生点，不读取 `characters.scene_id/x/y`。
- 角色位置的数据库写方法位于 `auth-http`，当前没有被 `game-server` 正式移动链路调用。
- `ScenePortal.csv` 会加载和 reload，但尚未进入 `SceneCatalog` 业务查询。
- `SceneCatalog` 在启动时构造，CSV reload 后不会自动重建。
- 当前 `RoomJoinReq` 允许客户端提交 `room_id` 和 `policy_id`，不适合作为正式 persistent world 入口。
- 当前移动、战斗、剧情限制和交互锁没有统一的角色运行时状态。
- 当前 `game-proxy` 绑定上游后主要透明转发，不支持同连接切换 upstream。
- 当前 Room Transfer 面向整个 Room，不能用于单个在线角色跨 Zone 迁移。
- 仓库内没有正式好友图谱、宗门成员关系、UGC 媒体存储、内容审核或推荐服务。

### 2.3 当前坐标语义

MyServer 当前权威移动使用二维 `x/y`；外部 mybevy 的 3D 表现采用 Y-up，并把逻辑二维坐标映射到 Bevy XZ 地面平面。

远行系统不应继续扩大 `x/y` 的歧义。新增位置与导航结构建议使用：

- `nav_x`：导航平面 X。
- `nav_z`：导航平面 Z。
- `nav_surface_key` 或 `floor_id`：多层导航面身份。
- 高度 Y：由当前导航面或场景高度查询得出，不由留影或客户端决定。

现有 `EntityTransform.x/y` 可在 demo 和既有移动协议中保留，正式大世界位置模型应在独立迁移任务中统一字段语义。

## 3. 第一阶段范围

### 3.1 必须覆盖

- 同一 `world_id`、当前 `epoch_id` 内的公共稳定区域。
- 系统配置并处于 active 状态的公共远行锚点。
- 服务端根据角色权威位置为留影绑定合法锚点。
- 公开和私人两种基础可见性。
- 留影创建、上传、提交、审核状态、浏览、收藏和举报。
- 已满足目的地区域准入角色的循迹远行。
- 吟唱、移动/攻击/施法/交互限制、有效伤害打断和失败冷却。
- 开始与完成双重校验。
- 场景 revision、锚点 revision 和安全落位校验。
- 简单容量限制、邻近锚点回退和明确失败码。
- 同进程跨 Room 迁移和跨 game-server redirect/reconnect 目标设计。
- 断线、重复请求和服务重启后的幂等恢复。

### 3.2 第一阶段不开放

- 跨世界或跨层级肉身远行。
- 未开拓区域盲传。
- 副本、解谜后方、首领房间和一次性剧情空间内部定标。
- 玩家自定义任意精确落点。
- 旧纪元锚点自动恢复。
- 私人洞府和宗门空间访客锚点。
- 好友和宗门可见性；正式社交图谱能力完成前不得使用聊天群模拟。
- 远行产生战斗强度、稀有资源或不可替代经济收益。
- 无感同连接跨 upstream 迁移。

## 4. 稳定世界地址

### 4.1 地址模型

长期可引用的目的地使用稳定地址，不包含运行时实例：

```text
WorldAddress {
  world_id
  epoch_id
  area_key
  scene_key
  scene_revision
  anchor_key
}
```

字段语义：

| 字段 | 说明 |
|------|------|
| `world_id` | 角色所属世界，当前已经随选角 ticket 进入连接上下文 |
| `epoch_id` | 当前世界纪元，由服务端世界状态解析，不从客户端或长期 ticket 直接信任 |
| `area_key` | 稳定区域身份，用于准入、探索和聚合 |
| `scene_key` | 资源和运行场景身份，不直接等于 Room |
| `scene_revision` | 场景规则和落点拓扑版本 |
| `anchor_key` | 稳定锚点身份，跨场景版本引用时需要重新解析 |

以下字段属于运行时路由，不得写入留影长期身份：

- `room_id`
- `zone_id`
- `instance_id`
- `game_server_id`
- upstream host / port
- 最终 `nav_x/nav_z/height`

### 4.2 纪元处理

`epoch_id` 不建议直接固化进 15 分钟有效的 game ticket。ticket 只证明账号、角色和世界身份；当前纪元、区域状态和锚点状态应在每次请求时由服务端读取。

纪元切换后：

- 留影内容、作者、收藏和历史统计可以继续存在。
- 留影绑定的旧 `anchor_key + revision` 默认不能继续远行。
- 可映射到新纪元的锚点进入待重校状态，由运营或世界事件恢复。
- 无法映射的留影仅作为历史档案展示。

## 5. 角色位置持久化

### 5.1 数据所有权调整

当前 `characters` 表同时保存身份资料和 `scene_id/x/y/dir_x/dir_y`。正式大世界接入前，建议把实时位置写入权转移到 `game-server`，并新增独立位置表：

```text
character_locations {
  character_id PK
  world_id
  epoch_id
  area_key
  scene_key
  scene_revision
  nav_surface_key
  nav_x
  nav_z
  facing_x
  facing_z
  last_safe_anchor_key
  location_version
  updated_at
}
```

设计原则：

- `game-server` 是在线位置的唯一写入方。
- `auth-http` 只读取位置摘要用于选角页，不更新在线位置。
- 不按每个移动 tick 写 PostgreSQL。
- 普通移动按周期、断线、离房或安全事件保存检查点。
- 远行提交、跨服迁移和安全回退必须作为关键事务立即保存。
- 所有更新使用 `location_version` 或等价 CAS，拒绝旧实例晚到写入覆盖新位置。

### 5.2 旧数据迁移

开发迁移建议：

1. 为现有角色创建 `character_locations`。
2. 将旧 `scene_id/x/y/dir_x/dir_y` 映射为当前场景 revision 下的导航位置。
3. 对位置执行场景存在、导航面和可行走校验。
4. 无法验证的位置回退到场景默认安全锚点。
5. 旧列短期只读保留用于对账，确认没有消费者后再移除。

当前没有线上持久库升级需求时，可以使用 bootstrap SQL 直接调整；进入生产迁移阶段后必须改为版本化 migration。

## 6. 远行锚点与场景数据

### 6.1 不直接复用 ScenePortal

`ScenePortal.csv` 表达固定场景入口到固定出生点的映射，缺少远行需要的长期身份、revision、状态、容量、候补、审核和纪元语义，因此不能直接充当远行锚点。

建议新增 `TravelAnchor.csv` 或等价配置：

| 字段 | 说明 |
|------|------|
| `AnchorKey` | 稳定业务键 |
| `AreaKey` | 所属稳定区域 |
| `SceneId` / `SceneKey` | 当前场景映射 |
| `SceneRevision` | 适用场景版本 |
| `ArrivalRegionId` | 安全落位范围 |
| `FallbackGroup` | 邻近或等价锚点组 |
| `AnchorType` | `public`、`entry`、`event` 等 |
| `AccessConditions` | 服务端条件 DSL |
| `CapacityPolicy` | 并发落位、排队和镜像策略 |
| `ManifestKey` | 客户端目标资源预加载标识 |
| `Tags` | safe、outdoor、no_combat 等规则标签 |
| `Enabled` | 配置级开关 |

公共静态锚点定义放配置；纪元和运营运行态放 PostgreSQL 或专用世界状态存储：

```text
travel_anchor_states {
  world_id
  epoch_id
  anchor_key
  anchor_revision
  status
  capacity_override
  redirect_anchor_key
  updated_at
}
```

`status` 至少包含：

- `active`
- `overloaded`
- `recalibrating`
- `disabled`
- `archived`

### 6.2 Catalog 热更

远行不能建立在当前“CSV reload 成功但 SceneCatalog 不重建”的状态上。需要把场景、区域、Portal、锚点和资源 manifest 构造成带 revision 的统一快照，并在 reload 成功后原子替换。

每次远行会话固定记录准备阶段看到的 `scene_revision` 和 `anchor_revision`。完成阶段必须读取当前快照重新校验：

- revision 未变：继续提交。
- 兼容变更：按当前规则重新求解落点。
- 锚点重定向：返回候补锚点并重新确认或按明确策略重定向。
- 不兼容或已禁用：失败，不能沿用旧坐标。

场景拓扑、坐标系或导航层结构变化仍应走 Room/实例灰度，不作为普通 CSV 热更。

## 7. 角色运行时状态

远行需要建立跨移动、战斗、剧情和交互系统的公共运行时状态，避免各玩法自行维护互相矛盾的布尔值。

建议最小结构：

```text
CharacterRuntimeState {
  character_id
  world_address
  room_id
  in_combat
  last_hostile_damage_at
  control_locks
  gameplay_tags
  active_travel_id
  arrival_protection_until
  runtime_version
}
```

`control_locks` 至少支持：

- movement
- attack
- skill
- interaction
- item_use
- room_leave

`gameplay_tags` 至少能表达：

- dead
- in_cutscene
- in_ranked_match
- boss_encounter
- pursuit
- no_travel_source
- no_travel_destination

公共状态只表达框架级约束。具体玩法仍负责产生状态变化和领域事件，不应让远行模块直接读取 combat demo 的私有 ECS 布局。

## 8. 远行状态机

### 8.1 状态定义

建议使用显式、可持久恢复的状态机：

```text
Prepared
  -> Channeling
  -> Committing
  -> AwaitingReconnect   # 仅跨 game-server
  -> Arrived

Prepared / Channeling
  -> Cancelled
  -> Interrupted
  -> Failed

Committing / AwaitingReconnect
  -> Recovering
  -> Arrived / RolledBack / Failed
```

终态必须可重复查询，客户端重发相同 `request_id` 时返回第一次结果，不重新执行远行。

### 8.2 准备阶段

客户端提交 `impression_id` 或允许直达的 `anchor_key`。服务端执行：

1. 从当前鉴权连接读取 `character_id` 和 `world_id`。
2. 读取角色权威运行时状态和当前世界地址。
3. 解析留影可见性；通过留影解析稳定锚点。
4. 校验来源场景和角色状态允许开始远行。
5. 校验脱战等待、冷却和请求限流。
6. 校验目标世界、纪元、区域准入和锚点状态。
7. 选择候选 Zone、Room、实例和资源 manifest。
8. 创建短 TTL 软容量预留。
9. 持久化 `travel_session=Prepared`。
10. 返回 `travel_id`、吟唱截止时间、manifest revision 和候选锚点摘要。

客户端不能在准备请求中提交：

- 目标坐标
- 目标 Room
- game-server 实例
- 自己计算的准入结果
- 自己计算的吟唱完成时间
- 自己声明的战斗状态

### 8.3 吟唱阶段

进入 `Channeling` 后，服务端施加控制锁并记录服务端截止时间。

中断来源包括：

- 玩家主动取消。
- 权威移动输入。
- 攻击、施法、主动交互或禁止使用的物品行为。
- 敌方、环境或规则认定的有效伤害事件。
- 角色死亡、进入演出、对局、追捕或其他禁止状态。
- 来源 Room/实例失去权威或进入不可恢复迁移状态。
- 目标锚点在吟唱中失效且没有可用候补。

被拒绝的非法客户端输入是否中断吟唱，应按行为类型明确配置。默认只有被服务端接受或规则认定为主动行为的输入才中断，避免网络重发或恶意垃圾包无条件改变玩法状态。

客户端可发送预加载完成通知，但该通知只表示资源准备状态，不授予远行权限。服务端可以在吟唱结束后等待有限的预加载宽限期，超过上限后失败或要求客户端重试。

### 8.4 提交阶段

吟唱截止后，服务端再次执行完整校验：

- 来源角色仍由当前实例持有权威。
- 角色仍满足来源状态、脱战和控制约束。
- 留影仍可见且未下架。
- 目标世界、纪元和区域准入仍成立。
- `scene_revision`、`anchor_revision` 和锚点状态仍有效。
- 目标容量预留可升级为正式 arrival lease。
- 存在可用安全落点。

任何一项失败都不能沿用准备阶段的旧结论。

## 9. 安全落位

### 9.1 输入与输出

安全落位求解器输入：

- 当前 `TravelAnchorDefinition`。
- 当前场景/导航 Catalog revision。
- 角色碰撞尺寸和移动能力。
- 动态阻挡、玩家占用和移动结构状态。
- 禁传区、危险区、活动规则和目标 Room 状态。
- `travel_id` 或服务器随机种子。

输出：

- 当前场景 revision 下的一次性落点。
- 目标导航面和朝向。
- 选择的锚点与候补链。
- 落位原因和诊断摘要。

### 9.2 候选规则

求解器至少校验：

- 位置位于锚点 arrival region 内。
- 导航面存在且角色可站立。
- 不在静态 block、禁传、坠落、封闭碰撞或移动结构中。
- 不在副本内部禁止区、解谜后方或一次性剧情空间。
- 与已有实体保留最小间距。
- 目标 Room 未冻结、退休或失去 owner。

候选选择建议以 `travel_id` 作为确定性种子，便于日志复现；不能把摄影师机位作为首选候选。

### 9.3 回退顺序

建议固定回退顺序：

1. 当前锚点 arrival region 内其它候选。
2. 同一 `FallbackGroup` 的邻近锚点。
3. 区域公共入口锚点。
4. 进入容量队列或镜像实例。
5. 返回明确失败，不修改角色位置。

落位后可授予极短 `arrival_protection`。保护由服务端计时，并在玩家攻击、施法、主动交互或离开安全区时提前取消，不能用于抢占目标或主动偷袭。

## 10. Room、路由与角色迁移

### 10.1 Persistent world 入口

正式大世界不应由客户端使用 `RoomJoinReq` 指定 Room。建议新增 `WorldEnterReq`：

1. `game-proxy` 根据已鉴权 `character_id` 查询 character route；没有 route 时选择可接入 game-server。
2. `game-server` 加载 `character_locations`。
3. 世界路由组件根据稳定地址选择 Zone、Room 和实例。
4. 服务端校验位置；失效时回退到最近安全锚点。
5. 服务端加入对应 persistent world Room，并返回权威世界状态。

现有 `RoomJoinReq` 保留给匹配、沙盒和 demo，不删除既有联调能力；服务端应拒绝它创建或加入正式 persistent world policy。

### 10.2 同进程迁移

同一 game-server 内跨 Room 远行不能简单拼接公开 `leave_room + join_room`，需要内部角色迁移事务：

1. 冻结来源角色输入。
2. 在目标 Room 预留成员位和落点。
3. 从来源 AOI 移除角色。
4. 原子切换 Room 索引和运行时位置。
5. 写入关键位置检查点。
6. 向来源和目标 AOI 分别广播离开/进入。
7. 向客户端发送新场景和权威位置。

任何失败都应在来源 Room 恢复角色，不能留下“数据库已到目标、运行时仍在来源”或反向状态。

### 10.3 跨 game-server 迁移

当前 proxy 不支持同连接换 upstream，第一阶段采用 redirect/reconnect：

```text
source game-server
  -> prepare destination arrival lease
  -> freeze source authority
  -> CAS character route to relocating
  -> persist travel session
  -> ServerRedirectPush + close old connection
client
  -> reconnect game-proxy
  -> AuthReq
  -> WorldEnterReq / WorldResumeReq
destination game-server
  -> consume arrival lease
  -> add target Room
  -> commit character location and active route
  -> mark travel Arrived
```

角色级远行不得调用 `FreezeRoomForTransfer`、`ExportRoomTransfer` 等整个 Room 迁移接口。

建议引入逻辑 `WorldRouteCoordinator`。第一阶段可以作为 `game-server + Redis` 内部模块实现；多实例规模和控制面复杂度上升后再拆独立服务。

角色 route 至少需要：

```text
CharacterWorldRoute {
  character_id
  world_id
  zone_id
  room_id
  game_server_id
  route_version
  authority_generation
  relocation_id
  state               # active / relocating / recovering
  expires_at
}
```

现有 game-server online route generation 可用于 fencing。proxy route store 和 game-server online route 当前是两套记录，正式远行开发前必须明确唯一控制面和一致性规则，不能再新增第三套互不协调的 route。

### 10.4 故障恢复

跨服提交阶段的故障不能依赖客户端重试猜测结果。需要持久化 `travel_sessions`，由恢复 worker 处理：

- 来源仍持有权威且 route 未切换：回滚到来源。
- route 已切目标且 destination 已消费 lease：完成到达。
- route 已切目标但 lease 未消费：重建 lease或回退到预先记录的安全来源。
- 来源和目标均声称 active：按 authority generation fencing，旧 generation 必须停止写入。
- 客户端长时间未重连：保留短期目标 route，超时后按明确规则离线落位或回退。

## 11. 留影领域

### 11.1 服务边界

目标态建议新增内部 `impression-service`，承载 UGC 元数据、媒体、审核和推荐读取。它不作为正式玩家直连服务；玩家控制请求仍通过 `game-proxy -> game-server`。

图片字节不应存入 PostgreSQL。建议使用 MinIO/S3 兼容对象存储：

- 短 TTL 上传会话和预签名 URL。
- 服务端限定 MIME、大小、像素尺寸和文件数量。
- 上传完成后校验对象 key、hash 和实际元数据。
- 审核通过后生成受控访问 URL 或 CDN 地址。
- 禁止客户端提交任意第三方图片 URL。

Core NATS 不是可靠持久队列。审核、聚合和推荐事件应先写 PostgreSQL outbox，再由 worker 投递。

### 11.2 留影数据

建议核心表：

```text
impressions {
  impression_id
  author_character_id
  world_id
  epoch_id
  area_key
  anchor_key
  anchor_revision
  media_asset_id
  caption
  visibility
  review_status
  review_version
  travel_status
  spoiler_level
  created_at
  archived_at
  deleted_at
}
```

留影不得保存可直接使用的长期角色落点。允许保存的视觉元数据包括：

- 客户端相机姿态和 FOV，用于展示或后续复现拍摄构图。
- 服务端记录的拍摄时区域、锚点和时间。
- 场景资源 revision 和审核版本。

相机姿态不能进入远行求解器。

辅助表或聚合包括：

- `impression_assets`
- `impression_collections`
- `impression_reports`
- `impression_moderation_actions`
- `impression_events`
- `impression_stats`
- `impression_outbox`

### 11.3 创建流程

```text
client -> ImpressionCaptureIntentReq
game-server
  -> 读取角色权威位置
  -> 校验当前区域允许拍摄/公开
  -> 选择附近合法 anchor_key
  -> 请求 impression-service 创建上传会话
client -> 对象存储上传图片
client -> ImpressionCommitReq
impression-service
  -> 校验对象、媒体元数据、caption 和可见性
  -> 写 impression=pending_review
worker -> 审核/缩略图/去重
impression-service -> approved / rejected / limited
```

客户端提交的当前位置只能用于诊断对比，不能覆盖服务端采样结果。

### 11.4 可见性与准入

第一阶段支持：

- `public`
- `private`

后续 `friends`、`organization` 必须读取正式好友或组织成员关系。聊天群成员关系不能作为宗门授权来源。

每次循迹远行需要依次判断：

1. 请求者能否查看该留影。
2. 留影是否审核通过、未下架且允许远行。
3. 请求者是否满足目标区域和锚点准入。

通过第 1 项不代表通过第 3 项。

### 11.5 统计与因果值

浏览、收藏、举报和实际到访使用幂等事件，不直接在热门行上同步累加所有计数。

- `view`：按角色、留影和时间窗口去重。
- `collect`：唯一约束保证重复收藏幂等。
- `visit`：只能由 `travel_session=Arrived` 产生。
- `report`：按角色、留影和有效状态限制重复提交。
- 因果值和推荐分数由异步聚合计算，不进入远行提交事务。

## 12. 玩家协议设计

### 12.1 消息边界

建议在 `packages/proto/game.proto` 中预留独立玩家消息段，例如 `1700-1799`。实际编号在实现前必须与协议文档统一确认，不复用现有消息号。

世界与远行消息建议包括：

| 消息 | 用途 |
|------|------|
| `WorldEnterReq/Res` | 服务端根据角色位置进入正式大世界 |
| `WorldResumeReq/Res` | 跨服 redirect/reconnect 或提交阶段恢复 |
| `TravelPrepareReq/Res` | 解析目标、首次校验并开始吟唱 |
| `TravelClientReadyReq/Res` | 上报 manifest 预加载状态 |
| `TravelCancelReq/Res` | 主动取消 |
| `TravelStatusReq/Res` | 重连或超时后查询幂等终态 |
| `TravelStatePush` | 吟唱、中断、提交、重连和到达状态 |
| `WorldRelocatedPush` | 下发目标场景、manifest 和权威到达摘要 |

留影消息建议包括：

| 消息 | 用途 |
|------|------|
| `ImpressionCaptureIntentReq/Res` | 服务端定标并创建上传会话 |
| `ImpressionCommitReq/Res` | 提交已上传媒体和展示字段 |
| `ImpressionListReq/Res` | 浏览受控列表 |
| `ImpressionGetReq/Res` | 查看详情和当前可用状态 |
| `ImpressionCollectReq/Res` | 收藏或取消收藏 |
| `ImpressionReportReq/Res` | 举报 |

### 12.2 TravelPrepareReq

建议字段：

```text
request_id
destination_ref oneof {
  impression_id
  anchor_key
}
```

不包含：

- `character_id`
- `world_id`
- 坐标
- Room/实例/host/port
- 客户端计算的权限或战斗状态

`request_id` 必须由客户端生成稳定幂等键，并限制长度和字符集。服务端唯一约束建议为 `(character_id, request_id)`。

### 12.3 TravelStatePush

建议字段：

```text
travel_id
state
server_time_ms
channel_deadline_ms
error_code
retry_after_ms
resolved_anchor_summary
manifest_key
manifest_revision
redirect_required
```

`resolved_anchor_summary` 只能包含用于 UI 展示的世界、区域和锚点信息，不包含最终落点。

### 12.4 错误码

错误码需要稳定、可统计并能区分重试策略，至少包括：

- `TRAVEL_SOURCE_NOT_ALLOWED`
- `TRAVEL_IN_COMBAT`
- `TRAVEL_COMBAT_COOLDOWN`
- `TRAVEL_ALREADY_ACTIVE`
- `TRAVEL_DESTINATION_NOT_VISIBLE`
- `TRAVEL_DESTINATION_NOT_ALLOWED`
- `TRAVEL_DESTINATION_ARCHIVED`
- `TRAVEL_ANCHOR_NOT_FOUND`
- `TRAVEL_ANCHOR_REVISION_CHANGED`
- `TRAVEL_ANCHOR_DISABLED`
- `TRAVEL_ANCHOR_OVERLOADED`
- `TRAVEL_NO_SAFE_ARRIVAL`
- `TRAVEL_INTERRUPTED_BY_MOVEMENT`
- `TRAVEL_INTERRUPTED_BY_ACTION`
- `TRAVEL_INTERRUPTED_BY_DAMAGE`
- `TRAVEL_PRELOAD_TIMEOUT`
- `TRAVEL_ROUTE_PREPARE_FAILED`
- `TRAVEL_RECONNECT_REQUIRED`
- `TRAVEL_RECOVERY_PENDING`
- `TRAVEL_REQUEST_CONFLICT`

失败响应可以带候补锚点或入口提示，但客户端不能自行把候补转换成坐标。

## 13. 数据一致性与存储分工

| 状态 | 权威存储 |
|------|----------|
| 在线角色移动和战斗 | 当前 owner `game-server` Room runtime |
| 角色稳定位置检查点 | PostgreSQL `character_locations` |
| 远行会话和提交结果 | PostgreSQL `travel_sessions` |
| 在线 route、arrival lease、容量预留、排队 | Redis，带 TTL、version 和 fencing |
| 场景与公共锚点定义 | CSV/场景资源构建出的 revision Catalog |
| 锚点纪元/运营状态 | PostgreSQL，热点快照可缓存 Redis |
| 留影元数据、审核、收藏、举报 | impression-service PostgreSQL |
| 图片和缩略图 | MinIO/S3 兼容对象存储 |
| 异步审核、统计、推荐事件 | PostgreSQL outbox + worker |

Redis 数据丢失不能导致角色被复制或永久卡在两个 Room。跨服提交以持久 `travel_session`、location version 和 authority generation 共同裁决。

## 14. 安全与滥用边界

### 14.1 客户端不可信输入

服务端不能信任：

- 拍摄坐标和附近锚点。
- 当前战斗、脱战和受击状态。
- 目标区域是否已解锁。
- 上传媒体类型、尺寸、hash 和审核结果。
- 预加载成功等同远行批准。
- 客户端本地时间和吟唱完成时间。

### 14.2 防重放与幂等

- 所有写请求限制 payload 大小和频率。
- 上传 token、arrival lease 和远行预留使用短 TTL 一次性凭证。
- `TravelPrepareReq`、留影 commit、收藏和举报均使用幂等键或唯一约束。
- 同一角色同一时刻最多一个 active travel。
- route mutation 必须校验 authority generation。
- 旧锚点 revision、旧 manifest revision 和已消费 arrival token 必须拒绝重放。

### 14.3 隐私和内容安全

- 列表和详情响应不暴露作者或目标的裸坐标、Room、实例和内部 route。
- 私人留影使用服务端 ACL，不依赖不可猜 ID 本身保密。
- 删除默认软删除并保留审核与审计记录；媒体按留存策略异步回收。
- 举报、下架、恢复和管理员修改均写审计。
- 日志不得记录上传签名、访问 token、ticket、图片正文或敏感 caption 全文。

## 15. 热点、容量和镜像

第一阶段可以先做简单锚点并发预留和邻近回退，但数据模型必须允许后续扩展：

- 锚点每秒到达速率限制。
- Zone/Room 最大在线人数。
- arrival region 同时落位上限。
- Redis 有界队列和预计等待时间。
- 同主题邻近锚点。
- 安全区镜像实例。
- recommendation 读取锚点压力快照，降低过载目标权重。

镜像分流只改变运行时实例，不改变留影绑定的稳定 `anchor_key`，也不修改作者、收藏或历史到访归属。

## 16. 管理后台与可观测性

### 16.1 管理能力

`admin-api + admin-web` 后续至少需要：

- 留影列表、详情、审核、下架、恢复和举报处理。
- 媒体 hash、审核版本和存储状态查询。
- 锚点定义、当前 revision、运行状态和重定向查询。
- 锚点启用、禁用、重校和容量 override。
- 远行会话按角色、锚点、状态、错误码和时间查询。
- 卡在 `Committing/AwaitingReconnect/Recovering` 状态的恢复操作。
- 角色当前位置、最近安全锚点和 location version 查询。

所有写操作需要独立 RBAC 权限和 PostgreSQL 审计。

### 16.2 指标

建议指标：

- `travel_prepare_total{result,error_code}`

- `travel_duration_seconds{stage}`
- `travel_recovery_total{result}`
- `travel_anchor_reservations{anchor_key}`
- `travel_anchor_queue_depth{anchor_key}`
- `travel_safe_arrival_attempts{result,fallback_level}`
- `impression_upload_total{result}`
- `impression_review_backlog`
- `impression_report_total{result}`
- `impression_visit_events_total`

日志应携带 `trace_id`、`travel_id`、`character_id`、稳定 anchor、revision、route version 和结果，不记录秘密或媒体正文。

## 17. 兼容策略

### 17.1 协议兼容

客户端尚未开发本功能，因此不提供“精确坐标留影传送”的旧协议兼容。

仍需遵守现有协议维护规则：

- 新消息使用新编号，不复用旧编号。
- Protobuf 已发布字段号不改变语义。
- 同步更新 `packages/proto`、game-server 生成代码、`tools/mock-client`、外部 mybevy 和协议文档。
- 新消息携带明确 schema/revision 字段，不能长期依赖无结构 JSON。

### 17.2 房间兼容

- 匹配、沙盒和 demo 继续使用现有 `RoomJoinReq`。
- 正式 persistent world 使用新 `WorldEnterReq`，不允许客户端选择 Room。
- `RoomReconnectReq` 保留现有对局恢复语义；大世界跨服恢复使用明确的 World/Travel resume 语义，不在空 body 上叠加不可见行为。

### 17.3 场景兼容

- `ScenePortal` 继续表达固定入口，不自动升级为远行锚点。
- 现有二维网格可支持单层公共锚点第一阶段。
- 如果正式场景存在桥梁、洞穴、多楼层、攀爬或飞行，必须先补分层导航面或等价服务端拓扑；单层二维网格不能安全表达重叠空间落位。

### 17.4 跨服兼容

- 第一阶段复用当前 `ServerRedirectPush` 和 reconnect 能力。
- 不把同连接 upstream swap 作为第一阶段前置条件。
- 后续 proxy 演进为 L7 session relay 后，可以在不改变稳定 WorldAddress 和 travel session 的前提下替换传输层迁移方式。

## 18. 分阶段开发路线

### 阶段 0：契约与数据基础

- 确认 WorldAddress、锚点、位置和 travel session schema。
- 明确 world/epoch/area 的权威来源。
- 新增正式专题 checklist。
- 分配协议消息号并同步协议文档。

完成定义：数据和协议评审通过，但不要求玩家可远行。

### 阶段 1：正式大世界入口与位置所有权

- 新增 `character_locations`。
- 将位置写入权迁移到 game-server。
- 新增 `WorldEnterReq/Res`。
- persistent world Room 改为服务端分配。
- 登录和断线恢复执行位置校验与安全回退。

完成定义：角色可通过正式入口进入服务端指定的 persistent world，并恢复权威位置。

### 阶段 2：公共锚点与同进程远行

- 新增 TravelAnchor Catalog 和 revision。
- 新增统一 CharacterRuntimeState 和控制锁。
- 实现远行状态机、双重校验、吟唱和打断。
- 实现安全落位求解器。
- 实现同 game-server 跨 Room 迁移。

完成定义：系统锚点之间可以稳定远行，重复请求、受击和失败不会复制或丢失角色。

### 阶段 3：跨 game-server 远行

- 明确统一 character world route 控制面。
- 实现 arrival lease、route CAS、authority fencing。
- 实现 redirect/reconnect 和 WorldResume。
- 实现 travel recovery worker 和多进程故障演练。

完成定义：跨实例远行在断线、重复请求和单点重启下能完成或明确回滚。

### 阶段 4：因果留影最小闭环

- 新增 impression-service 和对象存储。
- 实现权威定标、上传、commit、审核状态和详情。
- 实现 public/private、收藏和举报。
- 到访事件只由远行 Arrived 生成。

完成定义：审核通过的留影可以被浏览，并在当前准入允许时触发循迹远行。

### 阶段 5：热点和推荐

- 锚点压力快照、限流、排队、邻近回退和镜像。
- 留影统计异步聚合和基础推荐。
- 管理后台审核、锚点和恢复页面。

完成定义：热门留影不会把单个锚点或 Room 无界打满，失败有可解释替代路径。

### 阶段 6：扩展能力

- 正式好友/宗门 ACL。
- 私人洞府访客锚点。
- 纪元归档和锚点重校。
- 受控跨界展示与后续跨界移动设计。
- 同连接 upstream swap。

## 19. 测试与验收要求

### 19.1 单元测试

- 状态机合法和非法转换。
- 幂等 request 重放与 fingerprint 冲突。
- 开始/完成双重准入。
- movement/action/damage 中断。
- 锚点 revision 变化、禁用和重定向。
- 安全落位候选、动态阻挡和回退顺序。
- location version 与旧 authority 写入拒绝。
- 留影可见性与目的地准入独立判断。

### 19.2 集成测试

- WorldEnter 加载有效位置和非法位置回退。
- 同进程跨 Room 成功、目标加入失败和来源恢复。
- 预加载未完成、超时和断线重连。
- 留影下架后不能新建远行，进行中会话按当前规则重新校验。
- 对象上传伪造 MIME、超限、hash 不一致和重复 commit。
- 收藏、举报、到访事件幂等。

### 19.3 多进程故障测试

- source freeze 前后崩溃。
- route CAS 前后崩溃。
- redirect 已发但客户端未重连。
- destination consume lease 前后崩溃。
- Redis reservation 丢失。
- 旧 game-server 延迟写位置或 route。
- proxy route store reload 和陈旧 route。

### 19.4 联调验收

- `tools/mock-client` 覆盖 world enter、吟唱、取消、中断、到达、跨服恢复和错误码。
- 外部 mybevy 覆盖 manifest 预加载、场景切换和权威到达位置。
- 联调记录包含 room、route version、travel id、anchor revision 和最终状态。
- 启动 PostgreSQL、Redis、NATS、game-server、game-proxy、auth-http、对象存储或外部客户端前，仍须按仓库协作约定先取得用户确认。

## 20. 已确认决策记录

| 编号 | 决策 | 原因 |
|------|------|------|
| `TRAVEL-001` | 留影不保存可直接使用的长期裸坐标 | 防止跨版本、跨纪元和跨实例坐标失效或越权 |
| `TRAVEL-002` | 留影可见性与肉身准入分离 | 分享不能绕过剧情、世界层级和私人权限 |
| `TRAVEL-003` | game-server 是远行和落位最终权威 | 内容服务、客户端和推荐服务均不掌握角色运行时状态 |
| `TRAVEL-004` | persistent world Room 由服务端分配 | 防止客户端选择实例、绕过准入或加入内部 Room |
| `TRAVEL-005` | 跨服第一阶段使用 redirect/reconnect | 当前 proxy 不支持同连接 upstream swap |
| `TRAVEL-006` | 新导航字段采用 XZ + surface/floor 语义 | 对齐 mybevy Y-up 3D 表现，避免继续扩大二维 x/y 歧义 |
| `TRAVEL-007` | ScenePortal 与 TravelAnchor 分离 | 两者生命周期、状态、容量和权限语义不同 |
| `TRAVEL-008` | Core NATS 不作为 UGC 可靠任务队列 | 审核、统计和推荐需要持久 outbox 与可重试消费 |
| `TRAVEL-009` | 第一阶段不支持 friends/organization ACL | 当前没有正式好友图谱和宗门成员权威来源 |

## 21. 待确认问题

以下问题在正式拆 checklist 前需要产品、场景、客户端和服务端共同确认：

- `epoch_id` 的生成、切换和权威存储由哪个服务负责。
- `area_key` 与现有 `SceneRegion`、Zone、Room 的映射关系。
- 正式场景是否需要多层导航面、桥梁、洞穴、攀爬或飞行。
- 吟唱时长、脱战等待、失败冷却和预加载宽限期。
- 目标过载时默认排队、自动重定向还是返回用户确认。
- 到达保护时长和取消保护的主动行为集合。
- 留影图片大小、格式、缩略图规格、审核提供方和媒体留存周期。
- spoiler 等级由作者、场景配置、审核系统还是组合规则决定。
- 好友和宗门服务的正式边界与数据源。
- travel session 的离线超时后选择目标落位还是来源回滚。
- 是否允许同一账号多个角色同时在线；当前 P0 仍是一账号单在线连接。

这些问题未确认前，可以开发底层地址、位置、锚点、状态机和幂等基础，但不应把临时数值或权限假设固化为公开协议。
