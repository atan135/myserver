# 统一资产阶段 7：入口迁移与回滚

本文记录现有背包入口迁移到统一角色资产事务后的运行边界。当前代码和配置优先；奖励来源接入、流水查询和运营后台能力由后续阶段负责。

## 玩家协议

- `ItemEquipReq`、`ItemUseReq`、`ItemDiscardReq` 与 `WarehouseAccessReq` 只表达当前已鉴权角色的操作意图。
- 处理器不再修改 `PlayerData` 或 `ItemContainer`，而是调用 `PlayerManager` 的单角色资产事务。事务在持有角色锁时加载快照、执行预检、持久化，并且只在提交后发布在线副本和推送。
- `ItemAddReq/Res` 的 `1407/1408` 永久保留为废弃号。服务端返回 `MESSAGE_TYPE_DEPRECATED`，共享 protobuf、mock-client 和外部客户端都不能再依赖其 payload 或复用该消息号。
- `UseEffect=CharacterElementChange` 仍因角色四属性使用独立存储而返回 `ASSET_CROSS_STORE_ATOMICITY_UNAVAILABLE`；不得先消耗道具再写外部状态。

## 管理与邮件入口

- 普通玩家没有本地构造物品入口。测试或紧急操作只能通过 game-server admin 管理凭证进入，并由 admin audit 记录操作者和角色目标。
- `source=gm` 使用容量预演、请求指纹和 `character_inventory_grants` 兼容记录，且要求 `GM_ASSET_CONSTRUCTION_ENABLED=true`。
- `source=gm-emergency-correction` 还要求显式 actor 与 `GM_EMERGENCY_ASSET_CORRECTION_ENABLED=true`。它和普通奖励来源分开，避免将资产纠正混入玩家可触发奖励链路。
- `source=mail-claim` 继续使用相同的容量预演资产事务、原 request ID、query-first 和 lease fencing；其开关是 `MAIL_CLAIM_ASSET_TRANSACTIONS_ENABLED`，默认开启，兼容 v1 请求与 `character_inventory_grants`。
- 成就、任务、战斗、场景等普通奖励来源在后续阶段接入 `RewardDeliveryService`，并不得再直接调用 grant 或修改背包快照。

## 开关与回滚

| 入口 | 开关 | 默认值 | 关闭结果 | 对账与回滚 |
| --- | --- | --- | --- | --- |
| 玩家装备、使用、丢弃、仓库存取 | `PLAYER_ASSET_TRANSACTIONS_ENABLED` | `true` | `ASSET_TRANSACTION_MIGRATION_DISABLED`，不修改资产 | 按 request/角色快照 revision 对账；关闭只阻断新写入，绝不回退到旧旁路写入 |
| 邮件附件领取 | `MAIL_CLAIM_ASSET_TRANSACTIONS_ENABLED` | `true` | 管理 grant 返回稳定未提交失败，邮件仍保留 | 保留原 mail、附件指纹和 claim request；恢复开关后用同一 request query-first 重试 |
| 管理构造物品 | `GM_ASSET_CONSTRUCTION_ENABLED` | `false` | `ASSET_TRANSACTION_MIGRATION_DISABLED` | 对账 `character_inventory_grants`、资产 revision 和 admin audit；恢复只重新开启受审计通道 |
| 紧急资产纠正 | `GM_EMERGENCY_ASSET_CORRECTION_ENABLED` | `false` | `ASSET_TRANSACTION_MIGRATION_DISABLED` | 需具名 actor、admin audit 和独立 source；关闭后不允许降级到普通 GM 或玩家入口 |

关闭开关属于安全回滚：它停止新的写操作并保留已提交结果，不允许恢复历史 `ItemAddReq` 或任何直接 JSONB / `PlayerData` 写入路径。`npm run check:asset-write-boundaries` 会检查这一边界。
