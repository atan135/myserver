# game-server 帧同步与房间生命周期设计

## 1. 文档定位

本文描述 `apps/game-server` 当前已经落地的房间运行时、帧同步、断线恢复、观战和位移校正框架。

本文不是协议字段主文档：

- 服务边界以 [整体架构](../总览/整体架构.md) 为准。
- 消息号、包头和 proto 字段以 [协议设计](../协议与客户端/协议设计.md) 与 `packages/proto/game.proto` 为准。
- 更新、灰度、迁移口径以 [游戏服更新策略拆分](./游戏服更新策略拆分.md) 和 [空房接管式灰度规范](./空房接管式灰度规范.md) 为准。

当前代码优先于本文。如果本文与 `apps/game-server/src` 冲突，应以代码为准并同步修正文档。

## 2. 当前实现结论

当前 `game-server` 已经从早期“连接处理 + 简单房间逻辑”演进为 `RoomManager + RoomRuntimePolicy + RoomLogic` 的房间运行时框架。

已经落地：

- 房间级 tick task 与动态 fps。
- `PlayerInputReq.frame_id` 输入帧校验与缓存。
- 严格等待 / 乐观推进两种输入等待策略。
- 缺帧补偿策略：空输入、重复上一帧、连续缺帧后标记离线。
- `FrameBundlePush` 按帧广播输入集合和定时快照。
- `RoomFrameRatePush` 在房间运行 fps 实际变化时主动广播当前 fps。
- `RoomSnapshot.game_state` 轻量业务状态快照。
- 观战者加入与恢复数据。
- 断线重连恢复：snapshot、当前帧、最近输入、等待帧输入、`input_delay_frames`、移动恢复状态。
- 离线成员保留和 `offline_ttl_secs` 清理。
- `movement_demo` 的服务端权威位移、AOI 过滤、`MovementSnapshotPush` 和 `MovementRejectPush`。
- `ui_touch_room` 的 UI 触控输入帧同步，使用 `PlayerInputReq` / `FrameBundlePush` 同步 `ui_touch` 输入。
- `robot_sync_room` 的机器人同步输入帧转发，使用 `PlayerInputReq` / `FrameBundlePush` 同步 `robot_move`，服务端只校验 payload 和记录最近输入摘要，不计算机器人位置。
- `combat_demo` 的基础战斗逻辑和快照广播。

仍未完整落地：

- 完整增量状态同步。当前主链路仍是输入帧同步，位移校正是辅链路。
- OB 的暂停、快进、回放和回放持久化。
- 可用于跨服迁移的完整 room transfer。`RoomLogic::get_serialized_state()` 只是轻量快照钩子，不等同于灰度迁移 payload。

## 3. 代码分层

核心文件：

- `apps/game-server/src/core/room/mod.rs`
  - `Room`、`RoomMemberState`、`PlayerInputRecord`、输入缓存、输入历史、快照。
- `apps/game-server/src/core/runtime/room_manager.rs`
  - 房间创建、加入、离开、断线、重连、观战、tick task、fps 计算、输入推进、广播、清理。
- `apps/game-server/src/core/runtime/room_policy.rs`
  - 内置房间运行策略。
- `apps/game-server/src/core/logic/room_logic.rs`
  - `RoomLogic` 与 `RoomLogicBroadcast`。
- `apps/game-server/src/core/logic/factory.rs`
  - 框架层只依赖 `RoomLogicFactory` trait。
- `apps/game-server/src/gameroom/factory.rs`
  - 游戏侧按 `policy_id` 映射具体 `RoomLogic`。
- `apps/game-server/src/gameroom/*`
  - 具体玩法房间逻辑，如 `movement_demo`、`ui_touch_room`、`robot_sync_room`、`combat_demo`、`persistent_world`、`disposable_match`、`sandbox`、`test_room`。

依赖方向：

```text
server.rs / core/service
  -> RoomManager
    -> Room + RoomRuntimePolicy + RoomLogic trait
      -> GameRoomLogicFactory
        -> gameroom/* 具体实现
```

`core` 不直接反向依赖 `gameroom`，具体玩法装配放在游戏侧。

## 4. RoomRuntimePolicy

当前 `RoomRuntimePolicy` 不只是 fps 和销毁配置，也包含输入同步、重连保留、位移校正和 AOI 参数。

主要字段：

| 字段 | 含义 |
|------|------|
| `policy_id` | 房间策略 id，也用于选择具体 `RoomLogic` |
| `max_members` | 最大成员数，包含玩家和观战者 |
| `min_start_players` | 开局所需最少成员数 |
| `silent_room_fps` | 无在线成员时的低频 fps |
| `idle_room_fps` | 等待阶段 fps |
| `active_room_fps` | 游戏中普通 fps |
| `busy_room_fps` | 游戏中高负载 fps |
| `busy_room_player_threshold` | 进入高负载 fps 的在线人数阈值 |
| `destroy_enabled` | 是否允许销毁房间 |
| `destroy_when_empty` | 空房后是否进入销毁流程 |
| `empty_ttl_secs` | 空房保留时长 |
| `retain_state_when_empty` | 空房时是否保留房间状态 |
| `offline_ttl_secs` | 离线成员保留时长 |
| `snapshot_interval_frames` | 定时快照间隔 |
| `input_delay_frames` | 客户端最多可提前提交的帧数 |
| `wait_timeout_ms` | 当前帧等待输入的超时时间 |
| `wait_strategy` | `Strict` 或 `Optimistic` |
| `missing_input_strategy` | `Empty`、`RepeatLast` 或 `DropAfterMisses` |
| `movement_correction_interval_frames` | 位移权威校正周期 |
| `movement_correction_threshold` | 位移误差强校正阈值 |
| `movement_aoi_enabled` | 是否启用位移校正 AOI 过滤 |
| `movement_aoi_radius` | AOI 半径 |
| `movement_control_stop_frames` | 连续缺少真实移动控制输入后自动停步的帧数，`0` 表示禁用 |

当前内置策略：

| 策略 | 当前用途 |
|------|----------|
| `default_match` | 默认对局策略，严格等待输入，缺帧重复上一帧 |
| `persistent_world` | 常驻世界策略，不按空房销毁，开启 AOI |
| `disposable_match` | 临时对局策略，空房可销毁 |
| `sandbox` | 沙盒策略，乐观推进，缺帧补空输入 |
| `movement_demo` | 位移同步联调策略，乐观推进，开启 AOI 与权威校正 |
| `ui_touch_room` | UI 触控同步策略，乐观推进，缺帧补空输入，通过 `ui_touch` payload 同步归一化触控采样 |
| `robot_sync_room` | MyBevy `arena.robot_sync` 机器人同步策略，乐观推进，缺帧补空输入，只校验并转发 `robot_move` |
| `combat_demo` | 战斗联调策略，乐观推进，开启战斗快照 |

未知策略会通过 `RoomPolicyRegistry` 回退到默认策略。

`robot_sync_room` 当前显式注册在 `RoomPolicyRegistry::default()` 中，不应依赖未知策略回退。关键参数：

| 字段 | 当前值 |
|------|--------|
| `max_members` | `32` |
| `min_start_players` | `1` |
| `allow_join_in_game` | `true` |
| `silent_room_fps` / `idle_room_fps` / `active_room_fps` / `busy_room_fps` | `1` / `10` / `20` / `20` |
| `input_delay_frames` | `2` |
| `wait_timeout_ms` | `100` |
| `wait_strategy` | `Optimistic` |
| `missing_input_strategy` | `Empty` |
| `snapshot_interval_frames` | `20` |
| `movement_correction_interval_frames` | `0` |
| `movement_aoi_enabled` | `false` |
| `movement_control_stop_frames` | `0` |

## 5. 帧推进流程

### 5.1 输入进入

玩家通过 `PlayerInputReq` 提交：

- `frame_id`
- `action`
- `payload_json`

`RoomManager::accept_player_input` 会执行：

1. 校验房间和成员状态。
2. 拒绝观战者输入。
3. 拒绝离线成员输入。
4. 拒绝 `frame_id <= current_frame` 的过期输入。
5. 拒绝超过 `current_frame + input_delay_frames` 的过远未来帧。
6. 对同一玩家同一帧输入执行 upsert，后到输入会替换前一条。
7. 输入被框架接受后，调用 `RoomLogic::on_player_input`，让玩法层做统计、审计、临时缓存等非权威记录。

`on_player_input` 不应修改玩法权威状态。移动、战斗、技能等会影响房间结果的状态变更必须在 `RoomLogic::on_tick(frame_id, fps, inputs)` 中基于已解析出的帧输入集合统一完成。这样可以避免过期帧、过远未来帧或其他被拒绝的输入在收包阶段提前产生副作用，也便于回放、重连恢复和房间迁移。

`robot_sync_room` 当前只接受：

```text
action = "robot_move"
payload_json = {"version":1,"seq":1,"botTick":1,"dirX":1000,"dirY":0,"speed":10000}
```

payload 校验规则：

- payload 最大 `256` bytes，必须是 JSON object。
- 字段必须精确为 `version`、`seq`、`botTick`、`dirX`、`dirY`、`speed`，采用 camelCase，不允许多字段或少字段。
- 所有字段必须是整数；`version` 当前只能为 `1`。
- `seq` 和 `botTick` 必须在 `u32` 范围内。
- `dirX` / `dirY` 必须在 `-1000..1000`；当前服务端校验逐字段范围，不在服务端计算方向向量长度。
- `speed` 当前服务端合法范围为 `0..10000`。MyBevy 内置 bot 默认发送 `10000`。

常见拒绝码：

| 错误码 | 含义 |
|--------|------|
| `INVALID_ROBOT_MOVE_ACTION` | action 不是 `robot_move` |
| `INVALID_ROBOT_MOVE_JSON` | payload 不是合法 JSON |
| `ROBOT_MOVE_PAYLOAD_TOO_LARGE` | payload 超过 256 bytes |
| `ROBOT_MOVE_PAYLOAD_NOT_OBJECT` | payload 不是 object |
| `ROBOT_MOVE_FIELDS_MISMATCH` | 字段缺失或包含未知字段 |
| `ROBOT_MOVE_FIELD_NOT_INTEGER` | 字段不是整数 |
| `INVALID_ROBOT_MOVE_VERSION` / `UNSUPPORTED_ROBOT_MOVE_VERSION` | version 字段缺失、越界或不是 `1` |
| `ROBOT_MOVE_SEQ_OUT_OF_RANGE` | seq 越界 |
| `ROBOT_MOVE_BOT_TICK_OUT_OF_RANGE` | botTick 越界 |
| `ROBOT_MOVE_DIR_OUT_OF_RANGE` | dirX 或 dirY 越界 |
| `ROBOT_MOVE_SPEED_OUT_OF_RANGE` | speed 大于 `10000` |

### 5.2 每帧推进

`RoomManager::process_room_tick` 是核心推进点：

1. 非 `InGame` 房间不推进帧。
2. 没有在线角色输入参与者时暂停推进并保留等待状态。
3. 当前等待帧为 `current_frame + 1`。
4. 按策略判断是否可以推进：
   - `Strict`：所有在线角色输入都到齐，或等待超时。
   - `Optimistic`：所有输入到齐、至少一个输入到达，或等待超时。
5. 对缺失输入按 `missing_input_strategy` 补齐。
6. 更新 `current_frame`。
7. 调用 `RoomLogic::on_tick(frame_id, fps, inputs)`。
8. 取出玩法层 `take_pending_broadcasts()`。
9. 写入最近 300 帧输入历史。
10. 每 `snapshot_interval_frames` 生成一次 `RoomSnapshot`。
11. 广播 `FrameBundlePush`。

### 5.3 缺帧策略

| 策略 | 行为 |
|------|------|
| `Empty` | 缺失玩家使用空输入 |
| `RepeatLast` | 缺失玩家复用上一帧已应用输入；没有上一帧则补空输入 |
| `DropAfterMisses` | 缺失时补空输入，连续缺失达到阈值后标记玩家离线 |

当前连续缺帧离线阈值在代码中为 `MAX_MISSING_INPUT_STREAK_BEFORE_OFFLINE = 3`。

## 6. 房间生命周期

### 6.1 创建与开局

- `join_room` 会按请求的 `policy_id` 创建或加入房间。
- 新房创建时同步创建对应 `RoomRuntime`。
- `start_game` 校验 owner、人数和 ready 状态，通过后进入 `InGame`，清空运行时输入并启动 tick task。

### 6.2 动态 fps

`RoomManager::compute_room_fps` 当前按在线成员数量和房间阶段选择 fps：

- 在线成员数为 0：`silent_room_fps`
- `Waiting`：`idle_room_fps`
- `InGame` 且在线人数达到阈值：`busy_room_fps`
- 其他 `InGame`：`active_room_fps`

fps 变化会更新 `RoomRuntime.current_fps`，同时 `FrameBundlePush.fps` 会继续携带该帧 fps。
当 `previous_fps != target_fps` 时，`RoomManager` 会向该 room 当前在线成员广播 `RoomFrameRatePush { room_id, fps, reason }`，`reason` 当前固定为 `runtime_policy_changed`，表示由运行时策略、房间阶段或在线成员数量变化导致的 fps 重算结果发生变化。fps 未实际变化时不会重复推送。

`FrameBundlePush.fps` 仍保留，客户端可继续以每帧携带的 fps 作为帧同步上下文；`RoomFrameRatePush` 只用于提前感知运行时帧率变化，不替代 `FrameBundlePush.fps`。

### 6.3 离开、断线与清理

- `leave_room` 表示正常离房，会标记离线、调用玩法 hook，并让房间回到等待态。
- `disconnect_room_member` 表示连接断开，会保留 `InGame` 运行态，供断线重连恢复。
- cleanup task 会按 `offline_ttl_secs` 移除超时离线成员。
- 当策略允许销毁且房间满足空房条件时，cleanup task 会标记并移除房间。
- `persistent_world` 这类策略通过 `destroy_enabled=false` 和 `retain_state_when_empty=true` 保留状态。

### 6.4 观战与重连

观战者：

- 通过 `RoomJoinAsObserverReq` 加入。
- 成员角色为 `Observer`。
- 可接收快照和帧广播。
- 不能发送输入。

断线重连：

- `RoomReconnectReq` 根据离线成员恢复连接。
- 返回 `RoomRecoveryState` 对应协议响应字段：
  - `snapshot`
  - `current_frame_id`
  - `recent_inputs`
  - `waiting_frame_id`
  - `waiting_inputs`
  - `input_delay_frames`
  - `movement_recovery`

## 7. 协议关系

当前主链路协议：

| 消息 | 用途 |
|------|------|
| `PlayerInputReq/Res` | 玩家提交帧输入 |
| `FrameBundlePush` | 服务端按帧广播输入集合和可选快照 |
| `RoomFrameRatePush` | 房间 fps 实际变化时主动广播当前 fps |
| `RoomStatePush` | 房间状态变化广播 |
| `RoomJoinAsObserverReq/Res` | 观战加入与恢复 |
| `RoomReconnectReq/Res` | 断线重连与恢复 |
| `MovementSnapshotPush` | 位移权威快照或校正 |
| `MovementRejectPush` | 位移输入或客户端状态被拒绝后的校正 |
| `CombatActionReq/Res` | 战斗 demo 行为请求 |
| `CombatSnapshotPush` | 战斗 demo 状态快照 |

已定义但未形成完整运行链路：

| 消息 | 当前状态 |
|------|----------|
| `ServerRedirectPush` | 协议已定义并可通过已鉴权 admin/internal 控制入口触发；客户端自动断线重连与端到端 route 切换仍未闭环 |
| `FreezeRoomForTransfer*` / `ExportRoomTransfer*` / `ImportRoomTransfer*` / `RetireTransferredRoom*` | 已接入 `game-server` 已鉴权 internal/admin 通道，完成空房/全员离线 room 的 freeze/export/import/retire 最小闭环；不包含 proxy route 切换、同连接迁移或完整玩法 payload |
| `GetRolloutDrainStatus*` | 已接入 `game-server` 已鉴权 internal/admin 通道，返回本进程 room drain 快照、有限 route 样本和当前连接数；`empty_since_ms` 在当前实现中表示本进程内已空置时长，不是绝对时间戳；proxy / 控制面结合该真实状态自动停旧服仍未闭环 |

## 8. 位移同步

当前推荐并已部分落地的模型是：

```text
输入帧同步为主链路
低频权威位置校正为辅链路
```

`movement_demo` 当前负责：

- 根据输入更新服务端权威移动状态。
- 根据 `movement_correction_interval_frames` 周期性生成权威快照。
- 根据 `movement_correction_threshold` 对异常漂移触发强校正。
- 根据 `movement_aoi_enabled` 和 `movement_aoi_radius` 过滤下发对象。
- 根据 `movement_control_stop_frames` 在玩家持续缺少真实移动控制输入时自动停步，并用 `CONTROL_TIMEOUT` 校正通知客户端。
- 在重连或观战恢复时提供 `movement_recovery`。

客户端不应把当前位置当作权威状态直接提交给服务端。服务端应以输入和场景规则计算权威位置，再通过 `MovementSnapshotPush` 或 `MovementRejectPush` 做校正。

当前简化移动模型仍是固定速度：`MoveDir` 表示持续移动控制，`MoveStop` 表示显式停止，二者都会刷新移动控制状态；`FaceTo` 只改变朝向，不会被视为移动控制。框架补帧或复用上一帧产生的合成输入也不会刷新移动控制状态，因此客户端需要持续发送真实移动控制输入；超过策略阈值后，服务端会把该玩家权威状态置为停止。

更完整的算法背景见 [网络延迟补偿设计](./网络延迟补偿设计.md)。

## 9. 机器人同步房间

`robot_sync_room` 是为 MyBevy `arena.robot_sync` 场景提供的第一版远端 authority 房间。它和 `movement_demo` 的边界不同：

- `movement_demo` 在服务端计算权威位置，并可下发 `MovementSnapshotPush` / `MovementRejectPush`。
- `robot_sync_room` 不计算机器人坐标、不做边界 clamp、不做碰撞、不发送 movement snapshot；它只校验并转发 `robot_move` 输入，让客户端通过同一批 `FrameBundlePush` 做确定性 replay。

实现位置：

- policy：`apps/game-server/src/core/runtime/room_policy.rs`
- room logic：`apps/game-server/src/gameroom/robot_sync_room/mod.rs`
- factory 映射：`apps/game-server/src/gameroom/factory.rs`
- mock-client 场景：`tools/mock-client/src/scenarios/robot-sync.js`

`RobotSyncRoomLogic::on_tick` 当前只记录最近真实 `robot_move` 输入摘要，忽略合成空输入和其他 action。`get_serialized_state()` 返回轻量 debug JSON，包含：

```json
{
  "logicType": "robot_sync_room",
  "roomId": "robot-sync-room",
  "tickCount": 12,
  "lastFrame": 34,
  "recentInputs": [
    {
      "frameId": 34,
      "characterId": "chr_xxx",
      "seq": 7,
      "botTick": 7,
      "dirX": 1000,
      "dirY": 0,
      "speed": 10000
    }
  ]
}
```

该 snapshot 只用于 debug / recovery 可见性，不是机器人位置权威状态，也不是 room transfer 完整 payload。

联调时服务端侧至少关注这些字段：

- `room_id`、`policy_id`、`character_id`、账号 `player_id`、`guest_id`
- `frame_id`、`action=robot_move`、`input_delay_frames`
- `PlayerInputRes.ok`、`errorCode`
- `FrameBundlePush.frame_id`、`FrameBundlePush.fps`、输入列表中的 `payload_json`
- payload 校验错误码和拒绝原因

`npm run flow:mock-client -- --scenario robot-sync-room` 或 `node tools/mock-client/src/index.js --scenario robot-sync-room` 会覆盖双客户端加入同一 room、ready、start、发送合法 `robot_move`、两端收到包含两名角色输入的 `FrameBundlePush`，以及非法 action、非法 JSON、方向越界、速度越界四类拒绝。

## 10. 新房间类型开发流程

新增房间类型时建议按以下顺序：

1. 在 `apps/game-server/src/gameroom/<room_type>/mod.rs` 实现 `RoomLogic`。
2. 在 `apps/game-server/src/gameroom/mod.rs` 暴露模块。
3. 在 `GameRoomLogicFactory::create(policy_id)` 中注册映射。
4. 如果需要独立运行参数，在 `RoomRuntimePolicy` 中新增模板并注册到 `RoomPolicyRegistry`。
5. 如果需要专属消息入口，在 `core/service/` 或 `gameservice/` 中按领域新增 handler。
6. 如果需要新 CSV 表，在 `gameconfig/registry.rs` 装配，不要把表装配写回框架层。

房间运行默认值当前由 `RoomRuntimePolicy` 承载，新增玩法至少应明确这些字段：

- 人数: `max_members`、`min_start_players`
- tick: `silent_room_fps`、`idle_room_fps`、`active_room_fps`、`busy_room_fps`
- 输入: `input_delay_frames`、`wait_timeout_ms`、`wait_strategy`、`missing_input_strategy`
- 同步: `snapshot_interval_frames`
- 生命周期: `destroy_enabled`、`destroy_when_empty`、`empty_ttl_secs`、`offline_ttl_secs`

`RoomManager` 通过共享 `RoomPolicyRegistry` 读取这些默认值。后续如果把房间类型默认值迁到 `RoomType.csv` 或同类玩法表，应在 `RuntimeGameConfig::build(...)` 中构建新的 `RoomPolicyRegistry`，并沿用 CSV reload 的原子替换和失败保留旧版本语义。

约束：

- `core` 不直接 import `gameroom`、`gameservice` 或 `gameconfig`。
- `RoomLogic` 方法签名应谨慎变更，优先通过抽象补上下文。
- 通用移动、战斗、场景能力优先沉淀到 `core/system/`。
- 具体玩法逻辑只做规则编排和系统装配。

## 11. Room Transfer Timer 契约边界

跨服迁移的 runtime timer/scheduler 状态不属于 `get_serialized_state()` 的轻量 `game_state` 快照。当前 `RoomLogicTransferState.timer_state_json` 保留字符串兼容字段，但内容应优先使用 `core::logic::RoomRuntimeTimerTransferState` 序列化为 `room-transfer.runtime-timer-state.v1`；外层 `RoomTransferPayload.runtime_timers_json` 继续使用 `room-transfer.runtime-timers.v1` wrapper，并由 room manager 校验 schema/version、`runtimeSummary` 和内层 timer/scheduler entries。

现阶段该契约提供可导出的结构化骨架：runtime summary、可选 timer entries、可选 scheduler entries 和 metadata。combat_demo 已用周期快照 scheduler 证明导出、导入后可以继续按同一调度帧运行。它仍不是通用独立 timer wheel，也不表示 NPC / 行为树 / AI 定时器、路径状态或 RNG 状态已经完整迁移。

## 12. 后续重点

短期建议优先补：

1. 客户端对接 `RoomFrameRatePush`，并保留基于 `FrameBundlePush.fps` 的兼容处理。
2. 有界发送队列、慢连接处理和消息频率限制。
3. 更清晰的状态同步 / 增量快照模型。
4. 回放数据结构和持久化。
5. 与 room transfer 区分开的轻量 `game_state` 快照约定。

跨服迁移和空房接管灰度不在本文展开，详见 [空房接管式灰度规范](./空房接管式灰度规范.md) 和 [空房接管式灰度任务清单](./空房接管式灰度任务清单.md)。
