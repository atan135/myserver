# 统一资产模型与写入口基线

本文固定“统一资产事务与奖励交付”改造开始时的代码事实，以及阶段 1 新增的 Rust 领域模型。当前代码和配置优先；后续阶段应在此基线上收敛写入口，不以本文推断未落地能力。

## 首期边界

- 资产归属键始终是 `character_id`。账号 `player_id` 只用于登录、踢人和封禁等账号级操作，不能作为道具、背包、仓库或装备写入目标。
- 首期只操作现有物品资产。`AssetType::Currency` 仅是后续余额模型预留值，当前没有货币表、余额快照、发放/扣减入口或协议；不得据此实现货币结算。
- 玩家交易、拍卖、商城结算、完整掉落算法和货币经济均不在首期。交易/拍卖需要在冻结、双角色原子提交和审计模型落地后另行接入，不能复用当前单角色 JSONB 覆盖写入。
- 邮件附件是待交付状态，不能作为角色资产的最终容器。`direct` / `mail` 是交付方式；物品成功提交后的最终结算目标是 `inventory`；`push` 只是提交后的通知，失败不应回滚权威资产。

## 阶段 1 领域模型

`apps/game-server/src/core/inventory/asset.rs` 是后续资产模块的唯一基础类型入口，并由 `core/inventory/mod.rs` 重新导出：

| 概念 | Rust 类型 | 当前含义 |
| --- | --- | --- |
| 资产类别 | `AssetType` | `Item` 已落地；`Currency` 仅预留，当前不可操作。 |
| 已提交容器 | `AssetContainer` | `Inventory`、`Warehouse`、`Equipment`；邮件不在此枚举。 |
| 堆叠身份 | `AssetStackIdentity::Item(ItemStackIdentity)` | 包含物品配置 ID、绑定、冻结、配置版本、四属性与成长规则快照、成长历史标记。UID、数量和格位不属于身份。 |
| 绑定 | `AssetBinding` | 未绑定、绑定到角色，以及旧 JSONB 中 `binded` 与角色 ID 不一致的显式兼容状态。 |
| 锁定 | `AssetLockState` | 定义 `Unlocked` / `Frozen`；当前 JSONB 未保存该字段，读取现有物品时一律为 `Unlocked`。 |
| 配置版本 | `AssetConfigVersion` | 定义 `ItemTable` 的修订身份；当前 JSONB 未保存修订号，读取时标记为 `LegacyUnversioned`。 |
| 业务来源 | `AssetOriginType` + `AssetOrigin` | 来源类型与稳定业务 ID 分离，空来源 ID 被拒绝。 |
| 交付语义 | `AssetDeliverySemantics` | `direct` / `mail` 交付到唯一结算目标 `inventory`，随后可发 `push`。不包含奖励内容或请求 ID。 |

现有 `Item::can_stack_with` 已直接调用 `asset_stack_identity`，所以该模型不是旁置定义：它保持当前“属性/规则快照一致、无成长历史才可合并”的行为。阶段 4 增加配置修订与冻结字段后，必须从持久化快照构造 `ItemStackIdentity::from_item_state`，不能继续把其默认为未锁定、无版本。

### 稳定业务来源

`AssetOriginType` 的序列化值是审计和流水的稳定分类，不使用 UI 文案：

| 类型 | 稳定 `origin_id` 的示例 |
| --- | --- |
| `achievement` | 成就领取或结算 ID |
| `quest` | 任务结算 ID |
| `battle` | 战斗结算 ID |
| `scene_pickup` | 场景掉落实体 ID |
| `activity` | 活动结算 ID |
| `ranking` | 榜单结算批次与名次 ID |
| `world_event` | 世界事件结算 ID |
| `gm` | 管理操作 request ID |

另有 `mail_claim`、`player_operation` 和 `system` 用于现有内部链路。来源 ID 必须由服务端业务主键生成，能够在重试、审计和流水查询时稳定复现；客户端不能提供奖励内容或替代来源 ID。

## 当前持久化与并发事实

### 角色快照

- `db/init.sql:622` 和 `apps/game-server/src/core/player/db_player_store.rs:18` 定义 `character_inventory`。它以 `character_id` 唯一，并把 `inventory_data`、`warehouse_data`、`equipment_data`、`attr_base_data`、`visual_data`、`buffs_data` 与 `progress_data` 保存为 JSONB。
- `PlayerData` 同时在在线内存中保有背包、仓库、装备、属性、外观、Buff 和进度状态，见 `apps/game-server/src/core/inventory/player_data.rs:43`。`PlayerManager.players` 是以 `character_id` 为键的 `RwLock<HashMap<...>>`，见 `apps/game-server/src/core/player/player_manager.rs:89`。
- 普通 `save_player` 先替换在线内存副本，再调用 `PgPlayerStore::save` 对整份快照执行 upsert；写库失败仅记录日志，见 `apps/game-server/src/core/player/player_manager.rs:150` 与 `apps/game-server/src/core/player/db_player_store.rs:194`。它没有版本列、行锁或比较并交换，因此同一角色的普通路径与 grant 路径可以发生整体 JSONB 覆盖竞争。这是阶段 3 必须收敛的并发边界。

### 邮件 grant 事务

- `character_inventory_grants` 以 `request_id` 唯一，保存 `character_id`、来源、规范化物品、请求指纹和首次结果，见 `db/init.sql:647`。
- `PgPlayerStore::save_with_grant_record` 在同一个 PostgreSQL transaction 中先以 `ON CONFLICT DO NOTHING` 写 grant 记录，再 upsert 角色快照；冲突时读取既有结果并回滚，提交失败返回结果未知，见 `apps/game-server/src/core/player/db_player_store.rs:219`。
- `PlayerManager::grant_items_with_request` 额外使用 request ID 与 character ID 的进程内 keyed lock，只有数据库提交成功后才发布在线副本，见 `apps/game-server/src/core/player/player_manager.rs:190`。这只覆盖该 grant 路径，尚未覆盖玩家操作、GM 以外的直接写入及进度副作用。

## 已发现写入口基线

| 领域 | 当前写入口与代码位置 | 当前边界 / 风险 |
| --- | --- | --- |
| 背包、装备、仓库 | `PlayerData::add_item/remove_item/equip_item/unequip_item/use_item/warehouse_deposit/warehouse_withdraw`，`apps/game-server/src/core/inventory/player_data.rs:151`、`:223`、`:237`、`:374`。玩家处理器在 `apps/game-server/src/core/service/inventory_service.rs`，正式 dispatch 在 `apps/game-server/src/server.rs:1374`。 | 在内存副本修改后由请求处理器调用普通 `save_player`；不存在与邮件 grant 共享的事务边界。 |
| 直接背包 grant | `PlayerManager::grant_items`，`apps/game-server/src/core/player/player_manager.rs:161`。 | 逐物品修改后整体保存，未附 request ID、来源模型或数据库幂等。后续所有新发放入口不得继续调用它。 |
| GM 发物品 | `apps/game-server/src/admin_server/gm.rs` 的 `handle_grant_items`，通过 `PlayerManager::grant_items_with_request` 构造 `Item::from_config`。 | 已有 request ID、指纹和快照同事务提交；仍使用字符串来源，尚未迁移到 `AssetOrigin`。 |
| 邮件附件领取 | `apps/mail-service/src/mails/mails.service.ts:728` 调用 game-server 管理通道 `grantMailAttachments`；game-server 复用 `handle_grant_items`。 | 邮件工作流、邮件已领取状态与游戏库 grant 不是单一跨库事务，当前以请求 ID 查询/重试恢复；`blocked_capacity` 语义留待阶段 6。 |
| 职业学习与消耗 | 职业/流派学习走 `apps/game-server/src/core/character_discipline.rs` 与 `character_disciplines` 持久化服务。 | 当前未发现“扣物品作为职业学习成本”的正式入口；后续若接入，必须把扣物品和职业状态置于统一资产/业务事务，不能先写其中一方。 |
| 角色进度与奖励 | `CharacterProgressService::apply_for_identity` 在 `apps/game-server/src/core/character_progress.rs` 修改 `PlayerData.progress`，并调用四属性、职业和称号服务；协议服务处理器在 `core/service/character_progress_service.rs`。 | 当前进度奖励含四属性、职业 points、称号和学习资格，`progress_data` 随普通玩家快照保存；它不是通用物品奖励交付，也存在跨存储部分提交风险。 |
| 历史测试加物品 | 阶段 7 已移除 `ItemAddReq` handler、共享 protobuf payload 和 mock-client 常规场景；`MessageType::DeprecatedItemAddReq/Res` 保留 `1407/1408`。 | 这条历史旁路已关闭，收到废弃号固定返回 `MESSAGE_TYPE_DEPRECATED`，新消息不得复用该号码。 |

除上表外，`ItemContainer::add_item/remove_item`（`apps/game-server/src/core/inventory/container.rs:66`）与 `EquipmentSlots` 是可被内部模块直接调用的低层变更点。后续阶段应把它们限制在统一资产事务内部，业务模块不得直接写 JSONB 快照或绕过资产流水。

## 后续接线要求

1. 阶段 2 的奖励单和资产命令必须引用本文件的 `AssetOrigin`、`AssetDeliveryMethod`、`AssetSettlementTarget` 和 `AssetPostCommitNotification`，不要重新定义字符串常量。
2. 阶段 3 必须让普通保存、玩家物品操作、GM、邮件和后续来源模块共享每角色并发边界，并消除 grant 与普通快照的整体覆盖竞争。
3. 阶段 4 必须将配置版本和冻结状态写入资产快照；在此之前，任何冻结、交易或拍卖实现均不具备正确的持久化前提。
