# 角色体系与四属性设计

## 1. 文档定位

本文描述 MyServer 后续角色体系的目标设计，重点约束账号内角色、无等级成长、地火水风四属性、后天职业/流派学习、可成长道具和称号阶位。

当前实现状态：

- `auth-http` 当前已有账号登录、账号 access token、角色列表/创建/删除/恢复/资料查询/选择、选角后签发 character-bound game ticket 和 `player_accounts`。
- `game-proxy` 和 `game-server` 鉴权上下文已同时携带账号 `playerId`、角色 `characterId` 和 `worldId`，缺少 `characterId` 的 game ticket 默认拒绝。
- 房间成员索引、断线重连、匹配建房、输入、移动、战斗、背包和 transfer 链路已迁移为以 `characterId` / `character_id` 作为游戏内主体；账号 `playerId` 继续用于登录、安全、封禁、kick/revoke、ticket version 和 mail ownership。
- 背包持久化已移除传统 `level` 字段；物品使用、装备和区域判断不再以角色等级作为前置条件。
- 独立角色表、账号内多角色、软删除 / 恢复窗口、选角页基础资料查询和角色选择已落地；P1 四属性基础已落地角色 `affinity` / `mastery` 字段、约束、`character_element_logs`、`game-server` 统一变更服务、四属性查询协议和受控 debug 变更入口。
- P2 称号与职业阶位基础已落地 `character_disciplines`、`character_titles`、`character_title_logs`、`TitleTable.csv` 校验、称号授予 / 撤销 / 装备 / 过期处理、职业阶位触发称号解锁、玩家查询协议、受控 debug 入口、mock-client 场景和后台只读查询入口。
- 背包和道具使用已接入四属性变更入口：`ItemTable.csv` 可配置道具模板四属性、使用时 `affinity` / `mastery` delta、成长 / 交易 / 分解 / 继承规则边界、职业 / 流派条件标记和称号来源标记；`UseEffect=CharacterElementChange` 的物品使用会调用 `CharacterElementService.apply_change`，失败不消耗道具。
- 正式职业/流派学习流程、任务 / 成就 / 活动接入称号解锁、战斗公式联动、完整角色级背包/任务/后台 GM 闭环等仍属于后续阶段。

本文是后续开发口径，不代表当前代码已经全部实现。落地时应同步更新协议、数据库、`auth-http`、`game-proxy`、`game-server`、背包、战斗、任务和管理后台。

## 2. 核心设计结论

MyServer 后续角色体系采用以下原则：

- 账号身份和游戏角色身份分离。
- 创建角色时不选择职业、不选择阵营、不使用传统等级。
- 角色核心成长由地、火、水、风四属性、后天获得的职业/流派和称号组成。
- 职业不是出生选项，而是在大世界内通过学习、事件、道具或系统赋予获得；多职业架构第一阶段即落地。
- 道具四属性和人物四属性分开建模。
- 职业/流派自身带阶位，用于表达角色在某方面的级别高低。
- 称号可以来自职业阶位，也可以来自任务、事件、成就、活动或 GM 等其他系统。
- 所有永久成长变化必须走统一服务端入口，并记录来源和变更日志。

建议的身份分层：

```text
account / player_id
  负责登录、封禁、支付、安全审计和账号级状态

character_id
  负责游戏内角色、位置、四属性、背包、任务、战斗、社交展示和称号
```

## 3. 账号内角色模型

### 3.1 账号与角色关系

一个账号可以拥有一个或多个角色。后续推荐第一阶段支持：

- 普通玩家创建流程中，单账号最多 `6` 个角色。
- GM / 管理后台创建、恢复或迁移角色时允许绕过普通玩家角色数限制，但必须写入审计。
- 角色归属于一个 `world_id`。
- 角色名允许重复，不做同世界唯一约束。
- 角色删除使用软删除和冷却期，不做立即物理删除。

角色名不唯一时，所有服务端逻辑必须使用 `character_id` 识别角色，不能依赖角色名。

展示重名角色时采用区分机制：

- 普通场景可以只展示角色名。
- 排行榜、搜索结果、资料页、GM 后台等需要精确识别的场景必须展示 `character_id` 的短标识、称号、职业阶位或其他辅助信息。
- 当某个角色与排行榜玩家重名时，查看非排行榜同名角色资料应显示额外区分标记，避免玩家误认为该角色就是榜上玩家。

当前 `auth-http` 已落地的玩家侧生命周期接口：

- `POST /api/v1/characters/delete`：校验账号 session、账号归属、角色未删除且状态可删除后软删除，返回 `lifecycle`。删除后角色不再进入普通列表、不可选角、不可补发 game ticket、不可查询普通资料。
- `POST /api/v1/characters/restore`：校验账号 session、账号归属、角色为 deleted 状态、未超恢复窗口，并重新校验普通账号有效角色数量上限。
- `GET /api/v1/characters/{character_id}/profile`：查询当前账号未删除角色的基础信息、四属性、装备称号、职业阶位、短 ID 和同名区分信息。称号 / 职业数据源未接入或无数据时返回 `null` 与明确来源标记，不伪造默认称号或职业。

账号 / 选角相关接口归属 `auth-http`；进入游戏后的实时玩法资料查询继续由 `game-server` 玩家协议承载，例如四属性、称号、职业阶位查询和后续状态刷新。

### 3.2 角色创建输入

角色创建阶段只提供最小选择项：

- 角色名。
- 外观配置。
- 初始出生世界和出生场景。
- 初始四属性状态。

角色创建阶段不提供：

- 职业选择。
- 阵营选择。
- 等级选择。
- 初始流派选择。

### 3.3 初始四属性

角色创建时四属性锁定为均衡初始值，不提供偏向模板：

```text
earth = 2500
fire  = 2500
water = 2500
wind  = 2500
```

初始差异应由进入大世界后的行为、学习、道具、事件或系统赋予产生，不在角色创建阶段产生。

## 4. 四属性模型

### 4.1 属性命名

四属性固定为：

| 属性 | 英文字段 | 含义 |
|------|----------|------|
| 地 | `earth` | 稳定、防御、承载、采集、建筑、重型能力倾向 |
| 火 | `fire` | 爆发、锻造、燃烧、攻击、转化、破坏能力倾向 |
| 水 | `water` | 治疗、净化、流动、适应、冰冻、恢复能力倾向 |
| 风 | `wind` | 速度、位移、闪避、探索、感知、远行能力倾向 |

### 4.2 人物属性与道具属性分离

人物四属性表示角色自身的长期状态。

道具四属性表示道具实例或道具模板自身携带的元素属性。

两者不能直接混写到同一字段。道具只有在装备、使用、炼化、献祭、附魔或特定交互中，才会影响人物属性或当前有效属性。

### 4.3 亲和与掌握度

人物四属性建议拆成两组，原因是“倾向比例”和“实际积累值”表达的是两个不同问题。

```text
element_affinity
  亲和。可以理解为四属性的 normalized 比例，总和固定为 10000。
  它回答“这个角色当前更偏向哪种属性”。

element_mastery
  掌握。可以理解为四属性的实际成长值，总和不固定，无硬上限。
  它回答“这个角色在某种属性上积累了多少能力”。
```

示例：

```text
affinity:
earth 2500
fire  2500
water 2500
wind  2500
total 10000

mastery:
earth 120
fire  560
water 80
wind  240
```

两者职责不同：

| 模型 | 是否固定总和 | 用途 |
|------|--------------|------|
| `affinity` | 是，固定 `10000` | normalized 倾向比例；用于世界交互倾向、NPC 态度、区域亲和、转化方向、职业适性 |
| `mastery` | 否，无硬上限 | 实际成长值；用于职业学习条件、职业阶位推进、技能效果、道具使用门槛和长期积累 |

如果只保留一组四属性，会出现一个问题：角色火属性从 `2500` 提升到 `5000` 时，到底表示“火占比更高”，还是表示“火能力更强”。拆成两组后：

- `affinity` 表示比例变化，例如从均衡转为偏火。
- `mastery` 表示实际积累，例如火掌握从 `0` 增长到 `560`。

因此，亲和不是额外复杂成长线，而是四属性比例的归一化表达；掌握才是实际成长数值。

若后续实现阶段确认不需要比例概念，也可以只落地 `mastery`。但当前设计保留 `affinity`，因为大世界交互、NPC 态度、区域亲和、转化概率这类功能通常更适合用比例而不是绝对值判断。

### 4.4 当前有效属性

战斗和世界交互不应直接只读取人物基础属性，而应读取计算后的有效属性：

```text
effective_elements =
  character_base_elements
+ learned_path_modifiers
+ equipped_item_elements
+ active_buff_elements
+ scene_context_elements
+ temporary_system_modifiers
```

其中：

- `character_base_elements`：角色自身长期四属性。
- `learned_path_modifiers`：已学习职业、流派、修行路径提供的修正。
- `equipped_item_elements`：装备中的道具四属性。
- `active_buff_elements`：Buff、药剂、临时状态。
- `scene_context_elements`：场景、天气、区域、世界事件。
- `temporary_system_modifiers`：运营活动、GM 临时状态或特殊规则。

### 4.5 属性变化入口

所有人物永久四属性变化必须走统一入口。P1 已在 `game-server` 落地：

```text
CharacterElementService.apply_change(character_id, change, source, reason)
```

禁止任务、道具、职业、战斗或 GM 模块绕过统一入口直接改角色表字段。

当前 P1 入口负责：

- 校验 delta 合法性。
- 维护 `affinity` 总和。
- 保证 `mastery` 只按实际成长值增减，不受总和约束。
- 写入变更前后快照。
- 写入来源、原因和操作者。
- 写入 `character_element_logs`。

后续阶段再接入：

- 触发属性变化推送。
- 触发称号、职业、任务、成就等派生检查。
- 背包、道具、职业、战斗和场景系统的正式业务调用。

## 5. 道具四属性模型

### 5.1 道具属性分层

道具四属性建议拆成三层：

```text
template_elements
  配置表基础值，当前来自 ItemTable.csv 的 TemplateElementEarth/Fire/Water/Wind，后续可拆到 ItemElementTable。

growth_elements
  道具实例成长值，来自强化、培养、吸收、洗练等长期变化，当前随 Item 实例 JSON 持久化。

runtime_elements
  临时值，来自附魔、耐久、Buff、场景效果等短期变化，当前随 Item 实例 JSON 表达，尚未接入独立过期调度。
```

示例：

```text
一把火属性剑：
template_elements.fire = 300
growth_elements.fire = 120
runtime_elements.wind = 50
```

### 5.2 可成长道具

可成长道具是角色成长的重要来源，但不等同于角色本体属性。

可成长道具可以支持：

- 使用后永久提升角色 `mastery`。
- 装备时临时影响 `effective_elements`。
- 被炼化后改变角色 `affinity`。
- 作为学习职业/流派的条件。
- 作为称号解锁来源。
- 自身随使用次数、吞噬材料或世界事件成长。

道具成长需要记录：

- 道具实例 ID。
- 成长来源。
- 成长前后属性。
- 是否绑定角色。
- 是否可以交易。
- 是否可以被分解或继承。

当前代码已落地的边界：

- `apps/game-server/csv/ItemTable.csv` 增加模板四属性、使用 delta、`GrowthEnabled`、`GrowthSource`、`TradeRule`、`DecomposeRule`、`InheritRule`、`DisciplineConditionKey` 和 `TitleUnlockSource`。
- `apps/game-server/src/core/inventory/item.rs` 的 `Item` 保存 `template_elements`、`growth_elements`、`runtime_elements`、`bound_character_id`、`growth_rules` 和 `growth_records`，新增字段使用 `serde(default)` 兼容老 JSON。
- `UseEffect=CharacterElementChange` 只表达永久修改角色四属性的物品；使用成功路径调用 `CharacterElementService.apply_change(character_id, change, source, reason)`，source 固定为 `source_type=item_use`，`source_id=item_id:<id>:uid:<uid>`，operator 为当前账号玩家。
- 装备中的道具四属性通过 `PlayerData::effective_item_elements` 参与当前有效属性侧计算，不写回 `characters.affinity_*` 或 `characters.mastery_*`。
- `DisciplineConditionKey` 和 `TitleUnlockSource` 目前是配置 / 数据表达，供后续职业学习检查和称号解锁系统读取；本阶段不新增完整职业学习接口，也不自动授予称号。

## 6. 无等级成长设计

### 6.1 放弃传统等级

角色不再拥有传统意义上的 `level` 作为全局成长数字。

因此以下逻辑不应依赖角色等级：

- 装备穿戴。
- 技能学习。
- 区域进入。
- 副本开放。
- NPC 交互。
- 任务接取。
- 战斗伤害。

这些逻辑应改为依赖：

- 四属性 `affinity`。
- 四属性 `mastery`。
- 已学习职业/流派。
- 职业阶位。
- 称号或身份。
- 任务/世界事件状态。
- 道具持有或道具成长状态。

### 6.2 成长来源

角色成长可以来自：

- 学习职业或流派。
- 使用可成长道具。
- 完成任务、试炼或世界事件。
- 与 NPC、场景机关或特殊区域交互。
- 参与战斗、锻造、采集、治疗、探索等领域行为。
- 系统直接赋予。
- GM 或运营补偿。

### 6.3 成长表达

传统等级展示被拆分为多种领域表达：

```text
四属性倾向：我是谁
四属性掌握：我能做什么
职业/流派：我学过什么
职业阶位：我在哪方面有多强
称号：别人怎么看我
```

## 7. 职业、流派与后天学习

### 7.1 职业不是创建选项

职业或流派不在角色创建阶段选择，而是在大世界内通过行为、事件或系统授予获得。

更推荐的命名是：

- `discipline`
- `school`
- `path`
- `profession`

其中 `discipline` 或 `path` 更适合表达“修行路径”，避免传统职业锁死角色。

### 7.2 多职业架构

多职业需要从第一阶段就作为基础架构落地，而不是后续补丁式扩展。

设计规则：

- 角色可以学习多个职业/流派。
- 可学习职业数量上限由配置控制，例如 `MAX_LEARNED_DISCIPLINES`。
- 可同时激活的职业数量也由配置控制，例如 `MAX_ACTIVE_DISCIPLINES`。
- 第一版实现也应按多职业数据结构开发，即使配置上暂时只允许激活 `1` 个。
- 职业切换、激活、停用必须走服务端校验和持久化。
- 后续战斗模块开发前，职业不直接提供战斗属性；仅作为身份、能力、技能池、交互权限和称号来源。

建议配置项：

```text
MAX_LEARNED_DISCIPLINES=8
MAX_ACTIVE_DISCIPLINES=1
```

上述数值只是默认建议，实际可在配置中调整。

### 7.3 学习条件

职业/流派学习条件可以组合：

- `affinity` 倾向要求。
- `mastery` 掌握度要求。
- 其他职业阶位要求。
- 任务或事件完成。
- NPC 好感或组织身份。
- 特定道具持有或消耗。
- 场景位置或世界状态。
- 称号要求。

示例：

```text
学习“熔火锻造”：
fire_mastery >= 500
earth_mastery >= 300
完成 NPC 试炼
持有 熔炉印记

学习“风行术”：
wind_affinity >= 3000
wind_mastery >= 200
完成 风蚀峡谷探索事件
```

### 7.4 职业阶位

职业/流派自身带阶位，用来表示角色在该职业方向上的级别高低。

示例：

| 职业/流派 | 阶位示例 |
|-----------|----------|
| 锻造 | 学徒 / 熟手 / 匠人 / 大师 / 宗师 |
| 医术 | 见习 / 医者 / 名医 / 圣手 |
| 火法 | 入门 / 精通 / 掌御 / 化境 |
| 探索 | 旅人 / 远行者 / 开拓者 / 星图师 |

职业阶位用于规则判断，例如学习高级技能、解锁 NPC 交互、使用特定道具或进入特定区域。

称号可以来自职业阶位，但职业阶位不是称号本身。示例：

```text
职业：锻造
阶位：大师
可解锁称号：熔火匠师
```

### 7.5 职业/流派带来的能力

职业/流派可以提供：

- 可学习技能池。
- 可装备道具类型。
- `effective_elements` 修正。
- 特殊交互权限。
- 职业阶位成长路径。
- 称号解锁路径。
- NPC 对话分支。

## 8. 职业阶位与称号

### 8.1 称号的定位

由于角色没有传统等级，职业阶位承担“某方面级别高低”的表达职责，称号承担对外展示和身份表达职责。

称号不应只是装饰文本。它可以来自职业阶位，也可以来自任务、事件、成就、活动、组织身份、地区声望或 GM / 系统发放。

### 8.2 职业阶位和称号的关系

职业阶位是规则事实，称号是展示结果。

示例：

```text
职业：锻造
阶位：大师
称号：熔火匠师

职业：医术
阶位：名医
称号：沉水医者
```

后续系统判断角色能力时，应优先判断职业阶位和四属性，而不是只判断称号。

```text
是否能学习高级锻造：
discipline(forging).tier >= master
fire_mastery >= 500
earth_mastery >= 300
```

称号“熔火匠师”可以由这些条件解锁，但它不是唯一事实来源。

### 8.3 称号来源

称号来源包括：

- 职业阶位。
- 任务。
- 世界事件。
- 成就。
- 活动。
- 组织或地区身份。
- 排行榜。
- GM 或系统直接授予。

示例：

- `锻造` 职业达到 `大师` 阶位后，解锁“熔火匠师”。
- 完成风蚀峡谷远行事件后，解锁“风行旅者”。
- 治疗次数达到条件后，解锁“沉水医者”。
- 守护据点世界事件中贡献达标后，解锁“地脉守望者”。
- 限时活动排名达标后，解锁活动限时称号。

### 8.4 称号用途

- 角色展示。
- NPC 态度。
- 任务解锁。
- 社交识别。
- 特定场景交互。
- 排行榜和资料页身份标注。

第一阶段称号暂时不提供战斗属性。战斗相关加成留到后续战斗模块设计时单独定义，避免称号系统过早绑定战斗数值。

P2 当前实现中，`TitleTable.csv` 的 `Effects` 只允许非战斗展示或交互标记；配置加载会拒绝 `attack`、`defense`、`hp`、`crit`、`damage`、`buff`、`element_mastery`、`affinity` 等战斗或数值效果键。称号可作为后续任务、NPC、场景交互和外观展示条件，但不直接改变角色战斗面板。

### 8.5 称号装备规则

P2 当前规则：

- 一个角色可以拥有多个称号。
- 同时只能装备 `1` 个主展示称号。
- 装备称号必须已拥有、未过期，且玩家普通装备接口不能装备 `hidden=1` 的隐藏称号。
- 装备新称号时，`TitleService` 在事务 / 服务边界内先卸下旧展示称号，再装备目标称号；数据库不使用部分唯一索引表达跨库不兼容的“唯一已装备”约束。
- 称号默认不提供战斗属性。
- 称号可以作为任务、NPC、职业学习、场景交互的条件。
- 限时称号必须带 `expires_at` 授予；查询 / 装备前会处理已过期的已装备称号并写入日志。

P2 已支持的自动解锁规则：

- `manual`：手动 / GM / 系统授予，不会被自动检查器授予。
- `discipline_tier`：职业 / 流派达到指定阶位后授予，兼容 `{ "discipline": "forging", "tier": "novice" }` 形态。
- `element_mastery`：指定元素 `mastery` 达到阈值。
- `element_affinity`：指定元素 `affinity` 达到阈值。
- `all_of`：所有嵌套规则都满足才授予。

后续可扩展：

- 主称号。
- 副称号。
- 隐藏称号。
- 限时称号。
- 功能称号。
- 称号图标、颜色、边框和稀有度。

## 9. 推荐数据模型

### 9.1 角色表

建议放在 `myserver_game`，因为角色是游戏内状态，不是纯账号登录状态。

```sql
CREATE TABLE characters (
  id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  character_id varchar(64) NOT NULL,
  account_player_id varchar(64) NOT NULL,
  world_id bigint NOT NULL DEFAULT 0,
  name varchar(64) NOT NULL,
  status varchar(32) NOT NULL DEFAULT 'active',
  appearance_json jsonb NOT NULL,
  scene_id integer NOT NULL,
  x real NOT NULL DEFAULT 0,
  y real NOT NULL DEFAULT 0,
  dir_x real NOT NULL DEFAULT 0,
  dir_y real NOT NULL DEFAULT 1,
  affinity_earth integer NOT NULL DEFAULT 2500,
  affinity_fire integer NOT NULL DEFAULT 2500,
  affinity_water integer NOT NULL DEFAULT 2500,
  affinity_wind integer NOT NULL DEFAULT 2500,
  mastery_earth integer NOT NULL DEFAULT 0,
  mastery_fire integer NOT NULL DEFAULT 0,
  mastery_water integer NOT NULL DEFAULT 0,
  mastery_wind integer NOT NULL DEFAULT 0,
  created_at timestamptz NOT NULL DEFAULT current_timestamp,
  last_login_at timestamptz NULL,
  deleted_at timestamptz NULL,
  CONSTRAINT uk_characters_character_id UNIQUE (character_id),
  CONSTRAINT ck_characters_affinity_total CHECK (
    affinity_earth + affinity_fire + affinity_water + affinity_wind = 10000
  )
);
```

说明：

- `account_player_id` 指向账号侧 `player_id`。
- `character_id` 是游戏内角色身份。
- 当前代码和协议中仍存在账号级 `player_id`，但玩法主体和新业务设计应以 `character_id` 为核心。
- `affinity` 使用总和固定约束。
- `mastery` 不固定总和。
- 角色名允许重复，因此不设置 `(world_id, name)` 唯一约束。

### 9.2 四属性变更日志

```sql
CREATE TABLE character_element_logs (
  id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  character_id varchar(64) NOT NULL,
  source_type varchar(32) NOT NULL,
  source_id varchar(128) NULL,
  operator_type varchar(32) NULL,
  operator_id varchar(128) NULL,
  affinity_earth_delta integer NOT NULL DEFAULT 0,
  affinity_fire_delta integer NOT NULL DEFAULT 0,
  affinity_water_delta integer NOT NULL DEFAULT 0,
  affinity_wind_delta integer NOT NULL DEFAULT 0,
  mastery_earth_delta integer NOT NULL DEFAULT 0,
  mastery_fire_delta integer NOT NULL DEFAULT 0,
  mastery_water_delta integer NOT NULL DEFAULT 0,
  mastery_wind_delta integer NOT NULL DEFAULT 0,
  before_json jsonb NOT NULL,
  after_json jsonb NOT NULL,
  reason varchar(255) NULL,
  created_at timestamptz NOT NULL DEFAULT current_timestamp
);
```

常见 `source_type`：

- `quest`
- `item`
- `discipline`
- `scene`
- `combat`
- `title`
- `gm`
- `system`

### 9.3 角色职业表

```sql
CREATE TABLE character_disciplines (
  id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  character_id varchar(64) NOT NULL,
  discipline_id varchar(64) NOT NULL,
  points bigint NOT NULL DEFAULT 0 CHECK (points >= 0),
  tier varchar(32) NOT NULL DEFAULT 'novice',
  active boolean NOT NULL DEFAULT false,
  learned_at timestamptz NOT NULL DEFAULT current_timestamp,
  updated_at timestamptz NOT NULL DEFAULT current_timestamp,
  CONSTRAINT uk_character_disciplines_discipline UNIQUE (character_id, discipline_id),
  CONSTRAINT fk_character_disciplines_character_id
    FOREIGN KEY (character_id) REFERENCES characters(character_id)
);
```

说明：

- `discipline_id` 表示职业、流派或修行路径。
- `points` 表示该职业方向的长期积累值。
- `tier` 表示该职业方向的阶位。
- `active` 表示当前是否激活。激活数量由配置控制，不能只靠数据库约束表达。
- 同一角色可学习多个职业。
- P2 当前仅提供职业阶位持久化、查询、debug 设置和称号解锁触发；正式学习、升级、激活数量上限和技能池联动属于 P3。

示例 `discipline_id`：

- `forging`
- `healing`
- `fire_magic`
- `exploration`
- `alchemy`
- `fishing`
- `mining`

```sql
CREATE TABLE character_titles (

```

### 9.5 角色称号日志

```sql
CREATE TABLE character_title_logs (
  id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  character_id varchar(64) NOT NULL,
  title_id varchar(64) NOT NULL,
  action varchar(32) NOT NULL,
  source_type varchar(32) NULL,
  source_id varchar(128) NULL,
  operator_type varchar(32) NULL,
  operator_id varchar(128) NULL,
  before_json jsonb NULL,
  after_json jsonb NULL,
  reason varchar(255) NULL,
  created_at timestamptz NOT NULL DEFAULT current_timestamp,
  CONSTRAINT fk_character_title_logs_character_id
    FOREIGN KEY (character_id) REFERENCES characters(character_id)

```

说明：

- `action` 当前包含 `grant`、`revoke`、`equip`、`unequip`、`expire` 等服务动作。
- `source_type/source_id` 记录来源域，例如 `gm/debug-character-titles`、`discipline/forging`、`element/fire`。
- `operator_type/operator_id` 记录操作者；玩家协议和 debug 入口使用当前账号玩家 ID。
- `before_json/after_json` 记录状态快照，用于审计和排查。

### 9.6 称号定义

称号定义可以先用 CSV 配置，后续如需后台动态管理再迁入数据库。

P2 当前 CSV 字段：

```text
TitleId,Name,Description,TitleType,SourceDomainId,TierRequired,UnlockRules,Effects,Rarity,Icon,Color,Tags,Hidden,Limited,SortOrder
int,string,string,string,string,string,string,string,string,string,string,Array<string>,int,int,int
```

字段含义：

- `TitleId`：称号 ID。
- `Name`：展示名。
- `Description`：描述文本。
- `TitleType`：当前允许 `identity`、`discipline`、`event`、`honor`、`gm`、`system`。
- `SourceDomainId`：关联职业或其他来源模块，可为空。
- `TierRequired`：关联阶位，可为空。
- `UnlockRules`：解锁条件。
- `Effects`：第一阶段建议为空或只做非战斗效果。
- `Rarity` / `Icon` / `Color`：客户端展示元数据。
- `Tags`：稀有度、地区、职业、活动等标签。
- `Hidden`：隐藏称号标记；玩家普通装备接口不能装备隐藏称号。
- `Limited`：限时称号标记；授予时必须提供过期时间。
- `SortOrder`：称号列表排序。

## 10. 协议与流程建议

### 10.1 登录与选角流程

当前 P0 已落地流程：

```text
1. `auth-http` 登录账号或游客，返回 account access token，不返回可进游戏 ticket。
2. 客户端通过 `GET /api/v1/characters` 请求账号下角色列表。
3. 若无角色或玩家主动创建，通过 `POST /api/v1/characters` 创建角色。
4. 客户端通过 `POST /api/v1/characters/select` 选择角色。
5. `auth-http` 签发 character-bound game ticket。
6. game ticket 中携带 `playerId`、`characterId`、可选/现有 `worldId`、`exp`、`ver`。
7. 客户端连接 `game-proxy` 并发送 `AuthReq`。
8. `game-proxy` 和 `game-server` 鉴权后在连接上下文中同时保留账号 ID、角色 ID 和 world ID；游戏内房间、匹配、输入、移动、战斗、背包和 transfer 使用角色 ID。
```

### 10.2 ticket payload 建议

当前 P0 game ticket payload 至少包含：

```json
{
  "playerId": "plr_xxx",
  "characterId": "chr_xxx",
  "worldId": 0,
  "exp": "2026-06-25T12:15:00.000Z",
  "ver": 1
}
```

说明：

- `playerId` 继续表示账号玩家 ID。
- `characterId` 表示游戏内角色 ID，当前为必填；缺少 `characterId` 的 game ticket 没有兼容开关。
- `worldId` 来自角色数据，当前可选但选角签票路径会随角色写入。
- `exp` 是 ISO 时间字符串，过期后拒绝。
- `ver` 是账号级 ticket version；logout / 改密 / revoke 仍按账号玩家 ID 失效。
- 协议中的 `player_id` 只应出现在账号级登录、安全、GM kick/ban、mail ownership 等链路；游戏内角色身份统一使用 `character_id`。

### 10.3 角色相关协议

当前已落地的账号与选角 HTTP 接口：

- `GET /api/v1/characters`
- `POST /api/v1/characters`
- `POST /api/v1/characters/select`
- `POST /api/v1/game-ticket/issue`

当前已落地的 P1 / P2 角色体系玩家 TCP 协议：

| msgType | 名称 | 说明 |
|---------|------|------|
| `1413` | `GetCharacterElementsReq` | 查询当前已鉴权角色的 `affinity` 和 `mastery` |
| `1414` | `GetCharacterElementsRes` | 返回 `ok`、`error_code`、`character_id`、`elements` |
| `1415` | `DebugApplyCharacterElementChangeReq` | 受控 debug / GM 调试入口，提交 `affinity_delta`、`mastery_delta`、`reason`、`debug_token` |
| `1416` | `DebugApplyCharacterElementChangeRes` | 返回 `before` / `after` 四属性快照 |
| `1417` | `GetCharacterTitlesReq` | 查询当前已鉴权角色的称号定义、拥有状态和当前装备称号 |
| `1418` | `GetCharacterTitlesRes` | 返回 `ok`、`error_code`、`character_id`、`titles`、`equipped_title` |
| `1419` | `EquipCharacterTitleReq` | 装备一个已拥有、未过期、非隐藏称号 |
| `1420` | `EquipCharacterTitleRes` | 返回装备结果和新的 `equipped_title` |
| `1421` | `GetCharacterDisciplinesReq` | 查询当前已鉴权角色的职业 / 流派阶位状态 |
| `1422` | `GetCharacterDisciplinesRes` | 返回 `ok`、`error_code`、`character_id`、`disciplines` |
| `1423` | `DebugCharacterTitleReq` | 受控 debug / GM 调试入口，授予 / 撤销称号、设置职业阶位、触发解锁检查 |
| `1424` | `DebugCharacterTitleRes` | 返回 debug 动作结果、称号、职业阶位和本次解锁称号 |

说明：

- `GetCharacterElementsReq` 不带 body；服务端使用当前连接的 character-bound ticket 绑定角色，不接受客户端在 body 中声明任意 `character_id`。
- `GetCharacterTitlesReq` 和 `GetCharacterDisciplinesReq` 也不带 body；服务端始终使用当前连接鉴权角色。
- `DebugApplyCharacterElementChangeReq` 需要玩家已用合法 ticket 鉴权，并且 `debug_token` 等于 `game-server` 的 `GAME_ADMIN_TOKEN`；`tools/mock-client` 参数名为 `--element-debug-token`，默认可从 `MYSERVER_CHARACTER_ELEMENT_DEBUG_TOKEN` 或 `GAME_ADMIN_TOKEN` 读取。
- `DebugCharacterTitleReq` 需要玩家已用合法 ticket 鉴权，并且 `debug_token` 等于 `game-server` 的 `GAME_ADMIN_TOKEN` 或环境变量 `MYSERVER_CHARACTER_TITLE_DEBUG_TOKEN` / `GAME_ADMIN_TOKEN`；`tools/mock-client` 参数名为 `--title-debug-token`。
- debug 入口固定作为测试 / GM 调试能力使用，来源写入 `source_type=gm`、`source_id=debug-character-elements`，操作者写入当前账号玩家 ID，并记录到 `character_element_logs`。
- 称号 debug 入口来源写入 `source_type=gm`、`source_id=debug-character-titles`，操作者写入当前账号玩家 ID，并记录到 `character_title_logs`。
- 真实客户端应按 `messageType + seq` 关联 `GetCharacterElementsRes`、`DebugApplyCharacterElementChangeRes`、`GetCharacterTitlesRes`、`EquipCharacterTitleRes`、`GetCharacterDisciplinesRes` 和 `DebugCharacterTitleRes`，并把四属性、称号和职业阶位查询结果视为异步状态更新；不能假设发出查询或变更请求后收到的下一包一定是对应响应。
- 常见称号 / 职业错误码包括 `TITLE_NOT_OWNED`、`TITLE_EXPIRED`、`TITLE_CONFIG_NOT_FOUND`、`INVALID_TITLE_ACTION`、`TITLE_DB_UNAVAILABLE`、`CHARACTER_TITLE_DEBUG_FORBIDDEN`、`INVALID_TITLE_DEBUG_ACTION`、`TITLE_ID_REQUIRED`、`INVALID_DISCIPLINE_ID`、`INVALID_DISCIPLINE_TIER`、`NEGATIVE_DISCIPLINE_POINTS`。

当前阶段已落地的角色生命周期 / 资料 HTTP 接口：

- `DeleteCharacter`：`auth-http` 软删除。
- `RestoreCharacter`：`auth-http` 恢复窗口内恢复。
- `GetCharacterProfile`：`auth-http` 选角页基础资料查询。
- 管理后台创建角色：`admin-api` 的 `POST /api/v1/players/{playerId}/characters`，需要 `reason`，绕过普通数量限制并写入 `admin_audit_logs`。
- 管理后台恢复角色：`admin-api` 的 `POST /api/v1/players/characters/{characterId}/restore`，需要 `reason`，绕过普通数量限制并写入 `admin_audit_logs`。

后续仍需补齐或新增的角色协议 / HTTP 接口：

- 正式职业 / 流派学习和激活接口。
- 正式任务 / 成就 / 活动触发称号解锁接口。
- 管理后台迁移角色的安全写入口和后台 UI 闭环。

接口归属边界：

- 账号和选角相关放在 `auth-http`。
- 进入游戏后的实时角色资料放在 `game-server` 玩家协议。
- 管理后台账号维度的角色创建 / 恢复放在 `admin-api`，当前复用 `players.status.update` 权限；成功和失败都必须写 `admin_audit_logs`。

## 11. 与现有系统的关系

### 11.1 与背包系统

当前背包持久化已按无等级模型调整：

- 背包、仓库、装备、Buff 绑定 `character_id`。
- 不保存传统 `level` 语义。
- 物品使用中的 `UseEffect=CharacterElementChange` 已正式调用四属性成长服务；`CharacterElementService` 校验 `affinity` 总和、非负值和日志写入，失败时背包不消耗物品。
- 装备道具四属性参与 `effective_elements` / accessor 计算，但不直接改写角色基础四属性。
- 可成长道具拥有自己的 `growth_elements`、`growth_records` 和绑定 / 交易 / 分解 / 继承规则边界。

### 11.2 与战斗系统

战斗系统后续应读取：

- 角色自身 `mastery`。
- 角色当前激活职业/流派。
- 装备和道具四属性。
- Buff 和场景上下文。
- 称号或职业阶位带来的非直接战斗条件。

战斗结算不应使用传统等级压制。

### 11.3 与任务系统

任务条件后续应支持：

- `affinity` 条件。
- `mastery` 条件。
- 职业阶位条件。
- 称号条件。
- 职业/流派条件。
- 道具成长状态条件。

任务奖励也可以提供：

- `affinity` 调整。
- `mastery` 增长。
- 职业阶位进度增长。
- 称号解锁。
- 职业/流派学习资格。

### 11.4 与场景系统

场景交互后续应支持：

- 某些区域要求特定 `affinity`。
- 某些机关要求特定 `mastery`。
- 某些 NPC 根据称号和职业阶位改变对话。
- 场景上下文临时影响 `effective_elements`。

### 11.5 与管理后台

管理后台后续应支持：

- 查询账号下角色列表。
- 查询角色四属性、称号、职业阶位。
- GM 调整 `affinity` 和 `mastery`。
- GM 授予或移除称号。
- 查询四属性变化日志。
- 查询称号来源和过期时间。

所有 GM 调整必须写入审计和 `character_element_logs`。

## 13. P1 手动联调步骤

最终验收前的手动联调需要用户先确认可以启动依赖和服务。未确认前不要自动执行联调脚本或启动服务。

需要启动的依赖和服务：

- PostgreSQL：承载 `myserver_game.characters`、`character_element_logs` 等表。
- Redis：承载账号 session、game ticket、service registry 和相关安全状态。
- Core NATS：承载 session kick 和 metrics 等通道；本地可优先使用项目 `bin/` 下的 `nats-server.exe`。
- `auth-http`：登录、角色列表 / 创建 / 选择、签发 character-bound game ticket。
- `game-proxy`：玩家正式游戏接入入口，本地 TCP fallback 通常为 `14000`，以实际配置为准。
- `game-server`：四属性查询和 debug 变更 handler 所在服务，需要配置同一 `GAME_ADMIN_TOKEN`。

建议手动步骤：

1. 确认数据库已初始化到包含 P1 字段和 `character_element_logs` 的结构。
2. 设置 `GAME_ADMIN_TOKEN`，必要时设置 `MYSERVER_CHARACTER_ELEMENT_DEBUG_TOKEN` 供 mock-client 读取。
3. 启动 PostgreSQL、Redis、Core NATS、`auth-http`、`game-server` 和 `game-proxy`。
4. 使用 `tools/mock-client` 登录或自动创建角色，并通过 `character-elements-debug` 执行一次平衡 `affinity` delta 与非负 `mastery` delta。
5. 观察 mock-client JSON 输出中的 `before`、`change`、`after`，确认 `character_id` 来自 ticket 绑定角色。
6. 查询数据库，确认 `characters` 四属性已更新，`character_element_logs` 有 source、operator、delta、before_json、after_json 和 reason。
7. 使用不匹配的 `--element-debug-token` 或破坏 `affinity` 总和的 delta 执行负向验证，确认返回 `CHARACTER_ELEMENT_DEBUG_FORBIDDEN` 或 `INVALID_AFFINITY_TOTAL` 等错误，且非法变更不写入成功状态。

示例命令仅在依赖和服务已确认启动后执行：

```powershell
node tools/mock-client/src/index.js --scenario character-elements-debug `
  --http-base-url http://127.0.0.1:3000 `
  --host 127.0.0.1 --port 14000 `
  --login-name test001 --password Passw0rd! `
  --auto-create-character --character-name ElementDebug `
  --element-debug-token $env:GAME_ADMIN_TOKEN `
  --json-output
```

## 14. P2 手动联调步骤

最终验收前的手动联调需要用户先确认可以启动依赖和服务。未确认前不要自动执行联调脚本、不要启动服务、不要执行真实联网联调命令。

需要启动的依赖和服务：

- PostgreSQL：承载 `myserver_game.characters`、`character_disciplines`、`character_titles`、`character_title_logs` 等表。
- Redis：承载账号 session、game ticket、service registry、限流和安全状态。
- Core NATS：承载 session kick、metrics 等通道；本地可优先使用项目 `bin/` 下的 `nats-server.exe`。
- `auth-http`：登录、角色列表 / 创建 / 选择、签发 character-bound game ticket。
- `game-proxy`：玩家正式游戏接入入口，本地 TCP fallback 通常为 `14000`，以实际配置为准。
- `game-server`：称号、职业阶位查询、装备和 debug handler 所在服务，需要配置同一 `GAME_ADMIN_TOKEN`。
- 可选 `admin-api`：用于后台只读 API 查询角色称号和职业阶位，不是 mock-client 主链路必需依赖。
- `admin-web`：本阶段没有新增角色称号 / 职业阶位页面；即使启动也不能完成该项 UI 验收，后续阶段再补管理前端页面。

建议手动步骤：

1. 确认数据库已初始化到包含 P2 三张表、索引和触发器的结构。
2. 设置 `GAME_ADMIN_TOKEN`，必要时设置 `MYSERVER_CHARACTER_TITLE_DEBUG_TOKEN` 供 mock-client 读取。
3. 在用户确认后启动 PostgreSQL、Redis、Core NATS、`auth-http`、`game-server` 和 `game-proxy`；如需后台 API 验收，再启动 `admin-api`。
4. 使用 `tools/mock-client` 登录或自动创建角色，通过 `character-titles-debug` debug 授予 `9001`、装备称号并输出 JSON。
5. 观察 JSON 输出中的 `before`、`action`、`after`、`equippedTitle`、`request.debugTokenProvided`，确认响应通过 `messageType + seq` 匹配，`character_id` 来自 ticket 绑定角色。
6. 查询数据库，确认 `character_titles` 有 `9001`，`is_equipped=true` 且同角色最多一个称号处于装备状态，`character_title_logs` 有 `grant` 和 `equip` 日志。
7. 使用 `character-disciplines-debug` 设置 `discipline_id=forging`、`discipline_tier=novice`、`discipline_points=1` 并触发解锁检查。
8. 观察 JSON 输出中的 `discipline`、`unlockedTitles` 和 `after`，确认 `2001` 见习锻造者由职业阶位规则授予。
9. 查询数据库，确认 `character_disciplines` 写入 `forging/novice/points`，`character_titles` 写入 `2001`，`character_title_logs` 有来源为 `discipline/forging` 的授予日志。
10. 使用错误 `--title-debug-token` 执行负向验证，确认返回 `CHARACTER_TITLE_DEBUG_FORBIDDEN`；尝试装备未拥有、隐藏或过期称号时应返回 `TITLE_NOT_OWNED`、`INVALID_TITLE_ACTION` 或 `TITLE_EXPIRED` 等明确错误码。
11. 如启动 `admin-api`，通过只读 API 查询同一角色的称号和职业阶位，确认与 mock-client 和数据库状态一致；`admin-web` 本阶段没有对应 UI 页面，不作为该项验收入口。

## 15. 已确认规则与待定参数

### 15.1 已确认规则

当前已确认的角色体系规则：

- 普通玩家创建流程中，单账号最多 `6` 个角色。
- GM / 管理后台创建或恢复角色可以绕过普通玩家角色数限制，但必须写入 `admin_audit_logs`；当前最小入口在 `admin-api`，权限复用 `players.status.update`。
- 角色名允许重复。
- 排行榜、搜索结果、资料页和 GM 后台等精确识别场景必须展示额外区分信息。
- 初始四属性固定为均衡值：`earth/fire/water/wind = 2500/2500/2500/2500`。
- `affinity` 总和固定为 `10000`。
- `mastery` 无硬上限，不用数值上限卡成长。
- 背包、仓库、装备、Buff、任务、称号、职业等游戏内状态全部绑定角色；货币系统尚未正式落地，后续实现时也必须绑定 `character_id`。
- 背包持久化不保存传统 `level`，物品使用、装备和区域判断不得继续依赖角色等级。
- 新角色是完全新的游戏个体，只是由同一个账号登录。
- 多职业架构第一阶段即落地，P2 已有职业阶位持久化；最大职业数量和激活数量的正式规则留到 P3。
- 称号第一阶段不提供战斗属性；战斗相关加成留到后续战斗模块设计。
- P2 同一角色同一时刻只能装备一个主展示称号，装备唯一性由 `TitleService` 事务 / 服务逻辑保证。
- P2 自动解锁只覆盖 `discipline_tier`、`element_mastery`、`element_affinity` 和 `all_of`；`manual`、限时和事件类称号需要显式业务入口授予。
- 账号和角色相关流程由 `auth-http + game-server` 分担，不拆独立 `character-service`。
- `player_id` 是账号玩家身份；`character_id` 是游戏内角色身份。新功能不得为玩法请求增加 `player_id` fallback 或双字段兼容。

### 15.2 待定参数

后续实现前仍需具体定值：

1. `MAX_LEARNED_DISCIPLINES` 默认值和上限。
2. `MAX_ACTIVE_DISCIPLINES` 默认值和上限。
3. 职业阶位命名和阶位数量是否按职业分别配置。
4. `affinity` 调整是否允许负向变化，以及变化时如何保持总和 `10000`。
5. `mastery` 是否存在软上限、衰减、突破或阶段化显示。
6. 角色身份迁移后的外部客户端协议同步和真实联调验收范围。
