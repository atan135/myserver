# 角色体系 P0 身份拆分 Checklist

来源文档：`docs/游戏服与接入层/角色体系与四属性设计.md`

## 目标

完成 P0“身份拆分”阶段，把账号身份和游戏内角色身份拆开，为后续四属性、职业、称号、战斗联动和角色级业务提供稳定基础。

P0 完成后应达到：账号 `player_id` 和游戏内 `character_id` 在数据、协议和运行时上下文中明确区分；普通玩家可以在账号下创建、查看和选择角色；普通创建流程限制单账号最多 `6` 个角色；角色名允许重复，服务端所有逻辑使用 `character_id` 精确识别角色；game ticket 携带 `characterId`；game-server 鉴权后能同时拿到 `account_player_id` 和 `character_id`。

## 基础原则

- [ ] 数据库按新开发处理，不保留旧数据结构兼容要求，不编写历史数据迁移脚本。
- [ ] 数据库结构以空库初始化和 `db/init.sql`（或当前仓库实际初始化脚本）直接落新结构为准。
- [ ] 如果已有旧表、旧字段或旧约束与本 P0 模型冲突，本阶段可以直接废弃、替换或重建。
- [ ] `player_id` 只表示账号玩家 ID，不再混用为游戏内角色身份。
- [ ] `character_id` 是游戏内角色的唯一稳定标识，敏感操作不得只依赖角色名。
- [ ] 普通玩家创建角色默认最多 `6` 个，GM / 管理后台后续可以绕过限制，但必须保留审计入口设计。
- [ ] 角色名允许重复，不设置 `(world_id, name)` 唯一约束。
- [ ] 缺少 `characterId` 的 game ticket 不提供兼容开关，默认拒绝。
- [ ] P0 可以预留字段和接口边界，但不实现四属性成长、职业、称号、战斗公式和完整管理后台角色详情页。
- [ ] 背包、仓库、货币、任务等角色级归属的全面改造不放入 P0。
- [ ] 完成功能后同步协议文档、外部客户端接入说明、整体架构文档和 mock-client 说明。
- [ ] 模块开发完成后，先提示需要启动的服务和依赖，待确认后再执行集成测试或联调脚本。
