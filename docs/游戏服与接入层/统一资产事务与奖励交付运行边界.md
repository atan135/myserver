# 统一资产事务与奖励交付运行边界

## 1. 适用范围

本文记录当前已落地的物品资产事务和奖励交付边界，不替代背包、邮件、协议或运维专题设计。当前只覆盖物品资产；货币、交易、拍卖、商城和完整掉落算法仍不在实现范围内。

来源模块必须生成服务端权威 `RewardOrder`，携带稳定的 `request_id`、`character_id`、来源、原因、操作者和标准化道具。客户端不能构造发物品请求，也不能替换邮件领取的附件、request ID、来源或指纹。

## 2. 已落地链路

| 场景 | 当前结算路径 | 可靠性边界 |
| --- | --- | --- |
| 玩家背包、仓库、装备操作 | 玩家协议 -> `PlayerManager` 单角色资产事务 -> `character_inventory` / `character_asset_ledger` | 使用角色锁和 `asset_revision`，提交成功后才发送背包相关 push |
| GM 物品纠正 | game-server admin -> 资产事务 | `request_id` 幂等，写入来源、操作者和流水 |
| `MAIL_ONLY` 奖励 | `RewardDeliveryService` -> `reward_mail_outbox` -> mail-service 奖励邮件 | 奖励邮件和交付记录按 request ID 幂等 |
| `PREFER_INVENTORY` 奖励 | 先尝试 inventory；仅确定的未提交容量不足才转奖励邮件 | `unknown` 结果必须先查询原请求，禁止切换到邮件 |
| 邮件附件领取 | mail-service 鉴权 -> Redis 在线权威路由 + registry `game-server.admin` -> game-server admin -> inventory 资产事务 | `mail_claim:<mail_id>` 固定为领取 request ID；容量不足保留附件并进入 `blocked_capacity` |

`INVENTORY_REQUIRED` 适用于需要和扣款、材料消耗同步的业务。容量不足或其他失败必须让业务整体不提交，不允许转邮件。

## 3. 持久化与对账

`db/init.sql` 与 game-server runtime migration 都包含以下 additive schema：

- `character_inventory`：角色 JSONB 快照及 `asset_revision`。
- `character_asset_requests`：资产请求的确定结果，用于幂等查询。
- `character_asset_ledger`：append-only 资产 delta，关联 `request_id`、`origin_type`、`origin_id`、`delivery_method`、`delivery_id`、`mail_id`、`fallback_reason`、规则版本和操作者。
- `reward_delivery_records` 与 `reward_mail_outbox`：奖励 direct/mail 决策和奖励邮件创建意图。
- `mails`、`mail_claim_workflows` 与 `mail_notification_outbox`：邮件附件、领取冻结快照及通知 outbox。

流水表禁止 `UPDATE`、`DELETE` 和 `TRUNCATE`。纠正只能提交带新 request ID 的补偿资产事务，不能改写历史证据。

上线或灰度前应先运行根目录 `db/init.sql`，或先启动包含同等 runtime migration 的 game-server；回滚应用版本时保留所有新增列、索引、outbox 与流水记录。JSONB 兼容扫描必须只读，发现容量、UID、数量、绑定或配置问题后转人工处置，不在登录或扫描过程中改写快照。

## 4. 领取恢复与服务发现

领取工作流的状态包括 `processing`、`retryable_failure`、`blocked_capacity`、`reconciliation_pending`、`permanent_failure`、`manual_review` 和 `claimed`。

- `blocked_capacity` 不参加自动补发；玩家清理背包后以原邮件重试，附件和 request ID 不变。
- `reconciliation_pending` 先查询 game-server 上的原 request；只有确定 `not_seen` 才能重试原冻结请求。
- 进程中断、路由切换或响应丢失后，恢复 worker 只处理冻结的附件、指纹和 request ID，不能重建新邮件或新领取请求。
- `game-server.admin` 必须经 Redis service registry 的 `admin` endpoint 发现，并和 Redis 在线权威路由的实例 ID 交叉验证。测试、预发和生产不能用固定端口代替发现。

mail-service 的玩家 ticket 校验侧必须使用与 auth-http 一致的 `TICKET_SECRET`。本地联调若要验证领取链路，mail-service 必须启用 `REGISTRY_ENABLED=true`；不要把 `DISCOVERY_REQUIRED=true` 无差别注入整个开发栈，game-server 同时会据此要求 match-service 的 registry endpoint。需要严格发现演练时，应一并启动 match-service。

## 5. 客户端与可观测性

玩家协议保留既有背包、邮件领取和提交后 push 行为。废弃 `ItemAddReq/Res(1407/1408)` 固定拒绝，消息号不能复用。客户端把邮件领取响应的 `claimed`、`processing`、`blocked_capacity`、`reconciliation_pending` 作为状态展示，不以 push 是否到达判断资产是否成功。

game-server 上报 `asset_transaction_latency_ms`、`asset_transaction_count`、`asset_version_conflict_total`、`asset_capacity_fallback_total` 和 `reward_mail_created_total`。mail-service 上报邮件 outbox、领取失败分类、`blocked_capacity` backlog、恢复 worker、`reward_mail_created` 和 `reward_mail_idempotent_replay` 指标。所有指标保持低基数，不能使用角色 ID、邮件 ID、request ID 或 token 作为标签。

## 6. 验证基线

至少运行：

```powershell
cargo test --manifest-path apps/game-server/Cargo.toml inventory
cargo test --manifest-path apps/game-server/Cargo.toml reward
npm --workspace mail-service test
npm run check:proto
```

真实联调需启动 PostgreSQL、Redis、NATS、auth-http、game-proxy、game-server、mail-service 和 mock-client 所需入口。覆盖登录、在线角色邮件领取、同一邮件的幂等重试、服务发现、数据库 migration 与只读流水对账；测试数据应使用可识别的本地 fixture，不删除 append-only 流水。
