# 共享帧同步移动与战斗核心设计

## 1. 文档定位

本文描述一套客户端与服务端共用的移动、战斗确定性计算核心设计。目标是在帧同步场景中，把移动、技能、命中、伤害、Buff、状态 hash 等核心规则收敛到同一个 Rust crate，避免 `game-server` 和外部客户端 `mybevy` 各自实现一套逻辑后产生漂移。

本文是目标设计与落地路线，不表示当前代码已经全部完成：

- 当前代码优先于本文。
- 房间运行时、输入等待、断线重连和已有 demo 状态以 [帧同步与房间生命周期设计](./帧同步与房间生命周期设计.md) 为准。
- 已有战斗 ECS 实现状态以 [战斗 ECS 设计](./战斗ECS设计.md) 为准。
- 外部客户端路径和协作边界以 [外部客户端接入说明](../协议与客户端/外部客户端接入说明.md) 为准。
- 共享帧同步的自动化、真实联调、人工 smoke、失败证据和最终完成状态以 [共享帧同步调试验证 Checklist](./checklists/共享帧同步调试验证_checklist.md) 为统一验收入口。

当前本机 `local_help.txt` 指向的客户端仓库是：

```text
MYSERVER_CLIENT_ROOT=C:\project\mybevy
```

该路径只作为本机分析和联调示例；脚本、文档和代码不应把它写死为唯一依赖。需要访问外部客户端时优先使用 `MYSERVER_CLIENT_ROOT`。

## 2. 设计结论

已采用方案 A：在 MyServer monorepo 内维护共享 Rust crate：

```text
packages/sim-core/
```

`sim` 是 `simulation` 的缩写，`sim-core` 表示“确定性模拟核心”。这里的模拟不是客户端假模拟，而是客户端和服务端在相同初始状态、配置、随机种子和输入帧下共同执行的权威规则计算。

`sim-core` 是移动与战斗的确定性模拟核心。它被以下三类入口复用：

- `apps/game-server`：作为正式服务端房间 tick 的规则核心。
- `tools/lockstep-client`：服务端仓库内的 Rust 套壳 client，用于快速离线和在线验证。
- `mybevy` 外部客户端：通过 path dependency 或后续 git dependency 引用同一套计算代码，用于客户端帧同步测试场景和后续正式玩法预测。

初期不拆独立仓库。原因是当前 MyServer 已有 `packages/authority-core`，外部 `mybevy/project/Cargo.toml` 已通过 path dependency 引用：

```toml
authority-core = { path = "../../MyServer/packages/authority-core" }
```

这证明“服务端仓库内共享 crate + 客户端本地 path 引用”的协作方式当前可行。等 `sim-core` API 稳定、版本化和回归场景充分后，再评估是否拆成独立 repo。

## 3. 核心目标

### 3.1 功能目标

`sim-core` 应提供：

- 定点数数学基础。
- 世界状态模型。
- 帧输入模型。
- 移动推进。
- 简化碰撞和边界处理。
- 技能释放、命中、伤害、治疗、Buff 和死亡事件。
- 确定性状态 hash。
- scenario 离线回放能力。
- 快照、回滚和重放的基础数据结构。

### 3.2 工程目标

- 客户端和服务端共享同一套 Rust 计算代码。
- 服务端仍是权威边界：初始状态、配置版本、随机种子、输入校验、关键快照和异常纠正由服务端控制。
- 客户端只做本地预测、表现和一致性验证，不在核心规则上另写一套逻辑。
- 通过服务端仓库内的 Rust 套壳 client 快速跑场景，不依赖完整 MyServer 栈即可验证确定性。
- 通过 `mybevy` 的测试场景可视化同一批输入下的移动和战斗结果。

## 4. 当前现状与迁移关系

### 4.0 2026-07-11 当前实现快照

当前 MyServer 侧已经落地共享模拟最小闭环，但它仍是 demo / 验证入口，不等价于完整生产玩法能力。

已落地内容：

- `packages/sim-core` 已存在，当前 `SIM_CORE_SCHEMA_VERSION = 1`，提供 `Fp`、`Vec2Fp`、`QuantizedDir`、`SimWorld`、`SimInput`、`step()`、`hash_world()`、`snapshot()` 和 `restore()`。
- `sim-core` 当前支持固定帧推进、输入稳定排序、移动 / 停止 / 朝向、矩形边界 clamp、基础技能释放、伤害、治疗、Buff 添加、Buff tick、死亡事件、冷却推进和每帧 `SimHash`。
- `SimWorld` 实体按 `EntityId` 稳定排序；输入选择按 `frame + character_id + seq + original_index`，移动类和技能类各自选择同角色同帧最新输入。
- `static_obstacles` 只是可序列化配置保留字段，当前不会参与移动碰撞，也不会进入 `hash_world`。
- `apps/game-server` 已接入 `lockstep_sim_demo` room policy。该 policy ID 是 `lockstep_sim_demo`，当前策略为 `min_start_players=1`、`active_room_fps=20`、`missing_input_strategy=Empty`、`snapshot_interval_frames=10`，不启用旧 movement correction / AOI。
- `lockstep_sim_demo` 默认生成玩家实体 `1000 + roster_index`，训练目标实体 `9000`，默认玩家技能 `skillId=1`，冷却为当前 tick rate 帧，敌方目标固定伤害配置为 `15`，训练目标默认位于 `(8000, 0)` raw milli-units。
- 输入复用 `PlayerInputReq(action="sim_input", payload_json=...)`。当前 JSON payload 为 `version=1`、`seq`、`commands[]`；支持 `move`、`stop`、`face`、`castSkill`。`move` / `face` 使用 `dirX`、`dirY`，方向分量为 `-1000..=1000`，且 `dirX^2 + dirY^2 <= 1000^2`；`move` 的 `speed` 可选，单位为 raw milli-units per second，当前服务端输入层允许 `1..=12000`。
- 当前在线 wire 的 `castSkill` 支持 `skillId` 和可选 `targetEntityId`，`targetEntityId` 省略表示 `SkillTarget::None`，非零值表示 `SkillTarget::Entity`。`SkillTarget::Position` / `SkillTarget::Direction` 已在 `sim-core` 与 offline scenario 中存在，但 `lockstep_sim_demo` 的当前在线 JSON adapter 不接收这两类目标。
- 下行 / 恢复数据写入 `RoomSnapshot.game_state` JSON。`initialSnapshot` 包含可恢复的 `SimSnapshot`、`stateHash`、`configHash`、`rngSeed`、`entities` 和 `controlBindings`；`lastFrame` 与 `observerFrame.lastFrame` 携带最近一帧 `SimFrameEnvelope`；`observerFrame.stateHash` 是只读观测 hash，不应改变模拟状态。
- `SimFrameEnvelope` 的 schema 为 `myserver.lockstep-sim.frame-envelope.v1`，`schemaVersion=1`，包含 `frame`、`tickRate`、`configHash`、`stateHash`、`events`、`inputSources` 和 `debugSummary`。`stateHash` 是该帧应用后世界状态 hash；`events` 是本帧输出事件，不写回 `SimWorld`，也不参与 `hash_world`。
- `SimFrameEnvelope` 当前还包含 `eventCount`、`eventSummaries` 和 `debugState`。`eventSummaries` 是稳定字段摘要，包含 `kind`、`frame`、`sourceEntityId`、`targetEntityId`、`skillId`、`buffId`、`amount`、`sequence`；`debugState` 是最多 32 个实体的轻量诊断快照，只包含 `entityId`、`xRaw`、`yRaw`、`hp`、`maxHp`、`alive`，不携带完整 `SimWorld`。
- `tools/lockstep-client` 已提供 offline replay、online dry-run 和真实 online replay。online 模式会从 `RoomSnapshot.game_state.initialSnapshot.snapshot` 恢复本地世界，读取 `lastFrame` 或 `observerFrame.lastFrame`，用 `FrameBundlePush.inputs` 重放并对比服务端 `stateHash` 与 `events`。
- `scripts/online-lockstep-reconcile.ps1` 已统一编排 MyServer 的移动、近战、observer recovery，以及 mybevy 的单客户端、双客户端、重连/观战和 GUI visual smoke。plan 不落盘不联网，dry-run 只执行离线计划，execute 记录 ownership 并精确清理，`-DiagnosticFixture` 只验证 synthetic mismatch 证据链。
- 外部 `mybevy` 已落地 `arena.lockstep_sim`、`lockstep-sim-headless` 和 `mybevy.lockstep.telemetry` v1 JSONL。在线 headless 使用 `my_server_authority` hash；离线 fixture 使用 `offline_fixture_authority`，二者在报告 provenance 中严格区分。
- 统一 run 目录为 `logs/lockstep-online/<run-id>/`，`report.json` 同时索引 client JSONL、视觉报告/截图、run-owned 服务日志、`triage` 和 `diagnosticIndex`；适用 artifact 缺失会明确标记为 `missing`。

仍未落地或只保留目标态：

- 外部 `mybevy` 的共享帧同步代码仍不在 MyServer 仓库内维护；MyServer 只记录协议契约、`MYSERVER_CLIENT_ROOT` / `-ClientRoot` 路径约定和统一验收入口。当前相邻目录布局仍由 `mybevy/project/Cargo.toml` 的 `../../MyServer/...` path dependency 约束，环境变量不能单独解决任意非相邻布局。
- 外部客户端交接门禁和 MyServer 契约清单以 [外部客户端接入说明](../协议与客户端/外部客户端接入说明.md) 的“共享帧同步外部客户端交接门禁”为准；该清单覆盖 `policy_id=lockstep_sim_demo`、`sim_input` payload、`RoomSnapshot.game_state`、`stateHash.hex`、events、`inputSources`、`debugSummary`、常见错误码和联调启动确认要求。
- 当前没有专用 Protobuf `SimInputReq` / `SimSnapshotPush` / `SimHashPush`；`sim_input` 仍走 `PlayerInputReq.payload_json`。
- 当前没有复杂客户端预测、正式回滚策略、NavMesh、复杂物理、完整生产 AOI、正式玩法 UI / 动画、生产入口全链路验收、跨服 room transfer 的完整共享模拟迁移。现有 direct local online 和 GUI smoke 是调试验收，不等价于生产能力。
- 当前技能 / Buff 配置使用 `lockstep_sim_demo.fixed_v1` 固定 demo 定义，`configMigrationBoundary` 明确为 `skills_and_buffs_use_fixed_demo_definitions_until_sim_core_csv_mapping_is_complete`；完整 CSV 技能 / Buff 到 `sim-core` 的映射尚未完成。

`game-server` 当前已经具备完整房间运行时框架：

- `RoomManager + RoomRuntimePolicy + RoomLogic`

- `FrameBundlePush` 按帧广播输入集合

现有问题是：移动、战斗、客户端 replay 逻辑仍分散在不同模块和仓库中。

示例：

- `apps/game-server/src/core/system/movement/`
- `apps/game-server/src/core/system/combat/`
- `apps/game-server/src/gameroom/movement_demo/`
- `apps/game-server/src/gameroom/combat_demo/`
- `C:\project\mybevy\project\src\game\features\robot_sync\`

其中 `combat_demo` 仍使用 `f32` 位置、范围、速度等字段；这不适合作为长期帧同步确定性核心。

## 5. 边界原则

### 5.1 `sim-core` 负责什么

`sim-core` 只负责确定性计算：

- 解析已经过外层校验的帧输入。
- 从初始快照构建模拟世界。
- 按固定帧号推进世界。
- 输出新世界状态、事件列表和 hash。
- 提供序列化快照和回放辅助。

当前 `lockstep_sim_demo` 中，服务端 `game-server` 仍是房间权威和网络边界；`sim-core` 不拥有 room、连接、鉴权、输入等待或广播职责。

核心 API 形态：

```rust
pub fn step(
    world: &mut SimWorld,
    frame: SimFrame,
    inputs: &[SimInput],
    config: &SimConfig,
) -> SimStepResult;
```

### 5.2 `sim-core` 不负责什么

`sim-core` 不做：

- TCP / KCP / HTTP / gRPC 收发。
- `auth-http` 登录、ticket、session。
- Redis、PostgreSQL、NATS。
- `game-proxy` 路由。
- Bevy 渲染、动画、UI 和相机。
- 读取 CSV 文件、热更新文件监听。
- 系统时间、线程调度、异步 IO。
- 生产环境部署、服务发现和管理后台。

### 5.3 服务端仍是权威

服务端负责：

- 创建 room。
- 选择和下发初始状态。
- 下发配置版本、配置 hash 和随机 seed。
- 校验玩家身份、角色身份和输入窗口。
- 收集、排序和广播权威输入帧。
- 在服务端也调用 `sim-core` 推进权威状态。
- 周期性广播权威 hash 或快照。
- 检测客户端 hash 差异并触发回滚、重连恢复或强制校正。

在当前 `lockstep_sim_demo` 实现里，输入等待、缺输入补偿、`FrameBundlePush` 广播、`RoomSnapshot.game_state` 生成、重连和观战仍由 `RoomManager` / `RoomLogic` 承担；`sim-core` 只消费已经映射好的 `SimInput` 并返回 `SimStepResult`。

### 5.4 客户端只做预测与表现

客户端负责：

- 采集本地输入。
- 把输入量化成 `sim-core` 认可的确定性输入。
- 等权威输入帧或本地预测窗口驱动模拟。
- 使用 `sim-core` 状态渲染实体。
- 上报本地 hash 或在 HUD 中展示 hash。
- 收到权威快照后回滚并重放。

客户端不得：

- 在 `sim-core` 外重新实现移动、命中、伤害等核心规则。
- 把 Bevy `Time.delta_seconds()`、`Transform` 或物理引擎浮点结果写回权威模拟。
- 用本地计算结果决定永久属性、背包、职业、称号或关键战斗结算。

## 6. 定点数设计

### 6.1 基本类型

`sim-core` 应提供自定义定点数类型，作为核心模拟唯一的小数表达：

```rust
#[derive(Clone, Copy, Debug, Default, PartialEq, Eq, PartialOrd, Ord)]
pub struct Fp(i64);
```

推荐第一阶段采用十进制定标：

```text
FP_SCALE = 1000
1 sim unit = 1000 raw
```

理由：

- 与现有 `mybevy` robot sync 的 `FIXED_UNIT = 1000` 一致。
- 配置表和日志易读。
- 适合毫米级或 0.001 单位精度。
- 初期比 Q16.16 / Q32.32 更便于人工排查。

后续如有性能或范围压力，再评估二进制定点格式。

### 6.2 推荐单位

| 数据 | 类型 | 说明 |
|------|------|------|
| 位置 | `Fp` / `Vec2Fp` | 世界平面坐标 |
| 半径、距离 | `Fp` | 技能范围、碰撞半径、AOI 半径 |
| 速度 | `Fp` per second 或 per tick | 配置加载时统一换算 |
| 时间 | `u32 frame` / `u16 frames` | 不使用秒作为运行态时间 |
| 方向 | `Dir2` / `Vec2Fp` | 输入层量化后进入核心 |
| 伤害倍率 | basis points / `i32` | 优先整数比例 |
| 冷却 | `u16 frames` | 不使用浮点秒 |
| 血量、攻击、防御 | `i32` | 第一阶段保持整数 |

### 6.3 数学模块

建议结构：

```text
packages/sim-core/src/math/
├── fp.rs          # Fp 类型和基础运算
├── vec2.rs        # Vec2Fp
├── dir.rs         # 方向量化和查表
├── clamp.rs       # clamp、min、max
└── hash.rs        # raw 值 hash 辅助
```

`Fp` 至少提供：

- `from_raw`
- `raw`
- `from_i32`
- `from_milli`
- `to_f32_for_render`
- `checked_add` 或受控 saturating 行为
- `mul_ratio`
- `div_i64_trunc`

渲染转换只能在边界发生：

```rust
impl Fp {
    pub fn to_f32_for_render(self) -> f32 {
        self.0 as f32 / FP_SCALE as f32
    }
}
```

`to_f32_for_render()` 的结果不得再次进入 `sim-core` 计算。

### 6.4 方向量化

移动输入不应把平台浮点摇杆直接传入模拟核心。第一阶段建议支持两类输入：

#### 离散方向

用于键盘和简单 AI：

```text
Right      = (1000, 0)
Left       = (-1000, 0)
Up         = (0, -1000)
Down       = (0, 1000)
UpRight    = (707, -707)
DownRight  = (707, 707)
```

#### 量化摇杆

用于移动端摇杆：

```rust
pub struct QuantizedDir {
    pub x: i16, // -1000..1000
    pub y: i16, // -1000..1000
}
```

输入层负责把摇杆值量化，并保证：

```text
x * x + y * y <= 1000 * 1000
```

`sim-core` 可以再次校验并拒绝非法方向，避免客户端、mock 工具或脚本绕过约束。

### 6.5 禁止项

核心模拟中禁止：

- `f32` / `f64` 参与状态推进。
- 每帧调用平台相关 `sqrt`、`sin`、`cos` 后写入状态。
- 使用 `HashMap` 默认迭代顺序决定结算顺序。
- 使用系统时间或真实 delta time。
- 使用非固定 seed 的随机数。
- 使用多线程并行遍历后依赖完成顺序。

```text
    ├── hash.rs
    └── scenario.rs
```

### 7.1 依赖原则

`sim-core` 依赖应保持极小：

```toml
[dependencies]
serde = { version = "1", features = ["derive"] }
serde_json = "1"
```

可选：

- `thiserror`：如果项目接受该依赖，可用于错误类型。

不建议第一阶段引入：

- Bevy。
- Tokio。
- Redis / sqlx / NATS。
- 物理引擎。
- 平台相关输入库。

### 7.2 与现有 packages 的关系

`sim-core` 与 `authority-core` 的关系：

- `authority-core` 继续负责 authority endpoint、通用 input、snapshot 和 migration payload。
- `sim-core` 负责 gameplay 确定性模拟。
- 两者可以被同一个客户端或工具同时依赖，但不应互相强耦合。

`sim-core` 与 `packages/proto` 的关系：

- `packages/proto` 是网络协议事实源。
- `sim-core` 不直接依赖 Protobuf 类型。
- `game-server` 和 `mybevy` 的协议适配层负责把 `PlayerInputReq` / `FrameBundlePush` 转成 `SimInput`。

这样可以避免把网络协议演进直接绑死到核心模拟 API。

## 8. 数据模型

### 8.1 世界状态

```rust
pub struct SimWorld {
    pub schema_version: u16,
    pub frame: u32,
    pub rng: SimRngState,
    pub entities: Vec<SimEntity>,
}
```

第一阶段可使用 `Vec<SimEntity>`，但所有结算必须按稳定顺序进行。后续规模扩大后再拆 SOA。

### 8.2 实体模型

```rust
pub type EntityId = u32;

pub struct SimEntity {
    pub id: EntityId,
    pub kind: EntityKind,
    pub owner_character_id: Option<String>,
    pub team_id: u16,
    pub transform: SimTransform,
    pub movement: MovementState,
    pub combat: CombatState,
    pub alive: bool,
}
```

身份分层：

| 身份 | 用途 |
|------|------|
| `playerId` | 账号身份，仍归登录、session、封禁、ticket version 使用 |
| `characterId` | 游戏内角色身份，用于房间、输入、背包、职业、称号等玩法主体 |
| `entityId` | 模拟实体身份，玩家、NPC、怪物、投射物、召唤物统一使用 |
| `denseIndex` | 后续 SOA 优化时的内部连续数组下标 |

`sim-core` 内部不应使用 `playerId` 作为战斗实体主键。玩家控制实体的映射应由服务端初始快照下发：

```rust
pub struct PlayerControlBinding {
    pub character_id: String,
    pub entity_id: EntityId,
}
```

### 8.3 坐标和朝向

```rust
pub struct SimTransform {
    pub pos: Vec2Fp,
    pub facing: QuantizedDir,
    pub radius: Fp,
}
```

第一阶段只做 2D 平面逻辑。`mybevy` 渲染到 3D 时使用固定映射：

```text
sim.x -> world3d.x
sim.y -> world3d.z
world3d.y -> 地面高度 / 角色脚底高度
```

### 8.4 移动状态

```rust
pub enum MovementMode {
    Idle,
    Controlled,
    Dash,
    Knockback,
}

pub struct MovementState {
    pub mode: MovementMode,
    pub velocity: Vec2Fp,
    pub move_dir: QuantizedDir,
    pub speed_per_second: Fp,
    pub control_stop_frames: u16,
    pub lock_until_frame: u32,
}
```

第一阶段建议只实现：

- `Idle`
- `Controlled`
- 简单 `Knockback`

`Dash` 可作为战斗技能位移扩展。

### 8.5 战斗状态

```rust
pub struct CombatState {
    pub hp: i32,
    pub max_hp: i32,
    pub attack: i32,
    pub defense: i32,
    pub speed: i32,
    pub crit_rate_bps: u16,
    pub crit_damage_bps: u16,
    pub skill_slots: Vec<SkillSlot>,
    pub buffs: Vec<BuffSlot>,
}
```

第一阶段仍可沿用整数血量和 basis points 比例。涉及距离、半径、位移和速度的字段必须改为定点数。

## 9. 输入模型

### 9.1 帧输入

```rust
pub struct SimInput {
    pub frame: u32,
    pub character_id: String,
    pub entity_id: EntityId,
    pub seq: u32,
    pub command: SimCommand,
}
```

`entity_id` 可以由服务端协议适配层根据 `character_id -> entity_id` 映射填入。客户端也应从初始快照获得同一映射。

### 9.2 命令类型

```rust
pub enum SimCommand {
    Move(MoveCommand),
    Stop,
    Face(FaceCommand),
    CastSkill(CastSkillCommand),
}
```

### 9.3 移动输入

```rust
pub struct MoveCommand {
    pub dir: QuantizedDir,
    pub speed_per_second: Fp,
}
```

第一阶段允许客户端提交速度，但服务端应按角色状态、配置和 Buff 校验或重写速度。更稳妥的长期模型是客户端只提交方向，速度由 `SimConfig + EntityState` 决定。

### 9.4 技能输入

```rust
pub struct CastSkillCommand {
    pub skill_id: u16,
    pub target: SkillTarget,
}

pub enum SkillTarget {
    None,
    Entity(EntityId),
    Position(Vec2Fp),
    Direction(QuantizedDir),
}
```

不要让客户端提交“命中结果”或“伤害值”。客户端只提交释放意图。

### 9.5 输入排序

同一帧输入必须稳定排序：

```text
frame asc
character_id asc
seq asc
original_index asc
```

同一角色同一帧多条同类型输入的处理规则需要固定：

- 移动类：采用 seq 最大的一条；seq 相同采用原始输入列表最后一条。
- 技能类：第一阶段每帧每角色最多接受一条技能输入；多条时按 seq 最大。
- Stop 和 Move 冲突：按 seq 较大者生效。

该规则必须在服务端套壳 client、`game-server`、`mybevy` 场景中共用。

## 10. 配置模型

`sim-core` 不直接读取 CSV。服务端负责把 CSV 加载成运行时 `SimConfig`，再传入核心。

```rust
pub struct SimConfig {
    pub schema_version: u16,
    pub tick_rate: u16,
    pub movement: MovementConfig,
    pub skills: SkillCatalog,
    pub buffs: BuffCatalog,
    pub scene: SceneCollisionConfig,
}
```

### 10.1 配置版本

每个 room 创建时应绑定：

- `config_version`
- `config_hash`
- `sim_schema_version`

服务端在创建初始快照时下发这些信息。客户端如果本地 `sim-core` 或配置 hash 不匹配，应进入只观战、拒绝启动测试场景，或明确报错。

### 10.2 CSV 热更关系

已有 `game-server` 支持 CSV 热加载。对 `sim-core` 来说应遵守：

- 新 room 使用新配置。
- 运行中 room 默认继续使用创建时绑定的配置快照。
- 如果要运行中切配置，必须通过明确的帧边界事件，例如 `ConfigSwitchAtFrame`。
- 配置切换必须进入 hash 输入，否则双端可能在同一帧使用不同配置。

第一阶段不建议让运行中帧同步 room 自动吃热更配置。

## 11. Tick 推进

推荐单帧流程：

```text
step(world, frame, inputs, config)
  1. 校验 frame == world.frame + 1
  2. 稳定排序 inputs
  3. 应用控制输入到实体意图状态
  4. 推进移动
  5. 处理技能请求
  6. 推进技能冷却
  7. 推进 Buff / Dot / Hot
  8. 处理死亡、事件和派生状态
  9. 对实体和事件做稳定排序
 10. 更新 world.frame
 11. 计算 state_hash
```

输出：

```rust
pub struct SimStepResult {
    pub frame: u32,
    pub events: Vec<SimEvent>,
    pub state_hash: SimHash,
}
```

### 11.1 移动推进公式

第一阶段使用整数定点推进：

```text
denom = FP_SCALE * fps
delta_x = trunc_toward_zero(dir_x * speed_raw / denom)
delta_y = trunc_toward_zero(dir_y * speed_raw / denom)
pos = clamp(pos + delta, scene_bounds)
```

该公式与当前 `mybevy` robot sync 的思路一致，便于迁移和对照。

### 11.2 碰撞和阻挡

第一阶段不要直接接复杂物理引擎。建议先支持：

- 场景矩形边界。
- 圆形实体半径。
- 简单静态阻挡矩形或圆。
- 稳定顺序的实体间推开。

实体推开顺序必须固定，例如：

```text
按 entity_id asc 遍历实体对
同一对只处理一次
位移修正按固定比例和定点数截断
```

复杂寻路、NavMesh、斜坡和 3D 碰撞另开专题，不作为第一阶段目标。

### 11.3 战斗推进

第一阶段支持：

- 普通近战技能。
- 指向实体技能。
- 指向位置 AOE。
- 冷却。
- 固定伤害、攻击缩放伤害、真实伤害。
- 治疗。
- Buff 添加、持续、层数和周期伤害。

命中判定只使用 `sim-core` 中的定点距离：

```text
distance_squared(source.pos, target.pos) <= range * range
```

目标选择必须稳定排序：

```text
候选实体按 distance asc, entity_id asc 排序
```

如果 distance 相同，不能依赖 Vec 原始顺序或 HashMap 顺序。

## 12. 随机数

`sim-core` 如需随机，应使用固定算法和服务端下发 seed。

```rust
pub struct SimRngState {
    pub seed: u64,
    pub counter: u64,
}
```

随机调用必须：

- 只发生在 `step()` 中。
- 次数与输入和状态稳定相关。
- 不因日志、渲染或调试开关改变。
- 进入快照和 hash。

第一阶段能不用随机就不用随机。暴击可先关闭或使用确定性公式验证；如果打开暴击，必须通过 `SimRngState`。

## 13. Hash 与对账

### 13.1 Hash 内容

状态 hash 应覆盖：

- `sim_schema_version`
- `frame`
- `config_hash`
- `rng_state`
- 实体 id、类型、队伍、存活状态
- 位置 raw 值
- 朝向 raw 值
- 移动状态
- hp / max hp / stats
- 技能冷却
- Buff slot

不应覆盖：

- 渲染 Transform。
- 动画状态。
- HUD 状态。
- 日志序号。
- 网络延迟统计。
- HashMap 内存布局。

### 13.2 Hash 算法

第一阶段可以使用稳定的 FNV-1a 64-bit 或自定义固定 hash。关键不是加密强度，而是跨平台、跨端、跨运行稳定。

```text
entities 按 entity_id asc 排序
buffs 按 buff_id asc, source_entity asc 排序
events 按 frame, event_type, source, target, seq 排序
```

hash 输出建议：

```rust
pub struct SimHash {
    pub frame: u32,
    pub value: u64,
}
```

日志显示为 hex：

```text
frame=120 hash=4f63a1c9d20e887b
```

### 13.3 对账频率

建议：

- offline scenario：每帧比对。
- 服务端套壳 client online：每帧记录，可每 N 帧输出。
- `mybevy` 场景 HUD：每 10 或 20 帧显示一次 hash。
- 正式联机：每 10 到 30 帧上报一次本地 hash，避免过度占用带宽。

## 14. 服务端集成设计

### 14.1 新 room policy

建议新增策略：

```text
lockstep_sim_demo
```

职责：

- 使用现有 `RoomManager` 的输入等待、缺帧和广播能力。
- 在 `RoomLogic::on_tick(frame_id, fps, inputs)` 中把 `PlayerInputRecord` 转成 `SimInput`。
- 调用 `sim_core::step()`。
- 把 `SimEvent` 转成 `RoomLogicBroadcast` 或后续专用 Protobuf push。
- 在 `RoomSnapshot.game_state` 或专用快照中带上 `sim_hash` 和轻量 debug state。

### 14.2 与现有 demo 的关系

第一阶段不需要立即删除现有 demo：

- `robot_sync_room` 继续服务当前 `arena.robot_sync`。
- `movement_demo` 作为服务端权威位移样例保留。
- `combat_demo` 作为旧战斗 ECS 对照保留。
- `lockstep_sim_demo` 作为共享 `sim-core` 的新验证入口。

等 `lockstep_sim_demo` 覆盖移动、战斗、回放和客户端场景后，再逐步合并或替换旧 demo。

### 14.3 协议映射

短期可继续复用：

- `PlayerInputReq`
- `FrameBundlePush`
- `GameMessagePush`

输入 action 建议：

```text
sim_move
sim_stop
sim_face
sim_cast_skill
```

或者合并为：

```text
action = "sim_input"
payload_json = { "version": 1, "commands": [...] }
```

推荐第一阶段使用单一 `sim_input`，便于版本化和扩展：

```json
{
  "version": 1,
  "seq": 12,
  "commands": [
    {
      "type": "move",
      "dirX": 1000,
      "dirY": 0
    }
  ]
}
```

技能示例：

```json
{
  "version": 1,
  "seq": 13,
  "commands": [
    {
      "type": "castSkill",
      "skillId": 1,
      "targetEntityId": 2001
    }
  ]
}
```

后续稳定后再考虑专用 Protobuf 输入，减少 JSON 成本和字段歧义。

### 14.4 初始快照

服务端创建或开始 room 时应生成：

```rust
pub struct SimInitialSnapshot {
    pub schema_version: u16,
    pub room_id: String,
    pub start_frame: u32,
    pub tick_rate: u16,
    pub config_hash: String,
    pub rng_seed: u64,
    pub entities: Vec<SimEntitySnapshot>,
    pub control_bindings: Vec<PlayerControlBinding>,
}
```

该快照应下发给客户端：

- 开局。
- late join。
- 重连恢复。
- hash 不一致后的回滚基线。

### 14.5 缺帧策略

现有 `RoomRuntimePolicy` 已支持：

- `Strict`
- `Optimistic`
- `Empty`
- `RepeatLast`
- `DropAfterMisses`

`sim-core` 不直接决定等待策略。它只消费 `RoomManager` 已经决定应用到某一帧的输入集合。

但 `sim-core` 需要知道输入是玩家真实输入还是服务端合成输入。建议在适配层保留：

```rust
pub enum SimInputSource {
    Real,
    SynthesizedEmpty,
    SynthesizedRepeatLast,
}
```

这样移动控制超时、审计和排障可以区分真实输入与补帧输入。

## 15. 服务端套壳 client

### 15.1 工具定位

新增：

```text
tools/lockstep-client/
```

这是 Rust 套壳 client，不是正式玩家客户端。它用于快速验证 `sim-core` 在服务端仓库内的可重复性。

### 15.2 模式

#### offline 模式

不启动 MyServer 服务，不连接网络：

```text
tools/lockstep-client --mode offline --scenario move_straight
```

流程：

1. 读取 scenario 文件。
2. 构建 server_sim 和 client_sim 两个 `SimWorld`。
3. 喂入同一批初始状态和输入帧。
4. 每帧分别调用 `sim_core::step()`。
5. 比对事件和 hash。
6. 输出首个不一致帧。

offline 是第一优先级，因为它最快暴露确定性问题。

#### online 模式

连接本地 MyServer：

```text
tools/lockstep-client --mode online --room lockstep-test-room --policy lockstep_sim_demo
```

流程：

1. 走 auth / guest login 或使用测试 ticket。
2. 连接 `game-proxy` TCP fallback 或 KCP。
3. 加入 `lockstep_sim_demo` room。
4. 发送 `sim_input`。
5. 消费 `FrameBundlePush` 和服务端 hash。
6. 本地用同一 `sim-core` replay。
7. 比对本地 hash 和服务端 hash。

online 模式需要用户确认启动服务和依赖后再执行，不应被普通开发流程自动启动。

当前统一编排入口为：

```powershell
# 只输出 plan，不落盘、不读取 ticket、不启动服务
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/online-lockstep-reconcile.ps1 `
  -Client lockstep-client -Check all -StartDevStack -ProvisionDevTickets `
  -RunId <run-id>

# 用户确认后的真实移动、近战、observer recovery 对账
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/online-lockstep-reconcile.ps1 `
  -Execute -Client lockstep-client -Check all `
  -StartDevStack -ProvisionDevTickets `
  -RunId <run-id> -RedisKeyPrefix "lockstep-online:<run-id>:"
```

外部 mybevy 使用同一 wrapper 的 `-Client mybevy`，并把 `-Check` 限定为 `single-client`、`dual-client`、`reconnect-observer`、`visual-smoke` 或 `all`。完整命令、依赖、清理与 failure triage 见 [外部客户端接入说明](../协议与客户端/外部客户端接入说明.md) 和 `tools/lockstep-client/README.md`。

### 15.3 scenario 文件

建议放置：

```text
tools/lockstep-client/scenarios/
├── move_straight.json
├── move_stop.json
├── move_collision.json
├── melee_hit.json
├── aoe_hit.json
├── buff_dot.json
└── rollback_replay.json
```

示例结构：

```json
{
  "version": 1,
  "tickRate": 20,
  "config": {
    "movement": {
      "bounds": { "minX": -250000, "maxX": 250000, "minY": -250000, "maxY": 250000 }
    }
  },
  "initial": {
    "frame": 0,
    "seed": 12345,
    "entities": [
      {
        "id": 1001,
        "kind": "player",
        "characterId": "chr_a",
        "teamId": 1,
        "x": 0,
        "y": 0,
        "radius": 500,
        "hp": 100
      }
    ]
  },
  "inputs": [
    {
      "frame": 1,
      "characterId": "chr_a",
      "entityId": 1001,
      "seq": 1,
      "command": { "type": "move", "dirX": 1000, "dirY": 0 }
    }
  ],
  "assertions": {
    "finalFrame": 20,
    "finalHash": "0000000000000000",
    "events": []
  }
}
```

`finalHash` 在首次实现时可由 bless 流程生成，但 bless 必须是显式命令，不能默认覆盖。

### 15.4 输出要求

不一致时输出：

```text
scenario=melee_hit
first_mismatch_frame=17
server_hash=...
client_hash=...
entity_diff:
  entity=1001 pos server=(12000,0) client=(11500,0)
event_diff:
  missing client event DamageApplied(source=1001,target=2001,value=10)
```

这样方便快速定位是移动、战斗、排序、随机还是配置问题。

## 16. 客户端场景测试

### 16.1 场景定位

在 `mybevy` 中新增或演进一个测试场景：

```text
LockstepTestScene
```

建议场景 ID：

```text
arena.lockstep_sim
```

它不同于正式玩法场景，职责是验证帧同步移动和战斗效果。

### 16.2 与现有 `arena.robot_sync` 的关系

`arena.robot_sync` 已经验证：

- MyServer authority 接入。
- 双客户端进入同一 room。
- `FrameBundlePush` replay。
- fixed 坐标 checksum。
- HUD 可视化。

`arena.lockstep_sim` 可以复用这些能力，但模拟核心改为 `sim-core`：

```text
AuthorityEvent::FrameApplied
  -> mybevy protocol adapter
  -> SimInput
  -> sim_core::step()
  -> Bevy visual sync
  -> HUD hash / events
```

### 16.3 场景表现

客户端场景至少展示：

- 本地角色和远端角色。
- 训练假人或敌方实体。
- 角色位置、朝向、移动轨迹。
- 技能范围预览。
- 命中事件。
- 伤害数字或事件列表。
- 当前 frame。
- 本地 hash。
- 服务端 hash。
- mismatch 状态。
- 回滚次数。

可先使用 primitive mesh / 简单模型，不要求正式美术资源。

### 16.4 渲染边界

Bevy 只做表现：

```text
SimWorld entity position
  -> Fp::to_f32_for_render()
  -> Transform.translation
```

动画、插值、特效和 UI 不得反写 `SimWorld`。

### 16.5 输入边界

移动端或桌面输入流程：

```text
键盘 / 摇杆 / bot
  -> input quantization
  -> SimCommand
  -> PlayerInputReq(action="sim_input")
```

如果是触屏摇杆：

- UI 层可以用 `f32` 计算触控位置。

## 17. 回滚与重放

第一阶段可以只做“权威帧 replay”，不做激进客户端预测。后续需要更流畅手感时，再加入预测和回滚。

### 17.1 最小 replay

客户端只在收到 `FrameBundlePush` 后推进：

```text
收到 frame N 的权威输入
  -> step(world, N, inputs)
  -> 渲染
```

优点：

- 简单。
- 不容易不同步。
- 适合验证 `sim-core`。

缺点：

- 输入手感有延迟。

### 17.2 本地预测

后续可支持：

```text
本地输入 target_frame=N+delay
  -> 本地预测 step 到 target_frame
  -> 收到权威 frame
  -> 如果 hash 不同，回滚到最近快照并重放
```

需要数据：

- 最近 N 帧 world snapshot。
- 最近 N 帧本地输入。
- 最近 N 帧权威输入。
- 每帧 hash。

### 17.3 快照策略

建议：

- 每 10 或 20 帧保存一次完整 `SimWorld` 快照。
- 每帧保存 hash 和输入。
- 回滚到最近完整快照，再重放到当前帧。

`sim-core` 应提供：

```rust
pub fn snapshot(world: &SimWorld) -> SimSnapshot;
pub fn restore(snapshot: &SimSnapshot) -> SimWorld;
```

## 18. 与现有协议的演进

### 18.1 第一阶段

复用 `PlayerInputReq.payload_json`：

- 成本低。
- 便于 mock 和手工调试。
- 与 `robot_sync_room` 当前做法一致。
- 当前 `lockstep_sim_demo` 已落地该方案：`PlayerInputReq.action` 必须为 `sim_input`，`payload_json` 必须是 schema version `1` 的 JSON。

缺点：

- JSON 解析成本高。
- 字段约束需要额外校验。
- 客户端和服务端容易出现大小写或类型差异。

当前 `sim_input` wire payload：

```json
{
  "version": 1,
  "seq": 1,
  "commands": [
    { "type": "move", "dirX": 1000, "dirY": 0, "speed": 6000 },
    { "type": "stop" },
    { "type": "face", "dirX": 707, "dirY": -707 },
    { "type": "castSkill", "skillId": 1, "targetEntityId": 9000 }
  ]
}
```

字段语义：

- `version` 固定为 `1`。
- `seq` 是本 payload 内所有 commands 使用的客户端序号；`sim-core` 在同角色同帧冲突选择时按 `seq`，`seq` 相同再按原始输入顺序。
- `commands` 当前最多 `8` 条。
- `move.dirX/dirY` 和 `face.dirX/dirY` 使用 `QuantizedDir`：分量范围 `-1000..=1000`，且平方和不能超过 `1000000`；常用方向包括 `(1000,0)`、`(-1000,0)`、`(0,-1000)`、`(0,1000)` 和对角 `707/-707` 组合。
- `move.speed` 可选，单位为 raw milli-units per second。缺省时 `sim-core` 先复用实体当前正速度，再使用配置默认速度；当前 `lockstep_sim_demo` 输入层接受范围是 `1..=12000`。
- `stop` 把实体移动状态设为 `Idle`，后续缺输入不会继续移动。
- `face` 只更新朝向，不选择移动输入。
- `castSkill.skillId` 必须大于 `0`。当前在线 wire 只支持可选 `targetEntityId`：省略表示 `SkillTarget::None`，非零表示 `SkillTarget::Entity`。位置 / 方向目标已存在于 `sim-core` 和 offline scenario，但还不是 `lockstep_sim_demo` 在线 JSON 契约。
- `RawSimInputPayload` 和命令枚举使用 `deny_unknown_fields`。客户端不得在 payload 或 command 中提交 `entityId`、`hit`、`damage`、`buffs`、`finalState`、`stateHash` 等服务端权威结果字段；当前这类字段会被解析为 `INVALID_SIM_INPUT_JSON`。

当前下行 / 恢复写入 `RoomSnapshot.game_state`：

```json
{
  "logicType": "lockstep_sim_demo",
  "roomId": "room-lockstep",
  "worldFrame": 1,
  "tickRate": 20,
  "initialSnapshot": {
    "schema": "myserver.lockstep-sim.initial-snapshot.v1",
    "schemaVersion": 1,
    "startFrame": 1,
    "tickRate": 20,
    "configHash": "...",
    "rngSeed": 0,
    "stateHash": { "frame": 1, "value": 0, "hex": "0000000000000000" },
    "snapshot": {}
  },
  "lastFrame": {
    "schema": "myserver.lockstep-sim.frame-envelope.v1",
    "schemaVersion": 1,
    "frame": 1,
    "stateHash": { "frame": 1, "value": 0, "hex": "0000000000000000" },
    "eventCount": 0,
    "events": [],
    "eventSummaries": [],
    "inputSources": [],
    "debugSummary": {},
    "debugState": { "schemaVersion": 1, "entities": [] }
  },
  "observerFrame": {
    "worldFrame": 1,
    "stateHash": { "frame": 1, "value": 0, "hex": "0000000000000000" },
    "lastFrame": {}
  }
}
```

客户端恢复时以 `initialSnapshot.snapshot` 为准恢复 `SimWorld`，用 `stateHash` 校验快照；帧对账时优先读取 `observerFrame.lastFrame`，没有时读取顶层 `lastFrame`。`FrameBundlePush.snapshot` 携带的 `RoomSnapshot.game_state` 是在线 replay 的主要下行载体，客户端用其中 envelope 读取服务端 hash / events，再用 `FrameBundlePush.inputs` 重放同一帧输入。`SimFrameEnvelope.stateHash` 是应用该帧后世界状态 hash；恢复后继续 replay 应以服务端 `stateHash.hex` 为准判断本地状态是否一致。`events` 是本帧事件输出，事件不进入 `hash_world`，客户端需要单独对账事件列表。`eventCount` 是服务端记录的本帧事件数量，`eventSummaries` 是按稳定字段排序后的轻量事件摘要；`debugSummary` 和 `debugState` 只用于诊断输入数量、真实 / 合成输入数量、事件数量、实体数量和少量实体状态，它们不是权威模拟输入。

## 19. 与配置和角色系统的关系

### 19.1 角色属性

`sim-core` 不直接访问 PostgreSQL 角色数据。服务端在进入 room 时把角色当前有效属性转成 `SimEntity` 初始战斗状态。

来源可以包括：

- 永久四属性。
- 激活职业 / 流派。
- 装备道具。
- Buff。
- 场景临时上下文。
- 系统临时修正。

这与现有角色体系和战斗 ECS 文档保持一致：客户端不本地决定永久成长和最终战斗属性。

### 19.2 技能池

服务端根据当前角色激活职业 / 流派得到 active skill pool，再映射为 `sim-core` 技能 slot。

客户端只消费服务端下发的初始技能 slot，不自行把 inactive 职业技能加入战斗模拟。

### 19.3 场景

第一阶段 `sim-core` 可以只接简单边界和阻挡。后续接场景地图时：

- `game-server` 负责把场景配置转成 `SceneCollisionConfig`。
- `mybevy` 用同一配置 hash 检查本地资源一致性。
- 渲染层地图与模拟层碰撞数据必须分离。

## 20. 确定性风险清单

| 风险 | 处理方式 |
|------|----------|
| 浮点参与模拟 | 核心状态统一使用 `Fp`、整数和 basis points |
| HashMap 遍历顺序 | 结算前转 Vec 并按稳定 key 排序，或使用 `BTreeMap` |
| 输入重复或乱序 | 明确 seq 和 original_index 规则 |
| 平台时间差异 | 只用 frame 和 fps |
| 配置热更不同步 | room 绑定 config hash，运行中不自动切 |
| 随机次数漂移 | 统一 `SimRngState`，随机调用只在 step 中发生 |
| 客户端渲染反写 | 渲染转换单向，Transform 不回流 |
| JSON 字段差异 | payload version + 严格字段校验 |
| 服务端合成输入差异 | `SimInputSource` 标记真实/合成输入 |
| 伤害目标排序不稳定 | distance、entity_id 等稳定排序 |

#### `movement_demo`

处理决策：保留为旧服务端权威移动样例和回归基线，暂不在原地迁移到 `sim-core` movement，也不由 `lockstep_sim_demo` 直接替换。

- `RoomRuntimePolicy::movement_demo()` 和 `GameRoomLogicFactory::create("movement_demo")` 仍显式注册该策略。
- `MovementDemoLogic` 依赖 `core::system::movement::{RoomMovementState, tick_movement, decide_corrections, full_sync_broadcast, snapshot_broadcasts, reject_broadcast}`，职责包括角色出生点加载、服务端位移推进、移动拒绝、校正广播、离线停止、重连 / 观战恢复和 room transfer。

迁移风险：

- `sim-core` movement 当前以 `Fp` / `Vec2Fp` / `QuantizedDir` 推进确定性位置，和旧 movement 的 `f32` 协议、场景查询、校正快照并不是同一合同。
- 如果直接替换，可能破坏现有 `MovementSnapshotPush`、`MovementRejectPush`、客户端漂移校正、离线停止、重连恢复、room transfer payload、AOI 收件人过滤和场景配置读取行为。
- 旧 movement 有服务端权威校正语义；`lockstep_sim_demo` 当前更偏共享模拟和 hash 对账，不能直接等价覆盖所有 correction / recovery 场景。

后续建议：

- 新玩法优先接 `lockstep_sim_demo` / `sim-core`，不要再扩展 `movement_demo` 为新确定性核心。
- 补 `movement_demo` 到 `sim-core` 的字段映射评估表：旧 `f32` 位置 / 方向 / 速度 / AOI 半径 / drift threshold 到 `Fp` raw milli-units 的转换规则必须显式写清。
- 补 `MovementSnapshotPush`、`MovementRejectPush`、`MovementRecoveryState` 与 `RoomSnapshot.game_state.initialSnapshot` / `lastFrame` 的兼容差异说明，避免客户端误以为两条链路可混用。
- 只有在 `sim-core` adapter 覆盖旧 movement 的出生点、场景边界、拒绝原因、校正广播、AOI、重连 / 观战恢复和 transfer 回归后，才评估把 `movement_demo` 合并为兼容壳或标记废弃。

- 旧战斗系统的 `Position`、出生点、训练假人位置、技能 range、AOE radius、knockback / displacement distance、移动进度和位移速度等仍使用 `f32`。
- 旧战斗 CSV 解析会把 `Range`、`AoeRadius`、`DisplacementDistance` 等字段读成 `f32`，运行态事件和 snapshot 也围绕旧 ECS 数据结构组织。
- 旧事件名来自 `CombatEventKind::as_str()`，包括 `skill_cast`、`damage`、`heal`、`buff_applied`、`buff_expired`、`defeated`、`rejected` 等；`lockstep_sim_demo` 的 `eventSummaries.kind` 当前是 `skillCast`、`damage`、`heal`、`buffApplied`、`buffExpired`、`buffTick`、`death` 等，不是完全相同的 wire 合同。

迁移风险：

- `sim-core` combat 已有 `SkillDefinition`、`SkillTargetType`、`CombatEffect`、`DamageFormula`、Buff、事件和定点 range，但旧 `combat_demo` 的输入动作、事件命名、snapshot JSON、NPC / timer transfer 合同并不等价。
- 直接迁移会同时触碰坐标单位转换、技能配置转换、目标选择语义、Buff tick、冷却推进、训练假人生成、`GameMessagePush` payload 和 room transfer 校验，回归面较大。
- 旧 `f32` range / AOE / displacement / speed 和防御减伤中的浮点运算不能直接搬入确定性核心，必须先定义整数或定点等价算法，并确认旧 CSV 配置如何量化。
- 旧 ECS 仍可作为 `sim-core` 战斗迁移的对照样例；删除前必须证明 `lockstep_sim_demo` 和 offline scenarios 覆盖旧 demo 依赖的主要战斗合同。

后续建议：

- 不再把旧 `RoomCombatEcs` 作为长期帧同步核心扩展。
- 先为 `sim-core` 补齐旧 demo 等价场景：玩家对训练假人技能、近战命中/未命中、AOE、治疗封顶、Buff/Dot、冷却、目标排序和死亡事件。
- 补旧 `CombatSnapshot` JSON 与 `SimFrameEnvelope` / `SimSnapshot` 的字段对照，明确哪些字段用于表现，哪些字段参与 hash，哪些字段只用于 debug。
- 单独做 CSV 到 `sim-core::CombatConfig` 的转换评估，避免把 `f32` 配置直接泄漏进确定性核心；转换评估必须覆盖 `Range`、`AoeRadius`、`DisplacementDistance`、Buff duration / interval / max stack、伤害公式、冷却和目标类型。

#### `robot_sync_room`

处理决策：继续保留为轻量输入转发样例和客户端协议联调样例；后续可以评估并入 `lockstep_sim_demo` 的轻量模式，但本阶段不合并、不删除。

与 `sim-core` / `lockstep_sim_demo` 的关系：

- `robot_sync_room` 不调用 `sim-core`，它验证的是 `PlayerInputReq` / `FrameBundlePush` 输入广播、payload 校验和客户端本地 replay。
- `lockstep_sim_demo` 验证的是服务端权威 `sim-core` tick、`stateHash`、snapshot / restore、事件对账和 debug 状态。
- 两者当前价值不同。只有当 `lockstep_sim_demo` 或 `tools/lockstep-client` 提供不依赖完整战斗表现的轻量 `sim_input` 转发 / dry-run 场景后，才适合讨论合并。

当前代码关系：

- `RoomRuntimePolicy::robot_sync_room()` 和 `GameRoomLogicFactory::create("robot_sync_room")` 仍显式注册该策略，并允许游戏中加入。
- `RobotSyncRoomLogic` 只接受 `action="robot_move"`，校验 `version`、`seq`、`botTick`、`dirX`、`dirY`、`speed`，限制 payload 大小，并在 tick 中记录最近真实输入摘要。
- 它不推进服务端模拟状态，不发 movement snapshot，也不依赖 `sim-core`；客户端仍可用它验证 `FrameBundlePush` 输入广播和本地 replay。

保留价值：

- 它与 `lockstep_sim_demo` 的职责不同：`robot_sync_room` 是最小输入转发和客户端 replay 样例，`lockstep_sim_demo` 是共享 `sim-core` 的权威模拟样例。
- 在 `mybevy` 的 `arena.robot_sync` 或等价轻量场景仍作为协议联调入口时，删除它会降低排查输入帧广播、payload 校验和客户端本地 replay 的便利性。

后续建议：

- 保留到 `arena.lockstep_sim` 或等价客户端场景能稳定覆盖双客户端输入广播、hash 对账、重连恢复和基础表现后，再评估是否把它降级为文档样例或测试夹具。
- 若未来删除，应先提供等价的轻量 `sim_input` dry-run 或 mock-client 场景，不能只依赖完整战斗场景。
- 若未来并入 `lockstep_sim_demo`，应保留 `robot_move` 与 `sim_input` 的迁移说明、mock-client 场景替代命令、`arena.robot_sync` 下线条件和旧错误码对照，不让客户端从输入广播样例直接跳到完整战斗样例。

#### 迁移前验收条件

旧 demo 删除、合并或替换前至少满足以下回归条件：

- `hash`：`sim-core` 单元测试覆盖定点数、方向量化、输入排序、移动边界、基础战斗、Buff、snapshot / restore 和 hash 稳定性；offline scenario 必须记录 final hash，Windows debug / release 和重复运行 hash 一致。
- `snapshot`：`RoomSnapshot.game_state.initialSnapshot.snapshot` 能恢复 `SimWorld`，`initialSnapshot.stateHash.hex`、`lastFrame.stateHash.hex`、`observerFrame.stateHash.hex` 和恢复后本地 replay hash 对齐；snapshot schema / version / configHash / simSchemaVersion 不匹配时必须明确失败。
- `events`：`SimFrameEnvelope.events`、`eventCount`、`eventSummaries` 可稳定对账；事件不进入 `hash_world`，但客户端和 `tools/lockstep-client` 必须单独比对事件列表和摘要顺序。
- `inputSources`：`inputSources` 必须区分 `real`、`synthesizedEmpty`、`synthesizedRepeatLast`；`debugSummary` 必须能反映 inputCount、realInputCount、syntheticInputCount、synthesizedEmptyInputCount、synthesizedRepeatLastInputCount、eventCount 和 entityCount；`debugState` 只作为诊断快照，不得作为权威输入。
- `tools/lockstep-client` offline scenarios 覆盖移动直线、停止、边界 clamp、缺输入继续、非法输入、近战命中 / 未命中、AOE、治疗封顶、Buff / Dot、冷却、目标排序、死亡事件，并且 final hash 已显式确认。
- `tools/lockstep-client` online dry-run 能生成 `lockstep_sim_demo` 所需 `sim_input` payload，不打开 socket 即可检查 payload 结构、命令数量、方向范围、speed 范围和 `castSkill.targetEntityId` 规则。
- 在用户确认启动依赖和服务后，真实 online 模式能进入 `lockstep_sim_demo` room，读取 `RoomSnapshot.game_state` 中的 `initialSnapshot` / `lastFrame` / `observerFrame.lastFrame`，并完成服务端 hash、snapshot、events 与本地 replay 对账。
- 重连 / 观战：`RoomReconnectReq` 和 `RoomJoinAsObserverReq` 的恢复数据必须能让客户端从 `initialSnapshot.snapshot` 或最近 envelope 恢复；观战只读状态不得改变 `stateHash`、events 或房间模拟状态。
- 客户端场景：外部 `mybevy arena.lockstep_sim` 或等价场景必须能消费 `policy_id=lockstep_sim_demo`、`sim_input`、`RoomSnapshot.game_state`、`stateHash.hex`、events、`inputSources`、`debugSummary`、`debugState`；双客户端同 frame hash、服务端 hash 和客户端 hash 必须一致。`mybevy` 真实联调只有在用户确认启动依赖服务后执行。
- 旧协议兼容：`robot_sync_room` 的 `robot_move`、`movement_demo` 的 `MovementSnapshotPush` / `MovementRejectPush` / `MovementRecoveryState`、`combat_demo` 的 `combat_cast_skill` / `combat_apply_buff` / `GameMessagePush(event="combat", action="snapshot")` 在迁移期不得被静默改变。
- 回归命令：删除或替换前必须记录明确命令和结果，至少包括相关 Rust 单元测试、`tools/lockstep-client` offline replay、online dry-run、`tools/mock-client` 旧 demo 场景；需要 Redis / PostgreSQL / NATS / `auth-http` / `game-proxy` / `game-server` / 真实客户端的命令必须先经用户确认。
- `lockstep_sim_demo` 在 `RoomLogicFactory`、`RoomRuntimePolicyRegistry`、开始 / 结束、缺帧、观战只读状态、snapshot restore 和旧 demo 并存方面有测试覆盖。
- 若替换 `movement_demo`，必须额外覆盖 `MovementSnapshotPush`、`MovementRejectPush`、`MovementRecoveryState`、移动拒绝原因、client drift 校正、离线停止、AOI / correction 配置、场景出生点和 room transfer。
- 若替换 `combat_demo`，必须额外覆盖旧 `GameMessagePush` 事件 / snapshot 兼容、旧事件命名、训练假人 / NPC 状态、timer transfer、room transfer、CSV 技能配置转换和旧客户端 / 工具消费的 payload。
- 若删除 `robot_sync_room`，必须确认 `mybevy arena.robot_sync`、`tools/mock-client` 或替代轻量客户端样例不再依赖 `robot_move`，并补充等价的输入转发 / 协议联调路径。
- 文档索引、协议文档、外部客户端接入说明和相关 README 已同步说明新入口、旧入口状态和迁移方式。
- 删除前保留一轮明确的回归结果记录；如果没有等价能力和明确结果，默认继续保留旧 demo。

## 22. 开发约束

### 22.1 不自动启动服务

按照仓库协作约定，模块功能开发完成后，不直接自动运行项目检测、集成测试、联调脚本或启动相关服务。需要先提示用户：

- 需要启动哪些服务。
- 需要 Redis / PostgreSQL / NATS 哪些依赖。
- 需要 `mybevy` 是否参与。

用户确认后再执行测试或联调。

### 22.2 外部客户端修改边界

`mybevy` 是外部仓库。MyServer 侧文档可以记录需要同步的客户端改动，但实际修改外部客户端前应确认：

- `MYSERVER_CLIENT_ROOT` 是否指向正确仓库。
- 用户是否允许修改外部仓库。
- 是否只做文档、还是同步改代码。

### 22.3 版本兼容

`sim-core` 应显式暴露：

```rust
pub const SIM_CORE_SCHEMA_VERSION: u16 = 1;
```

所有 snapshot、scenario、input payload 都带 version。版本不兼容时应明确失败，不能静默 fallback。

## 23. 第一阶段验收标准

最小可用闭环应满足：

1. `packages/sim-core` 可被 `game-server`、`tools/lockstep-client` 和 `mybevy` 引用。
2. 核心模拟不使用 `f32/f64` 推进状态。
3. offline scenario 能验证移动和战斗 hash 一致。
4. 服务端 `lockstep_sim_demo` 能调用同一 `sim-core` 推进。
5. `tools/lockstep-client` online 能和服务端 hash 对账。
6. `mybevy` 有 `arena.lockstep_sim` 或等价场景可视化移动与战斗。
7. 同一初始状态、配置 hash、随机 seed 和输入帧下，服务端套壳 client、服务端 room、客户端场景得到一致 hash。

## 24. 不在第一阶段解决的问题

第一阶段不解决：

- 完整生产部署。
- 大世界 AOI 和区域切换。
- 复杂物理引擎。
- NavMesh 寻路。
- 高级动画状态机。
- FPS/TPS 命中回溯。
- 完整技能编辑器。
- 正式资源表现。
- 跨服 room transfer 的完整玩法状态迁移。
- 复杂客户端预测、自动回滚和网络抖动恢复的完整产品化体验。
- `mybevy` 正式玩法 UI、动画、复杂物理、资源表现和完整输入手感。
- 生产级跨实例共享模拟观战、断线恢复、长期状态持久化和多服务编排验收。

这些能力需要在 `sim-core` 最小闭环稳定后逐步拆专题设计。

## 25. 原始落地顺序与后续方向

以下 1 到 5 是本设计最初的落地顺序，当前均已有实现和验证证据；实际完成状态以 [共享帧同步调试验证 Checklist](./checklists/共享帧同步调试验证_checklist.md) 为准，不应把它们继续当作未开始任务：

1. 新建 `packages/sim-core`，只实现定点数、移动和 hash。
2. 新建 `tools/lockstep-client` offline，先跑 3 个移动 scenario。
3. 把最小近战技能加入 `sim-core`，补战斗 scenario。
4. 在 `game-server` 增加 `lockstep_sim_demo`，复用现有房间帧同步框架。
5. 在 `mybevy` 增加 `arena.lockstep_sim`，复用现有 `robot_sync` 的 authority 和 HUD 经验。
6. 再评估预测、回滚、Protobuf 专用协议和旧 demo 迁移。

当前后续方向是第 6 项以及生产入口、配置映射和产品化表现；在这些能力有等价自动化与人工证据前，继续保留现有旧 demo 和调试入口。
