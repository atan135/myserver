# 背包系统设计

## 1. 概述

背包系统管理角色的物品存储、装备穿戴和属性计算。角色物品分散存放于三个位置：**背包**（随身）、**仓库**（固定位置）、**装备栏**（穿戴中）。

### 当前实现状态

当前代码已经实现背包系统的主链路：

- 角色玩法数据以 `character_id` 为主键，包含背包、仓库、装备栏、属性、外观、Buff 和 dirty 标记。
- 背包默认容量 `48`，仓库默认容量 `64`。
- 协议已覆盖装备、使用、丢弃、测试加物品、仓库存取、背包拉取和增量推送。
- `ItemTable.csv` 已接入物品存在性、装备槽位、装备属性、消耗品效果、道具模板四属性、使用时四属性 delta、成长 / 交易 / 分解 / 继承边界、职业 / 流派学习条件标记和称号来源标记；传统 `LevelRequired` 列已移除，装备和使用不再依赖角色等级。
- `Item` 实例已支持 `template_elements`、`growth_elements`、`runtime_elements`、`bound_character_id`、`growth_rules` 和 `growth_records` 的 JSONB 持久化；老 JSON 缺失新增字段时按默认值兼容。
- `UseEffect=CharacterElementChange` 的物品使用会通过 `CharacterElementService.apply_change` 正式写入角色 `affinity` / `mastery`，失败时不消耗物品。
- 装备中的道具四属性可通过 `equipped_item_elements` / `effective_item_elements` 汇总，当前只作为 accessor 参与有效属性侧计算，不直接改写角色基础四属性。
- 角色背包、仓库、装备、属性、外观、Buff 已支持 PostgreSQL JSONB 持久化，`character_inventory` 不再保存 `level`。
- GM 发奖已通过 game-server admin 口接入，并支持 `request_id` 幂等记录。
- 正式货币系统当前没有 game-server 存储和协议实现；后续落地时必须按 `character_id` 独立持久化，不能挂在账号 `player_id` 下。

当前仍未落地或只保留设计目标的部分：

- 非关键操作延迟批量写库尚未实现；当前成功的物品操作会立即调用 `PlayerManager::save_player`。
- 仓库存取的位置校验尚未在 `inventory_service` 中执行，代码中仍由调用方负责。
- 物品堆叠当前按 `uid` 或 `item_id + binded + 元素/成长/绑定元数据一致` 合并，没有按 `ItemTable.MaxStack` 拆分多格。
- 交易、合成、商店、掉落、PK 掉落还没有主业务实现；本阶段只保存 `TradeRule`、`DecomposeRule`、`InheritRule` 等规则边界，供后续系统读取。
- 道具作为职业 / 流派学习条件和称号解锁来源当前只在配置和实例规则中表达，正式职业学习流程、称号自动触发和 UI 闭环仍未落地。
- `VisualChangePush` 当前发送给本人连接，尚未接入 AOI 视野广播。

### 1.1 核心设计原则

- **服务端权威**：所有物品操作以服务端数据为准
- **角色隔离**：背包、仓库、装备、Buff 和后续货币均绑定 `character_id`；同账号两个角色互不共享资产
- **预计算属性**：装备/Buff 变更时重新计算属性，战斗时直接读取
- **来源追踪**：属性变化记录具体来源，用于面板展示
- **延迟写库**：目标是非关键操作批量写库，当前实现仍是成功后立即保存
- **变更检测**：脏标记机制驱动通知和持久化

### 2.2 三种存储位置

| 位置 | 说明 | 访问限制 |
|------|------|----------|
| 背包 (Inventory) | 随身携带，存放道具、材料、任务物品 | 在线即可操作 |
| 仓库 (Warehouse) | 固定于主城/据点 NPC 处 | 目标是必须位于仓库 NPC 附近，当前服务层未校验位置 |
| 装备栏 (Equipment) | 穿戴中的装备，影响角色属性和外观 | 在线即可穿戴/卸下 |

### 2.3 物品堆叠规则

- 不同物品有各自的堆叠上限（MaxStack，当前表字段存在但容器合并逻辑尚未强制使用）
- 不可堆叠物品（装备）每个占用独立格子
- 堆叠物品达到上限后自动占用新格子

## 3. 数据结构

### 3.1 物品定义

```rust
struct Item {
    item_id: u32,           // 物品配置ID
    uid: u64,               // 唯一实例ID（用于区分同名物品）
    count: u32,             // 数量（堆叠物品）
    binded: bool,           // 是否绑定（绑定后不可交易）
    template_elements: ItemElementValues, // 配置表基础四属性快照
    growth_elements: ItemElementValues,   // 实例长期成长四属性
    runtime_elements: ItemElementValues,  // 运行期临时四属性
    bound_character_id: Option<String>,   // 绑定角色，绑定校验使用 character_id
    growth_rules: ItemGrowthRules,        // 成长 / 交易 / 分解 / 继承规则边界
    growth_records: Vec<ItemGrowthRecord> // 成长来源和前后属性记录
}

struct ItemDefine {
    id: u32,
    name: String,
    max_stack: u32,         // 最大堆叠数，1 表示不可堆叠
    type_: ItemType,        // 装备/道具/材料/任务物品
    quality: Quality,       // 白色/蓝色/紫色/橙色/红色
    // 装备属性（仅装备类型）
    equip_attr: Option<EquipAttr>,
    // 道具四属性与使用效果（来自 ItemTable.csv）
    template_elements: ItemElementValues,
    use_affinity_delta: ItemElementValues,
    use_mastery_delta: ItemElementValues,
    growth_enabled: bool,
    trade_rule: String,
    decompose_rule: String,
    inherit_rule: String,
    discipline_condition_key: Option<String>,
    title_unlock_source: Option<String>,
}
```

`uid` 是全局唯一 ID 的数字形式，使用统一 `41 bits time_ms + 10 bits origin_id + 6 bits worker_id + 6 bits sequence` 机制生成，可以通过全局 ID 解码工具追溯 origin。`item_id` 是配置表 ID，不接入全局发号器。

### 3.1.1 道具四属性边界

当前落地的道具四属性分三层：

| 层级 | 来源 | 存储位置 | 当前用途 |
|------|------|----------|----------|
| `template_elements` | `ItemTable.csv` 的 `TemplateElementEarth/Fire/Water/Wind` | `Item` 实例中保留快照；缺失时可回退当前配置表 | 装备有效属性、展示和后续条件判断 |
| `growth_elements` | 强化、培养、吸收、洗练等长期变化 | `Item` 实例 JSON | 可成长道具实例长期成长 |
| `runtime_elements` | 附魔、耐久、Buff、场景效果等短期变化 | `Item` 实例 JSON | 临时有效属性；当前没有独立过期调度 |

`Item::effective_elements` 会汇总三层。`PlayerData::effective_item_elements` 当前只汇总已装备道具，供后续战斗 / 条件系统读取，不会直接写入角色 `affinity` 或 `mastery`。

可成长道具的成长记录保存在 `growth_records`，记录成长来源、成长前后四属性、绑定角色、交易规则、分解规则、继承规则和原因。规则字段当前是边界表达，不代表交易、分解或继承系统已经实现。

## 4. 属性计算

### 4.1 计算公式

```
最终属性 = 基础属性 + Σ(装备附加) + Σ(Buff附加) + Σ(技能附加) + Σ(临时消耗品)
```

### 4.2 重算触发时机

| 触发时机 | 影响的dirty |
|----------|-------------|
| 穿/脱装备 | attr_dirty + visual_dirty |
| 升级/转职 | attr_dirty |
| 添加/移除Buff | attr_dirty + visual_dirty |
| 使用属性类消耗品 | attr_dirty |

### 4.3 重算逻辑

```rust
impl PlayerAttr {
    fn recalculate(&mut self, equipment: &EquipmentSlots, buffs: &[Buff]) {
        // 1. 收集所有来源
        let mut all_bonus: Vec<AttrRecord> = vec![];

        // 装备附加
        for (slot, item_opt) in &equipment.slots {
            if let Some(item) = item_opt {
                if let Some(equip_attr) = &item.equip_attr {
                    all_bonus.push(AttrRecord {
                        source: AttrSource::Equipment(item.uid),
                        attr_type: AttrType::Attack,
                        value: equip_attr.attack,
                    });
                    // ... 其他属性
                }
            }
        }

        // Buff附加
        for buff in buffs {
            for (attr_type, value) in &buff.attr_bonus {
                all_bonus.push(AttrRecord {
                    source: AttrSource::Buff(buff.id),
                    attr_type,
                    value: *value,
                });
            }
        }

        // 2. 聚合到 final
        let mut final_panel = self.base.clone();
        for record in &all_bonus {
            final_panel.add(record.attr_type, record.value);
        }

        // 3. 更新
        self.bonus = all_bonus;
        self.final = final_panel;
    }
}
```

## 6. 核心操作

### 6.1 装备穿戴/卸下

```rust
fn equip_item(&mut self, slot: EquipSlot, item: Item) -> Result<()> {
    // 1. 检查背包是否有该物品
    let idx = self.find_item(item.uid)?;
    if idx.is_none() { return Err(ErrCode::ItemNotFound); }

    // 2. 检查槽位是否匹配
    if !slot.matches_item_type(item.type_) {
        return Err(ErrCode::SlotMismatch);
    }

    // 3. 卸下当前装备到背包
    let old_equipment = self.equipment.slots.remove(&slot);
    if let Some(old) = old_equipment {
        self.inventory.add_item(old)?;
    }

    // 4. 穿上新装备
    self.inventory.remove_item(item.uid)?;
    self.equipment.slots.insert(slot, Some(item));

    // 5. 触发重算和脏标记
    self.attr.recalculate(&self.equipment, &self.visual.active_buffs);
    self.set_attr_dirty();
    self.set_visual_dirty();

    Ok(())
}
```

### 6.2 仓库存取

```rust
fn can_access_warehouse(player: &Player) -> bool {
    let dist = player.pos.distance(WAREHOUSE_POSITION);
    dist <= ACCESS_DISTANCE
}

fn warehouse_deposit(&mut self, item_uid: u64, count: u32) -> Result<()> {
    // 1. 位置校验
    if !can_access_warehouse(&self.player) {
        return Err(ErrCode::NotAtWarehouse);
    }

    // 2. 从背包移到仓库
    let item = self.inventory.remove_item(item_uid, count)?;
    self.warehouse.add_item(item)?;
    self.data_dirty = true;

    Ok(())
}

fn warehouse_withdraw(&mut self, item_uid: u64, count: u32) -> Result<()> {
    // 1. 位置校验
    if !can_access_warehouse(&self.player) {
        return Err(ErrCode::NotAtWarehouse);
    }

    // 2. 从仓库移到背包
    let item = self.warehouse.remove_item(item_uid, count)?;
    self.inventory.add_item(item)?;
    self.data_dirty = true;

    Ok(())
}
```

当前代码中的 `warehouse_deposit` / `warehouse_withdraw` 只做容器间转移，位置校验还没有接入请求处理链路。

### 6.3 物品使用

```rust
fn use_item(&mut self, item_uid: u64) -> Result<()> {
    let item = self.inventory.get_item(item_uid)?;

    match item.def.type_ {
        ItemType::Food => {
            // 消耗品：直接移除，产生Buff/属性效果
            self.inventory.remove_item(item_uid, 1)?;
            self.apply_food_effect(&item.def)?;
            self.set_attr_dirty();
        }
        ItemType::Quest => {
            // 任务物品：触发任务条件，不移除
            self.trigger_quest_condition(&item.def)?;
        }
        _ => {
            return Err(ErrCode::CannotUse);
        }
    }

    Ok(())
}
```

## 7. 数据库持久化

### 7.0 表归属

当前持久化表为 `character_inventory` 和 `character_inventory_grants`：

- `character_inventory.character_id` 是唯一角色归属键。
- `inventory_data`、`warehouse_data`、`equipment_data`、`visual_data`、`buffs_data` 使用 JSONB 保存当前角色状态。
- `hp` 保存当前角色生命值快照。
- `character_inventory_grants.request_id` 用于 GM 发物品幂等，`character_id` 记录发放目标。
- 表结构不再包含 `level`、`player_id` 或 `account_player_id`；背包和道具逻辑不得用账号 ID 或传统等级判断目标。

### 7.1 写库策略

| 操作类型 | 策略 |
|----------|------|
| 交易、合成、PK掉落 | 实时写库 |
| 物品使用、Buff变更 | 目标是延迟批量写库，当前实现为成功后立即写库 |
| 登录登出 | 强制写库 |

```rust
}
```

当前 `PlayerData` 和 `PlayerManager` 已有 dirty 标记与 dirty 查询接口，但还没有周期性 dirty collector / batch flush 循环。现有请求处理在成功修改后直接保存玩家数据。

## 8. 协议设计

### 8.1 背包相关协议

| 协议 | 方向 | 说明 |
|------|------|------|
| `WarehouseAccessReq/Res` | Client→Server | 仓库存取请求（当前未接入位置校验） |
| `ItemEquipReq/Res` | Client→Server | 装备穿戴/卸下 |
| `ItemUseReq/Res` | Client→Server | 物品使用 |
| `InventoryUpdatePush` | Server→Client | 背包增量更新 |
| `AttrChangePush` | Server→Client | 属性面板更新 |
| `VisualChangePush` | Server→Client | 外观/Buff变化 |

这些玩家协议请求不携带目标 `character_id`。服务端使用当前连接已鉴权 ticket 里的 `character_id` 读取和保存 `PlayerData`，客户端不能在请求体里指定其他角色。`tools/mock-client` 的背包场景会输出当前 `accountPlayerId` 和 `characterId`，用于人工确认本次操作目标。

GM 发物品走 admin 通道，`GrantItemsReq.character_id` 是唯一目标身份；`admin-api` HTTP 入口使用 `characterId` 字段并审计为 `targetType=character`。`playerId` 只用于账号级 kick / ban，不用于背包发放。

## 10. 设计决策汇总

| 问题 | 决策 |
|------|------|
| 背包数据存储位置 | 独立于 RoomState，登录时从 DB 加载到内存 |
| 属性计算方式 | 预计算 + 来源记录，变更时重算并标记 dirty |
| 仓库访问限制 | 目标是必须位于仓库 NPC 附近，当前服务层未校验位置 |
| 通知范围 | 属性→本人，外观目标→视野内玩家；当前外观推送先发本人连接 |
| DB 写入策略 | 当前成功操作立即保存；后续再补关键操作实时写、其余延迟批量 |
