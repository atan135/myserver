# 游戏业务模块开发规范

## 1. 文档定位

本文定义 `game-server` 后续具体游戏业务的模块划分、目录结构、依赖方向和开发准入规则。目标是让每一段业务代码都能明确回答：

- 属于哪个业务模块；
- 由谁调用；
- 拥有或修改哪一份业务状态；
- 可以依赖哪些模块；
- 应通过什么契约与其他模块交互。

本文适用于角色成长、背包、装备、任务、奖励、经济、活动等具体玩法和业务能力，不用于规定网络接入、协议编解码、房间运行时、帧同步、数据库驱动、配置表运行时或服务注册等底层实现。

当前代码和配置始终优先于本文。本文是新模块的强制开发基线，也是现有业务代码的目标结构；不要求在无关需求中一次性搬迁旧代码。

## 2. 当前结构与演进原则

当前业务实现主要分布在以下位置：

- `apps/game-server/src/core/inventory/`：背包、资产与奖励交付相关模型和逻辑；
- `apps/game-server/src/core/character_*.rs`：角色属性、职业、称号和进度等业务；
- `apps/game-server/src/core/service/`：部分协议请求对应的业务入口；
- `apps/game-server/src/core/system/`：移动、场景、战斗等运行系统，其中同时存在框架能力和具体规则；
- `apps/game-server/src/gameservice/`：游戏业务消息入口与协议适配。

这些路径是当前事实，不代表后续继续扩展时的目标结构。后续演进遵守以下原则：

1. 新增独立业务能力时，默认放入 `apps/game-server/src/business/<module>/`。
2. 修改现有模块时，若需求不包含结构迁移，不为追求目录一致性顺带大范围搬文件。
3. 旧模块开始迁移后，应以一个完整业务能力为单位迁移，避免同一状态长期由新旧两套入口共同修改。
4. 不再新增缺少业务归属的顶层 `character_xxx.rs`，也不继续扩大横向的通用 `service`、`manager`、`helper` 或 `common` 目录。
5. 迁移前后都必须保持公开契约、持久化语义、幂等规则和线上行为可验证。

## 3. 模块定义

业务模块是围绕一项稳定业务能力组织的代码单元。一个合格模块应同时具备：

- 清晰的职责与明确的非职责；
- 唯一的核心状态所有权；
- 可描述的业务不变量；
- 少量、稳定的公开命令、查询和事件；
- 单向且可审查的外部依赖。

代码通常应归入同一模块，当它们共同维护一份核心状态、遵守同一组业务不变量，并因同类需求一起变化。状态生命周期、规则和演进节奏都能独立时，才拆成不同模块。

不要仅按业务名词数量拆模块。模块过大会形成新的巨型服务，模块过小则会让一次简单操作跨越大量边界。默认采用中等粒度，以“状态所有权 + 业务不变量”作为最终判断依据。

## 4. 目标目录结构

```text
apps/game-server/src/
├─ business/
│  ├─ mod.rs
│  ├─ <module>/
│  │  ├─ mod.rs
│  │  ├─ MODULE.md
│  │  ├─ api/
│  │  │  ├─ mod.rs
│  │  │  ├─ command.rs
│  │  │  ├─ query.rs
│  │  │  └─ event.rs
│  │  ├─ application/
│  │  │  ├─ mod.rs
│  │  │  ├─ use_case.rs
│  │  │  ├─ subscriber.rs
│  │  │  └─ ports.rs
│  │  ├─ domain/
│  │  │  ├─ mod.rs
│  │  │  ├─ model.rs
│  │  │  ├─ rules.rs
│  │  │  ├─ service.rs
│  │  │  └─ event.rs
│  │  ├─ config.rs
│  │  └─ tests.rs
│  ├─ orchestration/
│  └─ shared/
├─ gameservice/       # 玩家协议到业务 API 的适配
├─ gameroom/          # 具体房间装配与运行时适配
├─ gameconfig/        # CSV 表注册与装配
└─ core/              # 现有框架、运行时及待渐进迁移代码
```

目录表示职责边界，不要求为空职责创建占位文件。小模块可以先使用 `api.rs`、`application.rs`、`domain.rs`，当单文件出现多个独立概念后再展开为子目录；展开前后依赖方向必须保持不变。

### 4.1 `api`

模块唯一对外入口，负责定义：

- Command：请求模块执行一个会改变状态的行为；
- Query：读取模块拥有的数据，不产生业务副作用；
- Event：模块完成提交后发布的业务事实；
- Facade：组合并暴露模块允许外部调用的能力。

`api` 不负责业务计算，不直接操作存储，不暴露领域对象的可变引用。协议生成类型、网络连接对象和数据库行类型不得成为业务 API 契约。

### 4.2 `application`

负责完成一次完整业务用例，包括权限上下文检查、调用领域行为、组织端口、保证幂等、确定提交顺序和返回结果。

`application` 可以编排本模块的多个领域对象，但不得成为核心数值和合法性规则的长期存放处。需要时间、随机数、存储、消息或外部服务时，在 `ports.rs` 定义业务所需接口，技术实现留在模块外部。

### 4.3 `domain`

负责实体、值对象、状态转换、业务不变量、纯计算规则和领域事件。该层不得依赖 Tokio、Protobuf、SQLx、Redis、NATS、HTTP 或具体 CSV 生成类型。

同一模块内跨多个领域对象的规则可以放入领域服务；如果一个服务只是转发调用或组织技术步骤，应放回 `application`。

### 4.4 `config`

负责把已加载并校验的配置表快照转换为模块可使用的业务配置。领域层依赖模块自己的配置值对象，不直接依赖 `csv_code` 生成结构或全局配置表运行时。

### 4.5 `orchestration`

只承载具有独立生命周期的跨模块流程。符合下列任一条件时可以建立编排：

- 流程跨越三个或更多业务模块；
- 需要独立的幂等、重试、超时或补偿状态；
- 任一参与模块都不适合作为完整流程的所有者；
- 流程需要保存阶段性结果并在进程重启后恢复。

简单的两个模块调用不自动建立编排层，应先判断是否存在明确的主模块。

### 4.6 `shared`

只允许存放稳定且不属于任何单一模块的业务基础概念，例如 `CharacterId` 等标识值对象。具体玩法规则、可变状态、数据库模型和“暂时不知道放哪里”的代码禁止进入 `shared`。

## 5. 依赖方向

```text
gameservice / admin_server / gameroom
                 |
                 v
          business::<module>::api
                 |
                 v
              application ------> application ports
                 |
                 v
               domain ----------> business::shared

orchestration ------> 各模块公开 API
模块 A 内部 --------X--------> 模块 B 内部
domain ------------X--------> 协议、存储、网络和运行时框架
```

必须遵守：

1. 模块外部只能依赖该模块从 `mod.rs` 明确导出的公开 API。
2. `api -> application -> domain` 为单向依赖，领域层不得反向依赖应用层。
3. 模块不得直接读取或修改另一个模块的内部状态。
4. 一个模块不得导出内部集合、锁、数据库连接或可变领域对象供外部操作。
5. Rust 可见性默认从严：内部模块使用私有或 `pub(super)`，只在模块根导出必要的 `pub(crate)` 契约。
6. 出现循环依赖时，必须通过确定状态所有者、改用事件、引入编排或重新划分模块解决，不得用 `common` 包装循环。

## 6. 状态所有权

每份可变业务状态必须有且只有一个所属模块。该模块负责状态的创建、合法性校验、修改、版本和业务事件；其他模块只能：

- 保存其 ID；
- 通过公开 Query 获取只读结果；
- 通过公开 Command 请求变更；
- 订阅已提交的 Event 维护非权威只读视图。

复制出来的查询快照不得成为第二写入口。跨模块操作需要强一致时，应由拥有事务边界的主模块或独立编排明确提交顺序、失败语义和幂等键。

模块设计阶段必须列出“明确不负责”的状态。例如奖励来源模块可以决定奖励为何产生，但不得直接写入背包；背包模块负责资产结算，但不得反向决定任务是否完成。

## 7. 跨模块交互规则

| 交互目的 | 允许方式 | 说明 |
|---|---|---|
| 请求执行行为 | Command API | 使用祈使动作命名，结果必须表达确定状态或未知状态 |
| 立即读取数据 | Query API | 不得借查询产生业务副作用 |
| 通知已发生事实 | Event | 使用过去式命名，只在业务提交后发布 |
| 跨模块长流程 | Orchestration | 显式记录阶段、幂等、失败和恢复策略 |
| 共享可变数据 | 禁止 | 指定唯一所有者后通过契约访问 |

Command 不等同于 Event。`GrantReward` 是请求，`RewardGranted` 是已经发生的事实；订阅方不得把可能失败的请求伪装成事实事件。

同步调用适合调用方必须立即获得结果的场景。事件适合解耦后续反应，但不能用来回避本应明确的事务边界。

## 8. 接入层与底层边界

### 8.1 玩家协议

`gameservice` 负责：

1. 完成协议解码和已鉴权身份提取；
2. 将 Protobuf 请求转换为业务 Command 或 Query；
3. 调用模块公开 API；
4. 将业务结果映射为协议响应和稳定错误码。

协议入口不得重复领域校验，不得直接修改 `PlayerData`。业务模块也不得接收原始连接、消息号或 Protobuf 请求对象。

### 8.2 身份与权限上下文

接入层从 `AuthenticatedSessionIdentity` 等可信会话对象构造业务 Actor Context。客户端提交的 `character_id`、权限、奖励内容或来源信息不得覆盖服务端可信上下文。

### 8.3 配置表

`gameconfig` 和配置表运行时负责加载、热更与全局快照。业务模块的 `config` 负责把相关行转换为领域可理解、已验证的配置对象。配置校验与运行时解释必须保持同一语义。

### 8.4 持久化与消息

业务模块在 `application/ports` 声明自己需要的仓储、时钟、随机数、消息发布等能力。SQLx、Redis、NATS 和远程服务客户端属于端口实现，不得渗入领域规则。

## 9. 文件归属判断

新增文件前按顺序判断：

1. 是否表达实体状态、业务不变量或纯计算规则？放入 `domain`。
2. 是否表达一次完整业务操作或提交顺序？放入 `application`。
3. 是否需要被其他模块调用或订阅？只把契约放入 `api`。
4. 是否把 CSV 行转换为业务配置？放入模块 `config`。
5. 是否实现存储、网络、消息或运行时适配？放在业务模块外部并实现对应 port。
6. 是否协调多个无明确主从关系的模块？满足编排准入条件后放入 `orchestration`。
7. 是否只是两个位置暂时重复使用？保留在真正的所有者中，不立即放入 `shared`。

仍无法归属时，应先修正模块边界，不得通过新增 `utils.rs`、`helper.rs`、`manager.rs` 或 `common.rs` 绕过判断。

## 10. 命名与文件规则

- 模块名使用稳定业务能力名和 Rust `snake_case`，不使用页面名、活动批次名或临时需求编号。
- Command 使用祈使动作，如 `ConsumeItem`、`GrantTitle`。
- Query 使用 `Get`、`List`、`Find`、`Can` 等读取语义。
- Event 使用过去式业务事实，如 `ItemConsumed`、`TitleGranted`。
- Use Case 使用 `<Verb><Object>UseCase`，Facade 使用 `<Module>Facade`。
- Port 按业务能力命名，如 `InventoryRepository`、`RewardEventPublisher`，不使用 `DataService`。
- 一个文件围绕一个主要概念组织；是否拆文件由职责决定，不以固定行数作为唯一标准。
- `mod.rs` 只负责模块声明、受控导出和少量装配，不承载复杂业务实现。

## 11. 模块说明模板

每个正式业务模块必须维护 `MODULE.md`：

```markdown
# <模块名称>

## 职责

## 明确不负责

## 核心状态与所有者

## 业务不变量

## 公开 Commands

## 公开 Queries

## 发布 Events

## 订阅 Events

## 允许依赖

## 核心用例

## 业务配置

## 幂等与事务边界

## 失败与恢复语义

## 相关协议和专题文档
```

`MODULE.md` 描述当前真实边界。目标设计、尚未落地能力和当前实现必须明确区分。

## 12. 测试规则

- `domain`：覆盖业务不变量、边界值、状态转换和纯计算规则，不依赖数据库或网络。
- `application`：使用 fake port 覆盖成功、拒绝、重复请求、部分失败和结果未知场景。
- `api`：覆盖公开契约、稳定错误码和兼容性要求。
- `orchestration`：覆盖重试、幂等、补偿、超时和进程恢复后的继续执行。
- 协议适配：验证可信身份映射、业务错误到协议错误的映射，以及客户端字段不能越权覆盖服务端上下文。

业务规则修复必须至少补充一个能在修复前失败的回归测试。涉及资产、奖励、货币、邮件附件等敏感状态时，还必须验证不会重复发放或无声丢失。

## 13. 新模块开发准入

开始编码前，至少完成以下内容：

- [ ] 模块职责和明确非职责已经写入 `MODULE.md`；
- [ ] 核心状态所有者和业务不变量已经确定；
- [ ] Commands、Queries、Events 已列出；
- [ ] 外部依赖已通过模块 API 或 application port 表达；
- [ ] 事务、幂等、失败和恢复语义已说明；
- [ ] 已确认无需直接访问其他模块内部状态；
- [ ] 已确定业务配置的加载、校验和热更语义；
- [ ] 已列出领域测试和应用层测试场景。

代码评审时，以下任一情况必须阻止合入：

- 新增第二个状态写入口；
- 从一个业务模块导入另一个模块的内部类型；
- 在协议处理器、数据库适配器或事件消费者中复制核心业务规则；
- 用通用目录隐藏无法解释的业务归属；
- 只记录成功路径，没有定义重复请求、失败或结果未知的行为；
- 文档声称为当前能力，但代码尚未落地。

## 14. 现有模块迁移模板

迁移现有代码前先建立模块档案：

```markdown
# <现有模块> 迁移档案

当前入口：
当前文件：
读写状态：
核心规则：
调用的外部能力：
被哪些代码调用：
当前公开契约：
当前持久化与幂等语义：
发现的越界代码：
目标模块归属：
迁移步骤：
兼容与回滚策略：
```

推荐迁移顺序：先固定契约和回归测试，再确定状态所有者，然后收拢领域规则和应用用例，最后切换入口并删除旧写路径。不要先搬目录再判断职责，也不要长期保留新旧双写。

## 15. 示例边界：任务完成与奖励交付

任务模块负责判断任务是否完成、记录任务状态并发布 `TaskCompleted`。奖励来源或流程编排根据已确认的业务事实生成奖励请求；背包或统一资产模块负责容量、堆叠、幂等和最终入账。

```text
gameservice
  -> quest::CompleteTask command
  -> quest 更新任务状态
  -> TaskCompleted event
  -> reward orchestration
  -> inventory 公开资产命令
  -> 资产提交后发布结果事件
```

任务模块不得直接修改背包集合，背包模块也不得反向判断任务条件。这种“来源决定为什么产生，所有者决定如何合法写入”的边界适用于任务、成就、战斗、活动、排行榜和 GM 奖励等场景。
